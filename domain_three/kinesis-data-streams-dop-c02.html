Kinesis Data Streams

Real-time data streaming — ingest, buffer, and replay ordered event data at scale.

01

Core Concepts

◈◈ HIGH

What It Is

  • Collect & store streaming data in real-time
  • Retention: 1 day (default) → up to 365 days
  • Data is immutable — cannot be deleted by consumers
  • Data expires automatically at retention end
  • Consumers can replay data within retention window
  • Max record size: 1 MB (not 10 MiB — PDF error)

Producers & Consumers

  • Producers: SDK, KPL, Kinesis Agent, IoT, CloudWatch Logs
  • Consumers: SDK, KCL, Lambda, Firehose, Managed Apache Flink
  • KPL (Kinesis Producer Library) — batching, compression, retry logic
  • KCL (Kinesis Client Library) — checkpointing, shard coordination via DynamoDB
  • Multiple consumers can read the same stream independently
Exam Tip
KCL uses a DynamoDB table to track shard checkpoints. If consumers fall behind, check DynamoDB throttling. Replay capability = key differentiator vs SQS (which deletes on consume).
02

Shards & Capacity Modes

◈◈ HIGH
A shard is the unit of capacity in KDS. All throughput limits are per shard. Partition key determines which shard a record goes to — same key = same shard = ordered delivery.
1 Shard ▲ 1 MB/s in ▲ 1,000 rec/s in ▼ 2 MB/s out ▼ 5 reads/s
5 Shards ▲ 5 MB/s in ▲ 5,000 rec/s in ▼ 10 MB/s out ▼ 25 reads/s
N Shards ▲ N MB/s in ▲ N×1,000 rec/s ▼ N×2 MB/s out ▼ N×5 reads/s

Provisioned Mode

  • You choose number of shards manually
  • Scale: shard splitting (add) or merging (remove)
  • Splitting doubles a shard → can only double at a time
  • Pay per shard-hour
  • Best when throughput is predictable

On-Demand Mode

  • No capacity planning needed
  • Default: 4 MB/s or 4,000 rec/s
  • Auto-scales based on peak throughput of last 30 days
  • Pay per stream/hour + data in/out per GB
  • Best for unpredictable or spiky traffic
Exam Tip
ProvisionedThroughputExceededException = producer writing too fast for the shard. Fix: increase shards, use partition key with better distribution, or enable KPL batching. Hot shard = too many records with the same partition key.
03

Ordering & Partition Keys

◈◈ HIGH

How Ordering Works

  • Records with the same partition key always go to the same shard
  • Within a shard, records are strictly ordered by sequence number
  • Across shards, there is no global ordering
  • Consumers read per-shard in order

Hot Shard Problem

  • All records share same partition key → all go to one shard
  • That shard gets throttled while others sit idle
  • Fix: Use high-cardinality partition keys (e.g. user ID, device ID)
  • Or add a random suffix to distribute load
Exam Tip
Ordering guarantee is per shard only. If you need ordered processing of a user's events → use their user ID as the partition key. All their events land on one shard, consumed in order.
04

Consumer Types: Standard vs Enhanced Fan-Out

◈◈ HIGH
Attribute Standard (Pull) Enhanced Fan-Out (Push)
Mechanism Consumer polls with GetRecords KDS pushes via HTTP/2 (SubscribeToShard)
Throughput 2 MB/s shared across ALL consumers per shard 2 MB/s per consumer per shard
Latency ~200ms avg ~70ms avg
Max consumers 5 reads/s per shard (shared) Up to 20 registered consumers per stream
Cost Included in shard cost Additional cost per consumer-shard-hour + data retrieval
Best for Low consumer count, cost-sensitive Multiple independent consumers, low-latency pipelines
Exam Tip
If multiple apps (Lambda + Firehose + custom app) need to read the same stream without starving each other → use Enhanced Fan-Out. Standard throughput is shared; EFO gives each consumer its own dedicated 2 MB/s.
05

Lambda Integration

◈◈ HIGH

Event Source Mapping

  • Lambda polls KDS on your behalf
  • Batch size: 1–10,000 records configurable
  • Batching window: up to 300 seconds
  • One Lambda invocation per shard (in order)
  • On failure: retries until data expires or DLQ configured
  • Supports bisect on error — splits failed batch to isolate bad record

Error Handling

  • On-failure destination → SQS or SNS for failed batches
  • BisectBatchOnFunctionError — halves the batch to find the poison record
  • MaximumRetryAttempts — limit retries before discarding
  • MaximumRecordAgeInSeconds — skip records older than threshold
  • Without error handling, a stuck shard blocks all new records
Exam Tip
A single bad record in a shard can block all processing because Lambda retries indefinitely. Always configure BisectBatchOnFunctionError + MaximumRetryAttempts + an on-failure destination for production.
06

KDS vs SQS vs SNS — Decision Guide

◈◈ HIGH
Feature Kinesis Data Streams SQS SNS
Pattern Streaming / replay Queue / decouple Pub/Sub fan-out
Ordering Per shard (partition key) FIFO queue only No ordering
Replay Yes — up to 365 days No No
Consumers Multiple, independent Competing consumers Multiple subscribers
Retention 1–365 days Up to 14 days No storage
Throughput 1 MB/s per shard (in) Unlimited (standard) Unlimited
Use when Real-time analytics, log ingestion, replay needed Task queues, decoupling, async workers Notifications, fan-out to multiple targets
Exam Tip
Replay needed → KDS. Multiple independent consumers on same data → KDS or SNS+SQS fan-out. Simple worker queue → SQS. "Real-time analytics" in the scenario almost always means KDS → Firehose or Flink.
07

Security & Encryption

◈ MEDIUM

Encryption at Rest

  • Server-side encryption with KMS
  • Enable per stream
  • Producers/consumers need KMS permissions

In Transit

  • HTTPS endpoints only
  • TLS enforced for all SDK calls

Access Control

  • IAM policies control who can PutRecord and GetRecords
  • VPC Endpoints (PrivateLink) for private access
  • CloudTrail logs all KDS API calls
08

Key Metrics to Know

◈ MEDIUM
Metric What It Means Action if High
GetRecords.IteratorAgeMilliseconds How far behind consumers are from the tip of the stream Scale consumers or shards; consumers are lagging
WriteProvisionedThroughputExceeded Producer throttled — shard write limit hit Add shards or fix hot partition key
ReadProvisionedThroughputExceeded Consumer throttled — shard read limit hit Switch to Enhanced Fan-Out
IncomingRecords / IncomingBytes Producer throughput per shard Monitor for capacity planning
Exam Tip
IteratorAgeMilliseconds approaching your retention period = data at risk of expiring before being processed. Alarm on this metric in production. It's the primary indicator of consumer lag.

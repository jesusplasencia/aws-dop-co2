API Gateway

Serverless proxy service to expose Lambda, HTTP backends, and AWS services via managed APIs.

01

Endpoint Types

◈ MEDIUM
Type Use Case Key Detail
Edge-Optimized (default) Global clients Routes via CloudFront edges. Gateway still in one region. TLS cert must be us-east-1.
Regional Clients in same region Can pair with CloudFront manually. TLS cert must match gateway region.
Private Internal VPC access only Accessed via Interface VPC Endpoint (ENI). Requires resource policy.
Exam Tip
Certificate location rule: Edge-Optimized → us-east-1. Regional → same region as API Gateway. Route53 CNAME or A-alias is required for custom domains.
02

Deployment Stages & Stage Variables

◈◈ HIGH
Key rule: Changes to an API take effect only after a deployment to a stage. Nothing is live until deployed.

Stages

  • Each stage is an independent environment (dev, staging, prod)
  • Each stage has its own config, throttling, caching, logging settings
  • Rollback is possible — deployment history is kept

Stage Variables

  • Act like environment variables per stage
  • Can point to different Lambda ARNs, HTTP endpoints, or mapping templates
  • Syntax: ${stageVariables.varName}
  • Enables one API definition across environments
Exam Tip
Stage Variables are the standard way to avoid duplicating API configs per environment. Know that they can point to different Lambda aliases.
03

Canary Deployments

◈◈ HIGH

How it works

  • Enable on any stage (typically prod)
  • Split traffic: configure % to canary channel
  • Canary gets new deployment; base stage keeps current
  • Metrics & logs are separate per channel

Key Properties

  • This is blue/green deployment for API Gateway + Lambda
  • Can override stage variables for the canary
  • Promote canary → becomes the new prod stage
  • Rollback by disabling canary or setting traffic to 0%
Exam Tip
Canary = blue/green at the API Gateway level. Questions often pair this with Lambda aliases and weighted routing for progressive delivery.
04

Security & Authentication

◈◈ HIGH
Method How When to use
IAM Roles/Policies SigV4 signed request headers Internal AWS services, cross-account access
Cognito User Pools JWT token validated by API Gateway Web/mobile apps with user identities
Lambda Authorizer Custom Lambda validates token/header; returns IAM policy Third-party auth (OAuth, SAML, custom logic)
Resource Policy JSON policy on the API itself Restrict by IP, VPC endpoint, or AWS account
Exam Tip
Lambda Authorizer result is cached by default (TTL configurable). Cognito handles authentication; IAM handles authorization. Know which layer does what.
05

Caching

◈ MEDIUM

Configuration

  • Defined at stage level
  • Default TTL: 300s (max 3600s, min 0s)
  • Capacity: 0.5 GB → 237 GB
  • Can override per method
  • Recommended for PROD only (has cost)

Cache Invalidation

  • Flush entire cache immediately from console
  • Client can invalidate via header: Cache-Control: max-age=0
  • Requires execute-api:InvalidateCache IAM permission
  • Without policy: any client can invalidate — security risk
06

Throttling

◈ MEDIUM
Level Default Notes
Account-level 10,000 req/s Shared across all APIs. Can be increased via support request.
Stage / Method Configurable Set lower limits per stage or per method to protect backend.
Usage Plans Per API key Throttle and quota individual customers/clients.
429 Too Many Requests — returned when throttle limit is exceeded. Clients should implement exponential backoff.
07

Logging, Tracing & Metrics

◈ MEDIUM

CloudWatch Logs

  • Enabled at stage level
  • Log levels: ERROR, DEBUG, INFO
  • Logs request/response body (careful with sensitive data)

X-Ray Tracing

  • Enable at stage level
  • API Gateway + Lambda = full end-to-end trace
  • Identifies latency bottlenecks

Key Metrics

  • CacheHitCount / CacheMissCount
  • Count — total API requests
  • IntegrationLatency — GW → backend roundtrip
  • Latency — full client roundtrip (includes GW overhead)
  • 4XXError (client) / 5XXError (server)
08

OpenAPI & Request Validation

◈ LOW–MED

OpenAPI

  • Import OpenAPI 3.0 spec (YAML or JSON) to create APIs
  • Export existing API as OpenAPI spec
  • AWS extensions available for Gateway-specific config
  • Infrastructure as code for API definitions

Request Validation

  • Validates before forwarding to backend
  • Returns 400 Bad Request on failure
  • Validate: request body, query params, headers
  • Saves Lambda invocations on invalid requests

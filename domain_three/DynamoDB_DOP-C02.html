Amazon DynamoDB

Serverless NoSQL key-value & document database — DevOps Professional exam scope

Architecture

Core Characteristics

Fundamentals

  • Fully managed — serverless, no cluster or patching
  • Multi-AZ replication by default — 3 AZs
  • Single-digit millisecond performance at any scale
  • Max item size: 400 KB
  • Schema-less — attributes can vary per item
  • All access control via IAM (no DB-level users)
  • Table classes: Standard Infrequent Access (IA)

Primary Keys — Two Options

  • Option 1 — Partition Key only (Simple PK): must be unique per item. e.g. user_id
  • Option 2 — Partition Key + Sort Key (Composite PK): combination must be unique. e.g. user_id + game_id
  • Partition Key determines which internal partition stores the item (hash function)
  • Sort Key enables range queries within a partition
  • Primary Key cannot be changed after table creation
  • Choose high-cardinality Partition Keys to avoid hot partitions
Partition Key (PK)
  • Determines storage partition
  • Hash function applied
  • Must be unique (simple PK)
  • High cardinality = even distribution
Sort Key (SK)
  • Optional — composite PK only
  • Items sorted within partition
  • Enables range queries (begins_with, between)
  • PK+SK combo must be unique
LSI — Local Secondary Index
  • Same PK, different Sort Key
  • Must be defined at table creation
  • Up to 5 LSIs per table
  • Shares provisioned throughput with base table
GSI — Global Secondary Index
  • Different PK and SK — query any attribute
  • Can add/delete GSIs anytime
  • Up to 20 GSIs per table
  • Has its own RCU/WCU — provision separately
  • Eventual consistency only
⚑ Exam Traps — Indexes
  • LSI: must be created at table creation — cannot add later. Throttling on the base table affects LSI reads.
  • GSI: if GSI WCU is undersized, writes to the base table will be throttled even if base table has capacity — GSI is a bottleneck.
  • GSI only supports eventual consistency reads — never strongly consistent.
Capacity

Read/Write Capacity Modes & Units

RCU — Read Capacity Unit

1 RCU = 1 Strongly Consistent Read/sec (up to 4 KB)
1 RCU = 2 Eventually Consistent Reads/sec (up to 4 KB)
2 RCU = 1 Transactional Read/sec (up to 4 KB)
  • Items >4 KB → ceil(size/4KB) × RCU
  • Example: 10 KB item, strongly consistent = 3 RCU
  • Scan consumes RCU for every item read (even filtered)

WCU — Write Capacity Unit

1 WCU = 1 Write/sec (up to 1 KB)
2 WCU = 1 Transactional Write/sec (up to 1 KB)
Items >1 KB → ceil(size/1KB) × WCU
  • Example: 2.5 KB item = 3 WCU per write
  • Conditional writes still consume WCU even if they fail
  • GSI WCU must be provisioned separately

Provisioned Mode

  • Specify RCU + WCU in advance
  • Pay for provisioned capacity whether used or not
  • Auto Scaling: set min/max RCU/WCU + target utilization % — DynamoDB adjusts automatically
  • Exceeding provisioned capacity → ProvisionedThroughputExceededException
  • Burst capacity: unused RCU/WCU saved for up to 300 seconds
  • Best for: predictable, steady workloads

On-Demand Mode

  • No capacity planning — scales instantly to any load
  • Pay per request (RRU / WRU)
  • ~2.5× more expensive per request than provisioned
  • No throttling — accommodates unlimited throughput
  • Can switch between modes twice per day
  • Best for: unpredictable, spiky, new apps with unknown traffic
⚑ Exam — Throttling
  • ProvisionedThroughputExceededException: caused by hot partition (poor PK design), or exceeding table RCU/WCU.
  • Fix: exponential backoff (built into SDK), improve PK distribution, switch to On-Demand, or increase provisioned capacity.
  • Adaptive capacity: DynamoDB automatically shifts capacity to hot partitions — but this has limits. Good PK design is always the root fix.

⚑ Exam — Query vs Scan
  • Always use Query over Scan when possible — Scan reads every item before filtering → wastes RCU and is slow on large tables.
  • Use Parallel Scan to speed up large table scans — but throttling risk increases.
  • Use ProjectionExpression to return only needed attributes — reduces response size but does NOT reduce RCU consumed on Scan.
DAX

DynamoDB Accelerator (DAX)

What DAX Does

  • Fully managed in-memory cache for DynamoDB only
  • Microsecond read latency (vs single-digit ms DDB)
  • No application code change — DAX is API-compatible with DynamoDB
  • Default TTL: 5 minutes (configurable)
  • Deployed as a multi-node cluster inside your VPC
  • Caches individual item reads + Query/Scan results
  • Write-through: writes go to DAX + DynamoDB simultaneously

DAX vs ElastiCache

  • DAX DynamoDB-specific, no code changes, item + query cache
  • ElastiCache General purpose, requires code changes, stores aggregations/computed results
  • DAX sits between your app and DynamoDB — transparent
  • ElastiCache stores aggregation results — e.g. sum/count queries already processed
  • Both can be used together: DAX for DDB item cache, ElastiCache for computed results
  • DAX: Not for write-heavy workloads — cache invalidation overhead
⚑ Exam — DAX
  • DAX reduces read load on DynamoDB — useful when hot-key reads are causing throttling.
  • DAX does not help with write throttling — writes still go directly to DynamoDB.
  • If app needs microsecond reads from DynamoDB with no code change → DAX. If it's a custom aggregation result → ElastiCache.
Streams

Stream Processing

DynamoDB Streams

  • Ordered stream of item-level changes: create, update, delete
  • Retention: 24 hours
  • Limited consumers (2 simultaneously)
  • Process with: Lambda Triggers or KCL Adapter
  • Stream records contain: KEYS_ONLY / NEW_IMAGE / OLD_IMAGE / NEW_AND_OLD_IMAGES
  • Required for Global Tables (cross-region replication uses streams)
  • Use cases: react to DB changes, send notifications, analytics, cross-region replication

Kinesis Data Streams (for DynamoDB)

  • Alternative to DynamoDB Streams — sends changes to Kinesis
  • Retention: up to 1 year
  • Higher number of consumers — fan-out
  • Process with: Lambda, Kinesis Data Analytics, Firehose, AWS Glue
  • Firehose destinations: S3, Redshift, OpenSearch
  • Better for long-term event processing pipelines
  • Enabled separately per table — does not replace DynamoDB Streams

Global

Global Tables

How It Works

  • Multi-region, multi-active replication (read AND write in any region)
  • Requires DynamoDB Streams enabled — streams power the replication
  • Changes replicate to all regions automatically
  • Replication latency: typically under 1 second
  • Conflict resolution: last-writer-wins (by timestamp)
  • Active-active — no concept of primary region

Use Cases & Caveats

  • Low-latency access for globally distributed users
  • Disaster recovery — another region available instantly
  • Must use On-Demand mode OR Provisioned with Auto Scaling enabled
  • Avoid writing to the same item from multiple regions simultaneously — conflict risk
  • LSIs/GSIs replicated to all regions automatically
  • TTL deletions replicated across regions
⚑ Exam Note
  • Global Tables = multi-active (every region accepts writes). This is different from Aurora Global (one primary writer region).
  • Prerequisite: DynamoDB Streams must be enabled before adding replicas.
  • Conflict resolution is always last-writer-wins — design your app to avoid concurrent conflicting writes.
TTL

Time-To-Live (TTL)

How TTL Works

  • Define an attribute that holds a Unix timestamp (epoch seconds)
  • DynamoDB automatically deletes items after expiry — no WCU consumed
  • Deletion is not immediate — items expire within ~48 hours of TTL
  • Expired items still visible in reads until physically deleted
  • Expired item deletions appear in DynamoDB Streams — can trigger Lambda
  • TTL attribute must be Number type

Use Cases

  • Web session store (expire inactive sessions)
  • Temporary data / one-time tokens
  • Regulatory data retention (delete after X days)
  • Event/log data with rolling window
  • Shopping cart expiry
  • Combine with Streams: on TTL delete → trigger Lambda → archive to S3
⚑ Exam Trap
  • TTL deletions do not consume WCU — they are free.
  • Items past their TTL are not immediately removed — can still be returned in reads for up to 48 hours. Filter in your app if exact expiry matters.
Backups

Backups, DR & S3 Integration

PITR — Point-in-Time Recovery

  • Must be explicitly enabled — not on by default
  • Continuous backups for up to 35 days
  • Restore to any second in the retention window
  • Restore creates a new table — does not overwrite existing
  • No performance impact on the source table
  • Required to enable Export to S3

On-Demand Backups

  • Full backup at any point — persists until manually deleted
  • Zero performance or latency impact
  • Managed via AWS Backup for cross-region copy and lifecycle policies
  • Restore always creates a new table
  • Can restore to same or different region / account
  • Does not count toward PITR retention

S3 Integration

DirectionRequirementDetailsCapacity Impact
Export → S3 PITR must be enabled Export any point in last 35 days. Formats: DynamoDB JSON, ION. Use Athena to query S3 data. No RCU consumed
Import from S3 None Formats: CSV, DynamoDB JSON, ION. Creates a new table. Errors logged to CloudWatch Logs. No WCU consumed
⚑ Exam Notes
  • Export to S3 does not affect table read capacity — uses a separate export process.
  • Import from S3 does not consume WCU — creates a new table, not an incremental load.
  • PITR restore and on-demand backup restore both create a new table — the original is untouched.
  • Use AWS Backup with DynamoDB for cross-account and cross-region backup policies with retention rules.
Security

Security & Access Control

IAM Access Control

  • All access via IAM policies — no DB-level users
  • Fine-grained access: restrict by table, item, or attribute
  • dynamodb:LeadingKeys condition: users can only access their own items (e.g. restrict by cognito-identity.amazonaws.com:sub)
  • IAM roles for EC2/Lambda accessing DynamoDB
  • VPC Endpoint: access DynamoDB without internet

Encryption

  • At-rest: AWS owned key (free), AWS managed KMS key, Customer managed KMS key
  • In-transit: TLS/HTTPS — always enforced
  • Encryption enabled by default — cannot disable
  • Customer managed keys: full audit trail in CloudTrail

VPC & Networking

  • DynamoDB is a public AWS service (not in your VPC)
  • Use VPC Gateway Endpoint for private access from VPC (no NAT, no internet)
  • Traffic stays within AWS network via endpoint
  • Access logs: CloudTrail for API-level audit

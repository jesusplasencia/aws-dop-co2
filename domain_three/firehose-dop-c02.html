Amazon Data Firehose

Fully managed delivery stream — load streaming data into storage and analytics destinations without writing consumer code.

01

What It Is & Core Characteristics

◈◈ HIGH
Mental model: Firehose is a managed pipe between a data source and a destination. You don't write consumer code. It buffers, optionally transforms, then delivers in batches.
ProducerKDS, SDK, CW, IoT…
→
Transform (optional)Lambda function
→
Buffersize or time trigger
→
DestinationS3, Redshift, OpenSearch…

Key Properties

  • Fully managed & serverless — no shards, no scaling config
  • Near real-time — minimum buffer: 60s or 1 MB
  • Automatic scaling based on throughput
  • Pay only for data volume ingested (per GB)
  • No data storage — not a streaming buffer like KDS
  • No replay — once delivered, data is gone from Firehose

Supported Formats

  • Input: JSON, CSV, Parquet, Avro, Raw text, Binary
  • Format conversion: JSON → Parquet/ORC for S3 (columnar = cheaper Athena queries)
  • Compression: GZIP, Snappy, ZIP, HADOOP_SNAPPY (for S3)
  • Custom transform: any logic via Lambda
Exam Tip
"Near real-time" = Firehose. "Real-time" = KDS. If the scenario says load into S3/Redshift/OpenSearch with no code → Firehose. If it says replay, custom consumers, or sub-second latency → KDS.

03

Destinations

◈◈ HIGH

AWS Destinations

  • Amazon S3 — primary destination; supports prefix, partitioning, compression
  • Amazon Redshift — via S3 COPY command (S3 is intermediate)
  • Amazon OpenSearch — log/search indexing

Third-Party

  • Splunk
  • Datadog
  • New Relic
  • MongoDB
  • Dynatrace, Coralogix, others

Custom

  • HTTP Endpoint — any custom API receiver
  • Useful for custom data lakes or external systems
  • Must return 200 OK or Firehose retries
Exam Tip
Redshift is NOT a direct destination — Firehose writes to S3 first, then issues a COPY command to load into Redshift. If that intermediate S3 step matters in the question, it's Firehose → S3 → Redshift.
04

Buffering & Delivery

◈◈ HIGH
Firehose delivers when either threshold is hit first — size OR time, whichever comes first.
Parameter Range Default Notes
Buffer Size 1 MB – 128 MB 5 MB Lower = more frequent (smaller) deliveries
Buffer Interval 60s – 900s 300s Minimum latency is 60 seconds
Exam Tip
Firehose minimum latency is 60 seconds — this is why it's "near real-time" not "real-time." If a question requires sub-minute delivery, KDS + Lambda is the right answer, not Firehose.
05

Data Transformation with Lambda

◈ MEDIUM

How It Works

  • Firehose invokes a Lambda function synchronously
  • Lambda receives a batch of records, returns transformed records
  • Each record must return: Ok, Dropped, or ProcessingFailed
  • Failed records → separate S3 error bucket (configurable)
  • Transformation adds latency — account for Lambda timeout

Common Transformations

  • CSV → JSON, JSON → Parquet
  • Data enrichment (add fields, lookup)
  • Filtering (drop unwanted records → Dropped)
  • PII masking / scrubbing
  • Format normalization before S3 ingestion

Dynamic Partitioning (S3)

Extract fields from incoming records (e.g. customer_id, event_date) and use them to build custom S3 prefixes. Example: s3://bucket/year=2024/month=04/customer=abc/. Enables efficient Athena/Glue queries. Requires inline parsing or Lambda extraction.

06

Error Handling & Backup

◈◈ HIGH

Delivery Failures

  • Firehose retries delivery for a configurable duration
  • S3: retry up to 24 hours
  • Redshift: retry up to 7200s (2 hrs), then moves to S3 error prefix
  • OpenSearch: retry 0–7200s, then optional S3 backup
  • After max retry → data goes to S3 error bucket

S3 Backup Bucket

  • Configure a source record backup to S3 alongside any destination
  • All records (or just failures) can be backed up
  • Useful for debugging transformation failures
  • Separate from the primary destination bucket
Exam Tip
Unlike KDS, there is no replay from Firehose itself. The S3 backup bucket is your only safety net. For replay, you'd need to re-ingest from the backup bucket. Design pipelines with backup enabled.
07

KDS vs Firehose — Decision Table

◈◈ HIGH
Attribute Kinesis Data Streams Amazon Data Firehose
Latency Real-time (~200ms) Near real-time (≥60s)
Management Provisioned shards or On-Demand Fully serverless, zero config
Consumer code Required (SDK, KCL, Lambda) None — Firehose handles delivery
Data storage Up to 365 days No storage (buffer only)
Replay Yes No
Destinations Any custom consumer S3, Redshift, OpenSearch, 3rd party, HTTP
Transformation In consumer code Via Lambda (managed)
Choose when Real-time processing, replay, custom logic Load to storage/analytics, no-code delivery
Common pattern: KDS (real-time processing + Lambda) → also fan out to Firehose (archiving to S3 data lake). Both can run simultaneously on the same stream.

PHASE 1
Full Load
Existing data bulk-copied to target. Source continues serving traffic. Takes minutes to days depending on size.
PHASE 2
Full Load + CDC
DMS captures changes from source (via CDC) while full load is still running. Ensures no data loss during migration.
PHASE 3
CDC Only
After full load completes, only ongoing changes are replicated. Use for near-zero-downtime cutover.

Key Facts

  • Source DB stays fully available during migration
  • Requires an EC2 Replication Instance — you provision it inside a VPC
  • Resilient and self-healing — auto-restarts failed tasks
  • Supports Homogeneous migrations (same engine)
  • Supports Heterogeneous migrations (different engines) — requires SCT first
  • Continuous replication via CDC (Change Data Capture)
  • Multi-AZ: provisions synchronous standby replica of replication instance

Migration Types

  • Homogeneous Same engine: Oracle → Oracle, MySQL → MySQL. SCT not needed.
  • Heterogeneous Different engine: SQL Server → Aurora, Oracle → PostgreSQL. SCT required first.
  • Full Load One-time bulk migration. Downtime at cutover.
  • Full Load + CDC Bulk + ongoing changes. Minimizes cutover window.
  • CDC Only Replicate ongoing changes only — source already synced.
⚑ Exam — Key Architecture Point
  • DMS is not serverless — you must provision and size the EC2 Replication Instance. Under-sizing it causes migration lag.
  • Multi-AZ DMS: creates a synchronous standby of the replication instance in another AZ — protects the migration process itself from AZ failure, not the data.
  • SCT runs on a server you provide (on-prem or EC2) — it is a software tool, not a managed service.
Endpoints

Supported Sources & Targets

Sources

  • On-premises / EC2: Oracle, MS SQL Server, MySQL, MariaDB, PostgreSQL, MongoDB, SAP, DB2
  • Cloud: Azure SQL Database, Google Cloud SQL
  • AWS: Amazon RDS (all engines including Aurora), Amazon S3, DocumentDB

Targets

  • On-premises / EC2: Oracle, MS SQL Server, MySQL, MariaDB, PostgreSQL, SAP
  • AWS Databases: Amazon RDS, Aurora, Redshift, DynamoDB, DocumentDB, Amazon Neptune
  • AWS Services: S3, OpenSearch Service, Kinesis Data Streams, Apache Kafka
  • Compatibility: Redis, Babelfish (Aurora PostgreSQL)
⚑ Exam — Notable Targets
  • Redshift as target = migrate OLAP workloads from Oracle/Teradata → data warehouse on AWS.
  • S3 as target = useful for data lake migrations or archiving DB snapshots.
  • Kinesis Data Streams as target = stream DB changes into a real-time pipeline.
  • DynamoDB as target = migrate relational data to NoSQL (schema redesign required).
SCT

AWS Schema Conversion Tool (SCT)

What SCT Does

  • Converts DB schema + stored procedures + views from one engine to another
  • Not needed for homogeneous migrations (same engine)
  • Installed on a local server or EC2 — it is a desktop/client tool, not a managed service
  • Generates a migration assessment report — shows complexity + manual effort
  • Automatically converts what it can — flags what needs manual intervention
  • Works before DMS starts the data migration

SCT Conversion Examples

  • OLTP Oracle / SQL Server → MySQL, PostgreSQL, Aurora
  • OLAP Teradata / Oracle → Amazon Redshift
  • SCT converts: tables, views, indexes, constraints, stored procedures, triggers, sequences
  • Complex Oracle PL/SQL or MSSQL T-SQL → may require manual rewrite
  • SCT can target Babelfish on Aurora PG to minimize code rewrite for SQL Server migrations
⚑ Exam — DMS vs SCT
  • SCT: converts the schema (DDL, stored procedures) — no data movement.
  • DMS: moves the data — does not convert schema.
  • Workflow for heterogeneous: SCT first → DMS second. Never the reverse.
  • For homogeneous migration (MySQL → RDS MySQL): use DMS alone — SCT is unnecessary.
CDC

Change Data Capture (CDC)

How CDC Works

  • Reads the source database's transaction log (e.g. Oracle Redo Logs, MySQL Binlog)
  • Captures INSERT, UPDATE, DELETE as they happen — in real time
  • Applies changes to target in near-real-time
  • Enables minimal-downtime migrations — run CDC until cutover, then flip DNS/connection
  • Source must have transaction log access enabled for CDC to work
  • Latency: typically seconds to minutes depending on load

CDC Prerequisites by Engine

  • Oracle: LogMiner or Binary Reader mode; supplemental logging must be enabled
  • SQL Server: MS-CDC or MS-Replication must be enabled on source DB
  • MySQL: Binary logging (binlog_format = ROW) must be enabled
  • PostgreSQL: Logical replication slot must be configured; wal_level = logical
  • Aurora MySQL: Binary logging enabled in parameter group
  • RDS sources: Automated backups must be enabled to use CDC
⚑ Exam — CDC Gotchas
  • If source does not support CDC (e.g. no binlog, no transaction logs): you can only do Full Load — requires downtime at cutover.
  • For Amazon S3 as source: CDC is simulated by reading new files added to a prefix — not true log-based CDC.
  • Replication Instance must have enough storage for CDC lag — if it falls behind, it needs to re-read logs that may have already been purged → task failure.
Instance

Replication Instance Sizing & HA

Instance Classes

  • Runs DMS tasks — sized like an EC2 instance (t3, r5, c5 families)
  • t3.micro / t3.small: dev/test, small migrations
  • r5 (memory-optimized): recommended for large, complex migrations
  • Deployed in a VPC subnet group — must have network access to both source and target
  • Single instance can handle multiple tasks (tables) in parallel
  • CPU/memory bottleneck = slower migration throughput

Multi-AZ Replication Instance

  • Provisions a synchronous standby replica in a second AZ
  • Automatic failover if primary replication instance fails
  • No data loss — standby is in sync
  • Does NOT improve migration throughput — only improves availability of migration process
  • Recommended for: large, long-running migrations, production cutovers
  • ~double the cost of single-AZ replication instance
⚑ Exam Note
  • Multi-AZ replication instance = HA for the DMS process itself, not for the source or target DB.
  • If the replication instance runs out of storage during CDC: migration fails. Always provision generous storage for long-running CDC tasks.

Monitoring

Monitoring & CloudWatch Metrics

Task & Table Status

  • Task Status: Creating, Running, Stopped, Failed, Starting
  • Task Status Bar: % progress estimate
  • Table State: Before load, Table completed, Table error, Table cancelled
  • Counts per table: inserts, updates, deletes, DDL statements completed

Host Metrics (Replication Instance)

  • CPUUtilization — high = resize instance
  • FreeableMemory — low = memory pressure, resize
  • FreeStorageSpace — low = risk of task failure
  • WriteIOPS / ReadIOPS — I/O throughput of instance disk
  • NetworkTransmitThroughput

Replication Task Metrics

  • FullLoadThroughputRowsSource — rows read/sec from source
  • FullLoadThroughputRowsTarget — rows written/sec to target
  • CDCThroughputRowsSource — CDC rows captured/sec
  • CDCThroughputRowsTarget — CDC rows applied/sec
  • CDCLatencySource — lag between source change and DMS reading it
  • CDCLatencyTarget — lag between DMS reading and applying to target
⚑ Exam — CDCLatency
  • CDCLatencySource high → DMS is falling behind reading source logs. Cause: under-sized replication instance or source generates too many changes.
  • CDCLatencyTarget high → DMS reads changes fast but applying to target is slow. Cause: target DB throughput bottleneck (write capacity, indexing).
  • Both high → overall replication lag is growing — risky for cutover timing.

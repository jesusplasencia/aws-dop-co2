Elastic Kubernetes Service

Managed Kubernetes control plane on AWS — ECS alternative for teams already using Kubernetes.

01

EKS vs ECS — Know the Difference

◈◈ HIGH
Attribute ECS EKS
Orchestrator AWS proprietary Kubernetes (open-source, CNCF)
Portability AWS-only Runs anywhere Kubernetes runs
Unit of work Task Pod
Config format Task Definition (JSON) YAML manifests (Deployment, Service…)
Learning curve Lower Higher — requires Kubernetes knowledge
When to choose AWS-first, simpler workloads Existing K8s workloads, multi-cloud, advanced scheduling
Exam Tip
If the scenario mentions "already using Kubernetes" or "multi-cloud portability" → choose EKS. If it asks for "least operational overhead on AWS" → ECS Fargate wins.
02

Key Kubernetes Concepts (EKS Context)

◈ MEDIUM
You don't need deep Kubernetes expertise for DOP-C02 — but you must map these terms to AWS components.
Pod
Smallest deployable unit. One or more containers sharing network/storage. Equivalent to ECS Task.
Node
EC2 instance (or Fargate) running pods. Equivalent to ECS container instance.
Deployment
Manages desired replica count of pods. Handles rolling updates and rollbacks.
Service
Stable network endpoint for a set of pods. Maps to a Load Balancer or ClusterIP.
Namespace
Logical isolation within a cluster. Used for multi-team or multi-env separation.
ConfigMap
Key-value config injected into pods (non-secret data).
Secret
Base64-encoded sensitive data in K8s. Best paired with AWS Secrets Manager via CSI driver.
Control Plane
Managed by AWS in EKS — API Server, etcd, scheduler, controller manager. You don't manage it.
03

Node Types

◈◈ HIGH

Managed Node Groups

  • AWS creates & manages EC2 nodes for you
  • Nodes part of an ASG managed by EKS
  • Supports On-Demand or Spot Instances
  • Automatic node updates & draining
  • Recommended for most workloads

Self-Managed Nodes

  • You create EC2s & register to cluster
  • Managed by your own ASG
  • Use Amazon EKS Optimized AMI
  • Full control over instance config
  • More operational overhead

AWS Fargate

  • No nodes to manage at all
  • Each pod runs in its own isolated compute
  • Requires a Fargate Profile (namespace + label selector)
  • No DaemonSets support
  • Best for serverless K8s
Exam Tip
Managed Node Groups = least overhead for EC2-backed EKS. Fargate on EKS = no nodes but requires Fargate Profiles and doesn't support DaemonSets (e.g., can't run node-level log agents this way).
04

IAM & Access Control

◈◈ HIGH

IAM Roles for Service Accounts (IRSA)

  • Assigns an IAM Role to a Kubernetes Service Account
  • Pods using that service account assume the role
  • Fine-grained pod-level AWS permissions (like ECS Task Role)
  • Uses OIDC provider on the EKS cluster
  • Replaces the need for node-level instance profiles

aws-auth ConfigMap

  • Maps IAM users/roles → Kubernetes RBAC
  • Required to grant kubectl access to cluster
  • Stored in kube-system namespace
  • Misconfiguration = locked out of cluster
  • EKS Access Entries (newer) replace this pattern
Exam Tip
IRSA is the EKS equivalent of ECS Task Role — always prefer it over instance profile-based permissions for pods. Know that it requires an OIDC identity provider to be associated with the cluster.
05

Auto-Scaling

◈◈ HIGH

Horizontal Pod Autoscaler (HPA)

  • Scales pod count based on CPU/memory or custom metrics
  • Built into Kubernetes
  • Equivalent to ECS Service Auto Scaling

Cluster Autoscaler

  • Scales EC2 node count in the ASG
  • Adds nodes when pods are unschedulable
  • Removes underutilized nodes
  • Requires proper IAM permissions on node role

Karpenter (AWS-native)

  • Next-gen node provisioner — faster than Cluster Autoscaler
  • Provisions right-sized nodes directly (no ASG required)
  • Supports diverse instance types & Spot automatically
  • Preferred for EKS on AWS
Exam Tip
HPA = scale pods. Cluster Autoscaler / Karpenter = scale nodes. For Fargate, only HPA applies — no nodes to scale. Karpenter may appear in newer exam questions as the recommended scaler.
06

Networking

◈ MEDIUM

VPC CNI Plugin

  • Each pod gets a real VPC IP address
  • Native VPC routing — no overlay network
  • Security Groups can be applied per pod
  • Must size subnets for pod IP density

Load Balancer Controller

  • Kubernetes Service (type LoadBalancer) → provisions NLB
  • Kubernetes Ingress → provisions ALB
  • Requires AWS Load Balancer Controller add-on
  • Annotations control LB behaviour (internal, SSL, etc.)
Exam Tip
Ingress → ALB. Service LoadBalancer → NLB. Security Groups for Pods allow pod-level network isolation without relying on node-level SGs.
07

Data Volumes (Storage)

◈ MEDIUM
Key concept: EKS uses Container Storage Interface (CSI) drivers. You define a StorageClass manifest pointing to the driver — Kubernetes handles the rest.
Storage Access Mode Fargate? Use Case
EBS Single node (ReadWriteOnce) No Databases, stateful single-pod workloads
EFS Multi-node (ReadWriteMany) Yes Shared storage across pods/AZs
FSx for Lustre Multi-node No HPC, ML training, high-throughput workloads
FSx for NetApp ONTAP Multi-protocol No Enterprise NFS/SMB workloads
Exam Tip
EFS = only CSI storage that works with Fargate. EBS requires EC2 nodes and is AZ-bound. Multi-pod shared storage → always EFS for EKS.
08

Logging & Observability

◈◈ HIGH

Control Plane Logs → CloudWatch

  • Disabled by default — must enable per log type
  • Log types: api, audit, authenticator, controllerManager, scheduler
  • audit logs → who did what via kubectl
  • Sent to /aws/eks/<cluster>/cluster log group

Node / Pod / Container Logs

  • Container logs stored at /var/log/containers on node
  • Use Fluent Bit (recommended) or Fluentd as DaemonSet
  • Ships logs to CloudWatch Logs
  • CloudWatch Agent collects node-level metrics

CloudWatch Container Insights

Full observability dashboard for nodes, pods, tasks, and services. Collects metrics + logs. Enable as an add-on. Provides CPU, memory, disk, and network metrics per pod/namespace. Use for cluster-wide performance monitoring.

Exam Tip
Control plane logs (especially audit) → security/compliance questions. Fluent Bit DaemonSet → application log shipping. Container Insights → monitoring dashboard. On Fargate, DaemonSets don't work — use Fluent Bit sidecar pattern instead.

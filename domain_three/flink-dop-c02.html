Amazon Managed Service for Apache Flink

Run Apache Flink applications on a fully managed cluster — stateful real-time stream processing at scale.

01

What It Is & Exam Context

◈ MEDIUM
DOP-C02 weight: Flink is a supporting service — you need to know when to choose it and how it fits in a streaming pipeline, not how to write Flink code.

What It Does

  • Runs Apache Flink applications on AWS-managed infrastructure
  • Real-time stateful stream processing (not just filtering)
  • Write in Java, Scala, Python, or SQL (Flink SQL)
  • Managed cluster: auto-scaling, parallel computation
  • Application state backed by checkpoints & snapshots to S3
  • Formerly: Kinesis Data Analytics for Apache Flink

When to Choose Flink

  • Complex streaming logic: windows, joins, aggregations
  • Stateful processing — maintain state across events
  • Pattern detection over a stream (CEP)
  • Need sub-second latency with custom business logic
  • Real-time dashboards, anomaly detection, fraud detection
Exam Tip
Flink = complex real-time processing with state. Lambda = simple per-event transform. Firehose = delivery only. If the scenario involves windowed aggregations, stream joins, or stateful logic → Flink is the answer.

Exam Tip
Sources = KDS or MSK. If a scenario shows Firehose as the Flink source → wrong answer. Firehose is a destination, not a source for Flink.
03

State, Checkpoints & Snapshots

◈ MEDIUM

Checkpoints (Automatic)

  • Periodic snapshots of application state to S3
  • Triggered automatically by Flink runtime
  • Used for automatic failure recovery
  • If a task fails, Flink restarts from last checkpoint
  • Configurable interval — trade-off between recovery time and overhead

Snapshots (Manual)

  • User-triggered full application state backup
  • Used for: version upgrades, code deployments, intentional restarts
  • Allows resume from exact point in stream
  • Stored in S3
  • Enables blue/green deployment of Flink apps
Exam Tip
Checkpoints = automatic recovery from failures. Snapshots = intentional save point for deployments or migrations. Both stored in S3. This is how Flink achieves exactly-once processing semantics.
04

Flink vs Lambda vs Firehose — Choose the Right Tool

◈◈ HIGH
Scenario Best Choice Why
Simple per-record transform (CSV → JSON) Lambda (via KDS/Firehose) Stateless, event-driven, low complexity
Load stream to S3/Redshift/OpenSearch Firehose No-code delivery, buffered batches
Count events per user over last 5 minutes Flink Requires time window + stateful aggregation
Detect fraud pattern across 3 events in sequence Flink Complex Event Processing (CEP) — stateful, ordered
Join two streams in real-time Flink Stream-to-stream join requires stateful processing
Real-time dashboard from IoT sensors Flink → OpenSearch / KDS Aggregations over time windows, low latency output
Archive logs with no processing Firehose Delivery only, no logic needed
05

Managed Infrastructure

◈ LOW

Compute

  • Provisioned KPUs (Kinesis Processing Units)
  • 1 KPU = 1 vCPU + 4 GB RAM
  • Auto-scaling based on CPU/memory
  • Parallel task execution across KPUs

Availability

  • Runs across multiple AZs
  • Automatic failure detection & restart
  • Recovers from last checkpoint
  • No manual cluster management

Deployment

  • Deploy Flink JAR or use Flink SQL (no-code)
  • Application code stored in S3
  • VPC support for private network access
  • IAM for access to KDS, S3, MSK

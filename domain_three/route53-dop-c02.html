Route 53

Authoritative DNS + Domain Registrar + Health Checks. The traffic manager for multi-region and DR architectures.

01

Records, Hosted Zones & Alias

◈◈ HIGH

Record Types (Must Know)

  • A — hostname → IPv4
  • AAAA — hostname → IPv6
  • CNAME — hostname → another hostname. Cannot be used on zone apex (root domain)
  • NS — name servers for the hosted zone
  • Alias — AWS-only extension of A/AAAA. Points to AWS resources. Works on zone apex. Free.

Hosted Zones

  • Public — routes internet traffic (e.g. example.com)
  • Private — routes within one or more VPCs (e.g. db.internal)
  • Cost: $0.50/month per hosted zone
  • Private zone must be associated with a VPC to resolve
  • Split-horizon DNS: same name → different answers for public vs private

CNAME vs Alias — Critical Distinction

  • CNAME: maps to any hostname. Cannot be at zone apex (example.com). Charged per query.
  • Alias: AWS-native, maps to AWS resource (ALB, CloudFront, S3, API GW, Elastic Beanstalk, VPC endpoints). Works at zone apex. No extra charge.
  • Alias target cannot be an EC2 DNS name
  • TTL is set automatically by Route 53 for Alias records
Exam Tip
"Route example.com to an ALB" → must use Alias, not CNAME. CNAME on zone apex is invalid. Alias to EC2 DNS → invalid. Alias to CloudFront, S3 website, ALB → valid.
02

Health Checks

◈◈ HIGH
Health checks are the engine behind Route 53 automated failover. Without them, Route 53 cannot detect failures and reroute traffic automatically.

Endpoint Health Checks

  • Monitor HTTP/HTTPS/TCP endpoints
  • ~15 global health checkers send requests
  • Healthy if ≥18% report healthy
  • Interval: 30s (or 10s fast — extra cost)
  • Can check response body for a string (first 5120 bytes)

Calculated Health Checks

  • Aggregate multiple health checks into one
  • Use AND / OR / NOT logic
  • Up to 256 child health checks
  • Define how many must pass
  • Use to monitor overall system health

CloudWatch Alarm Health Checks

  • Monitor a CloudWatch Alarm state
  • Enables health checks on private resources (inside VPC)
  • Route 53 health checkers can't reach private IPs
  • Pattern: CW Metric → CW Alarm → R53 Health Check
Exam Tip
Route 53 health checkers are outside your VPC — they cannot directly check private endpoints. Use CloudWatch Alarm → Health Check for private resources. This is a very common exam pattern.
03

Routing Policies

◈◈ HIGH
Simple
  • Route to a single resource
  • Can return multiple IPs (client picks randomly)
  • No health check association
  • No failover capability
Use when: single backend, no HA needed
Weighted
  • Split traffic by % across multiple resources
  • Weight 0 = stop sending traffic
  • All weights 0 = distributed equally
  • Can combine with health checks
Use when: A/B testing, blue/green, canary deployments
Latency
  • Routes to region with lowest latency to user
  • Latency is AWS-measured, not geographic
  • Can combine with health checks for failover
  • Same record name across multiple regions
Use when: multi-region, minimize response time
Failover (Active-Passive)
  • Primary receives all traffic
  • Secondary only used when primary health check fails
  • Health check on primary is mandatory
  • Secondary can be any endpoint (DR site)
Use when: DR, active-passive HA
Geolocation
  • Routes based on user's geographic location
  • Can define rules for continent, country, or US state
  • Must create a Default record for unmatched locations
  • Different from Latency — geo is location, not speed
Use when: compliance, content localization, region locks
Geoproximity
  • Routes based on geographic distance + bias
  • Bias: positive = expand region, negative = shrink
  • Requires Traffic Flow feature
  • Shift traffic between regions by adjusting bias
Use when: shift traffic to a region gradually
Multi-Value
  • Returns up to 8 healthy records randomly
  • Each record can have a health check
  • Client-side load balancing (not a replacement for ALB)
  • Unhealthy records filtered out automatically
Use when: client-side LB across healthy endpoints
IP-Based
  • Routes based on client's IP CIDR block
  • Define CIDR → endpoint mapping
  • Useful for ISP-based routing optimization
Use when: ISP-aware routing, known IP ranges
Exam Tip
Weighted = blue/green at DNS level. Failover = DR active-passive. Latency ≠ Geolocation (latency is measured speed; geo is literal location). Geoproximity requires Traffic Flow and uses a bias dial. Multi-Value ≠ load balancer.
04

Traffic Flow & Policy Records

◈ MEDIUM

Traffic Flow

  • Visual editor to build complex routing decision trees
  • Combine multiple routing policies (e.g. Latency → Weighted → Failover)
  • Required for Geoproximity routing
  • Policies are versioned — roll back to previous version
  • Reuse same policy across multiple hosted zones

Common Multi-Policy Pattern

  • Latency-based → routes user to nearest region
  • Within region → Weighted between blue/green
  • Each endpoint → has a Health Check
  • Failover → fallback to another region if all unhealthy
  • Traffic Flow makes this composable without writing code

Route 53 in DevOps Deployment Patterns

◈◈ HIGH
Pattern Route 53 Policy How
Blue/Green Weighted Blue = weight 100. Switch: Blue = 0, Green = 100. Or gradual shift.
Canary Release Weighted New version gets weight 5 (5% traffic). Monitor → increase gradually.
Active-Passive DR Failover Primary in us-east-1. Secondary in eu-west-1. Automatic failover on health check failure.
Active-Active Multi-Region Latency + Health Checks Each region serves traffic. Route 53 removes unhealthy regions automatically.
Geo Compliance Geolocation EU users → EU endpoint. US users → US endpoint. GDPR, data residency.
Exam Tip
Blue/green and canary at the DNS level = Weighted routing. Always pair with health checks so Route 53 can auto-remove unhealthy targets. TTL matters here — lower TTL before a deployment so clients pick up changes faster.
06

TTL (Time To Live)

◈ MEDIUM

How TTL Works

  • DNS resolvers cache the record for TTL seconds
  • Clients won't re-query until TTL expires
  • High TTL (e.g. 86400s / 24h) = fewer queries, stale risk
  • Low TTL (e.g. 60s) = more queries, faster propagation
  • Alias records: TTL set by Route 53, not configurable

TTL Strategy for Deployments

  • Before a planned change: lower TTL to 60s well in advance (wait for old TTL to expire)
  • Make DNS change — propagates in ~60s
  • After change is stable: restore high TTL
  • Forgetting this = clients stuck on old IP for hours
Exam Tip
Lower TTL before the deployment window — not during. If TTL is 24h, you need to lower it 24h before the change or clients will be stuck. This is a common operational scenario in the exam.
07

Private DNS & Hybrid Architecture

◈ MEDIUM

Private Hosted Zones

  • Resolve internal names (e.g. db.internal) within VPC
  • Must enable enableDnsHostnames and enableDnsSupport on VPC
  • Associate with multiple VPCs (even cross-account)
  • Overrides public DNS for same name (split-horizon)

Resolver Endpoints (Hybrid DNS)

  • Inbound Endpoint — on-prem DNS queries Route 53 private zones
  • Outbound Endpoint — VPC resources query on-prem DNS servers
  • Forwarding rules define which domains go where
  • Used in Direct Connect / VPN hybrid architectures
Exam Tip
On-prem → resolve AWS private names = Inbound Resolver Endpoint. VPC → resolve on-prem names = Outbound Resolver Endpoint. Both require Resolver Rules. Common in hybrid architecture questions.

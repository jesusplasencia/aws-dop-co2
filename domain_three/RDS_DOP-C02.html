Amazon RDS

Relational Database Service — DevOps Professional exam scope

Scalability

Read Replicas

Key Facts

  • Up to 15 Read Replicas per instance
  • Replication is ASYNC — eventual consistency
  • Replicas can be in Same AZ, Multi-AZ, or Cross-Region
  • Each replica has its own DNS endpoint — app must update connection string
  • Replicas are read-only — not for writes
  • Can be promoted to standalone DB (breaks replication)

Network Cost

  • FREE Same-region replication (cross-AZ within region)
  • BILLED Cross-region replication incurs data transfer fees
  • Replicas in another region also help with disaster recovery
  • Cross-region replica can serve as a standby for regional failover

⚑ Exam Traps
  • Read Replicas do NOT provide automatic failover — that's Multi-AZ.
  • You pay for cross-region data transfer even if same AWS account.
  • Analytics/reporting workloads offloaded to a replica = classic DOP-C02 scenario.
  • Read Replica can be configured as Multi-AZ itself for HA + read scaling.
Disaster Recovery

Multi-AZ

How It Works

  • Replication is SYNC — no data loss on failover
  • One DNS name — app always points to same endpoint
  • Automatic failover (60–120 sec) — no manual intervention
  • Standby is in a different AZ, same region
  • Standby is NOT readable — purely passive
  • Failover triggers: AZ outage, DB instance failure, OS patching, manual trigger

Multi-AZ vs Read Replica

  • Multi-AZ Purpose: High Availability
  • Read Replica Purpose: Read Scaling
  • Multi-AZ replication: SYNC → no lag
  • Read Replica replication: ASYNC → possible lag
  • Multi-AZ standby: Not readable
  • Read Replica: Readable
  • Multi-AZ failover: Automatic
  • Read Replica failover: Manual promotion

⚑ Exam Traps
  • Multi-AZ standby cannot serve read traffic — a common distractor answer.
  • Converting Single-AZ → Multi-AZ is a zero-downtime operation (takes a snapshot, restores in new AZ, enables SYNC).
  • Maintenance / patching applies to standby first, then failover, then patch the new standby — minimizes downtime.
  • Backups in Multi-AZ are taken from the standby — reduces I/O impact on master.

Storage

Storage Auto Scaling & Types

Storage Auto Scaling

  • Automatically increases storage when free space is low
  • Trigger: free space < 10%, low storage for 5 min, 6 hrs since last scale
  • Set a Maximum Storage Threshold to cap it
  • Supported for all engines: MySQL, PG, MariaDB, Oracle, MSSQL
  • Zero downtime — happens in the background

Storage Types

  • gp2 General Purpose SSD — older, burst IOPS
  • gp3 New gen SSD — independent IOPS/throughput config
  • io1 / io2 Provisioned IOPS — >32,000 IOPS, critical workloads
  • magnetic Legacy — not recommended
  • Can upgrade from gp2 → gp3 or gp2 → io1 online (no downtime)
Proxy

RDS Proxy

A fully managed, highly available database proxy that sits between your application and RDS. Essential for Lambda + RDS patterns.

Why Use It

  • Pool and share DB connections
  • Reduces "too many connections" errors
  • Serverless / Lambda workloads (short-lived functions)
  • Reduces DB CPU/RAM for connection overhead

Failover Benefit

  • Proxy keeps connections alive during Multi-AZ failover
  • Reduces failover time by up to 66%
  • App doesn't need to re-establish connections
  • Handles failover transparently

Security

  • Enforces IAM authentication
  • Stores credentials in Secrets Manager
  • Never publicly accessible — VPC only
  • Works with RDS and Aurora
⚑ Exam Scenario

Lambda functions are creating too many RDS connections and causing errors → use RDS Proxy. This is the canonical DOP-C02 scenario for this topic.

Backups & Snapshots

Backups, Restore & Snapshots

Automated Backups

  • Daily full backup during the backup window
  • Transaction logs backed up every 5 minutes
  • Point-in-time recovery (PITR) to any second within retention window
  • Retention: 1–35 days (0 = disable)
  • Stored in S3 (managed by AWS, not user-visible)
  • Deleted when DB instance is deleted (unless you create a final snapshot)

Manual Snapshots

  • User-triggered — persist indefinitely until manually deleted
  • Can copy snapshots across regions for DR
  • Can share snapshots with other AWS accounts
  • Restoring snapshot creates a new DB instance
  • Restore from snapshot + apply transaction logs → PITR
  • Cost: you pay for snapshot storage (first snapshot = full; incremental after)

Security

Encryption

Encryption At-Rest

  • Uses AWS KMS (AES-256)
  • Must be enabled at launch — cannot enable on existing unencrypted DB
  • Encrypts: data, automated backups, snapshots, logs, read replicas
  • To encrypt existing DB: snapshot → copy with encryption → restore
  • Read replicas of encrypted DB are automatically encrypted

Encryption In-Transit

  • Uses TLS/SSL by default
  • Enforce with DB parameter: rds.force_ssl=1
  • Can download AWS RDS CA certificate for client verification
  • IAM DB authentication available (MySQL, PostgreSQL) — token-based, 15 min TTL
  • VPC Security Groups control network access
⚑ Exam Trap

To encrypt an unencrypted RDS instance: you cannot enable encryption directly. Correct process: create an encrypted snapshot copy, then restore it as a new encrypted instance.

Monitoring

RDS Events & Enhanced Monitoring

RDS Events (SNS)

  • Subscribe to events via Amazon SNS
  • Event categories: availability, backup, configuration change, failover, failure, low storage, maintenance, notification, read replica, recovery, restoration, security
  • Can trigger Lambda or SNS → SQS → Lambda for automation
  • Events are emitted for: DB instances, snapshots, security groups, param groups

Enhanced Monitoring & Logs

  • Enhanced Monitoring: OS-level metrics (CPU, memory, disk I/O per process) — up to 1-second granularity
  • Sent to CloudWatch Logs (separate from standard CW metrics)
  • DB logs available: error log, slow query log, general log, audit log
  • Performance Insights: visualize DB load, identify bottleneck queries
  • CloudWatch Alarms on metrics like FreeStorageSpace, DatabaseConnections
Operations

Maintenance, Parameter Groups & Option Groups

Maintenance Window

  • Weekly scheduled window for patches
  • Minor version upgrades: can auto-apply
  • Major version upgrades: manual only
  • With Multi-AZ: standby patched first, then failover, then master patched
  • Can defer maintenance actions up to 35 days

Parameter Groups

  • DB engine configuration (e.g. max_connections, innodb settings)
  • Static params: require DB reboot
  • Dynamic params: apply immediately
  • Default param group: cannot modify — create a custom one
  • Apply at DB instance or cluster level

Option Groups

  • Enable additional features (e.g. Oracle TDE, MSSQL Transparent Data Encryption)
  • Some options persistent, some not
  • Must create custom option group — default is non-modifiable
  • Can specify port, VPC security group per option

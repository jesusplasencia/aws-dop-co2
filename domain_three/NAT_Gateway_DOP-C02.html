01

Core Concepts & Architecture

Managed service

AWS-managed NAT

No OS patches, no failover scripts, no instance management. AWS handles availability within the AZ automatically.

Addressing

Elastic IP required

Each NATGW is assigned one static Elastic IP (public IPv4). This is the source IP seen by internet destinations. Cannot be changed after creation.

Placement

Lives in public subnet

Created in a public subnet of a specific AZ. Private subnet instances route outbound internet traffic through it.

Bandwidth

5 Gbps baseline, auto-scales to 100 Gbps. No manual action needed.

Ports

Supports TCP, UDP, ICMP. Does not support IP protocols beyond those three.

IPv6

NAT Gateway supports IPv4 only. For IPv6 egress use an Egress-Only IGW instead.

Security groups

Cannot attach security groups to NATGW. Control access via NACLs and SGs on the private instances.

Critical rule

NATGW cannot be used to route traffic from within the same subnet it resides in. It must be in a different (public) subnet than the instances using it. Also: a NATGW cannot be used as a bastion host or to receive inbound-initiated connections — it is egress-only for private resources.

02

When to Use NAT Gateway

Private subnet resources need outbound internet access without being reachable from the internet. The private subnet has no route to an IGW — NATGW is the bridge.

Common use cases
  • EC2 instances pulling OS updates (yum, apt) from public repos
  • Lambda functions in a VPC calling external APIs or AWS public endpoints
  • ECS / EKS tasks needing to pull images from Docker Hub or public ECR
  • Private RDS instances downloading software patches
  • Instances calling AWS services in other regions not covered by VPC Endpoints
  • Sending outbound webhooks or notifications to internet URLs
When NOT to use NATGW
  • Traffic between two private subnets in the same VPC — use VPC routing directly
  • Traffic to AWS services that have VPC Interface Endpoints (cheaper, no NAT needed)
  • Inbound connections from the internet — use ALB/NLB in public subnet instead
  • IPv6 egress — use Egress-Only Internet Gateway
  • Inter-VPC traffic — use VPC Peering or Transit Gateway
Exam tip — Cost optimisation angle

If private instances only need to reach AWS services (S3, DynamoDB, SSM, Secrets Manager, ECR, etc.) — use VPC Endpoints instead of routing through a NATGW. VPC Gateway Endpoints (S3, DynamoDB) are free. Interface Endpoints cost less than NAT data transfer charges at scale. A common DOP-C02 scenario asks you to reduce NAT Gateway costs.

03

High Availability Pattern

A single NATGW is highly available within its AZ — AWS manages redundancy inside the AZ. However if the AZ fails, instances in other AZs lose internet access if they all route through one NATGW. The exam-correct HA pattern is one NATGW per AZ.

Exam tip — HA vs Cross-AZ cost trade-off
  • One NATGW for all AZs: single point of AZ failure + cross-AZ data transfer charges (adds cost per GB).
  • One NATGW per AZ: AZ-resilient + each AZ's traffic stays local (no cross-AZ charges), but you pay per-hour for each NATGW.
  • The exam-correct answer for production resilience is always one NATGW per AZ with AZ-local routing.

Exam tip — Common misconfiguration traps
  • NATGW in a private subnet — will not work. NATGW must be in a public subnet with a route to IGW.
  • Missing IGW on the VPC — NATGW cannot reach internet without an attached IGW.
  • Private subnet route points to wrong AZ's NATGW — works but incurs cross-AZ data transfer charges.
  • NATGW not yet in Available state — it takes ~1 min to provision; route exists but traffic drops.
  • NACL blocking outbound on the public subnet — NATGW traffic is affected by subnet-level NACLs (both inbound and outbound rules matter for return traffic).
05

NAT Gateway vs NAT Instance

Dimension NAT Gateway NAT Instance
Management Fully managed by AWS. No patching, no failover scripts. You manage: OS patches, monitoring, failover automation.
Availability Highly available within the AZ automatically. Single EC2 instance; you must script Multi-AZ failover.
Bandwidth 5 Gbps → auto-scales to 100 Gbps. Limited by EC2 instance type (e.g. c5n.18xlarge ≈ 100 Gbps max).
Public IPv4 ✔ Elastic IP assigned at creation ✔ Elastic IP or auto-assigned public IP
Private IPv4 ✔ Private IP in the public subnet ✔ Private IP in subnet
Security Groups ✘ Cannot attach SGs to NATGW ✔ Full SG support — attach inbound/outbound rules
Bastion Host ✘ Not possible — no SSH/RDP access ✔ Can serve dual purpose as bastion + NAT
Source/dest check Disabled automatically by AWS. Must be manually disabled on the EC2 instance — most common exam trap.
Cost Per hour + per GB data processed. Per hour (EC2) + EC2 instance type cost + data transfer.
Port forwarding ✘ Not supported ✔ Configurable via iptables
IPv6 support ✘ IPv4 only (use Egress-Only IGW for IPv6) ✔ Can handle IPv6 with custom config
Exam tip — The one thing about NAT Instances

If the exam mentions a NAT instance where traffic is failing, the answer is almost always: Source/Destination Check must be disabled on the EC2 instance. EC2 by default drops packets that are not addressed to/from its own IP. NAT instances forward packets on behalf of others — check must be off.

06

Cost Model

Hourly charge

Per NATGW per hour

Charged for every hour the NATGW exists, even with zero traffic. Delete unused NATGWs to stop billing.

Data processing

Per GB processed

Charged per GB of data that passes through the NATGW in either direction. Applies to all traffic regardless of destination.

Data transfer

Cross-AZ charges

If instances in AZ-B route through a NATGW in AZ-A, AWS charges cross-AZ data transfer fees in addition to NATGW processing fees.

Cost optimisation strategies

Use VPC Endpoints
  • Gateway Endpoints (S3, DynamoDB): free, no data processing charge
  • Interface Endpoints: hourly + per-GB, but cheaper than NAT for high-volume AWS service traffic
  • Eliminates NAT processing charge for all traffic to those services
Architecture choices
  • Place NATGW in same AZ as the majority of compute (avoid cross-AZ fees)
  • Delete NATGWs in dev/staging environments when not in use
  • Use one NATGW per AZ (not per subnet) — multiple subnets in same AZ share one NATGW
  • Monitor with CloudWatch BytesOutToDestination to identify high-traffic sources
07

Security & Monitoring

Security controls available

NACLs

Apply on the public subnet where NATGW lives. NACLs are stateless — you must allow both outbound (requests) and inbound (return traffic) on ephemeral ports 1024–65535.

Security Groups (on instances)

Control what traffic the private instances can send outbound. NATGW itself has no SG — the first security boundary is the instance SG.

DNS Firewall / Network Firewall

For content inspection or domain filtering on outbound traffic through NATGW, deploy AWS Network Firewall in a dedicated inspection VPC or between NATGW and IGW.

CloudWatch metrics for NATGW

MetricWhat it measuresExam relevance
BytesInFromDestination Bytes received from internet (inbound return traffic) Monitor data download volume; cost driver
BytesOutToDestination Bytes sent to internet by NATGW Identify high-egress workloads; cost driver

ConnectionEstablishedCount Successful outbound connections Baseline traffic patterns
ErrorPortAllocation Times NATGW could not allocate a port Port exhaustion — NATGW supports 55,000 simultaneous connections per unique destination. Alert on this metric.

Exam tip — Port exhaustion

NATGW supports 55,000 simultaneous connections to the same destination IP:port. If thousands of instances all hit the same endpoint, you may hit port exhaustion (ErrorPortAllocation rises). Solution: spread traffic across multiple destination IPs, or deploy multiple NATGWs and distribute instances across subnets.

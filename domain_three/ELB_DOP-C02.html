Gateway LB (GWLB)

Targets
Third-party virtual appliances (firewalls, IDS/IPS, deep packet inspection)
TLS
Passes traffic through unmodified; appliance handles TLS
Sticky sessions
5-tuple or 3-tuple flow stickiness
Use case
Security appliance chaining, inline inspection before traffic reaches app
Exam tip — Which LB to choose
  • HTTP routing rules, path-based, WebSocket, gRPC, Lambda target → ALB
  • Static IP, extremely low latency, TCP/UDP, PrivateLink service endpoint → NLB
  • Inline security appliances (firewall, IDS/IPS) for all VPC traffic → GWLB
  • NLB supports Elastic IPs per AZ — ALB does not. Critical for IP whitelisting requirements.
02

ALB Listener Rules

Rules are evaluated in priority order (lowest number first). The default rule has no conditions and is always last — it acts as the catch-all fallback. Each rule = one or more conditions + one action.

Priority 1
host-header is api.example.com AND path-pattern is /v2/*
→ forward TG-v2
Priority 2
path-pattern is /static/*
↗ redirect S3
Priority 3
http-request-method is OPTIONS
✦ fixed 200
Default
No conditions — matches everything not caught above
→ forward TG-main

Rule conditions (ALB only)

ConditionExample
host-header*.example.com, api.example.com
path-pattern/api/*, /images/*.jpg
http-headerCustom header name + value (exact or wildcard)
http-request-methodGET, POST, OPTIONS
query-stringKey=value pair, e.g. version=2
source-ipCIDR block, e.g. 192.0.2.0/24

Rule actions

forward

Route request to one or more target groups. Supports weighted forwarding across multiple TGs.

redirect

Return HTTP 301/302 redirect. Can rewrite protocol (HTTP→HTTPS), host, port, path, query string.

fixed-response

Return a fixed HTTP response (status code + content-type + message body). No target group needed.

Exam tip — Rules & conditions
  • Multiple conditions in one rule = AND logic (all must match).
  • Multiple values in one condition = OR logic (any value matches).
  • Max 100 rules per listener (including default). Max 5 conditions per rule.
  • fixed-response is useful for returning a custom maintenance page or CORS preflight without hitting backend.
  • redirect HTTP→HTTPS is the standard way to enforce HTTPS at the LB level.
03

Target Group Weighting (Weighted Routing)

On a single forward rule, assign an integer weight to each target group. Traffic is distributed proportionally. Weight range: 0–999. Weight 0 stops sending traffic to that TG without removing it from the rule.

Live example — blue/green canary rollout
Blue (stable) — 90%
10%
Target Group Blue (weight 9) — current production
Target Group Green (weight 1) — new version canary
Gradually shift weight to Green → 50/50 → 0/10 to complete cutover. Set Blue weight to 0 for instant rollback.
Use cases
  • Blue/green deployments — gradual traffic shift
  • Canary releases — test new version with % of traffic
  • A/B testing — split traffic between two app variants
  • Multi-region weighted failover (via Route 53 + ALB)
Key constraints
  • Only available on ALB forward action, not redirect/fixed-response
  • Max 5 target groups per rule
  • Weights are relative — total does not need to equal 100
  • Weight 0 = no new requests sent, existing connections complete
  • Sticky sessions per-TG are still respected within each TG's share
Exam tip — Weighted TGs vs CodeDeploy

ALB Target Group Weighting is self-managed — you update the rule manually or via API/CLI. For automated blue/green with health check gates and automatic rollback, use AWS CodeDeploy with ECSBlueGreenDeployment deployment type — CodeDeploy manages the ALB weight shifting automatically.

04

Target Groups & Health Checks

Target types

instance

EC2 instance ID. Traffic routed to instance's primary private IP on the port.

ip

Any routable IP in VPC, peered VPC, or on-premises via Direct Connect / VPN.

lambda

Lambda function ARN. ALB only. Request/response translated to JSON event. Supports multi-value headers.

alb

NLB only. Chain NLB → ALB for static IP with L7 routing behind it.

Health check configuration

ParameterDescriptionDefault / Typical
ProtocolHTTP, HTTPS, TCP (NLB). Must match target's capability.HTTP
PathALB/HTTPS only. The URL path pinged. Should return 2xx/3xx./
HealthyThresholdConsecutive successes to mark Healthy.5 (ALB), 3 (NLB)
UnhealthyThresholdConsecutive failures to mark Unhealthy and stop sending traffic.2
TimeoutSeconds to wait for response.5s
IntervalSeconds between health checks per target.30s
SuccessCodesHTTP codes that count as healthy. Range or comma-separated list.200
Exam tip — Health check nuances
  • ALB health checks originate from the ALB nodes' IPs — your target's SG must allow inbound from the ALB SG (not client SG).
  • NLB health checks for TCP targets use a TCP handshake — no HTTP path needed.
  • Lambda targets: health checks are disabled by default — Lambda is always considered healthy.
  • Unhealthy targets are removed from rotation but not terminated — ELB does not terminate instances.
  • ELB access logs to S3 do not record health check requests — only client requests.
05

SSL/TLS Termination & SNI

TLS Termination at ELB

Off-load from targets

ELB decrypts HTTPS from clients and forwards HTTP (or re-encrypted HTTPS) to targets. Certificate stored in ACM (recommended) or IAM.

Server Name Indication (SNI)

Multiple certs on one listener

ALB and NLB support SNI — multiple TLS certificates on a single HTTPS listener. ELB selects the certificate matching the client's hostname in the TLS handshake. Supports up to 25 certs per listener (default).

ACM Integration

Auto-renews certificates. Free for use with ELB. Certificate must be in same region as ELB.

Security Policy

Choose TLS protocol versions and cipher suites. Use ELBSecurityPolicy-TLS13-* for TLS 1.3 enforcement.

End-to-end encryption

Set HTTPS on both listener and target group. ELB re-encrypts to targets — requires valid cert on targets (self-signed OK for NLB pass-through).

Exam tip — CLB vs ALB for SNI

Classic Load Balancer (CLB) does not support SNI — only one certificate per CLB. ALB/NLB support SNI with multiple certificates. If a scenario requires multiple domains on one HTTPS endpoint, the answer is ALB (not CLB).

06

Sticky Sessions (Session Affinity)

Routes requests from the same client to the same target for the duration of a session. Enabled per target group.

TypeCookieLBDuration control
Duration-based AWSALB (ALB) / AWSALBTG ALB, CLB 1 sec to 7 days — set by ELB
Application-based Custom name (not AWSALB/AWSALBTG) ALB only App sets cookie TTL; ELB encodes TG info inside app cookie
Source IP (NLB) No cookie — flow-hash based on client IP NLB only Persistent until connection ends or target unhealthy
Exam tip — Sticky sessions and scaling

Sticky sessions break even load distribution during scale-out — new instances receive fewer requests if existing clients stick to old ones. For stateless apps prefer no stickiness + externalized session state (ElastiCache/Redis or DynamoDB). Stickiness is a workaround for stateful apps that cannot be refactored.

07

Connection Draining / Deregistration Delay

CLB — Connection Draining

1–3600 seconds (default 300s)

When a target is deregistered or unhealthy, CLB stops sending new requests but waits up to the drain period for in-flight requests to complete before removing it.

ALB / NLB — Deregistration Delay

0–3600 seconds (default 300s)

Same concept, different name. Target enters draining state. Set to 0 for instant removal (Lambda targets). Set lower for fast-rolling deployments.

Exam tip — Deployment speed vs graceful shutdown

For CodeDeploy or ASG rolling deployments where you need fast instance replacement, lower the deregistration delay. For long-running requests (video uploads, report generation), increase it to match max request duration. If requests are timing out during deployment, deregistration delay is likely too short.

08

DualStack Networking

What it does

IPv4 + IPv6 on one ELB

ELB receives traffic from both IPv4 and IPv6 clients on the same DNS name. Supported on ALB and NLB (not GWLB, not CLB).

IP version mapping

Client version = target version

IPv4 clients → forwarded to IPv4 targets. IPv6 clients → forwarded to IPv6 targets. ELB does not translate between IPv4 and IPv6.

AZ enablement

Each AZ where the ELB should accept traffic must be explicitly enabled. Instances in disabled AZs receive no traffic.

DNS name

ELB DNS returns both A record (IPv4) and AAAA record (IPv6). Clients automatically use the appropriate record.

Subnet requirement

VPC and subnets must have IPv6 CIDR blocks assigned. Targets must have IPv6 addresses if routing IPv6 traffic to them.

09

PrivateLink Integration (NLB → VPC Endpoint Service)

PrivateLink allows you to expose a service in your VPC to other VPCs privately — without VPC Peering, without overlapping CIDR constraints, without internet exposure. Requires an NLB as the endpoint.

Consumer VPC (10.0.0.0/16)
Private Subnet A (10.0.1.0/24)
EC2 → VPC Interface Endpoint (ENI with private IP)
Private Subnet B (10.0.2.0/24)
EC2 → VPC Interface Endpoint (ENI with private IP)
→
Provider VPC (10.0.0.0/16 — overlapping OK)
NLB (Endpoint Service)
Traffic received by NLB → forwarded to targets
Private Subnet A (10.0.1.0/24)
EC2 targets
Private Subnet B (10.0.2.0/24)
EC2 targets
Provider side
  • Create NLB in provider VPC
  • Create VPC Endpoint Service pointing to NLB
  • Optionally require acceptance of connection requests
  • Whitelist specific consumer AWS account IDs
Consumer side
  • Create VPC Interface Endpoint pointing to provider's service name
  • ENIs created in chosen subnets with private IPs
  • Route to service using endpoint DNS name or private DNS
  • Attach SG to Interface Endpoint (controls who can use it)
Why PrivateLink over Peering
  • Overlapping CIDRs are fine — no routing table complexity
  • One-way access — consumer cannot reach other resources in provider VPC
  • Scales to thousands of consumers without peering mesh
  • Used by all AWS services (S3, DynamoDB Interface Endpoints use this)
Exam tip — NLB is required for PrivateLink

Only NLB can back a VPC Endpoint Service. ALB cannot. If a scenario needs to expose a service via PrivateLink and currently uses an ALB, you must front it with an NLB (NLB → ALB chaining). PrivateLink is the answer whenever: overlapping CIDRs prevent peering, you need service isolation, or you're a SaaS provider exposing to customers.

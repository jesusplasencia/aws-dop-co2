Amazon ElastiCache

Managed in-memory caching — Redis & Memcached — DevOps Professional exam scope

Core

What ElastiCache Does

Purpose

  • Managed Redis or Memcached — AWS handles OS patching, backups, monitoring
  • In-memory key-value store — sub-millisecond latency
  • Offloads read-heavy workloads from RDS / Aurora
  • Requires application code changes — app must check cache before DB
  • Stateless architecture enabler — store sessions externally

Cache Patterns

  • Lazy Loading Cache miss → read DB → write to cache. Stale data possible.
  • Write-Through Write to DB + cache simultaneously. Fresh data, higher write cost.
  • Session Store Store user sessions with TTL. Enables stateless app tier (horizontal scaling).
  • Cache-Aside App manages cache manually — same as Lazy Loading terminology.
  • Set TTL (time-to-live) to avoid stale data buildup

⚑ Exam Note
  • ElastiCache does not automatically cache — your application code must implement the cache-check logic. This is a required code change.
  • Session handling pattern: user logs in → write session to ElastiCache → any app instance retrieves it → enables stateless EC2 fleet behind an ALB.
Engines

Redis vs Memcached

Redis

  • Multi-AZ with automatic failover
  • Read Replicas for scaling reads + HA
  • Data durability: AOF persistence (append-only file)
  • RDB snapshots — periodic point-in-time backup
  • Backup and restore supported
  • Supports Sets, Sorted Sets, Lists, Hashes — rich data structures
  • Pub/Sub messaging
  • Single-threaded (mostly) — atomic operations
  • Supports Lua scripting and Transactions
  • In-transit and at-rest encryption
  • HIPAA eligible, PCI DSS compliant

Memcached

  • Multi-node sharding — horizontal data partitioning
  • No replication — no Multi-AZ, no HA
  • Non-persistent — data lost on restart/failure
  • Backup & restore via Serverless tier only
  • Multi-threaded — better CPU utilization on multi-core
  • Simple key-value only — no complex data types
  • Lower overhead per connection
  • Scales out by adding nodes (sharding)
  • No encryption support (standard)
  • Use when: pure caching, simple data, max throughput

⚑ Exam — Pick Redis When
  • Need data durability / persistence (AOF)
  • Need Multi-AZ failover
  • Need Sorted Sets (e.g. gaming leaderboards, real-time ranking)
  • Need Pub/Sub (real-time notifications)
  • Need backup and restore
  • → Pick Memcached only for pure horizontal sharding, no HA, multi-threaded throughput, simple cache

⚑ Exam — Cluster Mode Decision
  • Need to scale writes → Cluster Mode Enabled (data partitioned across shards)
  • Need to scale reads only → Cluster Mode Disabled (add replicas)
  • Online resharding: Cluster Mode Enabled supports adding/removing shards without downtime
  • Cluster Mode Disabled: single shard — if node fails and no replicas, data is lost

  • Cluster Mode Enabled requires a cluster-aware Redis client — standard clients won't work correctly (they don't understand hash slot routing).
  • Using the Configuration Endpoint with a non-cluster-aware client → MOVED errors.
Scaling

Scaling Strategies

Redis — Cluster Mode Disabled

  • Horizontal scaling: add/remove Read Replicas (up to 5)
  • Vertical scaling: change node type (e.g. cache.r6g.large → xlarge)
  • Vertical scaling triggers a failover — plan for brief disruption
  • No data sharding — all data on single shard
  • Auto Scaling: scale replicas based on CPUUtilization or DatabaseMemoryUsagePercentage

Redis — Cluster Mode Enabled

  • Online resharding: add/remove shards without downtime
  • Shard rebalancing: redistributes hash slots across shards
  • Auto Scaling: scale shards (not just replicas) based on metrics
  • Target tracking policy: e.g. keep ElastiCachePrimaryEngineCPUUtilization at 60%
  • Scheduled scaling: predictable load patterns

Memcached Scaling

  • Horizontal: add/remove nodes (sharding — data spreads automatically)
  • Vertical: change node type — requires new cluster + data re-population
  • No replication — scaling out adds shards, not replicas
  • Auto Discovery: clients use config endpoint to find new nodes automatically
Security

Security

Redis AUTH & Access

  • Redis AUTH: password/token set at cluster creation
  • RBAC (Redis 6+): user-level access control with ACLs
  • IAM Auth: Not supported for Redis directly — use AUTH token
  • Secrets Manager: store Redis AUTH token
  • VPC Security Groups control network access

Encryption

  • In-transit: TLS/SSL (Redis only)
  • At-rest: KMS encryption (Redis only)
  • Encryption must be enabled at creation — cannot change later
  • Memcached: no native encryption support (standard clusters)
  • ElastiCache Serverless: encryption enabled by default

Network

  • Deployed inside a VPC — never public internet
  • Security Groups: control which EC2/Lambda can connect
  • Subnet Groups: define which subnets ElastiCache deploys into
  • No public endpoint — always private
  • Cross-VPC: VPC Peering or Transit Gateway
Serverless

ElastiCache Serverless

What It Is

  • No cluster or node management — fully managed capacity
  • Supports Redis and Memcached
  • Scales compute and memory automatically based on demand
  • High availability built-in — data replicated across multiple AZs
  • Encryption at rest and in transit enabled by default
  • Billed per ECU (ElastiCache Compute Unit) and GB stored

When to Use

  • Unpredictable or spiky cache workloads
  • Dev/test environments — no capacity planning needed
  • Variable cache sizes — avoid over-provisioning
  • New applications — start fast without sizing decisions
  • DOP-C02 Serverless = operational simplicity, auto-scaling, no patching

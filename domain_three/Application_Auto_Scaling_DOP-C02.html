01

What is Application Auto Scaling

A single API and control plane for scaling resources across 15+ AWS services. Unlike EC2 Auto Scaling Groups (EC2-only), Application Auto Scaling abstracts scaling across any supported scalable resource using a unified model: Scalable Target → Scaling Policy → CloudWatch Alarm → Scaling Action.

Unified API

Single control plane

One API (application-autoscaling) registers and manages scaling for any supported service. No per-service scaling consoles.

Discover resources

3 discovery methods

Point at resources via CloudFormation Stack, Resource Tags, or EC2 ASG name. AAS discovers all scalable resources within scope.

Policy support

3 policy types

Target Tracking, Step Scaling, and Scheduled Scaling. Predictive Scaling is not supported (EC2 ASG only).

Exam tip — AAS is the API, not the console

When you configure scaling directly in the DynamoDB or ECS console, those consoles call Application Auto Scaling API under the hood. The exam may describe it either way — recognise that any service-level scaling (DynamoDB RCU/WCU, ECS task count, Aurora replicas) is implemented by AAS, not EC2 ASG.

02

Scalable Target Anatomy

A Scalable Target is the registration of a specific resource with AAS. You must register it before attaching any policy. Three mandatory fields uniquely identify every scalable target:

ServiceNamespace
The AWS service. E.g. ecs, dynamodb, rds, lambda, sagemaker, custom-resource
ResourceId
Unique resource path within the service. E.g. for ECS: service/my-cluster/my-service; DynamoDB: table/my-table; Lambda: function:my-func:PROD
ScalableDimension
The specific metric dimension to scale. E.g. ecs:service:DesiredCount, dynamodb:table:ReadCapacityUnits, lambda:function:ProvisionedConcurrency
MinCapacity
Floor — AAS will never scale below this value. Required. Must be ≥ 0.
MaxCapacity
Ceiling — AAS will never scale above this value. Required. Must be ≥ MinCapacity.
RoleARN
IAM role AAS assumes to call the target service APIs and publish CloudWatch metrics. Service-linked role is created automatically for most services.
Exam tip — ScalableDimension is critical

One resource can have multiple scalable dimensions. DynamoDB table has ReadCapacityUnits AND WriteCapacityUnits — each is a separate scalable target registration. Lambda has ProvisionedConcurrency as its only AAS dimension (reserved concurrency is not managed by AAS). ECS has DesiredCount.

Exam tip — Most frequent confusion
  • ECS Service scaling = Application Auto Scaling (DesiredCount). The EC2 instances running ECS tasks = EC2 ASG. Both layers needed for ECS on EC2.
  • Predictive Scaling is EC2 ASG only — AAS does not support it.
  • Lifecycle hooks exist only on EC2 ASG — if a scenario needs pre-launch or pre-terminate hooks, answer is EC2 ASG.
05

Scaling Policies Deep Dive

Target Tracking Scaling

How it works

Define a target value for a metric. AAS automatically creates and manages CloudWatch alarms and calculates the required capacity. You never specify step adjustments.

Predefined metrics (per service)
  • ECS: ECSServiceAverageCPUUtilization, ECSServiceAverageMemoryUtilization, ALBRequestCountPerTarget
  • DynamoDB: DynamoDBReadCapacityUtilization, DynamoDBWriteCapacityUtilization
  • Lambda: LambdaProvisionedConcurrencyUtilization
  • Aurora: RDSReaderAverageCPUUtilization, RDSReaderAverageDatabaseConnections
  • SageMaker: SageMakerVariantInvocationsPerInstance
Custom metric target tracking

Any CloudWatch metric can be used as the tracking target by specifying a CustomizedMetricSpecification. The metric must represent load per unit of capacity (e.g. SQS ApproximateNumberOfMessagesVisible / DesiredCount). AAS cannot use metrics that don't reflect utilization-per-unit.

Exam tip — Target Tracking behaviour

AAS scale-out is aggressive (fast); scale-in is conservative (slow, with a longer cooldown) to avoid thrashing. You can disable scale-in on a Target Tracking policy (DisableScaleIn: true) — useful when you want another policy to manage scale-in independently (e.g. Scheduled scale-in at end of business day).

Step Scaling

How it works

You define a CloudWatch alarm and a set of step adjustments. Each step specifies a metric range and the capacity change (add N, remove N, or set to exact N). Multiple steps fire based on alarm breach magnitude.

Adjustment types
  • ChangeInCapacity — add or subtract absolute count (e.g. +2 tasks)
  • ExactCapacity — set to a specific value regardless of current
  • PercentChangeInCapacity — scale by % of current capacity (rounded up)

Scheduled Scaling

How it works

Set min/max/desired capacity at a specific time or recurrence using cron or at() expressions. Proactive — does not react to metrics.

Key fields
  • ScheduledActionName — unique name
  • Schedule — cron(0 9 * * ? *) or rate(1 hour) or at(2025-01-01T09:00:00)
  • StartTime / EndTime — optional bounded window
  • MinCapacity / MaxCapacity — sets new bounds at scheduled time
  • Timezone: all times in UTC by default
Exam tip — Combining policies

You can attach multiple policies to one scalable target. Common pattern: Scheduled Scaling raises MinCapacity before predicted peak (pre-warm) + Target Tracking handles real-time fluctuation within the new bounds. For Lambda: Scheduled pre-warms provisioned concurrency; Target Tracking scales it during the event window.

Cooldown Periods

Scale-out cooldown

Default: 300s. After a scale-out action, AAS waits this duration before evaluating another scale-out. Set shorter for bursty workloads.

Scale-in cooldown

Default: 300s. After a scale-in action, AAS waits before evaluating another scale-in. Target Tracking uses a longer default scale-in cooldown to avoid premature removal.

Scaling Plans (AWS Auto Scaling console)

A Scaling Plan is a higher-level abstraction in the AWS Auto Scaling service (distinct from Application Auto Scaling API). It discovers resources via CFN Stack, Tags, or EC2 ASG, then applies scaling strategies across all of them centrally.

Optimize for availability

High availability

Scales out aggressively. Keeps resources above the target utilization. Prioritises capacity over cost.

Balance availability & cost

Balanced (default)

Target tracking at 50% utilization. Moderate scale-in/out speed. Suitable for most general workloads.

Optimize for cost

Cost optimized

Scales in aggressively. Keeps utilization high. Accepts higher risk of under-provisioning under unexpected spikes.

Predictive scaling in plans

Available for EC2 ASG only

Within a Scaling Plan, EC2 ASG resources can use predictive scaling (ML forecast). Non-EC2 resources use reactive scaling only.

Resource discovery methods
  • CloudFormation Stack — all scalable resources within a stack
  • Tags — resources matching a key/value tag
  • EC2 ASG — a specific ASG by name
Exam tip — Scaling Plan vs direct AAS policy

Scaling Plans use Application Auto Scaling under the hood but add a discovery layer and strategy templates. If the exam scenario mentions managing scaling across a whole application stack from one place, or mentions the AWS Auto Scaling console, the answer is Scaling Plans. If it's per-resource fine-grained policy control, the answer is direct AAS policy.

07

IAM & Permissions

Service-Linked Roles (SLR)

Auto-created per namespace

AAS creates one SLR per service namespace automatically on first use. Named AWSServiceRoleForApplicationAutoScaling_[Service]. The role grants AAS permission to call the target service's APIs (e.g. ecs:UpdateService) and publish CloudWatch metrics.

Caller permissions

Who calls AAS

The IAM principal calling RegisterScalableTarget, PutScalingPolicy etc. must have application-autoscaling:* + iam:CreateServiceLinkedRole (for first-time registration) + target service describe/update permissions.

ServiceService-Linked Role nameKey permissions granted
ECS ...ApplicationAutoScaling_ECSService ecs:DescribeServices, ecs:UpdateService
DynamoDB ...ApplicationAutoScaling_DynamoDBTable dynamodb:DescribeTable, dynamodb:UpdateTable
Aurora ...ApplicationAutoScaling_RDSCluster rds:DescribeDBClusters, rds:CreateDBInstance, rds:DeleteDBInstance
Lambda ...ApplicationAutoScaling_LambdaConcurrency lambda:PutProvisionedConcurrencyConfig, lambda:GetProvisionedConcurrencyConfig
Exam tip — SLR creation failure

If a user tries to register a scalable target but gets an IAM permission error, they likely lack iam:CreateServiceLinkedRole. Once the SLR exists, this permission is no longer needed. This is a common scenario question.

DynamoDB throttles on reads during peak AAS Target Tracking on DynamoDBReadCapacityUtilization at ~70%. Or switch to On-Demand mode (bypasses AAS).

Predictive Scaling for non-EC2 service Not supported. Predictive Scaling is EC2 ASG only.
Lifecycle hooks on ECS scaling Not supported by AAS. ECS has its own task lifecycle events via EventBridge.
Disable scale-in on Target Tracking DisableScaleIn: true on the policy. AAS will not remove capacity; use another policy or manual action for scale-in.
Scale multiple services from one place AWS Auto Scaling Scaling Plan (discovers via CFN stack, tags, or ASG name).

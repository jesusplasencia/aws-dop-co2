01

ASG Core Concepts

Min Capacity

Minimum

Floor — ASG never terminates below this count. Scale-in stops here.

Desired Capacity

Target

The count ASG actively tries to maintain. Adjustable by policies at runtime.

Max Capacity

Maximum

Ceiling — ASG never launches beyond this count regardless of load.

Exam tip

ASG requires a Launch Template (preferred) or Launch Configuration (legacy, no versioning). Launch Templates support multiple instance types, Spot + On-Demand mix, and T2/T3 Unlimited credits. Launch Configurations cannot be modified — you must create a new one.

Key Integrations

ELB / ALB / NLB

Register instances automatically; ALB health checks replace EC2 health checks for app-level detection.

CloudWatch

Source of all alarm-based scaling triggers. Metrics published every 1 min (detailed) or 5 min (basic).

EC2 Health Checks

Default check. Instance is Unhealthy if EC2 reports status other than running. ELB check is stricter.

02

Scaling Policies

Policy Type How it works Best for Key constraint
Target Tracking Maintains a CloudWatch metric at a defined target value (e.g. CPU = 40%). ASG calculates required capacity automatically. Steady-state workloads. Simplest to configure. Can't scale to zero. Needs metric data — cold starts delay reaction.
Simple Scaling Single CloudWatch alarm triggers a fixed adjustment. Waits for cooldown before next action. Simple on/off alarms where 1 step is enough. Entire cooldown must expire between scaling actions — slow to react to rapid changes.
Step Scaling CloudWatch alarm triggers; multiple step adjustments based on alarm breach magnitude. No cooldown wait between steps. Variable load spikes needing proportional response. More complex to configure than Target Tracking.
Scheduled Scaling Changes min/max/desired at a cron-based schedule. Proactive — does not react to metrics. Known recurring patterns (e.g. business hours, weekly peaks). Requires accurate forecast. Over/under-provision if traffic varies.
Predictive Scaling ML-driven forecast using historical CloudWatch data. Pre-scales before the load arrives. Requires at least 24 h of data for forecasting. Cyclical patterns where reactive scaling causes latency. Forecast mode (observation only) vs Forecast + Scale mode. Min 1-day data needed.

Good Metrics to Scale On

CPU

CPUUtilization

Average across all instances. Target ~40–60% for headroom.

ALB

RequestCountPerTarget

Requests per instance behind ALB. Ideal for web tiers.

Network

NetworkIn / NetworkOut

Average bytes in or out. Useful for upload/download-heavy apps.

Custom

Custom Metric

Publish any metric to CloudWatch via PutMetricData (e.g. queue depth from SQS).

Exam tip

Step Scaling does not wait for the cooldown to expire before evaluating new scaling actions — it continues processing alarms. Simple Scaling does wait. For fast-scaling workloads, prefer Step or Target Tracking.

03

Lifecycle Hooks

Lifecycle hooks pause an instance at a transition state so you can run custom actions (install software, drain connections, collect logs) before the ASG proceeds. Maximum wait timeout: 7200s (2h), default 3600s (1h). Extend with record-lifecycle-action-heartbeat or complete with complete-lifecycle-action.

Hook Events & Use Cases

EC2_INSTANCE_LAUNCHING

Launch Hook actions

  • Install/configure application software
  • Pull config from Systems Manager Parameter Store
  • Run health/readiness checks before serving traffic
  • Register with external service discovery
EC2_INSTANCE_TERMINATING

Terminate Hook actions

  • Drain active connections / finish in-flight requests
  • Collect and ship logs (SSM Run Command → EventBridge)
  • Deregister from service discovery or external LB
  • Snapshot EBS volumes before deletion
Exam tip — Triggering from hooks

Hooks emit events that can trigger: EventBridge rule → Lambda (most common for automation), SNS, or SQS. The hook enters Wait state; your automation must call complete-lifecycle-action CONTINUE (or ABANDON) to proceed. If the heartbeat timeout expires, the default result (CONTINUE or ABANDON) is applied.

Hook Timeout & Heartbeat

HeartbeatTimeout

Max wait: 7200s. Default: 3600s. Extend with record-lifecycle-action-heartbeat before timeout.

DefaultResult

If timeout expires without complete-lifecycle-action call — either CONTINUE (default) or ABANDON.

ABANDON vs CONTINUE

ABANDON on launch = instance terminated. ABANDON on terminate = instance terminated immediately without waiting.

04

Scaling Cooldowns

Default Cooldown

300 seconds (5 min)

After any scaling activity (up or down), ASG waits this period before executing another scaling action. Applies to Simple Scaling only.

Scaling-Specific Cooldown

Override per policy

Set a shorter cooldown on scale-in policies to terminate instances faster. Set a longer cooldown on scale-out to avoid thrashing.

Exam tip — Reduce cooldown

Use a pre-baked (golden) AMI that includes all dependencies. Faster boot → instance reaches InService faster → cooldown can be shorter → ASG responds to load changes more quickly. Target Tracking and Step Scaling are not subject to the default cooldown (they have their own instance warmup period).

Instance Warmup (Step & Target Tracking)

Separate from cooldown. Newly launched instances are not counted toward ASG metrics until the instance warmup period expires. This prevents premature scale-out decisions caused by high CPU on a booting instance. Configurable per ASG.

05

Termination Policies

On scale-in, ASG first selects the AZ with the most instances to rebalance, then applies the termination policy to choose which instance within that AZ to terminate.

Policy Terminates Typical use case
Default AZ rebalance → oldest Launch Template/Config → closest billing hour General purpose. Applied when no policy specified.
AllocationStrategy Aligns remaining instances to allocation strategy (lowest-price Spot, lowest-priority On-Demand) Mixed instance groups with Spot.
OldestLaunchTemplate Instance using the oldest version of the Launch Template Rolling out a new Launch Template version.
OldestLaunchConfiguration Instance using the oldest Launch Configuration Migrating from LC to LT.
ClosestToNextInstanceHour Instance closest to the next billing hour boundary Minimise cost — avoid paying for partial hours.
NewestInstance Most recently launched instance Testing a new launch template — keep stable instances.
OldestInstance Oldest running instance (not based on template) Refreshing instance size/type without changing template.
Exam tip — Custom Termination Policy

You can define a Custom Termination Policy backed by a Lambda function. ASG calls Lambda with candidate instances; Lambda returns the instance(s) to terminate. Useful for stateful workloads (e.g., drain queue consumers with lowest job count). You can also chain multiple built-in policies — ASG evaluates them in order until it narrows to one instance.

Warm Pools

Eliminates scale-out latency by keeping a set of pre-initialized instances ready. When scale-out occurs, instances move from Warm Pool → ASG instead of booting from scratch.

Pool Size Settings

  • Minimum Warm Pool Size — guaranteed number always kept warm, even when ASG is at max capacity.
  • Max Prepared Capacity — absolute cap on total instances (ASG + pool). Prevents over-provisioning.

Instance Reuse Policy

On scale-in, instead of terminating instances, ASG can return them to the Warm Pool. Saves boot time on next scale-out. Configured via ReuseOnScaleIn: true. Only valid if Warm Pool exists.

Warm Pool Instance States

Running

Fastest

Instance is running but not in service. Zero boot delay. Higher cost — you pay for running EC2.

Stopped

Balanced

Instance is stopped. Restart time ~30–60s. Pay only for EBS storage while stopped. Recommended for most workloads.

Hibernated

Memory preserved

Instance RAM saved to EBS. Resume restores in-memory state. Useful for apps with expensive warm-up (e.g. large in-memory caches).

Exam tip — Warm Pool Lifecycle States

Warm Pool has its own lifecycle states: Warmed:Pending → Warmed:Pending:Wait (hook) → Warmed:Pending:Proceed → Warmed:Stopped / Running / Hibernated. When scaling out, instance re-enters: Pending:Wait (hook) → Pending:Proceed → InService. Lifecycle hooks apply at both the warm pool entry and the ASG entry transitions.

07

SNS Notifications

ASG can publish notifications to an SNS topic for 4 event types. Configure via put-notification-configuration. Each event generates an SNS message with AutoScalingGroupName, EC2InstanceId, and event details.

autoscaling:EC2_INSTANCE_LAUNCH
autoscaling:EC2_INSTANCE_LAUNCH_ERROR
autoscaling:EC2_INSTANCE_TERMINATE
autoscaling:EC2_INSTANCE_TERMINATE_ERROR
Exam tip — SNS vs EventBridge

SNS notifications are limited to 4 basic events and configured directly on the ASG. EventBridge supports a richer set of events (Instance Refresh, Checkpoint Reached, etc.) and enables more complex routing to Lambda, SQS, SSM Run Command, Step Functions, etc. For automation tasks, prefer EventBridge over SNS.

08

EventBridge Integration

ASG automatically emits events to EventBridge (source: aws.autoscaling). Create rules to match event patterns and route to targets.

Matchable Event Types

Instance Events
  • EC2 Instance Launching
  • EC2 Instance Launch Successful
  • EC2 Instance Launch Unsuccessful
  • EC2 Instance Terminating
  • EC2 Instance Terminate Successful
  • EC2 Instance Terminate Unsuccessful
Instance Refresh Events
  • EC2 Auto Scaling Instance Refresh Started
  • EC2 Auto Scaling Instance Refresh Succeeded
  • EC2 Auto Scaling Instance Refresh Failed
  • EC2 Auto Scaling Instance Refresh Cancelled
  • EC2 Auto Scaling Instance Refresh Checkpoint Reached

Common EventBridge Target Patterns

Terminating:Wait → EventBridge → Lambda

Log collection

Collect logs / flush metrics before instance terminates. Lambda calls complete-lifecycle-action when done.

Terminating:Wait → EventBridge → SSM

Run Command

Execute SSM Run Command on the terminating instance (e.g. sync logs to S3). Integrated directly — no Lambda needed.

Launch Success → EventBridge → Lambda

Post-launch automation

Trigger config management, register with DNS/CMDB, or send Slack notification on new instance InService.

Exam tip — EventBridge event payload

EventBridge events include "source": "aws.autoscaling" and "detail-type" such as "EC2 Instance Launch Successful". The detail object contains AutoScalingGroupName, EC2InstanceId, StatusCode, and Description. Use content-based filtering in EventBridge rules to match specific ASG names or status codes.

Instance Refresh (bonus — high exam weight)

Instance Refresh is the ASG-native way to roll out a new Launch Template version across all instances without manual intervention. It replaces instances incrementally, respecting health checks and configurable rollout parameters.

MinHealthyPercentage

% of instances that must remain healthy during refresh (e.g. 90%). Controls how many are replaced in parallel.

Checkpoints

Pause refresh at defined percentages to run smoke tests. EventBridge emits Checkpoint Reached; you resume via API or Lambda.

Rollback

If new instances fail health checks, Instance Refresh can automatically roll back to the previous Launch Template version.

Core Purpose

  • Hybrid cloud storage — bridges on-premises apps to AWS storage
  • Exposes AWS storage as familiar protocols: NFS, SMB, iSCSI, VTL
  • On-premises apps need no code changes — use existing interfaces
  • Data transferred over TLS (encrypted in transit)
  • Data stored encrypted at rest in S3, EBS, Glacier
  • Gateway runs as a VM appliance (VMware/Hyper-V/KVM) or hardware appliance
  • Hardware Appliance: physical device for sites without virtualization

Native AWS Storage (Reference)

  • BLOCK Amazon EBS, EC2 Instance Store
  • FILE Amazon EFS
  • OBJECT Amazon S3, Glacier
  • Storage Gateway extends these to on-premises via standard protocols
  • Use case: data center extension, cloud bursting, backup to cloud, disaster recovery
  • All three gateway types use a local cache for low-latency on-prem access to frequently used data
Gateway Types

Three Types of Storage Gateway

📁
S3 File Gateway
NFS v3/v4 · SMB
  • Files stored as objects in S3
  • Most recently used data cached on gateway
  • Access via NFS or SMB from on-prem
  • Supports S3 Standard, Standard-IA, One Zone-IA, Intelligent-Tiering
  • Lifecycle policies apply to objects in S3
  • SMB integrates with Active Directory for auth
  • Bucket access via IAM role on gateway
💾
Volume Gateway
iSCSI block storage
  • Presents iSCSI block volumes to on-prem apps
  • Cached Mode: primary data in S3, frequently accessed data cached locally
  • Stored Mode: primary data on-prem, async backup snapshots to S3 as EBS snapshots
  • Snapshots stored as EBS snapshots in AWS — can restore to EBS volume
  • Low-latency for recently accessed data (cached mode)
  • Use case: extend on-prem block storage to cloud with DR snapshots
📼
Tape Gateway
iSCSI VTL (Virtual Tape Library)
  • Presents a virtual tape library (VTL) to on-prem backup software
  • Compatible with: Veeam, NetBackup, Backup Exec, Commvault, etc.
  • Virtual tapes stored in S3 (active) or archived to S3 Glacier / Deep Archive
  • No physical tape infrastructure needed
  • Tapes in Glacier: retrieved on demand, minutes to hours
  • Use case: replace physical tape backup with cloud-backed VTL

⚑ Exam — Pick the Right Gateway
  • On-prem NFS/SMB file share backed by S3 → S3 File Gateway
  • On-prem app needs block storage (iSCSI), with cloud backups → Volume Gateway (Stored Mode)
  • On-prem app needs block storage with cloud as primary + local cache → Volume Gateway (Cached Mode)
  • Replace physical tape backup infrastructure → Tape Gateway
  • Backup software like Veeam/Commvault connecting to cloud → Tape Gateway VTL
File Gateway

S3 File Gateway — Deep Dive

How Files Become S3 Objects

  • Each NFS/SMB file share is mapped to an S3 bucket
  • Files written to the gateway → stored as S3 objects with same key structure
  • File metadata preserved as S3 object metadata
  • Objects visible directly in S3 — can be accessed via S3 API too
  • Supports S3 Lifecycle policies on the target bucket — auto-tier to Glacier
  • Multiple gateways / file shares can point to the same S3 bucket

Cache Refresh — Stale Data Problem

  • If S3 is updated directly (bypass gateway) → gateway cache becomes stale
  • Other gateway clients won't see updates made by other gateways automatically
  • Fix option 1: RefreshCache API — manually trigger a cache sync
  • Fix option 2: Automated Cache Refresh — periodic automatic refresh
  • Fix option 3: invoke RefreshCache via Lambda on S3 event (e.g. S3 PUT notification → Lambda → RefreshCache)
  • Notify all clients of updates via S3 Event Notifications

FSx File Gateway

  • Alternative to S3 File Gateway for Windows workloads
  • Provides local cache for Amazon FSx for Windows File Server
  • Protocol: SMB + NTFS — full Windows compatibility
  • AD integration, ACLs, DFS (Distributed File System) support
  • Use case: migrate Windows file servers to FSx with on-prem cache
  • Gateway caches frequently accessed data — reduces WAN latency

File Gateway Security

  • SMB auth: Active Directory or guest access
  • NFS auth: IP-based access control (POSIX)
  • Data encrypted in transit: TLS 1.2
  • Data encrypted at rest: S3-SSE (SSE-S3 or SSE-KMS)
  • Gateway IAM role controls which S3 bucket it can write to
  • VPC endpoint available for private connectivity to S3
⚑ Exam — RefreshCache Pattern
  • Multiple offices use separate File Gateways pointing to the same S3 bucket. Office A uploads a file — Office B won't see it until cache refresh.
  • Solution: S3 Object Created notification → SNS/Lambda → RefreshCache API on all other gateways → near-real-time consistency.
  • This is a high-frequency DOP-C02 architecture question — know the exact flow.
Volume Gateway

Volume Gateway — Cached vs Stored Mode

Cached Volume Mode

  • Primary data stored in S3 (not on-prem)
  • Only recently accessed data cached locally
  • Minimizes on-prem storage footprint
  • Low latency for hot data (cache hit)
  • Cache miss = data fetched from S3 (higher latency)
  • Volumes up to 32 TB each, 32 volumes = 1 PB total
  • Snapshots incremental, stored as EBS snapshots
  • Use case: large datasets, limited on-prem capacity, cloud as primary

Stored Volume Mode

  • Primary data stored on-premises
  • Entire dataset on-prem — low latency always
  • Async snapshots uploaded to S3 as EBS snapshots
  • EBS snapshots can be restored as EBS volumes in AWS
  • Volumes up to 16 TB each, 32 volumes per gateway
  • Requires on-prem storage capacity for full dataset
  • Use case: apps needing entire dataset locally with cloud DR
  • Classic hybrid DR pattern: on-prem primary, AWS failover
⚑ Exam — Cached vs Stored
  • "Minimize on-premises storage footprint while keeping low latency for hot data" → Cached Mode.
  • "Full dataset must be available locally at all times, cloud backup only" → Stored Mode.
  • In both modes: EBS snapshots in S3 → restore to EBS volume in AWS → attach to EC2 → cloud DR achieved without RDS or separate backup tool.
Tape Gateway

Tape Gateway — VTL Deep Dive

Virtual Tape Library (VTL)

  • Gateway presents a VTL — backup software sees it as physical tape hardware
  • Virtual Tape Drive: simulates a tape drive
  • Virtual Tape Shelf (VTS): archived tapes in Glacier
  • Active tapes: stored in S3 (accessible within minutes)
  • Archived tapes: moved to Glacier or Glacier Deep Archive
  • Max virtual tape size: 5 TB
  • VTL capacity: up to 1,500 virtual tapes, up to 150 TB

Tape Lifecycle

  • Backup software writes to virtual tape in VTL (S3)
  • Eject tape in backup software → tape archived to Glacier / Deep Archive
  • Retrieve archived tape: initiate retrieval from Glacier → returns to VTL in S3
  • Glacier retrieval: minutes to hours (Expedited/Standard/Bulk)
  • Deep Archive retrieval: up to 12 hours
  • Compatible backup apps: Veeam, Veritas NetBackup, Backup Exec, Commvault, IBM Spectrum, Quest NetVault
Deployment

Deployment Options & Connectivity

VM Appliance

  • Download OVA image from AWS console
  • Deploy on: VMware ESXi, Microsoft Hyper-V, Linux KVM
  • Minimum: 4 vCPUs, 16 GB RAM, 80 GB root disk
  • Separate cache disk and upload buffer
  • Most common deployment option
  • Activate via AWS console with activation key

Hardware Appliance

  • Physical AWS-provided device — order from AWS
  • For environments without virtualization infrastructure
  • Pre-installed gateway software
  • Rack-mounted 1U device
  • Supports all three gateway types
  • Same activation process as VM appliance

Connectivity Options

  • Internet: gateway uploads over public internet (TLS)
  • AWS Direct Connect: private dedicated link — consistent bandwidth
  • VPN: IPsec VPN over internet — encrypted tunnel
  • VPC Endpoint: gateway in VPC communicates privately with S3
  • Direct Connect recommended for large data transfers and consistent latency
⚑ Exam — Connectivity Choice
  • Large volume of data, consistent low-latency uploads → Direct Connect + Storage Gateway.
  • High bandwidth requirement but budget constraints → VPN.
  • All traffic must stay private, no internet exposure → VPC Endpoint + Direct Connect.

Elastic Container Service

Managed container orchestration — run Docker containers on EC2 or serverless via Fargate.

01

Launch Types: EC2 vs Fargate

◈◈ HIGH
Attribute EC2 Launch Type Fargate
Infrastructure You provision & manage EC2 instances Serverless — AWS manages everything
ECS Agent Must run on every EC2 instance to register with cluster Not needed — AWS handles it
Scaling Scale tasks + scale EC2 fleet separately Scale by increasing task count only
Cost model Pay for EC2 instances (always on) Pay per vCPU + memory per second
Best for Cost optimization at scale, GPU workloads Simplicity, variable workloads, microservices
Exam Tip
If a question asks for the least operational overhead → choose Fargate. EC2 Launch Type requires managing the ECS Agent, instance profile permissions, and capacity separately.
02

IAM Roles for ECS

◈◈ HIGH
Critical distinction: Two separate roles serve two separate purposes. Confusing them is a common exam trap.

EC2 Instance Profile

  • EC2 Launch Type only
  • Used by the ECS Agent itself
  • Pulls Docker images from ECR
  • Sends container logs to CloudWatch
  • Makes API calls to ECS service
  • Reads secrets from SSM / Secrets Manager

ECS Task Role

  • Applies to both EC2 and Fargate
  • Defined per task in the Task Definition
  • Grants the container permissions to AWS services (S3, DynamoDB…)
  • Different tasks can have different roles
  • Principle of least privilege per service
Exam Tip
Task Role = what the application inside the container can do. Instance Profile = what the ECS Agent on the EC2 host can do. On Fargate, only Task Role exists.
03

Auto-Scaling

◈◈ HIGH
ECS Service Auto Scaling (task count) ≠ EC2 Auto Scaling (instance count). On EC2 launch type you may need both.

Scaling Policies

  • Target Tracking — maintain a CloudWatch metric at target value (e.g. CPU 70%)
  • Step Scaling — trigger on a CloudWatch Alarm threshold
  • Scheduled Scaling — scale at a specific date/time for predictable load

Key Metrics for Scaling

  • ECSServiceAverageCPUUtilization
  • ECSServiceAverageMemoryUtilization
  • ALBRequestCountPerTarget (from ALB)

EC2 Cluster Capacity

  • Auto Scaling Group — scale based on CPU, add instances over time
  • ECS Cluster Capacity Provider (recommended) — automatically provisions EC2s when tasks can't be placed due to insufficient CPU/RAM
Exam Tip
Fargate Auto Scaling is simpler — no EC2 capacity to manage. For EC2, Capacity Provider is the recommended approach to automatically add instances when cluster is full.

05

Data Volumes (EFS)

◈ MEDIUM

EFS with ECS

  • Mount EFS onto ECS tasks
  • Works with both EC2 and Fargate
  • Tasks in any AZ share the same data
  • Persistent storage that survives task restarts
  • Recommended for shared/stateful workloads

What NOT to use

  • EBS — bound to a single AZ/instance, not shareable across tasks
  • EC2 local storage — ephemeral, lost when container stops
  • Fargate ephemeral storage — 20GB default, not shared
Exam Tip
EFS = multi-AZ shared persistent storage for ECS. If the question mentions "shared data between containers across AZs" → EFS is the answer.
06

Logging

◈◈ HIGH

Fargate Logging

  • Configure logConfiguration in Task Definition
  • Supported drivers: awslogs, splunk, awsfirelens
  • Permissions on the Task Role
  • awsfirelens = Fluent Bit/Fluentd for advanced routing

EC2 Launch Type Logging

  • Uses CloudWatch Unified Agent + ECS Container Agent
  • Config at /etc/ecs/ecs.config
  • Permissions on EC2 Instance Profile
  • Must enable ECS_AVAILABLE_LOGGING_DRIVERS in config

Sidecar Container Pattern

Deploy a dedicated logging container alongside your app container in the same task. It collects logs from other containers and the filesystem, then forwards to CloudWatch or other destinations. Common with awsfirelens / Fluent Bit.

Exam Tip
Fargate → permissions on Task Role. EC2 → permissions on Instance Profile. The awslogs driver sends directly to CloudWatch. awsfirelens enables routing logs to multiple destinations (S3, Kinesis, third-party).
07

Key Integrations

◈◈ HIGH

EventBridge → ECS

  • Trigger an ECS task in response to an AWS event
  • Example: S3 upload → EventBridge Rule → Run Fargate task to process file
  • Task uses Task Role to access S3, DynamoDB, etc.
  • Replaces need for always-running worker processes

EventBridge Scheduler

  • Run ECS tasks on a cron/rate schedule
  • Example: batch job every hour on Fargate
  • Serverless alternative to cron on EC2

SQS → ECS Auto Scaling

  • ECS tasks poll SQS queue for work
  • Scale ECS service based on ApproximateNumberOfMessages
  • Classic worker pattern — more messages = more tasks
  • Tasks scale in when queue drains
Exam Tip
EventBridge + Fargate = serverless event-driven container execution. SQS + ECS Auto Scaling = scalable async worker pattern. Both are very common DOP-C02 architecture scenarios.
08

Task Definitions — Key Exam Points

◈ MEDIUM

What's Defined

  • Docker image (from ECR or Docker Hub)
  • CPU & memory requirements
  • Environment variables (or from SSM/Secrets Manager)
  • Log configuration (logConfiguration)
  • Task Role + Task Execution Role
  • Port mappings

Task Execution Role vs Task Role

  • Task Execution Role — used by ECS agent to pull image, write logs (needed on Fargate)
  • Task Role — used by the app inside the container to call AWS services
  • Both must be set in the Task Definition

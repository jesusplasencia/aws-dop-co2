Amazon Aurora

AWS-optimized relational database — MySQL & PostgreSQL compatible — DevOps Professional exam scope

Architecture

Core Architecture

What Makes Aurora Different

  • AWS-proprietary engine — MySQL and PostgreSQL compatible
  • Storage auto-grows in 10 GB increments, up to 128 TB
  • Up to 15 read replicas with <10 ms replica lag
  • Failover is instantaneous — faster than Multi-AZ RDS
  • Costs ~20% more than RDS — but more efficient at scale
  • Data replicated 6 copies across 3 AZs
  • Self-healing: peer-to-peer replication, auto-repairs bad blocks
  • Only one AZ handles writes; all data on a shared storage volume

Storage Layer

  • 6 copies of data, spread across 3 AZs automatically
  • Needs 4/6 for writes, 3/6 for reads — tolerates AZ failures
  • Storage is decoupled from compute — all instances share one volume
  • No performance impact when adding replicas (shared storage)
  • Replication is at the storage layer, not log-shipping
  • Backups are continuous to S3 — no I/O impact on DB
  • Restore to any point within backup retention window
⚑ Exam Trap
  • Aurora storage is not attached to any single instance — it's a distributed shared volume. Adding replicas does NOT increase storage cost proportionally.
  • Aurora replication is storage-layer replication — not ASYNC log-shipping like RDS Read Replicas. Replicas can serve reads with minimal lag (<10ms).

4 Endpoint Types

  • Writer Single endpoint — auto-updates on failover. Always use for writes.
  • Reader Load-balances reads across all Read Replicas. Connection-level LB.
  • Custom Points to a user-defined subset of replicas. Use for analytics workloads on larger instance types.
  • Instance Direct connection to a specific instance. Rarely used in production.

Auto Scaling Replicas

  • Aurora can auto-add/remove replicas based on CPU or connections metric
  • Define min/max replica count in the Auto Scaling policy
  • New replicas are automatically added to the Reader Endpoint
  • When Custom Endpoint is defined, Reader Endpoint is typically not used — set this up carefully
  • Scaling out: ~minutes; scales in after cool-down period
⚑ Exam Note
  • After creating Custom Endpoints, you should stop using the Reader Endpoint for that workload — it may route to instances not in the custom set.
  • Writer Endpoint stays the same DNS during a failover — your app connection string doesn't change.

Aurora Global Database (Recommended)

  • 1 primary region (read/write)
  • Up to 10 secondary regions (read-only)
  • Up to 16 Read Replicas per secondary region
  • Replication via dedicated infrastructure — doesn't use DB compute
  • Replication lag: typically < 1 second
  • Promotes secondary region with RTO < 1 minute
  • Reduces read latency for globally distributed users

Cross-Region Read Replicas (Alternative)

  • Uses Aurora's own replication (not storage-layer)
  • Simpler to set up — just like standard Read Replica
  • Higher replication lag than Global Database
  • Each cross-region replica is an independent Aurora cluster
  • Useful for basic DR — promote manually
  • Not recommended vs Global DB for low-latency global reads
⚑ Exam — Global Aurora Key Numbers
  • Secondary regions: up to 10
  • Read replicas per secondary: up to 16
  • Replication lag: < 1 second
  • RTO for region promotion (DR): < 1 minute
  • Cross-region replication: storage-level (not application-level)
  • Write Forwarding: secondary regions can forward writes to primary — enables read + limited write from secondary
Serverless

Aurora Serverless

Aurora Serverless v2

  • Scales compute in fine-grained increments (0.5 ACU steps)
  • Scales from min to max ACU based on load — in seconds
  • Supports Multi-AZ and Read Replicas
  • Works with Aurora Global Database
  • Ideal for: unpredictable workloads, dev/test, multi-tenant apps
  • You pay per ACU-second used — no idle cost at minimum
  • v2 recommended over v1 for most use cases

Aurora Serverless v1 (legacy)

  • Scales in coarser increments — slower response to spikes
  • Can scale to zero (pause after inactivity) — cold start on resume
  • Does NOT support Multi-AZ or Read Replicas
  • Limited feature set compared to v2
  • Use case: infrequent, intermittent, unpredictable workloads
  • Avoid for production / consistent workloads

ACU (Aurora Capacity Unit): combination of ~2 GB RAM + corresponding CPU + networking. Set a min/max ACU range; Aurora handles the rest. v2 never scales to zero — minimum is 0.5 ACU.

⚑ Exam Scenario
  • Unpredictable DB workload with cost optimization requirement → Aurora Serverless v2
  • Application needs to pause DB when no traffic (overnight) → Aurora Serverless v1 (scales to zero)
  • v2 does NOT scale to zero — if zero cost at idle is needed, v1 or a different approach is required
High Availability

Failover & HA Behavior

ScenarioAurora BehaviorTime
Primary instance fails Aurora auto-promotes a Read Replica — uses Writer Endpoint (no DNS change needed) < 30 sec
No Read Replicas exist Aurora creates a new primary instance in another AZ ~minutes
Entire region fails Global Aurora promotes a secondary region — manual or automated RTO < 1 min
AZ failure Storage 6-copy redundancy absorbs it; replica in another AZ takes over writes Seconds
Failover priority Replica with highest priority tier (0) is promoted first; then largest instance size —

Failover Tiers

  • Each replica has a priority tier (0–15)
  • Tier 0 = highest priority → promoted first
  • Same tier → largest instance size wins
  • Same tier + same size → arbitrary selection
  • Set tiers explicitly in production for predictable failover

Aurora vs RDS Failover Speed

  • Aurora failover: <30 sec (storage-layer, no data re-sync needed)
  • RDS Multi-AZ failover: 60–120 sec
  • Aurora uses same Writer Endpoint DNS — no connection string change
  • RDS Multi-AZ uses same DNS too — but takes longer to re-point
  • Aurora replicas share storage — instant promotion, no catch-up needed
⚑ Exam Trap
  • If Aurora has no Read Replicas at time of failure, it must create a new instance — takes minutes, not seconds. Always maintain at least one replica for fast failover.
  • The Writer Endpoint is a DNS CNAME — Aurora updates it on failover. Apps using the writer endpoint don't need to change config.
Security

Security & Encryption

Encryption At-Rest

  • Uses AWS KMS (AES-256)
  • Enabled at cluster creation — cannot enable later
  • Encrypts: storage volume, backups, snapshots, replicas
  • To encrypt existing cluster: snapshot → copy encrypted → restore
  • All replicas in an encrypted cluster are automatically encrypted

Encryption In-Transit

  • TLS/SSL by default for all connections
  • Enforce with parameter: require_secure_transport=ON
  • Download AWS Aurora CA cert for client verification
  • IAM DB Authentication supported (MySQL + PostgreSQL)
  • Token-based, 15-minute TTL — no password in connection string

Network & Access

  • Deployed inside a VPC — private by default
  • Security Groups control inbound DB traffic
  • Credentials stored in Secrets Manager (recommended)
  • Audit logs → CloudWatch Logs
  • VPC Peering / Transit Gateway for cross-VPC access
Backups

Backups, Cloning & Restore

Automated Backups

  • Continuous backup to S3 — no I/O suspension
  • Retention: 1–35 days
  • Point-in-time recovery to any second in retention window
  • Cannot be disabled (unlike RDS — minimum is 1 day for Aurora)
  • Backups taken from shared storage — zero performance impact

Aurora Cloning

  • Creates a new Aurora cluster from existing one — copy-on-write
  • Faster and cheaper than snapshot restore
  • No data is copied initially — shares the same storage pages
  • New writes diverge — original cluster unaffected
  • Use case: create staging/test env from production data instantly
  • Recommended over snapshot for dev/test cloning
⚑ Exam Scenario
  • Need a test environment with production data immediately → Aurora Cloning (not snapshot restore — faster, cheaper)
  • Aurora automated backups cannot be disabled — minimum 1 day. This differentiates it from RDS where you can set retention to 0.

Amazon S3

Simple Storage Service — one of the highest-weight topics in DOP-C02 — DevOps Professional exam scope

Core

Fundamentals

Bucket & Object Basics

  • Objects stored in buckets — bucket names globally unique across all AWS accounts
  • Max object size: 5 TB. Single PUT: up to 5 GB
  • Objects >5 GB: must use Multipart Upload (recommended >100 MB)
  • Bucket region-specific — data does not leave region unless replicated
  • Flat namespace — folders are key prefixes, not real directories
  • No limit on number of objects in a bucket
  • Key = full path: folder/subfolder/file.txt

Performance

  • 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second per prefix
  • Spread requests across prefixes to multiply throughput
  • Multipart Upload: parallelizes uploads, improves throughput, enables resume
  • S3 Transfer Acceleration: routes upload through CloudFront Edge → AWS private network
  • S3 Byte-Range Fetches: parallelize GET requests — retrieve specific byte ranges
  • SSE-KMS limit: KMS quota (5,500–30,000 req/sec) can become bottleneck at scale
⚑ Exam — Performance
  • If S3 is slow for large uploads → Multipart Upload + Transfer Acceleration.
  • If KMS-encrypted S3 GETs are being throttled → request KMS quota increase, or use S3-managed keys (SSE-S3) to remove KMS from the path.
  • Prefix-based throughput: bucket/folder1/ and bucket/folder2/ are separate prefixes → each gets 5,500 GET/sec.

⚑ Exam — Storage Class Traps
  • One Zone-IA: data lost if AZ is destroyed — only use for re-creatable data.
  • Intelligent-Tiering: no retrieval fee, but has a monitoring/automation fee per object. Best for unknown access patterns.
  • Glacier Flexible: minimum 90-day storage charge — deleting earlier still charges for 90 days.
  • S3 Lifecycle rules can transition objects between classes automatically — reducing cost without code changes.
Lifecycle

Lifecycle Policies

Transition Actions

  • Move objects to cheaper storage class after N days
  • e.g. Standard → Standard-IA after 30 days → Glacier after 90 days
  • Can apply to current and/or noncurrent versions (with versioning)
  • Can filter by prefix, object tags, or object size
  • Minimum 30 days in Standard before transitioning to Standard-IA or One Zone-IA
  • Minimum 30 days in Standard-IA before transitioning to Glacier classes

Expiration Actions

  • Permanently delete objects after N days
  • Delete expired delete markers (versioned buckets)
  • Delete incomplete multipart uploads after N days — cost saving
  • Delete noncurrent versions after N days
  • Can clean up all noncurrent versions except the latest N versions
  • Use case: keep only last 30 days of logs, auto-expire temp files
⚑ Exam — Lifecycle Common Scenarios
  • App generates logs → Standard for 30 days → Standard-IA for 60 days → Glacier Deep Archive → delete after 7 years = classic lifecycle chain.
  • Incomplete multipart uploads silently accumulate cost — always add a lifecycle rule to abort them after 7 days.
  • With versioning: lifecycle must explicitly target noncurrent versions, or old versions accumulate storage cost indefinitely.
Versioning

Versioning

How Versioning Works

  • Enabled at bucket level — stores all versions of every object
  • Once enabled, can be suspended but never fully disabled
  • Deleting a versioned object adds a delete marker — object not removed
  • Permanently delete: delete the specific version ID
  • Objects uploaded before versioning was enabled have version ID null
  • Suspending versioning stops creating new versions — existing versions preserved

MFA Delete

  • Extra protection: requires MFA to permanently delete a version or suspend versioning
  • Only the root account can enable/disable MFA Delete
  • Must be enabled via CLI — not available in console
  • Versioning must be enabled before MFA Delete can be enabled
  • Regular PUT/GET/DELETE (with delete marker) do not require MFA
  • Use case: prevent accidental or malicious permanent data deletion
⚑ Exam Traps
  • Versioning is a prerequisite for S3 Replication — both source and destination buckets must have versioning enabled.
  • Deleting a versioned object via the console (without specifying version ID) creates a delete marker — the object still exists and incurs storage cost.
  • MFA Delete: only root account can manage it — not even admin IAM users.
Replication

S3 Replication (CRR & SRR)

Cross-Region Replication (CRR)

  • Replicates to a bucket in a different AWS region
  • Use cases: compliance, lower latency for global users, DR across regions
  • Can replicate across accounts — destination bucket policy must allow source
  • Reduces RTO for region-level DR
  • Objects replicated to destination with same key, metadata, ACLs

Same-Region Replication (SRR)

  • Replicates to a bucket in the same region
  • Use cases: log aggregation into one bucket, live copy between prod and test
  • No data transfer cost between regions (same region)
  • Useful for compliance: keep data in same region but with separate bucket

⚑ Exam Traps — Replication
  • Existing objects are NOT replicated when you enable replication — use S3 Batch Replication for pre-existing objects.
  • Delete markers NOT replicated by default — you must opt in. Permanent deletes are never replicated.
  • No chaining: if bucket A replicates to B, and B replicates to C — objects from A are NOT in C.
  • RTC: if the exam mentions "guaranteed 15 minute replication" → S3 Replication Time Control.
Security

Security & Access Control

Access Control Layers

  • IAM Policies Attach to users/roles — controls who can call S3 APIs
  • Bucket Policies JSON resource-based policy on the bucket — cross-account, public access
  • ACLs Legacy object/bucket-level — generally replaced by bucket policies
  • VPC Endpoint Policy Restrict S3 access to specific buckets from within VPC
  • Block Public Access: account-level or bucket-level setting — overrides bucket policies
  • Evaluation order: explicit Deny wins → then Allow from any policy

Bucket Policies — Key Use Cases

  • Grant public read access to a bucket (static website)
  • Grant cross-account access without IAM role assumption
  • Require encryption on all PUTs: s3:x-amz-server-side-encryption condition
  • Restrict access to specific VPC: aws:SourceVpc condition
  • Restrict access to specific IP ranges: aws:SourceIp condition
  • Deny all non-SSL requests: aws:SecureTransport: false → Deny
  • Allow only specific IAM roles/users from another account

Pre-Signed URLs

  • Temporary URL granting GET or PUT access to a specific object
  • Generated by: AWS CLI, SDK, or S3 Console
  • Default expiry: 3,600 seconds. Max: 7 days (via SDK/CLI with SigV4)
  • URL inherits the permissions of the IAM identity that generates it
  • Use case: allow user to download private object without making it public
  • Use case: allow user to upload directly to S3 without exposing credentials

S3 Access Points

  • Named endpoints — each with its own access point policy
  • Simplify bucket policy management for large teams
  • Each access point has its own DNS hostname
  • Can restrict access point to a specific VPC only
  • Access Point policy + Bucket policy both apply — most restrictive wins
  • Use case: different teams have different access points with scoped permissions
  • Object Lambda Access Point: transform object before returning to caller
⚑ Exam — Block Public Access
  • Block Public Access is a separate guardrail that overrides bucket policies. Even with a public bucket policy, if Block Public Access is ON → no public access.
  • For a fully private organization: enable Block Public Access at the account level — applies to all current and future buckets.
  • Pre-signed URL: if the IAM identity that created the URL loses permissions or the role is revoked → URL stops working immediately.

Enforce Encryption via Bucket Policy

  • Deny PUT if header x-amz-server-side-encryption is missing or wrong value
  • Example: deny all PUTs unless header = aws:kms → forces SSE-KMS
  • Default encryption setting also available at bucket level (no policy needed for SSE-S3/SSE-KMS default)
  • Bucket policy evaluated before default encryption is applied

In-Transit Encryption

  • S3 supports both HTTP and HTTPS
  • Enforce HTTPS via bucket policy: aws:SecureTransport: false → Deny
  • SSE-C: HTTPS required — AWS rejects SSE-C over HTTP
  • TLS 1.2+ recommended for all S3 traffic
⚑ Exam — SSE-KMS Throughput Issue
  • High-throughput S3 app with SSE-KMS → hitting KMS quota → ThrottlingException. Fix: request KMS quota increase, use S3 Bucket Keys (reduces KMS calls by ~99%), or switch to SSE-S3.
  • S3 Bucket Keys: generates a short-lived data key from KMS at the bucket level — dramatically reduces KMS API calls and cost.
Compliance

Object Lock, Legal Hold & WORM

S3 Object Lock

  • WORM model — Write Once Read Many
  • Must enable at bucket creation — cannot enable later
  • Requires versioning to be enabled
  • Compliance Mode: nobody can delete/overwrite — including root. Retention period cannot be shortened.
  • Governance Mode: users with special IAM permission can override. Root can delete.
  • Retention period set per object version

Legal Hold & Glacier Vault Lock

  • Legal Hold: no retention period — object protected indefinitely until removed. Any user with s3:PutObjectLegalHold can toggle.
  • Glacier Vault Lock: applies Object Lock policy to an entire Glacier vault. Policy becomes immutable once locked.
  • Use case: SEC 17a-4, HIPAA, CJIS compliance — immutable audit records
  • Compliance mode = strongest protection for regulatory requirements
⚑ Exam — Object Lock Modes
  • Compliance Mode: even the root account cannot delete — use for strict regulatory compliance where data must be provably unmodifiable.
  • Governance Mode: allows override with s3:BypassGovernanceRetention permission — flexible for internal policies.
  • Scenario: "prevent all deletion of audit logs for 7 years" → Object Lock in Compliance Mode.
Events

Event Notifications & S3 Select

S3 Event Notifications

  • Trigger on: s3:ObjectCreated, s3:ObjectRemoved, s3:ObjectRestore, s3:Replication
  • Destinations: SNS, SQS, Lambda, EventBridge
  • EventBridge: advanced filtering, multiple destinations, archive/replay
  • Delivery typically within seconds — not guaranteed ordering
  • One event per object — not per byte. Large multipart upload = one event when complete.
  • Must grant S3 permission to invoke the destination (resource policy)

S3 Select & Glacier Select

  • Query data inside S3 objects using SQL expressions
  • Filter rows/columns before data leaves S3 — reduce data transfer by up to 80%
  • Supports: CSV, JSON, Parquet (S3 Select); CSV, JSON (Glacier Select)
  • Lower cost than downloading full object and filtering in application
  • Use case: extract specific columns from large CSV without downloading full file
  • Cannot do JOINs or aggregations — single-object queries only
⚑ Exam — EventBridge vs Native S3 Notifications
  • If exam asks for advanced filtering, multiple destinations, or archiving/replaying S3 events → use EventBridge.
  • Simple Lambda trigger on object upload → native S3 notification is sufficient.
  • EventBridge also supports sending to Step Functions, Kinesis, API Gateway — native notifications only support SNS/SQS/Lambda.
Website

Static Website Hosting & CORS

Static Website Hosting

  • Enable in bucket settings — generates a public endpoint URL
  • Bucket must have Block Public Access OFF + public read bucket policy
  • URL format: bucket.s3-website.region.amazonaws.com
  • Can use CloudFront in front for HTTPS, custom domain, caching
  • Index document (e.g. index.html) + error document (e.g. 404.html)
  • Routing rules: redirect specific paths or error codes to other URLs

CORS (Cross-Origin Resource Sharing)

  • Browser security mechanism — controls cross-origin requests
  • If a website at domain A fetches from S3 bucket at domain B → CORS must be configured on bucket B
  • Configure via JSON CORS rules on the bucket
  • Specify allowed origins, methods (GET, PUT, POST), headers, max-age
  • Wildcard origin * allows any domain — use cautiously
  • Pre-flight OPTIONS request must be handled by S3 CORS config
Operations

S3 Batch Operations & Inventory

S3 Batch Operations

  • Run bulk operations on millions of S3 objects with a single request
  • Operations: copy, invoke Lambda, restore from Glacier, set tags, set ACLs, set Object Lock, replicate existing objects
  • Job uses a manifest (CSV or S3 Inventory report) as the object list
  • Manages retries, tracks progress, sends completion notifications
  • Use case: Batch Replication — replicate existing objects when replication is enabled after the fact
  • Use case: encrypt all existing unencrypted objects in a bucket

S3 Inventory

  • Generate daily or weekly CSV/ORC/Parquet reports of all objects in a bucket
  • Includes: encryption status, replication status, storage class, size, last modified, Object Lock status
  • Output goes to a destination S3 bucket (can be different account)
  • Query with Athena, Redshift Spectrum, or Presto
  • Use case: audit encryption compliance across all objects
  • Use case: find all unencrypted objects → run Batch Operations to encrypt
⚑ Exam — Batch + Inventory Pattern
  • Classic DOP-C02 pattern: use S3 Inventory to find all objects missing encryption → feed manifest into S3 Batch Operations to apply SSE-KMS → verify with next Inventory report.
  • To replicate existing objects when CRR/SRR was just enabled → use S3 Batch Replication (not regular replication rules, which only apply to new objects).

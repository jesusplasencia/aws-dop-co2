01Core

Overview

AWS Step Functions is a serverless workflow orchestration service. You model workflows as state machines written in Amazon States Language (ASL) — a JSON-based DSL. Step Functions manages state, handles retries, tracks execution history, and provides a visual workflow console.

⚡

Core mental model: Each execution moves through a graph of states. Each state receives JSON input, does something (or nothing), and produces JSON output. The output becomes the input of the next state. Step Functions durably tracks every state transition.

When to Use Step Functions

Long-Running Workflows

Order fulfillment, data processing pipelines, human approval flows. Executions can run for up to 1 year (Standard).

Error Handling & Retry

Built-in retry with exponential backoff and catch blocks. Eliminates complex retry logic from application code.

Parallel Coordination

Run branches simultaneously with Parallel state. Process arrays with Map state. Wait for all branches before continuing.

02Core

State Types

Step Functions has 8 state types. Every state in a state machine is one of these types. Each has specific required and optional fields.

Task
Invokes an AWS service or Activity. The workhorse — Lambda, ECS, SQS, DynamoDB, etc. Has timeout, heartbeat, retry, catch.
Choice
Conditional branching. Evaluates rules against input. No Next — each rule has its own Next. Must have a Default.
Wait
Pauses execution for a fixed duration or until a specific timestamp. Free — does not consume Lambda time.
Parallel
Runs N fixed branches simultaneously. Waits for ALL branches to complete. Output is an array of branch outputs.
Map
Iterates over an array, running a sub-workflow for each item. Configurable concurrency limit. Like forEach in parallel.
Pass
Passes input to output unchanged, or injects static JSON. Useful for testing, debugging, injecting defaults.
Succeed
Terminates execution successfully. Often used as the target of a Choice branch. No Next needed.
Fail
Terminates execution with an error. Requires Error and Cause fields. Cannot be retried or caught.

State Common Fields

FieldRequired?Description
Type✅ YesState type: Task, Choice, Wait, Parallel, Map, Pass, Succeed, Fail
CommentOptionalHuman-readable description (ignored at runtime)
NextYes (unless End or Fail/Succeed)Name of the next state to transition to
EndYes (if terminal)Set to true to mark as the last state
InputPathOptionalJSONPath to filter/select input before processing (default: $)
OutputPathOptionalJSONPath to filter/select output after processing (default: $)
ResultPathOptionalWhere to put the task result — merges into input (e.g., $.taskResult)
ParametersOptionalConstruct input payload; supports .$ suffix for dynamic references
ResultSelectorOptionalFilter/reshape the raw task output before ResultPath merge
03Core

Workflow Types: Standard vs Express

Standard Workflows
  • Max duration: 1 year
  • Execution model: Exactly-once — each state runs exactly once
  • Execution history: Full, queryable for 90 days
  • Pricing: Per state transition ($0.025 per 1,000)
  • Throughput: 2,000 executions/sec (default)
  • Idempotency: Built-in — safe to use for critical workflows
  • Use for: Order processing, human approvals, long ETL jobs
Express Workflows
  • Max duration: 5 minutes
  • Execution model: At-least-once (async) / At-most-once (sync)
  • Execution history: CloudWatch Logs only (not Step Functions console)
  • Pricing: Per execution + duration (much cheaper at high volume)
  • Throughput: 100,000+ executions/sec
  • Idempotency: Not guaranteed — must design for it
  • Use for: IoT data ingestion, high-volume event processing, streaming

Express Workflow Sub-Types

Asynchronous Express

  • Fire-and-forget — caller does not wait for result
  • At-least-once execution semantics
  • Use when caller doesn't need the response
  • Result via CloudWatch Logs or EventBridge

Synchronous Express

  • Caller waits for completion and gets the result back
  • At-most-once execution semantics
  • Can be called from API Gateway directly
  • Good for request-response patterns needing <5min
💡

Exam rule of thumb: If a question mentions long-running, human approval, exactly-once, or audit trail → Standard. If it mentions high-volume, IoT, streaming, 5 minutes max, or cost optimization at scale → Express.

04Language

Amazon States Language & I/O Processing

The Amazon States Language (ASL) is a JSON-based structured language for defining state machines. Every state machine is a JSON document with a States object and mandatory StartAt field.

state-machine-skeleton.json
{
  "Comment": "A simple order processing workflow",
  "StartAt": "ValidateOrder",        // first state to execute
  "TimeoutSeconds": 3600,             // overall execution timeout (optional)
  "States": {
    "ValidateOrder": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ValidateOrder",
      "Next": "ProcessPayment"
    },
    "ProcessPayment": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ProcessPayment",
      "End": true                    // terminal state
    }
  }
}

io-processing-example.json — ResultPath to preserve original input
{
  "ProcessOrder": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:...",

    // ① Only send $.order to the task (not the whole input)
    "InputPath": "$.order",

    // ② Construct exact payload sent to Lambda
    "Parameters": {
      "orderId.$": "$.orderId",      // .$ suffix means: take value from input
      "timestamp.$": "$$.Execution.StartTime",  // $$ = context object
      "environment": "production"    // static value (no .$)
    },

    // ③ Merge result into $.processResult (preserves original input)
    "ResultPath": "$.processResult",

    // ④ Reshape raw Lambda response before ResultPath merge
    "ResultSelector": {
      "status.$": "$.Payload.status",
      "orderId.$": "$.Payload.orderId"
    },

    // ⑤ Only output specific fields from the combined state
    "OutputPath": "$.processResult",

    "Next": "SendConfirmation"
  }
}
ℹ️

Context Object ($$): Step Functions provides execution metadata via the context object. Access with $$. prefix in Parameters: $$.Execution.Id, $$.Execution.Name, $$.Execution.StartTime, $$.StateMachine.Id, $$.State.Name.

Task State Examples

Integration Patterns

PatternResource ARN FormatBehavior
Request-Responsearn:aws:states:::lambda:invokeCalls service, waits for HTTP 200. Does NOT wait for task completion. Default for most services.
Sync (waitForTaskToken omitted)arn:aws:states:::ecs:runTask.syncCalls service and waits for it to fully complete before moving to next state. Uses .sync suffix.

task-examples.json — common task state patterns
{
  "States": {

    // ── Lambda (synchronous call, waits for response) ──
    "InvokeLambda": {
      "Type": "Task",
      "Resource": "arn:aws:states:::lambda:invoke",
      "Parameters": {
        "FunctionName": "arn:aws:lambda:us-east-1:123456789012:function:MyFn",
        "Payload.$": "$"           // pass entire input as payload
      },
      "ResultSelector": { "body.$": "$.Payload" },
      "TimeoutSeconds": 300,
      "Next": "NextState"
    },

    // ── ECS Task (sync — waits for container to finish) ──
    "RunECSJob": {
      "Type": "Task",
      "Resource": "arn:aws:states:::ecs:runTask.sync",  // .sync = wait for completion
      "Parameters": {
        "Cluster": "arn:aws:ecs:us-east-1:123456789012:cluster/MyCluster",
        "TaskDefinition": "arn:aws:ecs:us-east-1:123456789012:task-definition/MyTask:1",
        "LaunchType": "FARGATE",
        "Overrides": {
          "ContainerOverrides": [{
            "Name": "my-container",
            "Environment": [{ "Name": "INPUT", "Value.$": "$.inputFile" }]
          }]
        }
      },
      "Next": "NextState"
    },

    // ── DynamoDB PutItem directly (SDK integration) ──
    "SaveToDynamoDB": {
      "Type": "Task",
      "Resource": "arn:aws:states:::dynamodb:putItem",
      "Parameters": {
        "TableName": "OrdersTable",
        "Item": {
          "orderId": { "S.$": "$.orderId" },
          "status":  { "S": "PROCESSING" }
        }
      },
      "ResultPath": null,       // discard result, keep original input
      "Next": "NextState"
    },

    // ── SNS Publish ──
    "NotifyTeam": {
      "Type": "Task",
      "Resource": "arn:aws:states:::sns:publish",
      "Parameters": {
        "TopicArn": "arn:aws:sns:us-east-1:123456789012:AlertTopic",
        "Message.$": "States.Format('Order {} completed', $.orderId)"
      },
      "End": true
    },

    // ── Choice State ──
    "CheckOrderStatus": {
      "Type": "Choice",
      "Choices": [
        {
          "Variable": "$.status",
          "StringEquals": "APPROVED",
          "Next": "ProcessPayment"
        },
        {
          "Variable": "$.amount",
          "NumericGreaterThan": 1000,
          "Next": "RequireManagerApproval"
        }
      ],
      "Default": "RejectOrder"   // required fallback
    },

    // ── Wait State ──
    "WaitForShipping": {
      "Type": "Wait",
      // Option 1: fixed duration
      "Seconds": 86400,           // 24 hours
      // Option 2: from input
      // "SecondsPath": "$.waitSeconds",
      // Option 3: absolute timestamp from input
      // "TimestampPath": "$.shipByDate",
      "Next": "CheckDelivery"
    }
  }
}
06Language

Parallel & Map States

parallel-state.json
{
  "ProcessOrderInParallel": {
    "Type": "Parallel",
    "Branches": [
      {
        // Branch 1: charge card
        "StartAt": "ChargeCard",
        "States": {
          "ChargeCard": {
            "Type": "Task",
            "Resource": "arn:aws:lambda:...",
            "End": true
          }
        }
      },
      {
        // Branch 2: reserve inventory
        "StartAt": "ReserveInventory",
        "States": {
          "ReserveInventory": {
            "Type": "Task",
            "Resource": "arn:aws:lambda:...",
            "End": true
          }
        }
      }
    ],
    // Output = array of branch outputs: [chargeResult, inventoryResult]
    "Next": "ShipOrder"
  }
}
map-state.json
{
  "ProcessItems": {
    "Type": "Map",

    // Array to iterate over (from input)
    "ItemsPath": "$.orderItems",

    // Merge item + context into each iteration input
    "Parameters": {
      "item.$": "$$.Map.Item.Value",
      "index.$": "$$.Map.Item.Index",
      "orderId.$": "$.orderId"
    },

    // Max concurrent iterations (0 = unlimited)
    "MaxConcurrency": 5,

    // Sub-workflow run for each item
    "Iterator": {
      "StartAt": "ProcessSingleItem",
      "States": {
        "ProcessSingleItem": {
          "Type": "Task",
          "Resource": "arn:aws:lambda:...",
          "End": true
        }
      }
    },

    // Output = array of each iteration's output
    "ResultPath": "$.processedItems",
    "Next": "FinalizeOrder"
  }
}
ℹ️

Parallel vs Map: Parallel runs fixed, predefined branches simultaneously — you define each branch upfront. Map runs the same sub-workflow for each item in a dynamic array. Use Parallel for "do A and B at the same time"; use Map for "process each item in this list".

07Operations

Error Handling: Retry & Catch

Step Functions has built-in retry with exponential backoff and structured error catching. This eliminates retry logic from your application code entirely.

error-handling-complete.json
{
  "ProcessPayment": {
    "Type": "Task",
    "Resource": "arn:aws:states:::lambda:invoke",
    "Parameters": { "FunctionName": "ProcessPaymentFn", "Payload.$": "$" },

    // ── RETRY ── (applied before Catch)
    "Retry": [
      {
        // Retry transient Lambda errors with backoff
        "ErrorEquals": ["Lambda.ServiceException", "Lambda.TooManyRequestsException"],
        "IntervalSeconds": 2,      // first retry wait
        "MaxAttempts": 3,          // number of retries (not including original)
        "BackoffRate": 2.0          // multiply wait by 2 each retry: 2s, 4s, 8s
      },
      {
        // Also retry custom app errors
        "ErrorEquals": ["PaymentTemporaryError"],
        "IntervalSeconds": 5,
        "MaxAttempts": 2,
        "BackoffRate": 1.5,
        "MaxDelaySeconds": 30     // cap the wait time
      }
    ],

    // ── CATCH ── (runs after all retries exhausted)
    "Catch": [
      {
        // Catch specific payment error → send to refund flow
        "ErrorEquals": ["PaymentDeclinedError"],
        "Next": "HandleDecline",
        "ResultPath": "$.error"  // save error info into $.error
      },
      {
        // Catch ALL errors as fallback
        "ErrorEquals": ["States.ALL"],
        "Next": "NotifyFailure",
        "ResultPath": "$.error"
      }
    ],

    "TimeoutSeconds": 60,
    "HeartbeatSeconds": 10,     // fail if no heartbeat in 10s (for long tasks)
    "Next": "ConfirmOrder"
  }
}

Built-in Error Codes

Error CodeWhen it Triggers
States.ALLMatches any error — use as catch-all fallback (must be last in Catch array)
States.TimeoutTask exceeded TimeoutSeconds or heartbeat exceeded HeartbeatSeconds
States.TaskFailedTask threw an unhandled exception
States.PermissionsInsufficient IAM permissions to invoke the resource
States.ResultPathMatchFailureResultPath could not be applied to the input
States.HeartbeatTimeoutActivity or waitForTaskToken task didn't send heartbeat in time
Lambda.ServiceExceptionLambda service error (transient, worth retrying)
Lambda.TooManyRequestsExceptionLambda throttled — always add retry for this
Lambda.SdkClientExceptionNetwork error calling Lambda

The waitForTaskToken integration pattern pauses a workflow execution until an external process calls back with SendTaskSuccess or SendTaskFailure. Perfect for human approval flows, external system integration, or long-running jobs.

wait-for-callback.json + callback-handler.py
// State definition — send token via SQS, wait for callback
{
  "WaitForApproval": {
    "Type": "Task",
    "Resource": "arn:aws:states:::sqs:sendMessage.waitForTaskToken",
    "Parameters": {
      "QueueUrl": "https://sqs.us-east-1.amazonaws.com/123456789012/ApprovalQueue",
      "MessageBody": {
        "taskToken.$": "$$.Task.Token",    // $$.Task.Token = the unique token
        "orderId.$":   "$.orderId",
        "amount.$":    "$.amount"
      }
    },
    "HeartbeatSeconds": 3600,              // fail if no heartbeat in 1 hour
    "Next": "ProcessApprovedOrder"
  }
}

// Approval Lambda — called when human approves/rejects
import boto3

def handler(event, context):
    sf = boto3.client('stepfunctions')
    task_token = event['taskToken']
    approved = event['approved']

    if approved:
        sf.send_task_success(
            taskToken=task_token,
            output='{"approvedBy": "manager@company.com"}'
        )
    else:
        sf.send_task_failure(
            taskToken=task_token,
            error='ApprovalRejected',
            cause='Manager rejected the order'
        )
09Operations

Activities (Worker Polling)

Activities allow on-premises servers, EC2 instances, or any external worker to participate in a Step Functions workflow. The worker polls Step Functions for tasks, processes them, and sends results back. No inbound connectivity required from AWS.

Activity vs waitForTaskToken

  • Activities: Worker actively polls for tasks. Good for existing workers, on-premises systems.
  • waitForTaskToken: Step Functions pushes the token outward. Caller decides when to respond. More flexible.
  • Both pause execution waiting for an external process.
activity-worker.py
import boto3, json, time

sf = boto3.client('stepfunctions')
ACTIVITY_ARN = 'arn:aws:states:us-east-1:123456789012:activity:MyActivity'

while True:
    # Poll for available tasks (long-poll, up to 60s)
    response = sf.get_activity_task(
        activityArn=ACTIVITY_ARN,
        workerName='my-worker-1'
    )

    if response.get('taskToken'):
        token  = response['taskToken']
        input_ = json.loads(response['input'])

        try:
            result = process_work(input_)          # do the actual work

            # Send heartbeat every 30s while working
            sf.send_task_heartbeat(taskToken=token)

            sf.send_task_success(
                taskToken=token,
                output=json.dumps(result)
            )
        except Exception as e:
            sf.send_task_failure(
                taskToken=token,
                error='ProcessingError',
                cause=str(e)
            )
    else:
        time.sleep(1)  # no task available, wait a bit

12Advanced

IAM Permissions

Step Functions needs an IAM execution role to call other AWS services from Task states. The role must trust states.amazonaws.com.

trust-policy.json
{
  "Principal": {
    "Service": "states.amazonaws.com"
  },
  "Action": "sts:AssumeRole"
}
execution-role-policy.json
{
  // Lambda invocation
  "Action": "lambda:InvokeFunction",
  "Resource": "arn:aws:lambda:*:*:function:*"
}
// DynamoDB access
{ "Action": ["dynamodb:PutItem", "dynamodb:GetItem"], "Resource": "*" }
// SNS publish
{ "Action": "sns:Publish", "Resource": "*" }
// ECS tasks + pass role
{ "Action": ["ecs:RunTask", "iam:PassRole"], "Resource": "*" }
// CW Logs (required for Express)
{ "Action": ["logs:CreateLogDelivery", "logs:PutLogEvents"], "Resource": "*" }
⚠️

ECS requires iam:PassRole. When a Step Functions task runs an ECS task, it must pass the ECS task execution role and task role to ECS. The Step Functions execution role needs iam:PassRole on those ECS roles.

13Exam Focus

DOP-C02 Exam Tips

Must-Know Facts
  • Standard Workflow: Exactly-once, up to 1 year, full execution history (90 days), priced per state transition. Use for critical/long-running workflows.
  • Express Workflow: At-least-once (async) or at-most-once (sync), up to 5 minutes, CloudWatch Logs only, priced per duration. Use for high-volume/IoT/streaming.
  • Express Workflows REQUIRE CloudWatch Logs — they have no execution history in the SF console. Always configure logging.
  • waitForTaskToken pauses execution until an external system calls SendTaskSuccess/Failure. Used for human approvals, external systems, long-running jobs.
  • Activities = worker-polling pattern. Worker calls GetActivityTask, does work, calls SendTaskSuccess. Good for on-premises workers.
  • Parallel = fixed branches run simultaneously. Output = array of branch outputs. Map = same sub-workflow for each array item. MaxConcurrency controls parallelism.
  • Retry runs before Catch. BackoffRate multiplies IntervalSeconds each attempt. States.ALL in Catch = catch everything (must be last).
  • ResultPath: null discards task output and passes original input unchanged. Critical for "pass-through" tasks.
  • .$ suffix in Parameters means "evaluate as JSONPath from input". Without it, value is treated as a static string.
  • Step Functions execution role must trust states.amazonaws.com. ECS tasks require iam:PassRole in the execution role.
  • State machines are defined in Amazon States Language (ASL) — JSON format. The StartAt field is mandatory.

Common Exam Scenarios

ScenarioAnswer

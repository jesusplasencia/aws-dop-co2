01Foundation

Overview

AWS CDK (Cloud Development Kit) is an open-source framework that lets you define cloud infrastructure using real programming languages. Instead of writing YAML/JSON, you write code — with loops, conditions, functions, IDE autocomplete, and unit tests. CDK synthesizes your code into a standard CloudFormation template for deployment.

⚡

Core mental model: CDK is not an alternative to CloudFormation — it's a code-first abstraction layer on top of CloudFormation. Every CDK app ultimately becomes a CFN stack. You get the full power of a programming language to generate infrastructure definitions.

Key Advantages

Real Language Power

Loops, conditionals, functions, classes, inheritance — generate complex infrastructure patterns that are impossible in YAML.

Type Safety & IDE Support

TypeScript/Java give you compile-time errors before any deployment. Autocomplete for every AWS resource property.

Reusable Constructs

Package infrastructure patterns as npm/PyPI packages. Share your VPC or ECS service pattern across teams.

How CDK Works

CDK follows a synthesize → deploy model. Your code constructs a tree of Constructs. cdk synth converts that tree into one or more CloudFormation templates. cdk deploy uploads assets to S3 and creates/updates the CloudFormation stack.

03Foundation

Constructs — L1, L2, L3

A Construct is the fundamental building block of CDK. Every CDK component — from a single S3 bucket to an entire three-tier application — is a Construct. Constructs come in three abstraction levels.

L1
CFN Resources (Cfn*)
Direct 1:1 mapping to CloudFormation resources. Auto-generated from CFN spec. Every property is explicit. Prefix: Cfn (e.g., CfnBucket). Use when you need full CFN control.
L2
AWS Constructs
Higher-level AWS constructs with sensible defaults, helper methods, and automatic IAM grant methods. E.g., s3.Bucket, lambda.Function, ec2.Vpc. This is what you use most of the time.
L3
Patterns (Solutions)
Multi-service patterns combining multiple AWS resources. E.g., aws-solutions-constructs package provides LambdaToDynamoDB, ApiGatewayToLambda. Highest abstraction.
💡

Construct Tree: All constructs live in a tree. Every construct has a scope (parent) and an id (unique within parent). The root is the App. Children are Stacks. Inside stacks are Constructs. The tree path determines logical IDs in CloudFormation (e.g., MyStack/MyBucket/Resource).

L1 vs L2 Example — S3 Bucket

L1 (Cfn) — verbose, 1:1 CFN
import { CfnBucket } from 'aws-cdk-lib/aws-s3';

// Must specify everything manually
new CfnBucket(this, 'MyBucket', {
  bucketName: 'my-bucket',
  versioningConfiguration: {
    status: 'Enabled',
  },
  publicAccessBlockConfiguration: {
    blockPublicAcls: true,
    blockPublicPolicy: true,
    ignorePublicAcls: true,
    restrictPublicBuckets: true,
  },
});
L2 (Bucket) — smart defaults + methods
import * as s3 from 'aws-cdk-lib/aws-s3';

// Sensible defaults — public access blocked by default
const bucket = new s3.Bucket(this, 'MyBucket', {
  versioned: true,
  encryption: s3.BucketEncryption.S3_MANAGED,
  removalPolicy: RemovalPolicy.DESTROY,
  autoDeleteObjects: true,
});

// L2 grant methods auto-create IAM policies
bucket.grantRead(myLambdaFn);
bucket.grantReadWrite(myEcsTaskRole);

Custom Construct — Reusable Pattern

constructs/secure-bucket.ts — your own L2/L3 construct
import { Construct } from 'constructs';
import * as s3 from 'aws-cdk-lib/aws-s3';
import { RemovalPolicy } from 'aws-cdk-lib';

// Every construct extends Construct
export class SecureBucket extends Construct {
  public readonly bucket: s3.Bucket;

  constructor(scope: Construct, id: string) {
    super(scope, id);  // ← scope + id = position in construct tree

    this.bucket = new s3.Bucket(this, 'Resource', {
      versioned: true,
      encryption: s3.BucketEncryption.KMS_MANAGED,
      blockPublicAccess: s3.BlockPublicAccess.BLOCK_ALL,
      enforceSSL: true,
      removalPolicy: RemovalPolicy.RETAIN,
    });
  }
}

// Use anywhere in your app — same as any built-in construct
const myBucket = new SecureBucket(this, 'DataBucket');
04Foundation

App, Stack & Environment

App

Root of the construct tree. Creates the execution context. One App can contain multiple Stacks. Entry point of every CDK program.

Stack

Maps 1:1 to a CloudFormation Stack. Unit of deployment. Has an optional env specifying account + region. Can be environment-agnostic (resolved at deploy time) or explicit.

Environment (env)

Combination of account + region. Set explicitly or read from CDK environment variables. Required for cross-account/cross-region deployments.

bin/app.ts — multi-stack multi-environment CDK app
import * as cdk from 'aws-cdk-lib';
import { NetworkStack } from '../lib/network-stack';
import { AppStack }     from '../lib/app-stack';

const app = new cdk.App();   // root of construct tree

// Shared environment config
const devEnv  = { account: '111111111111', region: 'us-east-1' };
const prodEnv = { account: '222222222222', region: 'us-east-1' };

// Deploy same infrastructure to dev and prod accounts
const devNet  = new NetworkStack(app, 'NetworkStack-Dev',  { env: devEnv  });
const prodNet = new NetworkStack(app, 'NetworkStack-Prod', { env: prodEnv });

// AppStack depends on NetworkStack output (VPC ID)
const devApp  = new AppStack(app, 'AppStack-Dev', {
  env: devEnv,
  vpc: devNet.vpc,           // pass resource references between stacks
  isProd: false,
});

// Stack tags — applied to all resources in the stack
cdk.Tags.of(app).add('ManagedBy', 'CDK');
cdk.Tags.of(prodNet).add('Environment', 'Production');
⚠️

Cross-stack references: When you pass a resource (like a VPC) from one Stack to another, CDK automatically creates a CloudFormation Export/Import. The two stacks become linked — you cannot delete the exporting stack while the importing stack references it.

05Code

TypeScript — Full Stack Example

lib/api-stack.ts — Lambda + API Gateway + DynamoDB
import * as cdk      from 'aws-cdk-lib';
import * as lambda   from 'aws-cdk-lib/aws-lambda';
import * as apigw    from 'aws-cdk-lib/aws-apigateway';
import * as dynamodb from 'aws-cdk-lib/aws-dynamodb';
import { Construct } from 'constructs';

export class ApiStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // DynamoDB table — L2 construct with smart defaults
    const table = new dynamodb.Table(this, 'ItemsTable', {
      partitionKey: { name: 'pk', type: dynamodb.AttributeType.STRING },
      sortKey:      { name: 'sk', type: dynamodb.AttributeType.STRING },
      billingMode:  dynamodb.BillingMode.PAY_PER_REQUEST,
      removalPolicy:cdk.RemovalPolicy.RETAIN,
      pointInTimeRecovery: true,
    });

    // Lambda function — L2 with bundling
    const fn = new lambda.Function(this, 'ItemsHandler', {
      runtime:     lambda.Runtime.NODEJS_20_X,
      handler:     'index.handler',
      code:        lambda.Code.fromAsset('./src/lambda'),
      architecture:lambda.Architecture.ARM_64,
      environment: { TABLE_NAME: table.tableName },
      tracing:     lambda.Tracing.ACTIVE,
    });

    // Grant Lambda read/write to table — auto-creates IAM policy
    table.grantReadWriteData(fn);

    // API Gateway REST API — L2
    const api = new apigw.RestApi(this, 'ItemsApi', {
      restApiName: 'Items Service',
      deployOptions: { stageName: 'prod', tracingEnabled: true },
    });

    const items = api.root.addResource('items');
    items.addMethod('GET',  new apigw.LambdaIntegration(fn));
    items.addMethod('POST', new apigw.LambdaIntegration(fn));

    // Stack Output — visible in CFN outputs and cdk deploy
    new cdk.CfnOutput(this, 'ApiUrl', { value: api.url });
  }
}

stacks/api_stack.py — Lambda + DynamoDB + SQS
from aws_cdk import (
    Stack, Duration, RemovalPolicy, CfnOutput,
    aws_lambda    as _lambda,
    aws_dynamodb  as ddb,
    aws_sqs       as sqs,
    aws_lambda_event_sources as sources,
)
from constructs import Construct

class ApiStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs):
        super().__init__(scope, construct_id, **kwargs)

        # DynamoDB table
        table = ddb.Table(self, "ItemsTable",
            partition_key=ddb.Attribute(name="pk", type=ddb.AttributeType.STRING),
            billing_mode=ddb.BillingMode.PAY_PER_REQUEST,
            removal_policy=RemovalPolicy.DESTROY,
        )

        # Dead letter queue for failed messages
        dlq = sqs.Queue(self, "DLQ", retention_period=Duration.days(14))

        # Main processing queue
        queue = sqs.Queue(self, "WorkQueue",
            visibility_timeout=Duration.seconds(300),
            dead_letter_queue=sqs.DeadLetterQueue(queue=dlq, max_receive_count=3),
        )

        # Lambda worker — triggered by SQS
        worker = _lambda.Function(self, "WorkerFn",
            runtime=_lambda.Runtime.PYTHON_3_12,
            handler="worker.handler",
            code=_lambda.Code.from_asset("./src"),
            environment={"TABLE_NAME": table.table_name},
        )

        # Grant permissions using L2 grant methods
        table.grant_write_data(worker)
        queue.grant_consume_messages(worker)

        # Add SQS as event source
        worker.add_event_source(sources.SqsEventSource(queue, batch_size=10))

        CfnOutput(self, "QueueUrl", value=queue.queue_url)

07Code

Aspects, Context & Parameters

Aspects — Policy-as-Code

Aspects are a way to apply an operation to all constructs in a tree. Perfect for enforcing organization-wide policies (e.g., all S3 buckets must have encryption, all Lambda functions must have tracing).

aspects/enforce-encryption.ts
import { IAspect, Annotations } from 'aws-cdk-lib';
import { Bucket }                from 'aws-cdk-lib/aws-s3';
import { IConstruct }           from 'constructs';

// Aspect that enforces encryption on all S3 buckets in the tree
export class EnforceS3Encryption implements IAspect {
  visit(node: IConstruct) {
    if (node instanceof Bucket) {
      if (node.encryptionKey === undefined) {
        Annotations.of(node).addError('S3 Bucket must have encryption enabled');
      }
    }
  }
}

// Apply to entire app or specific stack
Aspects.of(app).add(new EnforceS3Encryption());

Context & Parameters

Context Values

Key-value pairs passed to CDK at synth time. Set in cdk.json, ~/.cdk.json, or via --context flag. Used for environment-specific values. Resolved at synth time — baked into the template.

// Read context in code
const stage = this.node.tryGetContext('stage') ?? 'dev';

// Pass at synth time
cdk synth --context stage=prod

CfnParameters

CloudFormation parameters that are resolved at deploy time (not synth time). Use sparingly in CDK — context is usually preferred. Required when value is truly unknown until deployment.

const param = new cdk.CfnParameter(this, 'DBPass', {
  type: 'String',
  noEcho: true,
});

// Deploy with parameter value
cdk deploy --parameters DBPass=secret

09CLI & Deploy

CDK Bootstrap

Before you can deploy CDK apps to an account/region for the first time, you must run cdk bootstrap. This creates a CloudFormation stack called CDKToolkit that provisions the infrastructure CDK needs to operate.

What Bootstrap Creates

🪣
S3 Bucket

Stores Lambda code, CFN templates, and other assets

🐳
ECR Repository

Stores Docker images used by ECS/Lambda container deployments

🔑
IAM Roles

Deployment role, CloudFormation execution role, lookup role

🔐
KMS Key

Encrypts assets stored in the CDKToolkit S3 bucket

💡

Bootstrap is per account + per region. If you deploy to us-east-1 and eu-west-1, you must bootstrap each region separately. In Organizations, you can configure trust policies to allow a central pipeline account to deploy into target accounts.

# Bootstrap current account/region (uses credentials from env)
cdk bootstrap

# Bootstrap specific account and region
cdk bootstrap aws://123456789012/us-east-1

# Bootstrap for cross-account pipeline (trust pipeline account)
cdk bootstrap aws://TARGET_ACCOUNT/us-east-1 \
  --trust PIPELINE_ACCOUNT_ID \
  --cloudformation-execution-policies arn:aws:iam::aws:policy/AdministratorAccess

# Upgrade bootstrap version (after CDK version upgrades)
cdk bootstrap --force
10CLI & Deploy

SAM CLI + CDK Local Testing

You can use the SAM CLI to locally invoke Lambda functions defined in CDK apps. Since CDK outputs CloudFormation templates, and SAM CLI can read CFN templates, the two work together seamlessly.

# Step 1: Synthesize the CDK app — produces cdk.out/MyStack.template.json
cdk synth MyStack

# Step 2: Use SAM to invoke a Lambda from the synthesized template
sam local invoke MyFunctionName \
  --template cdk.out/MyStack.template.json \
  --event events/test-event.json

# Start local API from CDK-synthesized template
sam local start-api \
  --template cdk.out/MyStack.template.json \
  --port 3000

# Watch mode — re-synth + restart SAM on changes
# Run in separate terminal:
cdk watch  # auto re-synths on changes
sam local start-api --template cdk.out/MyStack.template.json
✅

Key workflow for DOP-C02: cdk synth → sam local invoke -t cdk.out/StackName.template.json FunctionLogicalId. The function's logical ID in the synthesized template is CDK-generated (e.g., ItemsHandler1234ABCD) — use cdk synth and inspect the template or use cdk ls --long to find it.

11Advanced

CDK Pipelines — Self-Mutating CI/CD

CDK Pipelines is a high-level L3 construct for creating self-mutating CI/CD pipelines. "Self-mutating" means the pipeline updates itself whenever the CDK code defining the pipeline changes — you never need to manually update the pipeline infrastructure.

⚡

Self-mutating: On every commit, the pipeline first runs a UpdatePipeline step that redeploys the pipeline itself. Then the rest of the stages run with the latest pipeline definition. This means adding a new deploy stage to your CDK code automatically adds it to the real pipeline.

lib/pipeline-stack.ts — CDK self-mutating pipeline
import * as cdk      from 'aws-cdk-lib';
import * as pipelines from 'aws-cdk-lib/pipelines';
import { AppStack }    from './app-stack';
import { Construct }   from 'constructs';

// Stage = group of stacks deployed together
class AppStage extends cdk.Stage {
  constructor(scope: Construct, id: string, props?: cdk.StageProps) {
    super(scope, id, props);
    new AppStack(this, 'AppStack');
  }
}

export class PipelineStack extends cdk.Stack {
  constructor(scope: Construct, id: string) {
    super(scope, id);

    // Source from CodeCommit / GitHub / CodeStar
    const source = pipelines.CodePipelineSource.codeCommit(
      codecommit.Repository.fromRepositoryName(this, 'Repo', 'my-app'),
      'main'
    );

    const pipeline = new pipelines.CodePipeline(this, 'Pipeline', {
      pipelineName: 'MyAppPipeline',
      synth: new pipelines.ShellStep('Synth', {
        input: source,
        commands: ['npm ci', 'npx cdk synth'],  // produces cdk.out/
      }),
    });

    // Add deployment wave — dev (no approval)
    pipeline.addStage(new AppStage(this, 'Dev', {
      env: { account: '111111111111', region: 'us-east-1' },
    }));

    // Add production stage WITH manual approval gate
    pipeline.addStage(new AppStage(this, 'Prod', {
      env: { account: '222222222222', region: 'us-east-1' },
    }), {
      pre: [new pipelines.ManualApprovalStep('PromoteToProd')],
      post: [
        new pipelines.ShellStep('IntegTests', {
          commands: ['npm run test:integration'],
        }),
      ],
    });
  }
}

12Advanced

CDK Testing

CDK provides the aws-cdk-lib/assertions module for testing synthesized CloudFormation templates. Tests run with Jest (TS) or pytest (Python) — fast, no AWS calls needed.

Fine-Grained Assertions

Check that specific resources exist with specific properties. Most common test type.

Snapshot Tests

Save the entire synthesized template as a snapshot. Subsequent test runs compare against the saved snapshot — detect unexpected changes.

test/api-stack.test.ts — Jest + assertions
import { Template, Match } from 'aws-cdk-lib/assertions';
import * as cdk from 'aws-cdk-lib';
import { ApiStack }         from '../lib/api-stack';

describe('ApiStack', () => {
  let template: Template;

  beforeEach(() => {
    const app   = new cdk.App();
    const stack = new ApiStack(app, 'TestStack');
    template = Template.fromStack(stack);  // no AWS calls — pure synthesis
  });

  test('creates DynamoDB table', () => {
    template.hasResourceProperties('AWS::DynamoDB::Table', {
      BillingMode: 'PAY_PER_REQUEST',
      PointInTimeRecoverySpecification: { PointInTimeRecoveryEnabled: true },
    });
  });

  test('Lambda has correct runtime and tracing', () => {
    template.hasResourceProperties('AWS::Lambda::Function', {
      Runtime: 'nodejs20.x',
      TracingConfig: { Mode: 'Active' },
    });
  });

  test('exactly 2 Lambda functions created', () => {
    template.resourceCountIs('AWS::Lambda::Function', 2);
  });

  test('API has prod stage', () => {
    template.hasResourceProperties('AWS::ApiGateway::Stage', {
      StageName: 'prod',
    });
  });

  test('snapshot test — full template', () => {
    expect(template.toJSON()).toMatchSnapshot();
  });

  test('IAM policy grants DynamoDB access to Lambda', () => {
    template.hasResourceProperties('AWS::IAM::Policy', {
      PolicyDocument: {
        Statement: Match.arrayWith([
          Match.objectLike({
            Action: Match.arrayWith(['dynamodb:GetItem', 'dynamodb:PutItem']),
            Effect: 'Allow',
          }),
        ]),
      },
    });
  });
});
13Advanced

CloudFormation Escape Hatches

When CDK's L2 constructs don't expose a property you need, you can drop down to the underlying L1 (CFN) resource to set anything available in raw CloudFormation.

escape-hatch.ts — accessing the underlying CfnResource
// Get the underlying L1 (Cfn) resource from an L2 construct
const bucket = new s3.Bucket(this, 'MyBucket');
const cfnBucket = bucket.node.defaultChild as s3.CfnBucket;

// Set any CFN property not exposed by L2
cfnBucket.addPropertyOverride('NotificationConfiguration.TopicConfigurations', [...]);

// Override existing property set by CDK
cfnBucket.addPropertyOverride('BucketName', 'my-forced-name');

// Inject raw CFN — useful for new AWS features before CDK supports them
const cfnFn = fn.node.defaultChild as lambda.CfnFunction;
cfnFn.addPropertyOverride('SnapStart.ApplyOn'span>, 'PublishedVersions');

// Raw CFN resource override — full JSON merge
cfnBucket.cfnOptions.metadata = { Comment: 'Added via escape hatch' };

// Add CFN metadata / deletion policy overrides
cfnBucket.cfnOptions.deletionPolicy = cdk.CfnDeletionPolicy.RETAIN;
14Exam Focus

DOP-C02 Exam Tips

Must-Know Facts
  • CDK synthesizes into CloudFormation. cdk synth generates CFN templates. Every CDK resource becomes a CFN resource. cdk deploy creates/updates a CFN stack.
  • Bootstrap is required once per account + region. Creates CDKToolkit stack (S3 bucket, ECR repo, IAM roles). Must re-bootstrap when trust policies or CDK major versions change.
  • Three Construct levels: L1 (Cfn* — direct CFN mapping), L2 (smart defaults, grant methods), L3 (multi-service patterns).
  • CDK + SAM local testing: cdk synth → sam local invoke -t cdk.out/Stack.template.json FunctionId.
  • CDK Pipelines = self-mutating. The pipeline updates itself before deploying your app. Requires bootstrap trust between pipeline account and target accounts.
  • cdk hotswap bypasses CloudFormation for Lambda/ECS changes. Fast but causes drift. Dev only — never production.
  • Aspects = policy-as-code. Visit every construct in the tree. Use to enforce tagging, encryption, compliance rules across all stacks.
  • Context values are resolved at synth time (baked into template). CfnParameters are resolved at deploy time. Prefer context for most CDK use cases.
  • CDK unit tests use Template.fromStack() — no AWS calls, runs in milliseconds. hasResourceProperties, resourceCountIs, toMatchSnapshot.
  • Cross-stack references create CFN Export/Import. Cannot delete exporting stack while importing stack references it.

Common Exam Scenarios

01

Overview

AWS Service Catalog is a managed service that allows organizations to create and manage approved catalogs of IT resources on AWS. It enables governance, compliance, and standardization while giving end users a simple self-service portal to launch pre-approved infrastructure.

The core idea is separation of concerns: Admins define what can be deployed (Products in Portfolios), and Users launch those products without needing deep AWS or CloudFormation expertise.

Two-Layer Architecture

Admin Tasks

  1. Write CloudFormation templates (Products)
  2. Group products into Portfolios
  3. Define Constraints on each product
  4. Grant IAM access to Portfolios for teams/users

User Tasks

  1. Log in to the Service Catalog self-service portal
  2. Browse the product list (filtered by IAM access)
  3. Launch a product → CloudFormation stack is created
  4. Use the provisioned resource (VM, DB, etc.)
💡

Key mental model: Service Catalog is a vending machine for AWS infrastructure. Admins stock the machine with approved products; users select and deploy without touching the underlying AWS services directly.

Why Service Catalog Exists

Governance

Ensure only pre-approved, compliant infrastructure gets deployed.

Consistency

All teams deploy the same standardized configurations and templates.

Self-Service

Developers move fast without needing IAM privileges for every AWS service.

How It Works

When a user launches a product, Service Catalog uses an IAM Launch Role (Launch Constraint) to call CloudFormation on their behalf. The user doesn't need any direct IAM permissions to the underlying AWS services — only servicecatalog:ProvisionProduct.

End User
Has only servicecatalog:* permissions
No EC2, RDS, S3, or CFN permissions needed
→
assumes via
Launch Constraint
Launch Constraint IAM Role
Has cloudformation:*, ec2:*, s3:GetObject...
Actually creates the AWS resources
ℹ️

Under the hood: Every provisioned product is a CloudFormation stack. This means you get full CFN capabilities: rollbacks, update policies, deletion policies, drift detection — all inherited automatically.

Provisioned Product Operations

OperationWhat HappensRequires
LaunchCreates a new CloudFormation stack from the product templateLaunch Constraint role (if defined)
UpdateUpdates the CFN stack — can change version or parametersSame Launch Constraint role
TerminateDeletes the CloudFormation stack and all its resourcesSame Launch Constraint role
04

Portfolios

Portfolios are the primary access-control boundary in Service Catalog. You grant access to a portfolio, not to individual products. Anyone with access to a portfolio can see and launch all products in it.

Access Model

Who can access a Portfolio?

  • IAM Users — individual user accounts
  • IAM Groups — collection of users (recommended)
  • IAM Roles — for federated access / SSO

Portfolio Sharing

  • Share with other AWS Accounts
  • Share with entire AWS Organization or OUs
  • Recipient account can add products to the shared portfolio
  • Changes to the portfolio propagate to shared accounts
⚠️

Important: When you share a portfolio, the receiving account cannot edit the products in the shared portfolio. They can only launch them. To customize, the receiving account must import the product into their own portfolio.

05

Constraints

Constraints are rules attached to a product within a portfolio. They control how, where, and with what permissions a product can be launched. A constraint is applied at the Portfolio + Product level — the same product in different portfolios can have different constraints.

Launch Constraint
Assigns an IAM role that Service Catalog assumes when provisioning. Enables least-privilege for end users.
Notification Constraint
Sends stack event notifications to an SNS topic. Useful for monitoring and auditing launches.
Template Constraint
Restricts the parameter values a user can enter when launching a product (via rules). E.g., limit instance types to t2.micro only.
StackSet Constraint
Deploys the product as a StackSet across multiple accounts and regions using CloudFormation StackSets.
Tag Update Constraint
Controls whether users can update tags on provisioned products. Can be set to ALLOWED or NOT_ALLOWED.
Resource Update Constraint
Controls whether users can update resources of a provisioned product at all (ALLOWED or NOT_ALLOWED).
Exam Focus — Constraints
  • Constraints are per Portfolio + Product pair, not per product globally.
  • Only one Launch Constraint per portfolio/product combination.
  • If no Launch Constraint is defined, Service Catalog uses the user's own IAM permissions to provision — which usually means the user needs full permissions. This defeats the purpose.
  • Template Constraints are great for limiting parameter values (e.g., allowed EC2 instance types, allowed regions via parameter).
06

Launch Constraints

A Launch Constraint is the most important constraint type. It is an IAM Role that Service Catalog assumes when a user provisions a product. This means the end user only needs servicecatalog:* permissions — Service Catalog uses the Launch Role for all the actual AWS resource creation.

Required Permissions for the Launch Role

CloudFormation

Full access to create, update, delete, describe stacks and set stack policies.

Target AWS Services

All permissions needed to create the resources defined in the CFN template (e.g., ec2:*, rds:*).

S3 Read Access

Read access to the S3 bucket containing the CloudFormation template (tagged with servicecatalog:provisioning: true).

Launch Constraint IAM Role Policy (Example)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      // Statement 1: CloudFormation + target service permissions
      "Effect": "Allow",
      "Action": [
        "cloudformation:CreateStack",
        "cloudformation:DeleteStack",
        "cloudformation:DescribeStackEvents",
        "cloudformation:DescribeStacks",
        "cloudformation:GetTemplateSummary",
        "cloudformation:SetStackPolicy",
        "cloudformation:ValidateTemplate",
        "cloudformation:UpdateStack",
        "ec2:*",           // ← all services used in your CFN template
        "rds:*",
        "servicecatalog:*",
        "sns:*"
      ],
      "Resource": "*"
    },
    {
      // Statement 2: S3 access for templates (tag-based condition)
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          // Only allow S3 access to objects tagged as Service Catalog templates
          "s3:ExistingObjectTag/servicecatalog:provisioning": "true"
        }
      }
    }
  ]
}

Trust Policy for the Launch Role

The Launch Constraint IAM Role must have a trust policy allowing Service Catalog to assume it:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "servicecatalog.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
💡

Minimal User Permissions Pattern: End user only needs servicecatalog:DescribeProduct, servicecatalog:ProvisionProduct, servicecatalog:ListLaunchPaths, and servicecatalog:SearchProductsAsAdmin. All AWS resource creation is handled by the Launch Constraint IAM Role.

07

StackSet Constraints

StackSet Constraints let you configure a product so that when it's launched, it deploys as a CloudFormation StackSet — provisioning resources across multiple AWS accounts and regions simultaneously.

Accounts

Specify the target AWS account IDs where the product will be deployed.

Regions

Specify target regions with optional deployment order (sequential or parallel).

Permissions

Requires an IAM StackSet Administrator Role to manage target accounts.

💡

Region Restriction: Using a StackSet Constraint, you can enforce that users can only deploy to specific regions — a key governance feature for data residency and compliance requirements.

StackSet Constraint vs Standard Deployment

Standard Launch (no StackSet constraint)
  • Deploys to the current account & region only
  • One CloudFormation stack created
  • Faster, simpler
  • Use for single-account products
StackSet Constraint Launch
  • Deploys to multiple accounts & regions
  • CloudFormation StackSet created
  • Ideal for org-wide baseline resources
  • Requires Administrator role setup

IAM & Permissions

Three IAM Principals in Service Catalog

PrincipalTypical PermissionsNotes
Admin servicecatalog:* + cloudformation:* + iam:PassRole Creates products, portfolios, constraints. Needs iam:PassRole to assign Launch Constraints.
End User servicecatalog:ProvisionProduct, servicecatalog:ListLaunchPaths, servicecatalog:DescribeProduct No direct EC2/RDS/S3 permissions needed if Launch Constraint is set.
Launch Role cloudformation:* + all target services + s3:GetObject (conditional) Must trust servicecatalog.amazonaws.com to assume it.

IAM PassRole Requirement

When an admin creates a Launch Constraint, they must have iam:PassRole on the Launch Role ARN. This prevents privilege escalation — admins can only assign roles they are allowed to pass.

// Admin policy snippet — allows assigning a specific Launch Role
{
  "Effect": "Allow",
  "Action": "iam:PassRole",
  "Resource": "arn:aws:iam::123456789012:role/SCLaunchRole",
  "Condition": {
    "StringEquals": {
      "iam:PassedToService": "servicecatalog.amazonaws.com"
    }
  }
}

CI/CD Integration

Service Catalog products should be treated as code — versioned, reviewed, and automatically synced. The recommended pattern is to store CFN templates in a CodeCommit (or GitHub) repository and use a Lambda function triggered by repo events to keep products in sync.

CodeCommit Sync Pattern

01
Developer Pushes
product-a.yml to CodeCommit
02
CodeCommit Trigger
Invokes Lambda function
03
Lambda Reads mapping.yml
Maps files → SC Products
04
Lambda Calls SC API
UpdateProvisioningArtifact
05
Product Updated
New version available in SC

mapping.yml Example

# mapping.yml — maps template files to Service Catalog Products
Products:
  - Name: Product VPC
    Template: product-vpc.yml
    PortfolioId: port-abc123
    ProductId: prod-xyz456

  - Name: Product ALB
    Template: product-alb.yml
    PortfolioId: port-abc123
    ProductId: prod-uvw789

Lambda Sync Function (Key Logic)

# SyncServiceCatalogFunction — triggered by CodeCommit push
import boto3, yaml

def handler(event, context):
    sc = boto3.client('servicecatalog')
    cc = boto3.client('codecommit')
    
    # 1. Read the mapping file from CodeCommit
    mapping = yaml.safe_load(
        cc.get_file(repositoryName='my-repo', filePath='mapping.yml')['fileContent']
    )
    
    # 2. For each product in the mapping, update the artifact
    for product in mapping['Products']:
        template_url = get_s3_url(product['Template'])  # upload to S3 first
        
        # 3. Create a new provisioning artifact (product version)
        sc.create_provisioning_artifact(
            ProductId=product['ProductId'],
            Parameters={
                'Name': 'latest',
                'Type': 'CLOUD_FORMATION_TEMPLATE',
                'Info': {'LoadTemplateFromURL': template_url}
            }
        )

Alternative: CodePipeline Integration

For more complex workflows, use a full CodePipeline:

  • Source: CodeCommit / S3 / GitHub
  • Build: CodeBuild to lint/validate templates
  • Deploy: Lambda action or custom action to update Service Catalog products
  • Approval: Manual approval gate before publishing to production portfolio

Portfolio Sharing

Portfolios can be shared with other AWS accounts or entire AWS Organizations, enabling a central "hub" account to manage products that are consumed by many "spoke" accounts.

Account-Level Sharing
  • Share with specific account IDs
  • Recipient must accept the share
  • Works without AWS Organizations
  • Good for isolated cross-account scenarios
Org-Level Sharing
  • Share with Organization ID or OU
  • All accounts in OU automatically get access
  • Requires AWS Organizations integration
  • New accounts in OU automatically get portfolio

Sharing Mechanics

AspectBehavior
Product visibilityRecipient sees products in the imported portfolio but cannot edit them
Local customizationRecipient can add their own constraints to the imported products (e.g., their own launch role)
Updates propagationWhen the sharing account updates a product version, the update is available to all recipients immediately
Tag OptionsTagOptions associated with the portfolio are also shared and enforced in recipient accounts
11

Tagging & TagOptions

TagOptions is a Service Catalog feature that allows admins to define a key-value tag library that is applied to all provisioned products. This enforces consistent tagging across your organization — critical for cost allocation and compliance.

TagOptions Library

  • Centrally managed list of approved tag keys and allowed values
  • Associated with a Portfolio or Product
  • Enforced on all provisioned products automatically
  • Example: Environment: [dev, staging, prod]

Tag Update Constraint

  • ALLOWED — users can change tags after provisioning
  • NOT_ALLOWED — tags are locked after provisioning
  • Use NOT_ALLOWED to prevent tag tampering for cost tracking
💡

Exam tip: TagOptions are shared along with the portfolio when you share portfolios across accounts. Recipient accounts automatically inherit enforced tags.

DOP-C02 Exam Tips

Must-Know Facts
  • A Product is a CloudFormation template. Provisioned Products are CFN stacks.
  • A Portfolio is a collection of Products with access granted via IAM users/groups/roles.
  • Launch Constraint = IAM role that Service Catalog assumes to provision on behalf of the user. The user doesn't need direct AWS resource permissions.
  • StackSet Constraint = deploy a product across multiple accounts/regions. Also used to restrict which regions a product can be deployed to.
  • Template Constraint = restrict parameter values users can input (e.g., only allow t2.micro instances).
  • Admins need iam:PassRole to assign a Launch Constraint role to a product.
  • The Launch Role must trust servicecatalog.amazonaws.com in its trust policy.
  • Portfolio sharing with Organizations auto-deploys to new accounts added to the OU.
  • Service Catalog integrates with ServiceNow for ITSM-driven self-service.
  • CI/CD pattern: CodeCommit → Lambda (SyncServiceCatalogFunction) → Service Catalog.

Common Exam Scenarios

ScenarioSolution
"Developers need to launch EC2 instances but should not have direct EC2 permissions" Service Catalog with a Launch Constraint IAM role that has EC2 permissions. Users only need servicecatalog:ProvisionProduct.
"Ensure all provisioned resources are deployed only to us-east-1 and eu-west-1" Use a StackSet Constraint with Regions set to the allowed regions only.
"Prevent users from choosing large (expensive) EC2 instance types" Apply a Template Constraint that limits the InstanceType parameter to allowed values.
"Keep Service Catalog products in sync with CloudFormation templates stored in Git" CodeCommit + Lambda (SyncServiceCatalogFunction) pattern. Lambda updates provisioning artifacts when templates are pushed.
"Deploy a standard security baseline product to all AWS accounts in the organization" Share the portfolio at Organization level and use a StackSet Constraint to deploy to all accounts.
"Ensure all launched products have a mandatory 'CostCenter' tag" Create a TagOptions entry for CostCenter and associate it with the portfolio. Set Tag Update Constraint to NOT_ALLOWED.
"Allow requests for AWS resources through ITSM ticketing system" Service Catalog ServiceNow connector (or Jira Service Management connector).

Service Catalog vs Direct CloudFormation

Service Catalog
  • Self-service for non-technical users
  • Governance and constraint enforcement
  • Centralized product management
  • No need for deep AWS IAM knowledge
  • Consistent tagging across org
  • Portfolio sharing across accounts
Direct CloudFormation
  • Used by platform/infra engineers
  • Full flexibility — any template
  • Requires IAM permissions for services
  • No built-in governance layer
  • User manages their own stacks
  • Used under the hood by Service Catalog
🎯

One-line summary for the exam: Service Catalog = CloudFormation templates packaged with governance. When you see "self-service", "least privilege for developers", "governance/compliance", "approved templates", or "non-technical users provisioning AWS resources" — think Service Catalog.

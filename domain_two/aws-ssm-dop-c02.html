Domain
Domain 2
Topic Weight
~High

01

Overview & How It Works

Know Well
Purpose
Manage EC2 and on-premises servers at scale — patching, configuration, secrets, remote access
SSM Agent
Must be installed on every managed node. Pre-installed on Amazon Linux 2023 and some Ubuntu AMIs
Prerequisites
SSM Agent running + correct IAM Instance Profile attached to EC2
Node Prefixes
i- for EC2 instances · mi- for hybrid/on-premises managed nodes
Integrations
CloudWatch Metrics / Dashboards, AWS Config, EventBridge, CloudTrail, S3, SNS
★
If an instance is not showing up in SSM, it's always either the SSM Agent not running or a missing IAM permissions (missing AmazonSSMManagedInstanceCore policy on the instance role).
02

Documents

Know Well
What is a Document?
  • JSON or YAML definition of actions (scripts) to execute on managed nodes
  • Support parameterization via {{PARAM_NAME}} placeholders
  • AWS provides pre-built documents; you can create custom ones
  • Schema version 2.2 for Command documents, 0.3 for Automation runbooks
Run Command
Command documents — run shell/PowerShell scripts on instances
Automation
Automation documents — act on AWS resources (restart EC2, create AMI…)
State Manager
Command/Policy documents — keep instances in desired state on a schedule
Patch Manager
Uses AWS-RunPatchBaseline document under the hood
03

Run Command

Know Well
  • Execute a Command document or ad-hoc shell command on one or many instances
  • Target by instance ID, tag, or Resource Group
  • Controls: Rate Control (concurrency/error thresholds), Timeout
  • Output: optional S3 bucket or CloudWatch Logs
  • Notifications via SNS on command state changes
  • Can be invoked programmatically via EventBridge — key exam scenario
  • Full audit trail via CloudTrail
★
Common exam scenario: "Run a script on all EC2 instances tagged Env=Prod at once" → Run Command using tag-based targeting, not SSH/Ansible.
04

Automation (Runbooks)

High Priority
What it Does
  • Automates tasks against AWS resources (not just inside an OS)
  • Uses Automation Runbook documents (type: Automation)
  • Pre-built AWS runbooks available (e.g. AWS-RestartEC2Instance)
Trigger Methods
  • Manually — Console, CLI, SDK
  • Amazon EventBridge schedule or event
  • Maintenance Windows
  • AWS Config rules remediation
Exam-Relevant Use Cases
  • Auto start/stop EC2 and RDS to reduce costs (scheduled via EventBridge or Maintenance Windows)
  • Build a Golden AMI pipeline
  • Downsize underutilized instances automatically
  • Auto-remediate AWS Config rule violations
★
Run Command vs Automation: Run Command = actions inside OS on managed nodes. Automation = actions on AWS resources (EC2 API level, AMI creation, RDS, etc.).
05

Parameter Store

High Priority
Purpose
Secure, hierarchical storage for configuration values and secrets
Encryption
SecureString type encrypted via KMS; String / StringList are plaintext
Standard tier
10,000 parameters, no cost, no parameter policies, max 4KB value
Advanced tier
100,000 parameters, $0.05/parameter/month, supports Parameter Policies, max 8KB
Versioning
Every update creates a new version; retrieve specific version by appending :N
Notifications
Via EventBridge — trigger on put, delete, expiration events
CloudFormation
Reference parameters directly in CF templates with SSM dynamic references
Parameter Policies (Advanced only)
  • Expiration — automatically delete a parameter at a timestamp
  • ExpirationNotification — EventBridge alert N days before expiry
  • NoChangeNotification — alert if parameter hasn't changed in N days
Special Built-in Paths
# Access Secrets Manager secrets via Parameter Store /aws/reference/secretsmanager/<secret-name> # Latest Amazon Linux 2 AMI ID (per region) /aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
★
Parameter Store vs Secrets Manager: Parameter Store is free for standard params and great for config. Secrets Manager adds automatic rotation, cross-account sharing, and costs more. The exam tests knowing when each is appropriate.
06

Patch Manager

High Priority
Patch Baseline
  • Defines which patches are approved/rejected for installation
  • AWS managed baselines per OS (e.g. AWS-AmazonLinux2DefaultPatchBaseline)
  • Default: only Critical + Security patches auto-approved
  • Auto-approval delay: N days after release
  • Custom baselines can specify: OS, allowed patches, rejected patches, custom repos
Patch Group
  • Groups instances to a specific Patch Baseline
  • Instance must have tag key Patch Group
  • One instance can belong to only one Patch Group
  • One Patch Group can be registered with only one baseline
SSM Document
AWS-RunPatchBaseline — applies OS and application patches
Supported OS
Linux, macOS, Windows Server — both EC2 and on-premises
Scheduling
On-demand or via Maintenance Windows
Reporting
Compliance scan reports — exportable to S3
★
Exam scenario: "Ensure dev instances get all patches but prod only gets critical patches" → create two Patch Baselines, two Patch Groups, tag instances accordingly, schedule with Maintenance Windows.
07

Session Manager

High Priority
  • Secure shell access to EC2 instances and on-premises servers — no SSH, no open ports, no bastion host needed
  • Access via: AWS Console, AWS CLI (aws ssm start-session), Session Manager SDK
  • All session activity (commands + output) logged to S3 and/or CloudWatch Logs
  • CloudTrail records StartSession API calls for audit
  • IAM controls which users/roles can start sessions to which instances (tag-based restrictions supported)
  • Works with VPC Interface Endpoints for instances with no internet access
★
The exam loves this: "How to provide shell access to private EC2 instances without opening port 22 or maintaining a bastion host?" → Session Manager + VPC Endpoint.
i
For port forwarding through Session Manager (e.g. RDP to Windows, access internal web UI), use aws ssm start-session --document-name AWS-StartPortForwardingSession.
08

Maintenance Windows

Know Well
Purpose
Schedule when patching, updates, and automation tasks run on instances
Components
Schedule (cron/rate) · Duration · Registered Instances · Registered Tasks
Supported Tasks
Run Command, Automation runbooks, Lambda functions, Step Functions
Typical Use
OS patching during low-traffic hours, driver updates, software installs, scheduled AMI creation
09

Default Host Management Config (DHMC)

Know Well
  • Automatically registers new EC2 instances as SSM managed nodes — no Instance Profile required
  • Uses an Instance Identity Role (minimal IAM role that only identifies the instance to AWS)
  • Automatically passes AWSSystemsManagerDefaultEC2InstanceManagementRole to the instance
  • Requirements: IMDSv2 enabled + SSM Agent installed
  • Automatically enables: Session Manager, Patch Manager, Inventory
★
DHMC eliminates the need to manually attach an instance profile for SSM. If a question describes "automatically managed without instance profiles", the answer is DHMC.
10

Compliance & OpsCenter

Awareness
Compliance
  • Aggregates patch and State Manager association compliance data
  • Sync to S3 via Resource Data Sync → analyze with Athena / QuickSight
  • Can forward data to Security Hub
OpsCenter
  • Central place to view, investigate, and remediate operational issues (OpsItems)
  • Sources: EventBridge, Config, CloudTrail, Security Hub, DevOps Guru
  • Suggests relevant Runbooks to resolve each issue

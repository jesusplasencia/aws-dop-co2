01

Overview

AWS Elastic Beanstalk is a Platform as a Service (PaaS) that handles all infrastructure provisioning automatically. You upload application code, Beanstalk orchestrates EC2 instances, Auto Scaling Groups, Load Balancers, security groups, and more.

⚡

Core principle: Developer provides application code. Beanstalk provides everything else — capacity provisioning, load balancing, scaling, health monitoring, instance configuration. You retain full control over underlying resources but none of the management burden.

What EB Manages vs What You Manage

Elastic Beanstalk Manages
  • EC2 instance provisioning & configuration
  • Auto Scaling Group creation & policies
  • Load Balancer setup (ALB / CLB / NLB)
  • Security Group rules
  • Application health monitoring
  • Platform patching & managed updates
  • Log rotation & CloudWatch integration
You Are Responsible For
  • Application code
  • Application version management
  • Environment configuration (instance type, VPC)
  • Custom .ebextensions configurations
  • RDS / database management (if external)
  • IAM roles (Service Role + Instance Profile)

Supported Platforms

Go Java SE Java + Tomcat .NET on Windows .NET on Linux Node.js PHP Python Ruby Packer Builder Docker (Single) Docker (Multi) Custom Platform

Deployment Modes

Single Instance

One EC2 instance with an Elastic IP. No load balancer. Good for development / test environments. Includes RDS Master (if configured).

High Availability with Load Balancer

ALB + ASG across multiple AZs. Optional RDS Master + RDS Standby. Production-grade setup with auto-failover.

02

Components

ComponentDefinitionKey Detail
Application Top-level container. Holds environments, versions, and configurations. Like a "project" — you can have multiple environments per application.
Application Version A specific iteration of application code. Points to an S3 object (ZIP or WAR). Versions are stored in S3. Old versions can be cleaned by Lifecycle Policy.
Environment A running set of AWS resources hosting an application version. Separate envs for dev, test, prod. Each env = its own CFN stack.
Configuration Saved environment configuration (instance type, scaling settings, etc.). Saved configs can be applied to new environments for quick replication.
Platform The OS + runtime + web server combination (e.g., Amazon Linux 2 + Node.js 18). Platforms receive managed updates. You can pin a platform version.

Application Version Workflow

[ Create Application ]
  → Upload Version # ZIP/WAR → S3 → EB App Version
  → Launch Configuration # instance type, VPC, scaling, etc.
  → Create Environment # EB provisions AWS resources via CloudFormation
  → Deploy / Update # choose a deployment strategy
  → Manage Environment # scale, swap URLs, monitor health
  → [ Iterate — upload new version → deploy ]
03

Environment Tiers

Web Server Tier

  • Handles HTTP/HTTPS requests
  • Architecture: ELB → ASG → EC2 instances
  • Publicly accessible via URL (myapp.us-east-1.elasticbeanstalk.com)
  • EC2 instances run the web application
  • Periodic tasks defined in cron.yaml
  • Can push messages to SQS → Worker Tier

Worker Tier

  • Handles long-running background tasks
  • Architecture: SQS Queue → ASG → EC2 workers
  • Scales based on number of SQS messages
  • No public URL — not directly accessible
  • Periodic tasks defined in cron.yaml
  • Receives messages pushed by Web Tier
💡

Decoupling pattern: Web Server Tier handles user requests and queues heavy work. Worker Tier processes the queue asynchronously. Both define periodic tasks via cron.yaml. Worker Tier auto-scales based on queue depth — not CPU or request count.

cron.yaml — Worker Tier periodic tasks
version: 1
cron:
  - name: "process-reports"
    url: "/process-reports"
    schedule: "0 */6 * * *"       # every 6 hours

  - name: "cleanup-old-data"
    url: "/cleanup"
    schedule: "0 0 * * 0"         # every Sunday midnight
04

Service Role

The Service Role is assumed by the Elastic Beanstalk service itself — not by your EC2 instances. It gives EB the permissions to manage AWS resources (ASG, ELB, EC2, CloudWatch) on your behalf.

🔑

Critical distinction: Service Role = permissions for EB the service. Instance Profile = permissions for your application code running on EC2. They are different IAM roles with different trust policies.

Service Role Permissions Summary

autoscaling:*

Create, update, delete ASG. Set desired capacity. Attach/terminate instances.

ec2:Describe*

Describe instances, status, security groups, subnets, VPCs. Run & terminate instances.

elasticloadbalancing:*

Register/deregister targets. Describe health. Create/delete load balancers.

cloudwatch:*

Describe, put, delete CloudWatch alarms. Get metric statistics for scaling decisions.

iam:PassRole

Pass the EC2 Instance Profile role to newly launched EC2 instances.

s3:GetObject

Download application version ZIP from S3 elasticbeanstalk-* bucket.

Trust Policy (Service Role)

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Principal": { "Service": "elasticbeanstalk.amazonaws.com" },
    "Action":    "sts:AssumeRole"
  }]
}
05

EC2 Instance Profile

An Instance Profile is a container for an IAM Role that is attached to EC2 instances. This lets your application code call AWS services (S3, DynamoDB, SQS, etc.) without embedding credentials.

What the Instance Profile Enables

Application AWS Access

  • Read / write to S3 buckets
  • Publish to SNS topics
  • Send / receive SQS messages
  • Read/write DynamoDB tables
  • Call other AWS APIs your app needs

EB Internal Operations

  • Download app bundle from S3
  • Write logs to CloudWatch Logs
  • Report health status
  • Enhanced health reporting data

Instance Profile Trust Policy

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    // EC2 instances (not EB service) assume this role
    "Action":    "sts:AssumeRole"
  }]
}

Example Instance Profile Policy

ec2-instance-profile-policy.json
{
  "Statement": [
    {
      "Sid": "S3AppAccess",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"],
      "Resource": ["arn:aws:s3:::my-app-bucket", "arn:aws:s3:::my-app-bucket/*"]
    },
    {
      "Sid": "DynamoDBAccess",
      "Action": ["dynamodb:GetItem", "dynamodb:Query", "dynamodb:PutItem"],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/MyAppTable"
    },
    {
      "Sid": "EBMinimumRequired",       // EB needs these for internal operations
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::elasticbeanstalk-*"
    },
    {
      "Sid": "CloudWatchLogs",
      "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}
⚠️

Always include elasticbeanstalk-* S3 access in the Instance Profile. Without it, EB cannot download your application bundle to the instance during deployment.

Deployment Options

Elastic Beanstalk offers six deployment strategies. Choosing the right one requires trading off speed, availability, cost, and rollback complexity.

All at Once
Downtime Fastest Free
Deploys to ALL instances simultaneously. Application is unavailable during the update. Fastest deployment — no extra cost. Not suitable for production. Rollback requires another full re-deploy.
Rolling
Reduced Capacity No Downtime Free
Deploys to a configurable batch size at a time. Old and new versions run simultaneously. Capacity is reduced during deployment (bucket size is removed from pool). Long deployment if many instances.
Rolling with Addl. Batches
Full Capacity No Downtime Small Extra Cost
Like Rolling but spins up new instances first to maintain full capacity throughout. Old application stays fully available. Slightly more expensive. Good for production with steady traffic.
Immutable
Zero Downtime Best Rollback Highest Cost
Launches a new temporary ASG with new instances running the new version. When all healthy, moves them to original ASG and terminates old instances. Longest deployment. One-click instant rollback (just terminate new ASG).
Blue / Green
Zero Downtime URL Swap 2x Cost
Create a completely new environment (green), deploy there, then swap URLs using Route 53 weighted routing or EB URL swap. Full rollback by swapping URLs back. Not natively "one button" in EB — done manually.
Traffic Splitting
Canary A/B Testing Medium Cost
Sends a configurable % of traffic to new version (canary testing). EB creates new ASG with same capacity. ALB routes traffic based on percentage. If unhealthy, traffic automatically reverts — automated rollback.
07

Deployment Comparison

Strategy Downtime? Capacity During Deploy Deployment Speed Rollback Extra Cost Use Case
All at Once ✅ Yes Zero capacity ⚡ Fastest Re-deploy None Dev / quick iterations
Rolling ❌ No Reduced (batch removed) Medium Re-deploy None Low-traffic production
Rolling + Batches ❌ No Full capacity Medium-Slow Re-deploy Small Production w/ stable capacity
Immutable ❌ No Full + temp extra ASG 🐢 Slowest ⚡ Instant (terminate new ASG) 2x for duration Production, safety critical
Blue / Green ❌ No Full (2 envs) Slow (full env) ⚡ URL swap back 2x ongoing High-stakes releases
Traffic Splitting ❌ No Full + canary group Medium ⚡ Auto (health-based) Moderate Canary / A-B testing
Deployment Decision Tree
Need zero downtime?
├─ NO → All at Once (dev only)
└─ YES
   ├─ Need instant rollback + best safety?
   │ └─ YES → Immutable or Blue/Green
   ├─ Need canary / gradual traffic shift?
   │ └─ YES → Traffic Splitting
   ├─ Need full capacity during deploy?
   │ └─ YES → Rolling with Additional Batches
   └─ Capacity reduction is OK?
      └─ YES → Rolling

.ebextensions Configuration

The .ebextensions/ directory at the root of your application source bundle lets you customize and configure the environment using YAML/JSON configuration files. Files must have a .config extension.

📁

Location: .ebextensions/*.config — placed in the root of your application ZIP. Executed during environment creation and each deployment, before the application is started.

Key .ebextensions Sections

SectionPurposeExample Use
option_settingsSet EB configuration options (same as EB console)Environment variables, instance type, scaling settings
ResourcesCreate AWS resources via CloudFormation syntaxAdd SQS queue, SNS topic, DynamoDB table to the env
packagesInstall OS packages via yum/apt/rpmInstall nginx, git, custom packages
filesCreate/write files on the EC2 instanceNginx config, app config files, SSL certs
commandsRun shell commands before app deploymentDB schema migration, folder creation
container_commandsRun commands after app is extracted but before it startsRun migrations, leader_only: true for single-instance commands
servicesStart/stop Linux servicesEnsure nginx, cron, custom daemon starts
.ebextensions/environment.config
option_settings:
  aws:elasticbeanstalk:application:environment:
    NODE_ENV: production
    DATABASE_URL: "{{resolve:ssm:/myapp/db-url}}"
  aws:autoscaling:asg:
    MinSize: 2
    MaxSize: 8
  aws:ec2:instances:
    InstanceTypes: "t3.medium,t3.large"  # mixed instance types
.ebextensions/packages-and-files.config
packages:
  yum:
    git: []
    nginx: []

files:
  "/etc/nginx/conf.d/proxy.conf":
    mode: "000644"
    owner: root
    content: |
      server {
        listen 80;
        location / { proxy_pass http://localhost:3000; }
      }

container_commands:
  01-run-migrations:
    command: "python manage.py migrate"
    leader_only: true   # only runs on the leader instance (1 instance)
  02-collect-static:
    command: "python manage.py collectstatic --noinput"

services:
  sysvinit:
    nginx:
      enabled: true
      ensureRunning: true
.ebextensions/custom-resource.config — Add SQS queue to environment
Resources:
  MyWorkerQueue:                      # CloudFormation syntax inside EB
    Type: AWS::SQS::Queue
    Properties:
      QueueName: my-worker-queue
      VisibilityTimeout: 300

option_settings:
  aws:elasticbeanstalk:application:environment:
    QUEUE_URL:
      Fn::GetOptionSetting:          # reference CFN resource output
        Namespace: aws:cloudformation:outputs

Docker on Elastic Beanstalk

Single Container Docker

  • Uses Dockerfile or Dockerrun.aws.json v1
  • Runs a single Docker container on each EC2 instance
  • EB manages the Docker daemon directly
  • Good for simple, single-service apps
  • No ECS involved — Docker runs on EC2 directly

Multi-Container Docker

  • Uses Dockerrun.aws.json v2
  • Runs multiple containers per EC2 instance
  • Uses Amazon ECS under the hood
  • Requires images pre-built in ECR or Docker Hub
  • Good for microservices or sidecar patterns
Dockerrun.aws.json v2 — Multi-Container
{
  "AWSEBDockerrunVersion": 2,
  "containerDefinitions": [
    {
      "name":    "web",
      "image":   "123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:latest",
      "memory":  512,
      "portMappings": [{ "hostPort": 80, "containerPort": 3000 }],
      "essential": true
    },
    {
      "name":   "nginx-proxy",
      "image":  "nginx:latest",
      "memory": 128,
      "links":  ["web"]
    }
  ]
}

Application Version Lifecycle Policy

EB can store up to 1,000 application versions. If you don't clean up old versions you can hit this limit and be unable to deploy. A Lifecycle Policy automates cleanup.

Time-Based

Delete versions older than N days (e.g., keep last 90 days of versions).

Count-Based

Keep only the N most recent versions (e.g., keep last 200 versions).

💡

Important: By default, EB does NOT delete the source bundle from S3 when it deletes a version. You can optionally set the policy to also delete the S3 source bundle, saving storage costs. Versions currently deployed to an environment are never deleted by lifecycle policy.

Cloning & Migration

Environment Cloning

  • Clone an existing environment to create an exact copy
  • Preserves: environment options, environment variables, installed software, config files
  • Useful for: promoting staging → production, Blue/Green setup
  • RDS data is NOT cloned — new empty database created
  • CLI: eb clone my-env --clone_name my-env-prod

RDS Decoupling (Migration)

  • RDS created inside EB environment is tied to the env lifecycle
  • If env is terminated → RDS is deleted (even with Retain policy if not careful)
  • Best practice: Create RDS outside EB, reference via env variable
  • To decouple existing RDS: take snapshot → create new external RDS from snapshot → update env variable → delete old RDS

Swapping Environment URLs (Blue/Green)

# Using EB CLI to swap environment CNAME (Blue/Green URL swap)
eb swap blue-env --destination_name green-env

# Or via AWS CLI
aws elasticbeanstalk swap-environment-cnames \
  --source-environment-name my-app-blue \
  --destination-environment-name my-app-green
ℹ️

URL swapping is a DNS-level change (CNAME swap). It takes effect within seconds for most clients, but DNS TTL means some clients may still hit the old environment briefly. Route 53 weighted routing gives more precise control for gradual shifts.

Health & Monitoring

Health Reporting Modes

Basic Health Reporting

  • Reports a single color: Green Yellow Red Grey
  • Based on ELB health checks and EC2 status checks
  • Free — no additional cost
  • Limited visibility into root cause

Enhanced Health Reporting

  • Detailed per-instance metrics (CPU, memory, load, disk)
  • HTTP response codes (2xx, 3xx, 4xx, 5xx rates)
  • Requires enhanced health agent on instance (via AMI or config)
  • Publishes metrics to CloudWatch (additional cost)
  • Required for managed platform updates

Health Status Colors

ColorMeaningCommon Causes
GreenAll healthy — environment functioning normally—
YellowDegraded — some checks failed recentlyHigh latency, occasional health check failures, recent scale events
RedSevere — environment not functionalInstance crashes, deployment failures, health check failures for multiple instances
GreyUnknown — no data from environmentEnvironment updating, no instances running, EB cannot reach environment

Managed Platform Updates

EB can automatically apply platform patches (OS, runtime) during a configurable maintenance window. Requires Enhanced Health Reporting to be enabled. You choose whether to allow minor and patch version updates automatically.

# .ebextensions to configure managed updates
option_settings:
  aws:elasticbeanstalk:managedactions:
    ManagedActionsEnabled: true
    PreferredStartTime: "Tue:09:00"   # UTC
  aws:elasticbeanstalk:managedactions:platformupdate:
    UpdateLevel: minor                  # minor | patch
    InstanceRefreshEnabled: true

Notifications & Events

Elastic Beanstalk emits events to Amazon EventBridge. You can create rules to react to environment and resource state changes, trigger Lambda, SNS, SQS, or other targets.

EventBridge Event Categories

Environment Operations

  • Environment create — start, success, fail
  • Environment update — start, success, fail
  • Environment terminate — start, success, fail

Resource Status

  • Auto Scaling Group events
  • ELB events
  • EC2 Instance events

Managed Updates

  • Managed update started
  • Managed update failed
  • Managed update completed

Environment Health

  • Health transitions (Green → Yellow → Red)
  • Trigger SNS notification or Lambda
  • Drive downstream alerting pipelines
EventBridge rule — alert on deployment failure
{
  "source": ["aws.elasticbeanstalk"],
  "detail-type": ["Elastic Beanstalk resource status change"],
  "detail": {
    "Status": ["Environment update failed", "Failed"],
    "EnvironmentName": ["my-app-production"]
  }
}
// Target: SNS → send email/PagerDuty alert
// Target: Lambda → auto-rollback, notify Slack

DOP-C02 Exam Tips

Must-Know Facts
  • Service Role = permissions for EB service to manage EC2/ASG/ELB. Trusts elasticbeanstalk.amazonaws.com.
  • Instance Profile = IAM role for your app code on EC2. Trusts ec2.amazonaws.com. Must include elasticbeanstalk-* S3 access.
  • Immutable = best rollback (terminate new ASG), highest cost, safest for prod.
  • Traffic Splitting = canary testing with automated rollback based on health metrics.
  • Blue/Green = swap environment URLs, NOT a native single-click EB feature — done via CNAME swap.
  • .ebextensions = .config files in application ZIP root. CloudFormation syntax in Resources: section. container_commands supports leader_only: true.
  • Worker Tier scales on SQS queue depth, not CPU/requests.
  • cron.yaml defines periodic tasks in the Worker Tier (and Web Tier).
  • Enhanced Health Reporting is required for Managed Platform Updates.
  • Version Limit: 1,000 max. Use Lifecycle Policy (time-based or count-based) to avoid hitting the limit.
  • RDS inside EB env = tied to env lifecycle. Best practice: create RDS outside EB.
  • Multi-Container Docker uses ECS under the hood (Dockerrun.aws.json v2).

Common Exam Scenarios

ScenarioAnswer
"Deploy with zero downtime and ability to instantly revert if issues found" Immutable deployment (or Blue/Green). Immutable = instant rollback by terminating temp ASG.
"Test new version with 5% of real production traffic before full rollout" Traffic Splitting (canary deployment). Configurable % with automated rollback on failures.
"App needs to process long-running background jobs triggered by web requests" Web Tier + Worker Tier with SQS decoupling. Web tier queues tasks; Worker tier scales on SQS depth.
"Run DB migration only on one instance during deployment" .ebextensions container_commands with leader_only: true.
"App failed to deploy. Need to investigate without losing the failed state" Deploy with --on-failure DO_NOTHING (disable rollback). Inspect the instance logs.
"Prevent accidental deletion of production database when EB env is terminated" Create RDS outside of EB environment. Reference it via environment variable. Never put prod RDS inside EB.
"Reached 1000 application version limit and cannot deploy" Configure an Application Version Lifecycle Policy (count-based or time-based).
"Need to get notified when a production deployment fails" EventBridge rule on EB events → SNS topic → email/PagerDuty. Or Lambda for auto-rollback.
"Application needs to read/write to an S3 bucket from EC2 instances" Add S3 permissions to the EC2 Instance Profile IAM role, not the Service Role.

EB vs Other AWS Services

ServiceWhen to Choose
Elastic BeanstalkDeveloper wants PaaS simplicity. Application code only. Standard web apps. Full control still available.
ECS / FargateNeed containers with fine-grained control over task definitions, networking, IAM, and cluster configuration.
CloudFormationNeed full IaC control of every resource. Multi-service stacks. Beanstalk actually uses CFN under the hood.
CodeDeployNeed deployment control over existing EC2 fleets or Lambda. Not opinionated about platform/stack.
App RunnerFully managed, no config. Simpler than EB. Good for containers with zero infrastructure management.

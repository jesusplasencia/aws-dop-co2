01Core

Overview

AWS AppConfig is a capability of AWS Systems Manager that lets you deploy runtime configuration to applications separately from code deployments. Changes take effect without restarting the application or pushing new code.

💡

Key insight: AppConfig solves the "config in code" problem. Instead of hardcoding feature flags or tuning parameters and redeploying, you push config changes through a safe deployment pipeline with validation and automatic rollback.

What it solves

Feature Flags

Turn features on/off for all users or a subset without deploying code. A/B testing, gradual rollouts, kill switches.

Tunable Parameters

Change rate limits, timeout values, connection pool sizes, logging levels at runtime — no redeployment.

Allow/Block Lists

Maintain dynamic lists of IPs, user IDs, or regions to allow or deny. Update instantly without redeployment.

04Deploy

Deployment Strategies

A deployment strategy defines how fast the new config is rolled out and how long to bake (wait after 100% rollout before marking complete).

StrategyGrowth TypeBehaviorUse Case
AppConfig.AllAtOnce All at once 100% of clients receive new config immediately. No gradual rollout. Dev/test environments only. Not recommended for prod.
AppConfig.Linear50PercentEvery30Seconds Linear 50% → 100% over 60 seconds. 1-minute bake time. Testing / staging environments.
AppConfig.Canary10Percent20Minutes Exponential 10% immediately, then 100% after 20 minutes if no alarms fire. Production — safe gradual rollout with alarm monitoring.
Custom Linear or Exponential Define your own growth factor, interval, and bake time. Any specific requirement.

Custom Strategy Parameters

Growth Factor

% of clients to add each interval. E.g., 10 = add 10% each step.

Deployment Duration

Total time for the full rollout (in minutes). Controls step interval.

Bake Time

Minutes to wait after 100% rollout. Alarms can still trigger rollback during bake.

custom deployment strategy — CloudFormation
MyDeploymentStrategy:
  Type: AWS::AppConfig::DeploymentStrategy
  Properties:
    Name: Prod-Canary-10Percent-10Min
    DeploymentDurationInMinutes: 10
    FinalBakeTimeInMinutes: 10      # wait 10 min after 100% for alarm monitoring
    GrowthFactor: 10                  # 10% → 20% → ... → 100%
    GrowthType: LINEAR               # or EXPONENTIAL
    ReplicateTo: NONE                 # or SSM_DOCUMENT
05Deploy

Validation

AppConfig validates configuration before deploying it. If validation fails, the deployment is blocked. Two validator types exist:

JSON Schema — Syntactic Check
  • Validates config against a JSON Schema
  • Checks structure and data types
  • Runs automatically on every save and deploy
  • No Lambda needed — purely declarative
  • Best for: required fields, type checking, format validation
Lambda — Semantic Check
  • AppConfig invokes your Lambda with the config
  • Lambda can check business logic
  • E.g., validate feature flag % sum = 100%
  • E.g., verify referenced resource ARNs exist
  • Lambda must return HTTP 200 for validation to pass
json-schema-validator.json + lambda-validator.py
// JSON Schema validator — attached to Configuration Profile
{
  "$schema": "http://json-schema.org/draft-04/schema#",
  "title": "FeatureFlags",
  "type": "object",
  "required": ["dark_mode_enabled", "max_upload_mb"],
  "properties": {
    "dark_mode_enabled": { "type": "boolean" },
    "max_upload_mb": {
      "type": "integer",
      "minimum": 1, "maximum": 100
    }
  }
}

# Lambda validator — checks business logic
import json

def handler(event, context):
    config = json.loads(event['content'])
    # Semantic check: rollout percentages must sum to 100
    rollouts = config.get('rollout_percentages', {})
    total = sum(rollouts.values())
    if total != 100:
        raise ValueError(f"Rollout percentages sum to {total}, not 100")
    # Validation passes — return nothing (HTTP 200 implicit)
⚠️

Lambda validator must throw an exception to fail validation. If it returns normally (any value), validation passes. AppConfig grants the Lambda invoke permission via the configuration profile's IAM setup.

06Deploy

Rollback & CloudWatch Alarms

You can attach CloudWatch Alarms to an AppConfig Environment. If any alarm enters the ALARM state during a deployment (including bake time), AppConfig automatically rolls back to the previous config version.

🚀
Deploy starts
10% of clients get new config
→
📊
CW Alarm monitors
Error rate, latency, custom metric
→
🚨
Alarm fires
Metric threshold breached
→
⏪
Auto rollback
Previous version restored to ALL clients
💡

Rollback is automatic and immediate. AppConfig monitors alarms throughout the entire deployment and bake period. You can also trigger a manual rollback via StopDeployment API at any time during an active deployment.

appconfig-environment-with-alarm.yaml
MyProdEnvironment:
  Type: AWS::AppConfig::Environment
  Properties:
    ApplicationId: !Ref MyApp
    Name: production
    Monitors:
      - AlarmArn: !GetAtt ErrorRateAlarm.Arn
        AlarmRoleArn: !GetAtt AppConfigAlarmRole.Arn  # role to read CW alarms
07Usage

Config Retrieval Patterns

Applications retrieve config by polling the AppConfig API. They cache the result locally and only call the API again when the cache expires. AppConfig does not push config to apps.

AppConfig Agent (Recommended)

The AppConfig Agent is a sidecar/daemon that handles polling, caching, and token management. Your app reads from a local HTTP endpoint — no SDK needed.

app reads config from local agent (any language)
# Agent runs locally — polls AppConfig on a configurable interval
# Your app just reads from localhost:2772
import urllib.request, json

def get_config():
    url = "http://localhost:2772/applications/my-app/environments/prod/configurations/feature-flags"
    with urllib.request.urlopen(url) as r:
        return json.loads(r.read())

config = get_config()
if config['dark_mode_enabled']:
    enable_dark_mode()

AWS SDK (Direct API)

For Lambda and other environments where a sidecar isn't available, use the SDK directly. Always use GetLatestConfiguration with a token for efficient polling.

lambda-sdk-retrieval.py
import boto3, json, os

appconfig = boto3.client('appconfigdata')

# Cache the session token between invocations (Lambda /tmp or global)
_session_token = None

def get_config():
    global _session_token
    if not _session_token:
        resp = appconfig.start_configuration_session(
            ApplicationIdentifier='my-app',
            EnvironmentIdentifier='prod',
            ConfigurationProfileIdentifier='feature-flags',
            RequiredMinimumPollIntervalInSeconds=30
        )
        _session_token = resp['InitialConfigurationToken']

    resp = appconfig.get_latest_configuration(ConfigurationToken=_session_token)
    _session_token = resp['NextPollConfigurationToken']  # rotate token each poll

    if resp['Configuration'].read():   # empty = no change since last poll
        return json.loads(resp['Configuration'].read())
    return None  # no change — use cached config
ℹ️

Token rotation is mandatory. Each call to GetLatestConfiguration returns a new token — you must use it for the next call. If the response body is empty, config hasn't changed since last poll — use your cached version. AppConfig charges per API call, so caching is important.

08Exam Focus

Exam Tips

Must-Know Facts
  • AppConfig is for runtime config, not code deployment. No restarts or redeployments needed — apps poll for the new config.
  • Two validator types: JSON Schema = syntax/structure check. Lambda = semantic/business logic check. Both block deployment on failure.
  • CloudWatch Alarms trigger automatic rollback during deployment and bake time. Attach alarms to the Environment (not the config profile).
  • Deployment strategies control rollout speed: AllAtOnce (dev only), Linear, Canary (recommended for prod). Bake time gives alarms time to fire after 100% rollout.
  • AppConfig Hosted Config Store is the recommended source — built in, versioned, no extra resources. SSM and S3 are alternatives.
  • Lambda Extension is the recommended retrieval method for Lambda — runs as an extension alongside function, caches config locally.
  • AppConfig is part of AWS Systems Manager — important for exam categorization questions.

Scenario Patterns

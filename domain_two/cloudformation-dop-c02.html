01 / Overview

CloudFormation Overview

AWS CloudFormation is a declarative Infrastructure-as-Code (IaC) service. You describe what you want, and CloudFormation figures out how to create it — handling dependencies, ordering, and rollback automatically.

📄 Template (.yaml/.json)
→
🪣 S3 Bucket
→
⚙️ AWS CloudFormation
→
📦 Stack
→
☁️ AWS Resources
💡

Templates must be uploaded to S3 before CloudFormation references them. To update a template you must upload a new version — you cannot edit the previous one in-place.

Key Benefits

Infrastructure as Code

No manual resource creation. Version-controlled in Git. Infrastructure changes go through code review.

Cost Management

Every resource in a stack gets a stack identifier tag — making cost attribution effortless. Estimate costs before deploying.

Productivity

Destroy and recreate entire environments on the fly. Automated diagram generation. Declarative programming model.

Separation of Concerns

Separate stacks per app, per layer (network, app, DB). Outputs let stacks share references safely.

Stack Lifecycle

OperationBehaviorExam Note
CREATEProvisions all resources. On failure, rolls back everything by default.Set OnFailure=DO_NOTHING to keep resources for debugging.
UPDATECompares current vs. desired template; executes only the diff.Use ChangeSets to preview before applying.
DELETEDeletes all resources created by the stack (respects DeletionPolicy).Non-empty S3 buckets block deletion by default.
⚠️

Deleting a stack deletes every artifact CloudFormation created. Use DeletionPolicy: Retain or TerminationProtection to prevent accidental data loss.

02 / Structure

Template Anatomy

A CloudFormation template is a YAML or JSON document composed of several top-level sections:

AWSTemplateFormatVersion: '2010-09-09'   # Optional. Only valid value.

Description: 'What this stack does'          # Optional. Max 1024 chars.

Metadata:                                    # Optional. Arbitrary objects.
  AWS::CloudFormation::Interface:
    ParameterGroups: [...]

Parameters:                                  # Optional. User inputs.
  EnvType:
    Type: String
    AllowedValues: [dev, prod]

Mappings:                                    # Optional. Fixed key-value lookup.
  RegionMap:
    us-east-1:
      HVM64: ami-0ff8a91507f77f867

Conditions:                                  # Optional. Conditional logic.
  IsProd: !Equals [!Ref EnvType, prod]

Transform:                                   # Optional. SAM / Macros.
  - AWS::Serverless-2016-10-31

Resources:                                   # MANDATORY — the only required section!
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub 'my-bucket-${AWS::AccountId}'

Outputs:                                     # Optional. Export values for other stacks.
  BucketArn:
    Value: !GetAtt MyBucket.Arn
    Export:
      Name: !Sub '${AWS::StackName}-BucketArn'
🎯 Exam Focus

Resources is the only mandatory section. Everything else is optional. Know which section serves which purpose and the difference between Parameters (dynamic input) vs Mappings (static lookup).

Resource Type Format

Resource Type Identifier Pattern
service-provider::service-name::data-type-name

# Examples:
AWS::EC2::Instance
AWS::S3::Bucket
AWS::RDS::DBInstance
AWS::Lambda::Function
AWS::IAM::Role
Custom::MyCustomResource   # Custom Resource

CloudFormation supports over 700 resource types. The full reference is at docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/.

03 / Parameters

Parameters

Parameters let users provide dynamic inputs at stack creation/update time. Use them whenever a value might change across deployments.

💡

Decision rule: Ask yourself — "Is this CloudFormation resource configuration likely to change in the future?" If yes, make it a parameter.

Parameter Settings

PropertyDescriptionExample
TypeData type (required)String, Number, List<Number>, CommaDelimitedList, AWS::EC2::KeyPair::KeyName, AWS::SSM::Parameter::Value<String>
DefaultValue used if none providedt2.micro
AllowedValuesWhitelist of valid values[dev, staging, prod]
AllowedPatternRegex the value must match[a-zA-Z0-9]*
MinLength / MaxLengthString length constraints8 / 64
MinValue / MaxValueNumeric range constraints1 / 999
NoEchoMasks value in console/API (for secrets)true
ConstraintDescriptionError message shown when validation fails"Must be t2.micro or t2.small"
 
Parameters:
  InstanceType:
    Description: Choose an EC2 Instance Type
    Type: String
    AllowedValues: [t2.micro, t2.small, t3.medium]
    Default: t2.micro
    ConstraintDescription: "Must be a valid EC2 instance type."

  DatabasePassword:
    Description: Database Admin Password
    Type: String
    NoEcho: true               # never shown in console or logs
    MinLength: 8
    MaxLength: 64

# Referencing a parameter:
Resources:
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType

04 / Mappings

Mappings

Mappings are static, hardcoded lookup tables embedded in the template. Use them when values are known at design time — typically for environment-based, region-based, or AMI-based lookups.

Mappings:
  RegionAMIMap:
    us-east-1:
      HVM64: ami-0ff8a91507f77f867
      HVMG2: ami-0a584ac55a7631c0c
    us-west-2:
      HVM64: ami-a0cfeed8
      HVMG2: ami-0e09505bc235aa82d
    eu-west-1:
      HVM64: ami-047bb4163c506cd98

  EnvironmentMap:
    prod:
      InstanceType: m5.large
      MultiAZ: true
    dev:
      InstanceType: t2.micro
      MultiAZ: false

Resources:
  MyEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      # !FindInMap [ MapName, TopLevelKey, SecondLevelKey ]
      ImageId: !FindInMap [RegionAMIMap, !Ref "AWS::Region", HVM64]
      InstanceType: !FindInMap [EnvironmentMap, !Ref EnvType, InstanceType]
Mappings — Use When
  • Values are known at template-write time
  • Need region → AMI lookups
  • Environment-based config (dev/prod)
  • No user input needed
Parameters — Use When
  • Values are decided at deploy time
  • Need user/pipeline input
  • Values differ per deployment
  • Reusing the same template across teams

05 / Outputs & Exports

Outputs & Cross-Stack References

Outputs let you expose values from a stack so they can be consumed by the console, CLI, or — via Export — by other stacks in the same region & account.

# Stack A — Network Stack — exports the VPC and Security Group
Outputs:
  VPCID:
    Description: The VPC ID
    Value: !Ref MyVPC
    Export:
      Name: !Sub '${AWS::StackName}-VpcId'   # unique per region

  SSHSecurityGroupId:
    Value: !GetAtt SSHSecurityGroup.GroupId
    Export:
      Name: SSHSecurityGroup               # export names are region-scoped

---
# Stack B — App Stack — imports the security group
Resources:
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      SecurityGroupIds:
        - !ImportValue SSHSecurityGroup    # must match Export Name exactly
      SubnetId: !ImportValue NetworkStack-VpcId
⚠️

You cannot delete a stack that has exported outputs being consumed by another stack. You must first delete the importing stack (or update it to stop importing).

ℹ️

Export names must be unique within a region + account. Best practice: prefix with !Sub '${AWS::StackName}-ExportName' to avoid collisions.

06 / Conditions

Conditions

Conditions allow conditional creation of resources, properties, or outputs. They are evaluated at deployment time based on parameter values, mappings, or other conditions.

Conditions:
  IsProd:       !Equals [ !Ref EnvType, prod ]
  IsNotProd:    !Not    [!Condition IsProd]
  IsUsEast1:    !Equals [ !Ref "AWS::Region", us-east-1 ]
  IsProdUsEast1: !And
    - !Condition IsProd
    - !Condition IsUsEast1

Resources:
  # Entire resource conditionally created
  ProdDatabase:
    Type: AWS::RDS::DBInstance
    Condition: IsProd
    Properties:
      DBInstanceClass: db.m5.large
      MultiAZ: true

  # Conditional property value using !If
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !If [IsProd, m5.large, t2.micro]
      DeleteOnTermination: !If [IsNotProd, true, !Ref "AWS::NoValue"]

Condition Functions

FunctionUsage
!Equals [a, b]Returns true if a == b
!Not [condition]Logical NOT
!And [c1, c2, ...]All conditions must be true (2–10 conditions)
!Or [c1, c2, ...]At least one condition must be true (2–10 conditions)
!If [condition, valueIfTrue, valueIfFalse]Conditional value selection — can use !Ref AWS::NoValue to omit a property

08 / Rollbacks

Rollbacks

ScenarioDefault BehaviorOverride Option
Stack Create FailureEverything rolls back and is deletedSet --on-failure DO_NOTHING to keep failed resources for debugging
Stack Update FailureRolls back to the last known good stateSet DisableRollback to preserve state for investigation
Update Rollback FailureStack in UPDATE_ROLLBACK_FAILEDUse ContinueUpdateRollback after manually fixing the issue

UPDATE_ROLLBACK_FAILED — Common Causes

  • Resources were changed outside of CloudFormation (manual console changes) — drift
  • Insufficient IAM permissions during rollback
  • A dependent resource was deleted outside the stack
  • The AWS service had a temporary limit or quota issue

Fix: Manually correct the out-of-band change, then call aws cloudformation continue-update-rollback.

🎯 Exam Focus

Know the difference between ROLLBACK_COMPLETE, UPDATE_ROLLBACK_FAILED, and UPDATE_ROLLBACK_COMPLETE. The fix for UPDATE_ROLLBACK_FAILED is always to manually resolve the issue then call ContinueUpdateRollback.

09 / Capabilities

Capabilities & IAM

When your template creates or modifies IAM resources, you must explicitly acknowledge this by passing capability flags. Without them, CloudFormation throws InsufficientCapabilitiesException.

Does template contain AWS::IAM::* resources?
├─ YES → Does it use RoleName / UserName / GroupName / PolicyName?
│ ├─ YES → CAPABILITY_NAMED_IAM
│ └─ NO → CAPABILITY_IAM
└─ NO → No IAM capability needed
Does template use Transform: or AWS::CloudFormation::Stack?
├─ YES → CAPABILITY_AUTO_EXPAND
└─ NO → No auto-expand capability needed
# IAM resources without explicit names
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-stack \
  --capabilities CAPABILITY_IAM

# IAM resources WITH explicit RoleName/PolicyName
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-stack \
  --capabilities CAPABILITY_NAMED_IAM

# SAM/Nested Stacks — both are often needed together
sam deploy \
  --template-file template.yaml \
  --stack-name my-stack \
  --capabilities CAPABILITY_IAM CAPABILITY_AUTO_EXPAND

Service Role

A Service Role is an IAM role that CloudFormation assumes to create/update/delete stack resources on your behalf. This enables the least privilege principle:

  • The user deploying the stack only needs cloudformation:* and iam:PassRole permissions.
  • The Service Role holds the permissions to create the actual resources (EC2, S3, RDS, etc.).
  • Without a service role, CFN uses your identity's permissions directly.
💡

The user deploying the stack must have iam:PassRole to pass the service role to CloudFormation.

10 / DeletionPolicy

DeletionPolicy & UpdateReplacePolicy

Controls what happens to a resource when the stack is deleted or when the resource is replaced during an update.

PolicyOn Stack Delete / Resource RemoveSupported Resources
DeleteDefault. Resource is deleted. Does NOT work on non-empty S3 buckets.All
RetainResource is kept. CloudFormation releases ownership — resource persists outside the stack.All
SnapshotCreates a final snapshot before deleting. Then deletes the resource.EBS, ElastiCache, RDS, Redshift, Neptune, DocumentDB
 
Resources:
  ProductionDB:
    Type: AWS::RDS::DBInstance
    DeletionPolicy: Snapshot         # snapshot before delete
    UpdateReplacePolicy: Snapshot    # snapshot if resource must be replaced
    Properties:
      DBInstanceClass: db.t3.medium
      Engine: mysql

  ImportantBucket:
    Type: AWS::S3::Bucket
    DeletionPolicy: Retain           # keep bucket even after stack delete
ℹ️

UpdateReplacePolicy is the sibling of DeletionPolicy — it controls what happens when a resource must be replaced during an update (e.g., changing a DB engine forces replacement). Same three options: Delete, Retain, Snapshot.

11 / Stack Policy

Stack Policy & Termination Protection

A Stack Policy is a JSON document that defines which update actions are allowed on specific resources during a stack update — protecting critical resources from accidental changes.

// Example: Allow all updates EXCEPT to ProductionDatabase
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "LogicalResourceId/ProductionDatabase"
    }
  ]
}

Update Actions

Update Actions

Update:Modify — in-place updates
Update:Replace — replacement updates
Update:Delete — removes resource
Update:* — all of the above
Termination Protection

Prevents accidental stack deletion entirely. Enable it on all production stacks. Can be toggled via console or CLI after creation.

aws cloudformation update-termination-protection --enable-termination-protection --stack-name my-stack
🎯 Exam Focus

Stack Policy ≠ IAM policy. Stack Policy controls which CloudFormation update actions are permitted on resources. IAM controls who can call CloudFormation APIs. Once a Stack Policy is applied, all resources not explicitly allowed are denied by default.

12 / Custom Resources

Custom Resources

Custom Resources let you run arbitrary logic (via Lambda or SNS) during CloudFormation Create/Update/Delete events. Use them when:

  • The AWS resource type isn't yet supported by CloudFormation
  • You need to provision on-premises or third-party resources
  • You need to run custom provisioning logic (e.g., empty an S3 bucket before deletion)
  • You need to lookup values at deploy time (e.g., latest AMI ID from Parameter Store)
Resources:
  # Step 1: The Lambda function that handles CFN events
  CustomResourceLambda:
    Type: AWS::Lambda::Function
    Properties:
      Handler: index.handler
      Runtime: python3.12
      Role: !GetAtt LambdaRole.Arn
      Code:
        ZipFile: |
          import cfnresponse, boto3
          def handler(event, context):
              request_type = event['RequestType']
              bucket = event['ResourceProperties']['BucketName']
              if request_type == 'Delete':
                  s3 = boto3.resource('s3')
                  s3.Bucket(bucket).objects.all().delete()
              cfnresponse.send(event, context, cfnresponse.SUCCESS, {})

  # Step 2: Reference the custom resource in your template
  EmptyBucketOnDelete:
    Type: Custom::S3BucketCleaner          # Custom:: prefix required
    Properties:
      ServiceToken: !GetAtt CustomResourceLambda.Arn  # Lambda or SNS ARN
      BucketName: !Ref MyBucket
    DependsOn: MyBucket
💡

The Lambda must respond to the CloudFormation pre-signed S3 URL with either SUCCESS or FAILED. Use the cfnresponse module for Python/Node. If Lambda times out or fails to respond, CloudFormation will wait up to 1 hour before timing out the stack operation.

13 / Dynamic References

Dynamic References & Secrets

Dynamic References resolve values from external stores at stack create/update time — keeping secrets out of your templates entirely.

SyntaxSourceNotes
{{resolve:ssm:/param/name:version}}SSM Parameter Store (plaintext)Version is optional; omit for latest
{{resolve:ssm-secure:/param/name:version}}SSM Parameter Store (SecureString)Not supported in all resource types
{{resolve:secretsmanager:secret-id:SecretString:key}}Secrets ManagerCan reference specific JSON key in secret
 
Resources:
  MyDBInstance:
    Type: AWS::RDS::DBInstance
    Properties:
      # Pull username from SSM
      MasterUsername: '{{resolve:ssm:/prod/rds/username:1}}'
      # Pull password from Secrets Manager
      MasterPassword: '{{resolve:secretsmanager:MyDBSecret:SecretString:password}}'
      DBInstanceClass: db.t3.micro
      Engine: mysql

  # RDS Native Secret Rotation — ManageMasterUserPassword
  MyCluster:
    Type: AWS::RDS::DBCluster
    Properties:
      Engine: aurora-mysql
      MasterUsername: masteruser
      ManageMasterUserPassword: true   # CFN creates & manages secret in Secrets Manager

Outputs:
  SecretArn:
    Value: !GetAtt MyCluster.MasterUserSecret.SecretArn

14 / Helper Scripts

CloudFormation Helper Scripts

Helper scripts run on EC2 instances to configure them as part of a CloudFormation stack operation. They are installed via the aws-cfn-bootstrap package.

cfn-init

Reads AWS::CloudFormation::Init metadata from the template and configures the instance. Logs to /var/log/cfn-init.log.

cfn-signal

Signals CloudFormation that the instance is done configuring. Used with CreationPolicy and WaitCondition.

cfn-hup

Daemon that polls for stack metadata changes every 15 minutes and re-runs cfn-init on change. Config in /etc/cfn/cfn-hup.conf.

cfn-get-metadata

Retrieves CloudFormation metadata for a specific resource. Useful for debugging.

AWS::CloudFormation::Init Sections

KeyPurpose
packagesInstall pre-packaged apps via yum, apt, rpm, python, rubygems
groupsDefine Linux user groups
usersCreate users and assign to groups
sourcesDownload tarballs/zips from URLs and unpack them
filesCreate/manage files — inline content or from S3/URL
commandsRun shell commands in order; supports test, ignoreErrors
servicesStart/stop/enable sysvinit/upstart/systemd services
 
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Metadata:
      AWS::CloudFormation::Init:
        config:
          packages:
            yum:
              httpd: []
              php: []
          files:
            "/var/www/html/index.php":
              content: !Sub |
                <?php echo "Hello from ${AWS::StackName}"; ?>
              mode: "000644"
              owner: apache
          services:
            sysvinit:
              httpd:
                enabled: true
                ensureRunning: true
    Properties:
      UserData:
        !Base64
          !Sub |
            #!/bin/bash -xe
            # Install cfn-bootstrap tools
            yum install -y aws-cfn-bootstrap
            # Run cfn-init — reads Metadata and configures instance
            /opt/aws/bin/cfn-init -v \
              --stack ${AWS::StackName} \
              --resource WebServer \
              --region ${AWS::Region}
            # Signal success/failure back to CloudFormation
            /opt/aws/bin/cfn-signal -e $? \
              --stack ${AWS::StackName} \
              --resource WebServer \
              --region ${AWS::Region}
    CreationPolicy:
      ResourceSignal:
        Timeout: PT15M      # wait up to 15 minutes for signal
        Count: 1

15 / Nested Stacks

Nested Stacks & DependsOn

Nested stacks embed one CloudFormation stack inside another using AWS::CloudFormation::Stack. Use them to isolate reusable patterns (VPC, security groups, ALB) into modular, shareable templates.

# Root Stack — references nested stacks stored in S3
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/bucket/network.yaml
      Parameters:
        EnvType: !Ref EnvType
      TimeoutInMinutes: 20

  AppStack:
    Type: AWS::CloudFormation::Stack
    DependsOn: NetworkStack
    Properties:
      TemplateURL: https://s3.amazonaws.com/bucket/app.yaml
      Parameters:
        # Pass outputs from NetworkStack to AppStack
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
Nested Stacks
  • Single account/region
  • Parent manages lifecycle
  • Good for modular templates
  • Update parent to update nested
  • Outputs passed via !GetAtt StackName.Outputs.Key
StackSets
  • Multi-account / multi-region
  • Independent stack instances
  • Good for org-wide baselines
  • Update StackSet propagates to all instances
  • Requires admin + execution roles

DependsOn

Forces a resource to be created/deleted after another. Applied automatically when using !Ref or !GetAtt — only use explicit DependsOn when there's an implicit dependency CFN can't detect.

Resources:
  MyEC2Instance:
    Type: AWS::EC2::Instance
    DependsOn: MyInternetGatewayAttachment  # IGW must be attached first
    Properties:
      ImageId: ami-0a3c3a20c09d6f377
      InstanceType: t3.micro

  MyBucket:
    Type: AWS::S3::Bucket
    DependsOn: MyEC2Instance   # bucket created after instance

16 / StackSets

StackSets

StackSets allow you to deploy the same CloudFormation template across multiple accounts and/or regions with a single operation. Ideal for organization-wide security baselines, compliance, and governance.

Permission Models

Self-Managed Permissions
  • You manually create IAM roles in admin and target accounts
  • Admin account: AWSCloudFormationStackSetAdministrationRole
  • Target accounts: AWSCloudFormationStackSetExecutionRole
  • Deploy to any account you have role access to
  • Works without AWS Organizations
Service-Managed Permissions
  • StackSets create IAM roles on your behalf
  • Requires trusted access with AWS Organizations
  • Auto-deploy to new accounts added to the org
  • Can target entire OU (Organizational Unit)
  • Simpler setup; recommended for Organizations

Operation Preferences

aws cloudformation create-stack-instances \
  --stack-set-name security-baseline \
  --accounts 222222222222 333333333333 \
  --regions us-east-1 us-west-2 eu-west-1 \
  --operation-preferences \
    FailureToleranceCount=1,\          # fail at most 1 account before stopping
    MaxConcurrentCount=2,\             # deploy to 2 accounts in parallel
    RegionConcurrencyType=PARALLEL    # deploy all regions simultaneously per account
ℹ️

Auto Deployment: With service-managed permissions, StackSets can automatically deploy to new accounts as they're added to an OU — and remove stack instances when accounts leave the OU.

17 / ChangeSets & Drift

ChangeSets & Drift Detection

ChangeSets

ChangeSets let you preview exactly what changes will occur before applying an update — giving you confidence that only the intended resources will be modified.

# 1. Create a ChangeSet (does NOT apply changes yet)
aws cloudformation create-change-set \
  --stack-name my-stack \
  --template-body file://updated-template.yaml \
  --change-set-name my-changes

# 2. Review the ChangeSet
aws cloudformation describe-change-set \
  --change-set-name my-changes \
  --stack-name my-stack

# 3a. Execute the ChangeSet (applies changes)
aws cloudformation execute-change-set \
  --change-set-name my-changes \
  --stack-name my-stack

# 3b. Or delete it (abandons changes)
aws cloudformation delete-change-set \
  --change-set-name my-changes \
  --stack-name my-stack
💡

ChangeSets for Nested Stacks show changes across all child stacks — you see the complete picture before committing.

Drift Detection

Drift occurs when a stack's actual resource configurations differ from what CloudFormation expects based on the template. This typically happens when someone makes manual changes outside of CloudFormation.

Drift StatusMeaning
IN_SYNCResource matches the template
MODIFIEDResource exists but differs from template
DELETEDResource was deleted outside CloudFormation
NOT_CHECKEDDrift detection hasn't been run for this resource
⚠️

Changes made through CloudFormation to a stack instance are NOT considered drift. Only out-of-band changes (console, CLI, SDK acting directly on the resource) cause drift.

# Detect drift on entire stack
aws cloudformation detect-stack-drift --stack-name my-stack

# Check drift status
aws cloudformation describe-stack-drift-detection-status \
  --stack-drift-detection-id <id-from-above>

# See drifted resources
aws cloudformation describe-stack-resource-drifts \
  --stack-name my-stack \
  --stack-resource-drift-status-filters MODIFIED DELETED

18 / Troubleshooting

Troubleshooting

Common Error States

DELETE_FAILED

  • S3 bucket is not empty — use a Custom Resource Lambda to empty it first
  • Security Group in use — EC2 instances referencing it must be deleted first
  • RDS with deletion protection — must disable deletion protection first
  • Consider DeletionPolicy: Retain for resources that block deletion

UPDATE_ROLLBACK_FAILED

  • Resources were changed outside CloudFormation (drift)
  • Insufficient IAM permissions during rollback
  • A service quota was hit
  • Fix: Manually resolve the issue → aws cloudformation continue-update-rollback
  • Skip specific resources with --resources-to-skip flag

StackSet Failures

  • Insufficient permissions in the target account
  • Template creates globally-unique resources (S3 bucket names) that already exist
  • Admin account doesn't have a trust relationship with the target account
  • Target account hasn't enabled AWSCloudFormationStackSetExecutionRole

EC2 / Instance-Related

  • Can't set EC2 PrivateDnsName in CFN — not a supported attribute
  • Rule of thumb: If you can't configure something in the AWS Console, you can't do it in CloudFormation either
  • UserData changes don't take effect until instance replacement (UpdateReplacePolicy)
  • cfn-init failures — check /var/log/cfn-init.log on the instance

DOP-C02 Exam Tips

🔑 Must-Know Concepts
  • Resources is the only mandatory template section.
  • !Ref on a Parameter returns the parameter value; on a Resource returns the physical ID (e.g., bucket name).
  • !GetAtt returns a specific attribute of a resource (e.g., !GetAtt MyBucket.Arn).
  • NoEcho: true on a Parameter masks the value — use for passwords and API keys.
  • Export names must be unique per region + account. You can't delete a stack whose exports are in use by another stack.
  • CAPABILITY_NAMED_IAM when IAM resources have explicit names; CAPABILITY_IAM when auto-named.
  • EC2 UserData must be passed through !Base64 or Fn::Base64.
  • cfn-signal + CreationPolicy = wait for instance to be properly configured before marking stack CREATE_COMPLETE.
  • StackSets with service-managed permissions support automatic deployment to new org accounts.
  • Drift = out-of-band changes. CFN changes to a stack instance are NOT drift.

Capability Selection Decision

Template has AWS::IAM::* resources?
├─ Yes + named (RoleName, UserName) → CAPABILITY_NAMED_IAM
├─ Yes + no explicit names → CAPABILITY_IAM
└─ No → no IAM capability needed

Template has Transform: or AWS::CloudFormation::Stack?
├─ Yes → CAPABILITY_AUTO_EXPAND
└─ No → no auto-expand needed
⚡ Common Scenario: Prevent Accidental Deletion

Question asks how to protect a production stack from accidental deletion AND prevent accidental updates to a critical database:

  • Enable Termination Protection on the stack → prevents stack deletion
  • Add a Stack Policy with Deny Update:* on the database resource → prevents updates
  • Set DeletionPolicy: Snapshot on the DB resource → snapshot before deletion if somehow triggered

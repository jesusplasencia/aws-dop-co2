01 Foundation

Overview

AWS SAM (Serverless Application Model) is an open-source framework that extends CloudFormation with a simplified syntax specifically for serverless applications. A SAM template is transformed into a standard CloudFormation template at deploy time — meaning you get all CFN power with far less verbosity.

🔑

Core mental model: SAM is syntactic sugar on top of CloudFormation. Every SAM resource (AWS::Serverless::Function) expands into multiple standard CFN resources (Lambda function + IAM role + event source mappings + aliases). You write 10 lines; CFN sees 80.

What SAM Gives You

Simplified Syntax

Define Lambda functions, APIs, and tables in a fraction of the CloudFormation code. SAM handles the boilerplate.

Local Development

Run and test Lambda, API Gateway, and DynamoDB locally using Docker containers — no AWS account needed for dev.

Safe Deployments

Built-in CodeDeploy integration for Lambda traffic shifting with automated rollback on CloudWatch alarms.

SAM vs Pure CloudFormation

Pure CloudFormation
  • Verbose: Lambda needs Function + Role + Permission resources
  • Manual IAM policy writing for common patterns
  • No local testing toolchain
  • No built-in traffic shifting
  • Works for any AWS resource
SAM (extends CFN)
  • Concise: one AWS::Serverless::Function expands to all of the above
  • Policy templates for common permissions (S3, DynamoDB, SQS)
  • sam local invoke and sam local start-api
  • Built-in DeploymentPreference with CodeDeploy
  • Serverless-focused; falls back to full CFN when needed
02 Foundation

How SAM Works

SAM operates as a two-phase process: build locally, then package and deploy to AWS. The sam build command compiles and prepares your application. The sam deploy command transforms the SAM template into CloudFormation, uploads artifacts to S3, and creates/updates the CloudFormation stack.

STEP 01
✍️
Write SAM Template
YAML with AWS::Serverless::* resources
template.yaml
→
STEP 02
🔨
sam build
Compile code, resolve dependencies, prepare artifacts
sam build
→
STEP 03
🔄
Transform → CFN
SAM expands resources to full CloudFormation template
Transform macro
→
STEP 04
📦
Package & Upload
ZIP code to S3, update template refs
sam package
→
STEP 05
🚀
Deploy to AWS
Creates/executes CloudFormation ChangeSet
sam deploy
ℹ️

Under the hood: SAM uses the Transform: 'AWS::Serverless-2016-10-31' CloudFormation macro. When CloudFormation processes the template, it calls this macro which expands all AWS::Serverless::* resources into standard CloudFormation resources. You need CAPABILITY_IAM and CAPABILITY_AUTO_EXPAND when deploying SAM templates.

03 Foundation

Template Anatomy

template.yaml — complete SAM template structure
# ① REQUIRED: declare the SAM transform
AWSTemplateFormatVersion: '2010-09-09'
Transform: 'AWS::Serverless-2016-10-31'   # ← makes this a SAM template
Description: My serverless API

# ② Standard CFN sections all work — Parameters, Mappings, Conditions, Outputs
Parameters:
  Stage:
    Type: String
    Default: prod
    AllowedValues: [dev, staging, prod]

# ③ Global defaults applied to all serverless resources
Globals:
  Function:
    Runtime: python3.12
    Timeout: 30
    MemorySize: 256
    Environment:
      Variables:
        STAGE: !Ref Stage
    Tracing: Active               # X-Ray on all functions
  Api:
    Cors: "'*'"
    TracingEnabled: true

# ④ Resources — SAM + standard CFN resources can co-exist
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: app.handler
      CodeUri: ./src/
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /items
            Method: GET

# ⑤ Standard CFN Outputs
Outputs:
  ApiUrl:
    Value: !Sub 'https://${ServerlessRestApi}.execute-api.${AWS::Region}.amazonaws.com/${Stage}/'
💡

Globals section is SAM-specific — it sets default property values for all Function, Api, HttpApi, and SimpleTable resources. Individual resource properties override globals. Reduces repetition across many functions.

04 Resource Types

All SAM Resource Types

SAM provides six resource types that expand into multiple CloudFormation resources. You can mix these with standard AWS::* CloudFormation resources in the same template.

AWS::Serverless::Function
Lambda function + IAM execution role + event source mappings + aliases + versions. The most commonly used SAM resource.
AWS::Serverless::Api
REST API Gateway (v1). Generates ApiGateway::RestApi + Deployment + Stage. Supports OpenAPI inline or S3.
AWS::Serverless::HttpApi
HTTP API Gateway (v2). Cheaper, faster alternative to REST API. Supports JWT authorizers, CORS config.
AWS::Serverless::SimpleTable
DynamoDB table with a single primary key. Expands to AWS::DynamoDB::Table. Good for simple use cases.
AWS::Serverless::LayerVersion
Lambda Layer containing shared code or dependencies. SAM handles building and publishing the layer version.
AWS::Serverless::StateMachine
Step Functions state machine. Supports event triggers, IAM policies, and definition inline or via file.
⚠️

SAM resources expand into multiple CFN resources. A single AWS::Serverless::Function generates: AWS::Lambda::Function + AWS::IAM::Role + any AWS::Lambda::Permission entries for each event. This is why CAPABILITY_IAM is always required when deploying SAM templates.

05 Resource Types

AWS::Serverless::Function

function-complete.yaml — full feature showcase
Resources:
  MyApiFunction:
    Type: AWS::Serverless::Function
    Properties:
      # Code location
      Handler: src/app.handler        # file.exportedFunction
      Runtime: python3.12
      CodeUri: ./src/                 # local path or S3 URI
      Architectures: [arm64]         # arm64 is cheaper than x86_64

      # Performance
      MemorySize: 512
      Timeout: 30
      EphemeralStorage:
        Size: 1024                    # /tmp storage in MB (512-10240)

      # Environment & config
      Environment:
        Variables:
          TABLE_NAME: !Ref MyTable
          SECRET_ARN: '{{resolve:secretsmanager:MySecret}}'

      # Networking (for VPC access)
      VpcConfig:
        SubnetIds: [!Ref PrivateSubnet1]
        SecurityGroupIds: [!Ref LambdaSG]

      # IAM — use managed policy templates OR custom policies
      Policies:
        - AmazonDynamoDBFullAccess      # managed policy ARN or name
        - DynamoDBCrudPolicy:           # SAM policy template (shorthand)
            TableName: !Ref MyTable
        - Statement:                    # inline policy
            - Effect: Allow
              Action: ssm:GetParameter
              Resource: '*'

      # Observability
      Tracing: Active                  # X-Ray tracing
      Layers:
        - !Ref MyLayer                  # reference a Lambda Layer

      # Versioning & Aliases (required for DeploymentPreference)
      AutoPublishAlias: live           # auto-creates alias + publishes version on each deploy

      # Event sources — SAM creates all trigger permissions automatically
      Events:
        RestApiEvent:
          Type: Api
          Properties:
            Path: /items/{id}
            Method: GET
        ScheduledEvent:
          Type: Schedule
          Properties:
            Schedule: rate(5 minutes)
        SQSEvent:
          Type: SQS
          Properties:
            Queue: !GetAtt MyQueue.Arn
            BatchSize: 10
        S3Event:
          Type: S3
          Properties:
            Bucket: !Ref MyBucket
            Events: s3:ObjectCreated:*
        DynamoDBStream:
          Type: DynamoDB
          Properties:
            Stream: !GetAtt MyTable.StreamArn
            StartingPosition: TRIM_HORIZON

06 Resource Types

AWS::Serverless::Api & HttpApi

REST API (v1)
MyRestApi:
  Type: AWS::Serverless::Api
  Properties:
    StageName: prod
    TracingEnabled: true
    AccessLogSetting:
      DestinationArn: !GetAtt LogGroup.Arn
    MethodSettings:
      - ResourcePath: "/*"
        HttpMethod: "*"
        ThrottlingRateLimit: 1000
        ThrottlingBurstLimit: 500
    Auth:
      DefaultAuthorizer: MyCognitoAuth
      Authorizers:
        MyCognitoAuth:
          UserPoolArn: !GetAtt UserPool.Arn
    # Use OpenAPI definition from file
    DefinitionUri: ./openapi.yaml
HTTP API (v2) — cheaper & faster
MyHttpApi:
  Type: AWS::Serverless::HttpApi
  Properties:
    StageName: prod
    CorsConfiguration:
      AllowOrigins:
        - "https://example.com"
      AllowMethods:
        - GET
        - POST
    Auth:
      DefaultAuthorizer: JWTAuth
      Authorizers:
        JWTAuth:
          JwtConfiguration:
            issuer: "https://cognito-idp..."
            audience: [my-client-id]
          IdentitySource: "$request.header.Authorization"
ℹ️

Implicit vs Explicit API: If you define an Events block with Type: Api and don't define an explicit AWS::Serverless::Api, SAM creates an implicit API called ServerlessRestApi. Use explicit API definitions when you need custom domain names, throttling, authorizers, or access logging.

SimpleTable, LayerVersion & StateMachine

all-other-resources.yaml
Resources:

  # SimpleTable — DynamoDB with a single primary key
  MyTable:
    Type: AWS::Serverless::SimpleTable
    Properties:
      PrimaryKey:
        Name: pk
        Type: String
      ProvisionedThroughput:
        ReadCapacityUnits:  5
        WriteCapacityUnits: 5
      SSESpecification:
        SSEEnabled: true

  # LayerVersion — shared dependencies / code
  MyDependencyLayer:
    Type: AWS::Serverless::LayerVersion
    Properties:
      LayerName: my-dependencies
      ContentUri: ./layer/           # SAM builds this directory
      CompatibleRuntimes: [python3.12, python3.11]
      RetentionPolicy: Retain        # keep old versions when updating
    Metadata:
      BuildMethod: python3.12       # sam build uses this

  # StateMachine — Step Functions with event trigger
  MyWorkflow:
    Type: AWS::Serverless::StateMachine
    Properties:
      DefinitionUri: ./statemachine/workflow.asl.json
      DefinitionSubstitutions:
        ProcessFunctionArn: !GetAtt ProcessFunction.Arn
      Policies:
        - LambdaInvokePolicy:
            FunctionName: !Ref ProcessFunction
      Events:
        HourlyTrigger:
          Type: Schedule
          Properties:
            Schedule: rate(1 hour)
💡

SimpleTable limitation: AWS::Serverless::SimpleTable only supports a single primary key (no sort key, no LSIs, no GSIs). For anything complex, use a full AWS::DynamoDB::Table resource alongside your SAM template.

09 CLI & Deploy

SAM Accelerate (sam sync)

SAM Accelerate speeds up the development feedback loop by bypassing CloudFormation for code-only changes. Instead of the full CFN update cycle (minutes), code changes deploy in seconds by updating Lambda directly.

CommandWhat SyncsUses CloudFormation?Speed
sam sync Both code AND infrastructure changes Yes — full CFN update Slower (~minutes)
sam sync --code Code changes only (Lambda function code) No — bypasses CFN entirely ⚡ Seconds
sam sync --code --resource AWS::Serverless::Function All Lambda functions and their dependencies No ⚡ Seconds
sam sync --code --resource-id HelloWorldFunction Only the specific named function No ⚡ Fastest
sam sync --watch Detects file changes automatically and syncs Config changes → Yes; Code changes → No Auto-chooses
⚠️

sam sync --code bypasses CloudFormation. This means it does NOT create a ChangeSet or go through the standard CFN pipeline. It's only safe for development — never use --code in production pipelines since infrastructure drift can occur.

💡

sam sync --watch smart detection: It monitors your project for file changes. If only Lambda function code changed → uses sam sync --code (seconds). If template.yaml or other config changed → uses full sam sync (minutes). Best for inner dev loop.

Local Testing

SAM CLI uses AWS Lambda Runtime Interface Emulator (RIE) via Docker to run Lambda functions locally. Requires Docker to be running on your machine.

local-testing-examples.sh
# ── Invoke a function with a local event file ──
sam local invoke MyFunction --event events/api-event.json
sam local invoke MyFunction --event events/sqs-event.json --env-vars env.json

# ── Generate a sample event for testing ──
sam local generate-event apigateway aws-proxy > events/api-event.json
sam local generate-event s3 put > events/s3-put.json
sam local generate-event sqs receive-message > events/sqs.json
sam local generate-event sns notification > events/sns.json

# ── Start local REST API on port 3000 ──
sam local start-api
sam local start-api --port 8080 --env-vars env.json --warm-containers EAGER

# ── Start Lambda endpoint (for SDK testing) ──
sam local start-lambda
# Then invoke via SDK:
# aws lambda invoke --function-name MyFunction --endpoint-url http://localhost:3001 out.json

# ── Run unit tests with DynamoDB Local ──
docker run -d -p 8000:8000 amazon/dynamodb-local
AWS_SAM_LOCAL=true pytest tests/

What Can Be Tested Locally

✅ Lambda Functions

Full Lambda runtime emulation. Same execution context as AWS. Environment variables, layers, timeouts.

✅ API Gateway

Local HTTP server that routes requests to Lambda functions. Supports path params, query strings, headers.

✅ DynamoDB

Use DynamoDB Local (Docker) or DynamoDB Local CLI with SAM. Point your app at localhost:8000.

⚠️

Local limitations: SAM local does NOT emulate SQS, SNS, EventBridge, or Step Functions locally. For those, you need mocking libraries (moto for Python, aws-sdk-mock for Node) or integration tests against real AWS services in a dev account.

11 Advanced

SAM + CodeDeploy Integration

When you specify DeploymentPreference on a AWS::Serverless::Function, SAM automatically creates a CodeDeploy Application, Deployment Group, and CloudWatch Alarms to manage Lambda traffic shifting — routing traffic from the old function version to the new one gradually.

💡

How it works: SAM sets AutoPublishAlias which creates a Lambda Alias (e.g., live). On each deploy, a new version is published. CodeDeploy gradually shifts traffic from the old version to the new one. If CloudWatch alarms trigger, CodeDeploy automatically rolls back to 100% old version.

Deployment Preference Types

Canary Canary10Percent5Minutes / Canary10Percent30Minutes
Send 10% of traffic to new version, wait N minutes monitoring alarms, then shift remaining 90% at once. Best for validation with real traffic before full cutover.
Linear Linear10PercentEvery1Minute / Linear10PercentEvery10Minutes
Shift traffic in equal increments every N minutes (10% every minute = fully shifted in 10 minutes). More gradual than Canary. More monitoring points to catch problems early.
AllAtOnce AllAtOnce
Shifts 100% of traffic immediately to new version. No gradual rollout. Still uses CodeDeploy pipeline — alarms can trigger rollback. Fastest, highest risk.
codedeploy-integration.yaml — full DeploymentPreference example
Resources:
  MyLambdaFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: index.handler
      Runtime: nodejs20.x
      CodeUri: s3://my-bucket/code.zip

      # Required: creates versioning + alias for traffic shifting
      AutoPublishAlias: live

      DeploymentPreference:
        Type: Canary10Percent10Minutes   # Canary | Linear | AllAtOnce

        # CloudWatch alarms — trigger automatic rollback if breached
        Alarms:
          - !Ref AliasErrorRateAlarm
          - !Ref LatencyAlarm

        # Hooks — Lambda functions run before/after traffic shift
        Hooks:
          PreTraffic:  !Ref PreTrafficValidatorFn   # smoke test before shift
          PostTraffic: !Ref PostTrafficValidatorFn  # integration test after shift

        # Enable/disable without removing config
        Enabled: true

        # Deploy to specific CodeDeploy deployment group
        DeploymentGroupName: MyDeploymentGroup

  # Alarm that watches the function alias error rate
  AliasErrorRateAlarm:
    Type: AWS::CloudWatch::Alarm
    Properties:
      AlarmDescription: Lambda alias errors
      Namespace: AWS/Lambda
      MetricName: Errors
      Dimensions:
        - Name: FunctionName
          Value: !Ref MyLambdaFunction
        - Name: Resource
          Value: !Sub '${MyLambdaFunction}:live'   # watch the alias
      Statistic: Sum
      Period: 60
      EvaluationPeriods: 1
      Threshold: 0
      ComparisonOperator: GreaterThanThreshold

  # Pre-traffic hook — must signal CodeDeploy with success/failure
  PreTrafficValidatorFn:
    Type: AWS::Serverless::Function
    Properties:
      Handler: pre_traffic.handler
      Policies:
        - Version: '2012-10-17'
          Statement:
            - Effect: Allow
              Action: codedeploy:PutLifecycleEventHookExecutionStatus
              Resource: '*'

Traffic Hook Lambda Pattern

pre_traffic.py — must call PutLifecycleEventHookExecutionStatus
# Pre-traffic hook — runs BEFORE any traffic shifts to new version
import boto3

def handler(event, context):
    codedeploy = boto3.client('codedeploy')

    # Run your smoke tests / health checks against the new function version
    deployment_id        = event['DeploymentId']
    lifecycle_event_hook = event['LifecycleEventHookExecutionId']

    try:
        # Invoke new function version directly (not the alias)
        result = run_smoke_tests()
        status = 'Succeeded' if result else 'Failed'
    except Exception:
        status = 'Failed'

    # MUST call this — CodeDeploy waits for it
    codedeploy.put_lifecycle_event_hook_execution_status(
        deploymentId=deployment_id,
        lifecycleEventHookExecutionId=lifecycle_event_hook,
        status=status    # 'Succeeded' → continue; 'Failed' → rollback
    )

13 Advanced

Serverless Application Repository (SAR)

The AWS Serverless Application Repository (SAR) is a managed repository for serverless applications. You can publish SAM applications to SAR to share them within your organization or publicly, and consume pre-built applications from the public catalog.

Publishing to SAR

  • Package your SAM application: sam package
  • Add Metadata block to template.yaml with AWS::ServerlessRepo::Application
  • Publish: sam publish
  • Set as public (anyone) or private (your org / specific accounts)
  • Versioning with semantic versioning support

Consuming from SAR

  • Deploy directly from AWS Console → Serverless Application Repository
  • Reference in CloudFormation/SAM via AWS::Serverless::Application
  • Common use cases: authorizers, data transformers, monitoring components
  • Requires CAPABILITY_AUTO_EXPAND when deploying nested SAR apps
template.yaml — publishing metadata + consuming a SAR app
# ── Publishing: add metadata to your template ──
Metadata:
  AWS::ServerlessRepo::Application:
    Name: my-image-resizer
    Description: Automatically resizes images uploaded to S3
    Author: My Company
    SpdxLicenseId: Apache-2.0
    LicenseUrl: LICENSE.txt
    ReadmeUrl: README.md
    Labels: [image, s3, processing]
    HomePageUrl: https://github.com/myco/my-image-resizer
    SemanticVersion: 1.2.0
    SourceCodeUrl: https://github.com/myco/my-image-resizer

---
# ── Consuming: reference a SAR application as a nested app ──
Resources:
  NestedApp:
    Type: AWS::Serverless::Application
    Properties:
      Location:
        ApplicationId: arn:aws:serverlessrepo:us-east-1:123456789012:applications/my-image-resizer
        SemanticVersion: 1.2.0
      Parameters:
        BucketName: !Ref MyBucket
14 Exam Focus

DOP-C02 Exam Tips

Must-Know Facts for the Exam
  • SAM = CloudFormation with serverless syntactic sugar. The Transform: 'AWS::Serverless-2016-10-31' line makes it SAM. Deploying requires CAPABILITY_IAM + CAPABILITY_AUTO_EXPAND.
  • AutoPublishAlias is required for DeploymentPreference (CodeDeploy traffic shifting). It creates a Lambda Alias and automatically publishes a new version on each deploy.
  • Three DeploymentPreference types: Canary (10% then rest), Linear (gradual equal increments), AllAtOnce (instant shift). All support Alarms for auto-rollback.
  • Pre/Post traffic hooks are Lambda functions that run before/after the traffic shift. They MUST call codedeploy:PutLifecycleEventHookExecutionStatus with Succeeded or Failed.
  • sam sync --code bypasses CloudFormation. For development only. sam sync --watch automatically uses --code for code-only changes.
  • SAM local testing requires Docker. Supports Lambda, API Gateway (REST), and DynamoDB Local. Does NOT emulate SQS, SNS, or EventBridge locally.
  • Globals section sets default values for all functions/APIs. Individual resources override globals.
  • SimpleTable only supports a single primary key. No sort key, no GSIs. Use raw AWS::DynamoDB::Table for anything complex.
  • SAM Policy Templates (DynamoDBCrudPolicy, S3CrudPolicy, etc.) expand to full IAM policies. Exam may ask about least-privilege Lambda permissions.
  • SAR = AWS Serverless Application Repository. Publish with sam publish. Consume as AWS::Serverless::Application. Nested SAR apps need CAPABILITY_AUTO_EXPAND.

Common Exam Scenarios

ScenarioAnswer
"Deploy a Lambda update with 10% traffic routed to new version for 10 minutes, then all traffic shifts" SAM + CodeDeploy with DeploymentPreference: Type: Canary10Percent10Minutes and AutoPublishAlias: live
"Automatically roll back a Lambda deployment if error rate exceeds 1%" Add a CloudWatch::Alarm watching the alias error metric and reference it in DeploymentPreference.Alarms
"Run integration tests against the new Lambda version before any production traffic is shifted" Use DeploymentPreference.Hooks.PreTraffic pointing to a validation Lambda function
"Developer wants to test code changes without waiting for CloudFormation" sam sync --code (bypasses CFN) or sam sync --watch (auto-detects code vs config changes)
"Test Lambda function and API Gateway locally before deploying to AWS" sam local invoke for functions, sam local start-api for API Gateway emulation (requires Docker)
"What capabilities are needed to deploy a SAM template?" CAPABILITY_IAM (SAM creates IAM roles) + CAPABILITY_AUTO_EXPAND (SAM uses a Transform macro)
"Share a serverless application across multiple AWS accounts in the organization" Publish to AWS Serverless Application Repository (SAR) as a private app with org-level permissions
"SAM template needs to grant Lambda write access to a DynamoDB table with least privilege" Use SAM Policy Template DynamoDBCrudPolicy with the TableName parameter instead of AmazonDynamoDBFullAccess

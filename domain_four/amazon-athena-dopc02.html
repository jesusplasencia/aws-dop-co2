★ Columnar formats — strongly preferred for cost and performance

⚡ EXAM TIP If a question asks how to reduce Athena query cost → answer is Parquet or ORC (columnar). Convert with AWS Glue.

The 4 Levers — memorize these
TechniqueWhat to DoWhy it Helps
Columnar Format Convert data to Parquet or ORC using AWS Glue Only relevant columns are read → massive scan reduction
Compression bzip2, gzip, lz4, snappy, zlip, zstd Smaller files = less data transferred and scanned
Partitioning Organize S3 data in Hive-style partition paths Athena prunes partitions — only scans matching folders
Larger Files Keep files > 128 MB Reduces per-file overhead; avoids the "small files problem"
Partitioning — S3 Path Structure

Hive-compatible paths allow Athena to skip irrelevant partitions entirely:

s3://yourBucket/pathToTable
  /<PARTITION_COL>=<VALUE>
    /<PARTITION_COL>=<VALUE>
      /<PARTITION_COL>=<VALUE>
        /data.parquet

# Real example:
s3://athena-examples/flight/parquet/year=1991/month=1/day=1/
⚡ EXAM TIP Query on a partitioned column (e.g. WHERE year=2023 AND month=6) and Athena only scans that partition. This is the most impactful cost/performance optimization for large datasets.
Workgroups
  • Isolate query execution and costs per team, application, or environment
  • Set data usage controls — max data scanned per query or per workgroup
  • Separate query result locations in S3 per workgroup
  • Integrates with IAM, CloudWatch metrics, and SNS alerts on data usage limits
⚡ EXAM TIP Workgroups enforce cost controls at the query level. A common scenario: limit a workgroup to X GB per query to prevent runaway scans.
FFederated Query
What it is
  • Extend Athena SQL queries to data sources beyond S3 — relational, non-relational, and custom
  • Uses Data Source Connectors that run as AWS Lambda functions
  • Results can be stored back to S3 for further analysis
  • Query across multiple data sources in a single SQL statement
Supported Data Sources via Connectors
CloudWatch Logs DynamoDB RDS / Aurora Redshift ElastiCache DocumentDB HBase on EMR SQL Server MySQL / PostgreSQL On-Premises DBs Custom (any JDBC)
Athena SQL → Lambda Connector → Data Source → Results → S3
⚡ EXAM TIP Federated Query scenario: "query DynamoDB AND S3 in a single SQL query without moving data" → answer is Athena Federated Query with Lambda Data Source Connector.
↔Key Integrations
AWS Glue
  • Glue Data Catalog = central metadata store (databases, tables, schemas)
  • Glue Crawlers auto-discover S3 data and populate the catalog
  • Use Glue ETL jobs to convert CSV/JSON → Parquet/ORC before querying
  • Athena uses Glue catalog by default — no separate schema setup needed
⚡ EXAM TIP Athena + Glue Crawler + S3 = a fully serverless data lake query stack. This architecture appears frequently in DOP-C02 scenarios.
Amazon QuickSight
  • Native BI / visualization layer on top of Athena
  • QuickSight → Athena → S3 = serverless reporting pipeline
  • No data movement required — QuickSight queries Athena directly
  • Supports SPICE (in-memory cache) to reduce repeated Athena costs

🔒Security
Access Control
  • IAM policies control who can run queries and access workgroups
  • S3 bucket policies / ACLs control access to the underlying data
  • Workgroup-level permissions — restrict workgroup access per team
  • Supports Lake Formation for fine-grained column/row-level security on Glue catalog tables
Encryption
  • Query results in S3 can be encrypted with SSE-S3, SSE-KMS, or CSE-KMS
  • Data at rest in S3 follows the bucket's encryption settings
  • In-transit encryption: TLS between Athena and S3
  • Cross-account S3 access supported with proper bucket policies + IAM

What Athena is NOT
  • Not a database — it does not store data; S3 is the store
  • Not for real-time streaming queries — use Kinesis Data Analytics for that
  • Not a replacement for Redshift on large-scale petabyte analytics with complex joins
  • Not suited for frequent small queries — cost adds up vs. a cached DB
⚠ WATCH OUT Athena charges per TB scanned — a single SELECT * on an unpartitioned, non-columnar dataset can be expensive. Always recommend partitioning + Parquet when cost is mentioned.

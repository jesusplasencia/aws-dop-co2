MCloudWatch Metrics
Core Concepts
MetricA variable to monitor (e.g. CPUUtilization, NetworkIn). Belongs to a namespace.
NamespaceContainer that isolates metrics by service (e.g. AWS/EC2).
DimensionKey/value attribute used to filter/segment metrics (e.g. InstanceId, Environment). Max 30 per metric.
Custom Metrics
  • Publish with API call PutMetricData
  • Use cases: RAM usage, disk space, app-specific counters
  • Supports up to 30 dimensions per metric
  • Accepts data points 2 weeks in the past and 2 hours in the future — ensure EC2 time is correct
⚡ EXAM TIP RAM and disk metrics are NOT collected by default — you must publish them via a Custom Metric or the Unified Agent.

Metric Streams

  • Destinations: Kinesis Data Firehose Datadog Splunk New Relic

  • Option to filter by namespace — stream only the metrics you need

Logs Insights
  • Interactive query language (similar to SQL) against log groups
  • Query across multiple log groups including different AWS accounts
  • Supports time range filtering, aggregations, sorting, limiting
  • Results can be exported or pinned to a CloudWatch Dashboard
  • Not real-time — designed for historical/analytical queries
Metric Filters
  • Apply filter patterns to log data to extract metric values
  • Examples: count ERROR occurrences, extract response time
  • Max 3 dimensions per filter
  • Can trigger CloudWatch Alarms
  • Filters only apply to new log events (not retroactive)
EC2 + CW Agent → CW Logs → Metric Filter → CW Alarm → SNS
Logs Aggregation (Multi-Account / Multi-Region)
  • Use Subscription Filters in each account/region pointing to a central Kinesis Data Streams
  • Pattern: Account A (Region 1) + Account B (Region 2) + Account B (Region 3) → Kinesis Data Streams → Kinesis Data Firehose → S3 (near real-time)
  • Enables centralized log aggregation architecture across an organization
⚡ EXAM TIP Cross-account log aggregation uses Subscription Filters with cross-account IAM permissions — a common DevOps architecture scenario.

ACloudWatch Agent
⚡ KEY CONCEPT By default, no logs and no OS-level metrics leave an EC2 instance. You must install and configure the CW Agent. Requires appropriate IAM role/permissions.
CloudWatch Logs Agent OLDER
  • Only sends logs to CloudWatch Logs
  • No system-level metrics collection
  • Limited configuration options
CloudWatch Unified Agent PREFERRED
  • Collects logs AND system-level metrics
  • Additional metrics: CPU (per-core), RAM, disk I/O, net stats, processes, swap
  • Centralized config via SSM Parameter Store
  • Works on EC2 and on-premises servers
  • More granularity, same agent for logs + metrics
Unified Agent — Metrics Collected
CPU (per core) Disk I/O Disk Usage RAM Net stats Processes Swap space
⚡ EXAM TIP RAM, swap, and per-process metrics are NOT in standard EC2 CloudWatch metrics. Any question about monitoring memory on EC2 → answer is CloudWatch Unified Agent + Custom Metric.
!CloudWatch Alarms
Alarm States
OK INSUFFICIENT_DATA ALARM

INSUFFICIENT_DATA = not enough data to evaluate the condition (e.g. metric just created or EC2 stopped)

Period
  • Time window (in seconds) over which the metric is evaluated
  • Supported intervals: 10s 30s 60s (multiples of 60 for standard)
  • High-resolution alarms require high-resolution custom metrics
Alarm Targets (Actions)
  • EC2 Actions: Stop, Terminate, Reboot, Recover instance
  • Auto Scaling: Scale out / Scale in (ASG)
  • SNS notification → trigger further automation
  • Alarms can be created from Metric Filters
Composite Alarms
  • Combine multiple alarms using AND / OR logic
  • Reduces alarm noise — only alert when multiple conditions are met
  • Example: ALARM only if CPU > 80% AND IOPS > threshold
⚡ EXAM TIP Composite Alarms are key for avoiding alarm storms. If the question is about reducing unnecessary alert notifications, the answer involves Composite Alarms with AND logic.
Testing Alarms via CLI
aws cloudwatch set-alarm-state \
  --alarm-name "myAlarm" \
  --state-value ALARM \
  --state-reason "testing purposes"

Useful for testing alarm actions in CI/CD pipelines without waiting for actual threshold breaches.

~Anomaly Detection
  • Uses ML algorithms to continuously analyze metrics and establish a normal baseline band
  • Creates a model based on metric's historical data (patterns, seasonality, trends)
  • Highlights data points that fall outside the normal range on graphs
  • Allows creating alarms based on expected value instead of a hard static threshold
  • Can exclude specific time windows from training (maintenance periods, deployments)
⚡ EXAM TIP When a question mentions dynamic thresholds, variable workloads, or avoiding false alarms from predictable load patterns → answer is CloudWatch Anomaly Detection.
SCloudWatch Synthetics Canary
What it is
  • Configurable scripts that monitor APIs, URLs, and websites from the outside, simulating real user behavior
  • Detects issues before customers are impacted
  • Checks availability and latency of endpoints; stores screenshots and load time data
  • Integrates with CloudWatch Alarms → trigger Route 53 failover automatically
  • Scripts written in Node.js or Python; uses headless Chromium browser
  • Runs once or on a recurring schedule
Built-in Blueprints
BlueprintPurpose
Heartbeat MonitorLoads a URL, stores a screenshot and HTTP archive file
API CanaryTests basic read and write operations of REST APIs
Broken Link CheckerChecks all hyperlinks within a target URL
Visual MonitoringCompares current screenshot vs a baseline — detects visual regressions
Canary RecorderRecords your actions on a site and auto-generates a canary script
GUI Workflow BuilderVerifies multi-step actions work (e.g. login form → dashboard)
⚡ EXAM TIP Synthetics Canary + CloudWatch Alarm + Route 53 Health Check is a common automated failover pattern in DOP-C02. The canary detects the failure → alarm fires → Route 53 DNS failover to a secondary region.

AWS QuickSight

Serverless BI · ML Insights · Interactive Dashboards · Per-Session Pricing

📊
What is AWS QuickSight?
A serverless, ML-powered Business Intelligence (BI) service that lets you create interactive dashboards, perform ad-hoc analysis, and share insights — without managing any infrastructure. Scales automatically, charges per-session, and can be embedded in applications.
  • Not a DevOps-core service — tested in DOP-C02 primarily in the context of monitoring dashboards, cost visualization, and operational analytics fed by AWS data sources
  • Fully managed — no servers, no clusters, no patching; scales to tens of thousands of users automatically
  • ML-powered — built-in anomaly detection, forecasting, and natural language queries (Q feature)
  • Embeddable — dashboards can be embedded in custom applications via the GetEmbedUrl API

SPICE — In-Memory Engine
SPICE = Super-fast, Parallel, In-memory Calculation Engine. QuickSight's proprietary in-memory data store that dramatically accelerates query performance.
  • Data is imported into SPICE from a source (S3, RDS, Redshift, etc.) and cached — queries run against SPICE, not the source database
  • Reduces load on underlying data sources and speeds up dashboard rendering
  • Auto-scales to handle concurrent readers without performance degradation
  • Refresh schedule — SPICE datasets can be refreshed on a schedule (hourly, daily, weekly) or via the CreateIngestion API call
  • 10 GB per user included in Standard Edition; Enterprise adds more capacity
  • Direct Query mode (no SPICE) is also supported — queries go live to the source each time; slower but always fresh
Exam trap: If the scenario says "near-real-time data" → use Direct Query mode or schedule frequent SPICE refreshes. SPICE alone is not real-time.

Pricing Concept
  • Authors — create/edit dashboards; pay per-user/month (higher rate)
  • Readers — view dashboards only; pay per-session (capped at ~$5/month max per reader)
  • Per-session pricing makes QuickSight very cost-efficient for large audiences with occasional access

Security & Access Control
Authentication
  • IAM users — federated access via IAM roles
  • QuickSight-managed users — email/password accounts
  • Active Directory — Enterprise only, via AWS Directory Service
  • SAML 2.0 — SSO federation (Enterprise)
Data Security
  • Row-Level Security (RLS) — restrict rows per user/group via a dataset rule (Enterprise)
  • Column-Level Security (CLS) — restrict specific columns (Enterprise)
  • VPC connection — connect to private RDS/Redshift without public exposure (Enterprise)
  • Encryption — TLS in transit; SPICE encrypted at rest (Enterprise)
Exam pattern — RLS: Scenario says "users should only see data relevant to their department/region" → answer is QuickSight Row-Level Security, requires Enterprise Edition.

⭐ Exam Focus — DOP-C02 Scenario Patterns
  • "Visualize AWS Cost & Usage data by team/project" → Export CUR (Cost & Usage Report) to S3 → Athena → QuickSight
  • "Business stakeholders need dashboards on S3 data without writing SQL" → S3 → Athena → QuickSight (with SPICE for performance)
  • "Each sales rep should only see their own region's data" → QuickSight Enterprise + Row-Level Security (RLS)
  • "Dashboard is slow because it queries Redshift on every load" → Enable SPICE import to cache the dataset
  • "Detect anomalies in deployment metrics automatically" → QuickSight ML Insights / Anomaly Detection (no SageMaker needed)
  • "Embed dashboards in an internal portal for 10,000 employees" → QuickSight Enterprise + GetEmbedUrl API + per-session reader pricing
  • "Connect QuickSight to private RDS inside a VPC" → Enterprise Edition + VPC connection (private subnet)
  • QuickSight ≠ real-time — for real-time operational metrics use CloudWatch or Grafana; QuickSight is for BI/reporting workloads

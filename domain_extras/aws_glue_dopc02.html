What is AWS Glue?
A fully serverless, managed ETL (Extract, Transform, Load) service that prepares and transforms data for analytics. No infrastructure to provision, patch, or manage.
  • Core job: move data from sources → transform/enrich it → load into a target (Redshift, S3, RDS, etc.)
  • Scripted in Python or Scala — Glue auto-generates ETL scripts which you can customize
  • Runs on Apache Spark under the hood (distributed processing, no cluster management)
  • Glue Data Catalog — central metadata repository; acts as a Hive-compatible metastore for Athena, Redshift Spectrum, and EMR
  • DOP-C02 weight: moderate — tested mainly through data pipeline architecture scenarios

  • Why Parquet? — columnar format; dramatically reduces Athena query cost (charged per TB scanned) and speeds up reads
  • Trigger options: S3 Event Notification → Lambda → start Glue Job, or EventBridge rule → Glue Job directly
  • Glue Crawler runs after conversion to update the Data Catalog with the new Parquet schema and partitions
  • Partitioning — Glue can write output partitioned by date/region (e.g., year=2024/month=01/) to further cut Athena scan costs
Parquet vs ORC: Both are columnar. Parquet is preferred for Athena/Spark; ORC is preferred for Hive/EMR. Either reduces cost vs CSV/JSON.

Glue Data Catalog — Central Metadata Store
The Glue Data Catalog is a persistent metadata repository containing table definitions, schema, partition info, and connection details. It is the single source of truth for data discovery across your AWS analytics stack.
Services that use it
  • Amazon Athena — queries tables defined in the catalog against S3
  • Amazon Redshift Spectrum — queries external S3 tables via catalog
  • Amazon EMR — uses catalog as Hive metastore
  • AWS Lake Formation — governs access to catalog resources
Glue Crawler details
  • Automatically infers schema and data types from raw files
  • Detects partitions from S3 folder structure
  • Handles schema evolution — adds new columns on next run
  • Supports incremental crawling to skip unchanged S3 objects
Exam note: When a scenario says "make S3 data queryable by Athena" → the answer involves Glue Crawler + Glue Data Catalog, not manual table creation.

Trigger Modes
  • Scheduled — cron expression (e.g., every 6 hours)
  • On-Demand — manually started via console, CLI, or API
  • Job Event Trigger — start Job B when Job A succeeds/fails; enables job chaining without Lambda
  • EventBridge — trigger a Glue job from any AWS event (S3 PUT, DynamoDB stream, etc.)

Glue Streaming ETL
Glue Streaming ETL consumes data from Kinesis Data Streams or Apache Kafka (MSK) and processes it in micro-batches — enabling near-real-time pipeline processing without a persistent cluster.

  • Runs continuously — the job does not terminate; processes new data as it arrives
  • Supports checkpointing to resume from last position on job restart
  • Can join streaming data against static reference tables in the Data Catalog
  • Billed per DPU-second while the job is running (continuous cost)
Exam pattern: "Near-real-time transformation of Kinesis stream data before loading to S3 or Redshift" → Glue Streaming ETL (not Lambda; Lambda has 15-min limit and no Spark scale).

Pricing Model & Key Limits
Pricing
  • DPU (Data Processing Unit) — Glue's compute unit; 1 DPU = 4 vCPU + 16 GB RAM
  • Billed per DPU-hour (rounded to the nearest second); minimum 1 minute
  • Crawlers billed per DPU-hour separately from ETL jobs
  • Glue Data Catalog: first 1M objects free; then per 100K objects/month
  • No charge when jobs are idle (serverless — only pay when running)

⭐ Exam Focus — DOP-C02 Scenario Patterns
  • "Convert CSV files in S3 to Parquet for cost-efficient Athena queries" → Glue ETL Job (CSV → Parquet) + Glue Crawler to update catalog
  • "Make S3 data discoverable/queryable without writing DDL manually" → Run a Glue Crawler; it populates the Data Catalog automatically
  • "ETL pipeline should start automatically when new data lands in S3" → S3 Event Notification → EventBridge → trigger Glue Job (or Lambda → StartJobRun API)
  • "Chain multiple ETL steps — crawler, then transform, then load" → Use Glue Workflow with job event triggers (no Lambda needed)
  • "Transform Kinesis stream data before storing in Redshift" → Glue Streaming ETL job (continuous Spark Structured Streaming)
  • "Glue vs EMR" → Glue for serverless simplicity; EMR for full Spark/Hadoop control, custom libraries, iterative ML
  • "Glue ETL script language" → Python (PySpark) or Scala — auto-generated but customizable
  • Glue Data Catalog = shared metastore → Athena, Redshift Spectrum, and EMR all read from it; one schema update propagates everywhere
  • "Schema validation on Kinesis stream" → Glue Schema Registry — define and enforce Avro/JSON schemas on producers/consumers

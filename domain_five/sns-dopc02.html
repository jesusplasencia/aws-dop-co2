Publish Methods

TOPIC PUBLISH (SDK)

  • Create a topic → create subscriptions → publish to topic ARN
  • All subscriptions receive the message (subject to filter policies)
  • Standard API: Publish with TopicArn

DIRECT PUBLISH (MOBILE SDK)

  • Create platform application (iOS/Android) → create platform endpoint → publish to endpoint ARN
  • Bypasses topic — sends directly to one mobile device
  • Platform support: Apple APNs, Google FCM/GCM, Amazon ADM
FSNS + SQS Fan-Out
Why Fan-Out?
  • Publish once to an SNS topic → message delivered to all subscribed SQS queues simultaneously
  • Fully decoupled — new consumers just subscribe; the producer doesn't change
  • Each SQS queue can have its own processing logic, scaling, and DLQ
  • Cross-region delivery: SNS can deliver to SQS queues in other AWS regions
  • SQS adds durability — messages are retained in queue even if consumer is down
⚠ REQUIRED SQS queue must have a resource-based policy allowing SNS to call sqs:SendMessage on the queue ARN. Without this, delivery silently fails.
Fan-Out Architecture Patterns

S3 EVENT → MULTIPLE QUEUES

  • S3 can only send one event notification per event type per bucket
  • Solution: S3 → SNS → multiple SQS queues (fan-out)
  • This is the canonical exam pattern for S3 multi-consumer scenarios

SNS → KINESIS DATA FIREHOSE → S3

  • SNS cannot write directly to S3
  • Route via Kinesis Data Firehose subscriber
  • Pattern: SNS → KDF → S3 / Redshift / OpenSearch

~SNS Message Filtering
  • Each subscription can have a filter policy (JSON) — SNS only delivers messages that match
  • Filter applies to message attributes (not the message body)
  • Subscription with no filter policy receives ALL messages
  • Supports: exact match, prefix, numeric ranges, anything-but, exists
  • Filter scope can be set to MessageBody (newer) or MessageAttributes (default)
// Subscription filter — only deliver "Placed" orders
{
  "State": ["Placed"]
}

// Only deliver orders where quantity > 10
{
  "Quantity": [{ "numeric": [">", 10] }]
}

// Anything BUT cancelled
{
  "State": [{ "anything-but": ["Cancelled"] }]
}
⚡ EXAM TIP Classic scenario: one SNS topic, multiple SQS subscribers each with a different filter policy — "Placed orders → Queue A", "Cancelled orders → Queue B + Email", "All orders → Queue C". This fan-out + filter pattern is heavily tested.
OSNS FIFO
FIFO Topic Characteristics
OrderingStrictly preserved per Message Group ID
DeliveryExactly-Once — deduplication built-in
Throughput300 msg/s (3,000 with batching) — same as SQS FIFO
Topic nameMust end in .fifo
SubscribersOnly SQS Standard or SQS FIFO queues — no Email, SMS, HTTP, Lambda
FIFO Deduplication
  • 5-minute deduplication window (same as SQS FIFO)
  • Content-based: SHA-256 hash of message body
  • Explicit: provide a MessageDeduplicationId
⚡ EXAM TIP SNS FIFO → SQS FIFO = ordered + exactly-once fan-out. Only SQS queues can subscribe to SNS FIFO — no Lambda, no HTTP. This subscriber restriction is a common distractor.

How SNS DLQ Differs from SQS DLQ

⚡ EXAM TIP SNS DLQ is at the subscription level — each subscriber can have its own DLQ. If HTTP endpoint fails delivery retries → messages go to that subscription's DLQ, not affecting other subscribers.
SNS Delivery Retry Policy
  • SNS retries failed deliveries with exponential backoff
  • HTTP/HTTPS: up to 3 immediate retries, then 20 retries over ~1 hour (configurable)
  • After retry policy is exhausted → message is discarded unless DLQ is configured
  • SQS/Lambda subscribers: failures handled by those services' own retry mechanisms

🔒Security
Encryption
  • In-flight: HTTPS API enforced
  • At-rest: SSE-KMS (AWS-managed or customer-managed key)
  • Client-side: encrypt payload before publishing
Access Control
  • IAM Policies — control which IAM principals can publish / subscribe
  • SNS Topic Policy (resource-based) — required for cross-account access and for AWS services to publish (e.g. S3, CloudWatch)
  • To allow S3 to publish SNS notifications → SNS topic policy must allow sns:Publish from s3.amazonaws.com with a condition on the source bucket ARN

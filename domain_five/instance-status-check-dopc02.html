✓The Three Check Types
System Status Check
AWS INFRASTRUCTURE LAYER
  • Monitors the physical host (AWS side)
  • Detects: hardware failure, loss of network connectivity, system power, software issues on the host
  • Problem is AWS's responsibility to fix
  • Check AWS Health Dashboard for scheduled host maintenance

RESOLUTION

Stop & Start the instance → migrates to a new physical host. (Reboot keeps same host — won't fix system issues)

Instance Status Check
GUEST OS / SOFTWARE LAYER
  • Monitors the software/network config inside the instance (your responsibility)
  • Detects: exhausted memory, corrupted file system, incompatible kernel, invalid network config, failed OS boot
  • Problem is the customer's responsibility

RESOLUTION

Reboot the instance or change instance config (security groups, instance type, user-data fix)

Attached EBS Status Check
STORAGE LAYER
  • Monitors EBS volumes attached to the instance
  • Detects: unreachable volume, I/O operations failing or stalling
  • Separate from instance health — an instance can be healthy while its EBS fails

RESOLUTION

Reboot the instance or replace the affected EBS volume

⚡ EXAM TIP — STOP vs REBOOT System check fail → Stop & Start (moves to new host). Instance check fail → Reboot (same host, restarts OS). Reboot does NOT fix system-level failures because the host doesn't change.

↻Recovery Options
Option 1 — CloudWatch Alarm Action: Recover
  • Alarm on StatusCheckFailed_System → action: EC2 Recover
  • Migrates instance to a new healthy physical host
  • Preserves: same private IP, public IP, EIP, instance ID, metadata, Placement Group
  • In-memory data (RAM) is lost
  • Pair with SNS to notify team on recovery
  • Only works for instances backed by EBS (not instance store)
CW Alarm
StatusCheckFailed_System → EC2 Recover + SNS notify
Option 2 — Auto Scaling Group (ASG)
  • Set min/max/desired = 1 — ASG replaces a terminated/failed instance automatically
  • Uses EC2 health checks or ELB health checks to detect unhealthy instances
  • Launches a brand new instance — does not preserve private IP, EIP, or instance ID
  • Better for stateless workloads; less suitable when IP/identity must be preserved
  • Can span multiple AZs for higher availability
ASG Health Check
EC2 or ELB → Terminate failed → Launch new

EC2 Recover — What is and isn't preserved

PRESERVED ✓

  • Instance ID
  • Private IPv4 address
  • Public IPv4 address (if assigned at launch)
  • Elastic IP (EIP)
  • IPv6 address
  • Placement Group membership
  • All instance metadata & tags
  • EBS volumes (reattached)

NOT PRESERVED ✗

  • RAM / in-memory data
  • Instance store volumes (ephemeral)
  • Physical host (migrates to a new one)

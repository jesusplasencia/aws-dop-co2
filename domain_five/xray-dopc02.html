XOverview
What it is
PurposeDistributed tracing — follow a request end-to-end across all microservices and AWS resources
LevelRequest level (vs CloudWatch = metric/log level)
OutputService map, trace timeline, latency histograms, error breakdown
RequirementApplication must be instrumented with the X-Ray SDK + X-Ray Daemon running
IAMInstance profile / execution role must include xray:PutTraceSegments and xray:PutTelemetryRecords
Problems X-Ray Solves
  • Find latency bottlenecks across distributed services
  • Identify which microservice is causing errors or throttling
  • Understand request flow across Lambda, API Gateway, EC2, ECS, DynamoDB, SQS, SNS
  • Debug "why is this request slow?" without reading individual logs
  • Visualize service dependencies via the Service Map
⚡ EXAM TIP "Debug latency / errors across microservices" or "trace a request end-to-end" → AWS X-Ray. CloudWatch shows metrics; X-Ray shows request flow.
CCore Concepts
Trace
End-to-end record of a single request across all services. Composed of one or more Segments. Identified by a unique Trace ID.
Segment
Work done by a single service/resource for one request. Contains: timing, HTTP data, errors, annotations, metadata. Sub-tasks = Subsegments.
Subsegment
Granular unit of work within a segment. Examples: outbound HTTP call, DB query, AWS SDK call. Required to trace downstream dependencies.
Annotations
Key-value pairs indexed for searching/filtering in the X-Ray console. Max 50 per trace. Use for: user ID, order ID, environment, feature flag.
Metadata
Key-value pairs attached to segments for debugging context — NOT indexed, not searchable. Use for rich objects, payloads, non-filterable data.
Service Map
Visual graph of all services involved in requests — shows avg latency, error rates, and request volume per node and edge. Auto-generated from trace data.
⚡ EXAM TIP — ANNOTATIONS vs METADATA Annotations = indexed, searchable (find all traces where user_id = "abc123"). Metadata = not indexed, stored only for context. If the question says "filter traces by a custom attribute" → Annotations. If it says "attach debug payload" → Metadata.
DX-Ray Daemon
How It Works
Application + X-Ray SDK → UDP :2000 X-Ray Daemon
(local agent) → HTTPS batch X-Ray Service
(AWS managed)
  • The SDK sends raw segment data via UDP to port 2000 on localhost — daemon collects and batches
  • Daemon sends batches to the X-Ray API over HTTPS (handles retries, buffering)
  • Must be running alongside your application — not built-in except where AWS manages it
  • Required IAM permissions on the instance/role: xray:PutTraceSegments, xray:PutTelemetryRecords
⚡ EXAM TIP "X-Ray not receiving traces from EC2" → check: (1) X-Ray Daemon is running, (2) EC2 Instance Profile has xray:PutTraceSegments permission, (3) SDK is instrumented in code. All three are needed.

↔Service Integrations
Lambda
  • Enable Active Tracing on Lambda function (console or TracingConfig: Active)
  • Lambda runs daemon — no setup needed
  • Execution role must have xray:PutTraceSegments
  • Traces Lambda invocations and downstream calls (DynamoDB, SNS, etc.)
API Gateway
  • Enable X-Ray tracing per stage (not per method)
  • Traces API Gateway + propagates trace header to Lambda/backend
  • Shows latency at API layer separately from backend latency
  • No daemon required — API GW is fully managed
Elastic Beanstalk
  • Enable via .ebextensions/xray-daemon.config
  • Or toggle in EB console under Software configuration
  • Instance profile must include X-Ray permissions
option_settings:
  aws:elasticbeanstalk:xray:
    XRayEnabled: true
ECS / Fargate
  • Add X-Ray daemon as a sidecar container in task definition
  • App container sends to localhost:2000 (UDP)
  • Task execution role must include X-Ray permissions
  • Use AWS_XRAY_DAEMON_ADDRESS env var if daemon is in a separate container port
%Sampling Rules
Why Sampling?
  • Tracing every request at high traffic = high cost and overhead
  • Sampling rules control what percentage of requests generate traces
  • Default rule: first request per second + 5% of additional requests
  • Custom rules can be set per service, URL path, HTTP method, or host
  • Rules are managed in the X-Ray console or API — no code deployment needed to change sampling
Sampling Rule Parameters
ParameterDescriptionDefault
ReservoirSizeFixed number of traces per second regardless of traffic rate1 req/sec
FixedRatePercentage of requests sampled beyond the reservoir5%
PriorityRule evaluation order — lower number = evaluated first—
Host / URL / MethodMatch criteria — scope the rule to specific traffic patternsWildcard (*)
⚡ EXAM TIP "Reduce X-Ray costs but still capture all errors" → create a custom sampling rule with higher rate for error responses (5xx) and lower rate for 2xx. Sampling rules are updated centrally without redeploying code.

X-Ray Groups
  • Filter expressions to group traces into named subsets (e.g. "errors", "slow requests")
  • Groups can have their own sampling rules
  • Used to create focused views in the Service Map and Traces console
  • Example group filter: responsetime > 5 AND http.status = 500
  • Can generate CloudWatch metrics per group (error rate, latency) → trigger CW alarms
⚡ EXAM TIP X-Ray Groups + CloudWatch Metrics = alarm on high error rates detected in traces. This bridges X-Ray observability into CloudWatch automation.
OAWS Distro for OpenTelemetry (ADOT)
  • AWS-supported distribution of the open-source OpenTelemetry project
  • Collects traces, metrics, and contextual data from applications
  • Vendor-agnostic — send data to multiple destinations simultaneously
  • Destinations: X-Ray, CloudWatch, Amazon Managed Service for Prometheus, partner monitoring tools (Datadog, New Relic, etc.)
  • Supports same instrumentation libraries as X-Ray SDK (auto-instrumentation for Java, Python, Node.js, .NET)
Application (instrumented) → ADOT Collector → X-Ray + CloudWatch + Prometheus / Partners
⚡ EXAM TIP "Send traces to X-Ray AND a third-party monitoring tool simultaneously" → migrate from X-Ray SDK to AWS Distro for OpenTelemetry. X-Ray alone can only send to X-Ray.

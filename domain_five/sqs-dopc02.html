Delay Queue
  • Delay delivery of new messages — consumer doesn't see them immediately
  • Default: 0 seconds · Maximum: 15 minutes
  • Set at queue level (DelaySeconds on CreateQueue)
  • Override per message using DelaySeconds in SendMessage — message-level delay
  • Standard queues only — FIFO queues don't support per-message delay
⚡ EXAM TIP "Messages must not be processed until X minutes after being sent" → Delay Queue. Common use case: notify a customer 10 minutes after an order is placed.
Extended Client (Large Messages > 256 KB)
Producer → large payload (up to 2 GB) → S3 (stores payload) + SQS (stores S3 pointer metadata) → Consumer → retrieves from S3

Uses the SQS Extended Client Library (Java). The SQS message carries only the S3 object reference, not the actual payload.

VMessage Visibility Timeout
  • When a consumer polls a message, it becomes invisible to all other consumers for the duration of the visibility timeout
  • Default: 30 seconds · Configurable: 0 seconds → 12 hours
  • If not deleted within the timeout, the message becomes visible again and can be delivered to another consumer (duplicate processing risk)
  • Consumer can extend visibility using ChangeMessageVisibility API while still processing
  • If visibility timeout is too short → duplicate processing. If too long → slow re-processing on consumer crash.

⚡ EXAM TIP Consumer gets a message but needs more than 30 seconds → call ChangeMessageVisibility to extend. If it crashes without deleting, the message reappears after timeout — this is how SQS guarantees at-least-once delivery.
DDead Letter Queue (DLQ)
How It Works
  • Consumer fails → message returns to queue after visibility timeout
  • Set MaximumReceives threshold (e.g., 3) — if exceeded, message moves to DLQ
  • DLQ is a separate SQS queue — Standard DLQ for Standard, FIFO DLQ for FIFO
  • DLQ retention: up to 14 days — set high to allow time to debug
  • Useful for: isolating poison pill messages, debugging, alerting on failures
⚡ EXAM TIP DLQ type must match source queue: Standard → Standard DLQ, FIFO → FIFO DLQ. This is a common distractor.
DLQ Redrive to Source
  • After fixing the consumer bug, send failed messages back to the source queue
  • Uses Redrive Task — no custom code needed
  • Messages reprocessed in batches from DLQ → source queue
  • Can redrive to the original source or a custom destination queue
DLQ → (after fix) Redrive Task → Source Queue → Consumer (fixed)
⚡ EXAM TIP Set DLQ retention to 14 days — you want messages to survive long enough to debug and redrive. The default 4 days may be too short.
CloudWatch Alarm on DLQ — Operational Pattern
DLQ — ApproximateNumberOfMessages > 0 → CW Alarm → SNS → Ops Alert

Monitor ApproximateNumberOfMessagesNotVisible and ApproximateNumberOfMessages on the DLQ for alerting.

PShort vs Long Polling
Short Polling (default)
  • Returns immediately, even if queue is empty
  • May return empty responses → wasted API calls + cost
  • ReceiveMessageWaitTimeSeconds = 0
Long Polling PREFERRED
  • Consumer waits up to 20 seconds for messages to arrive
  • Reduces empty responses → fewer API calls → lower cost
  • Decreases latency of message processing
  • Set ReceiveMessageWaitTimeSeconds = 1–20 (queue level or per-call)
⚡ EXAM TIP "Too many empty SQS API responses / high cost from polling" → enable Long Polling (ReceiveMessageWaitTimeSeconds = 20). Always prefer Long Polling for consumers that don't need sub-second response.
FSQS FIFO
FIFO Fundamentals
OrderingStrict First-In-First-Out within a Message Group
DeliveryExactly-Once — deduplication built-in
Throughput300 msg/s without batching · 3,000 msg/s with batching (batch size 10)
Queue nameMust end in .fifo
In-flight limitMax 20,000 in-flight messages
Deduplication (5-minute window)
  • Duplicate messages sent within the 5-minute deduplication interval are discarded
  • Method 1 — Content-based: SQS hashes the message body with SHA-256; identical bodies = duplicate
  • Method 2 — Explicit ID: producer sets a MessageDeduplicationId; same ID within 5 min = duplicate
⚡ EXAM TIP Exactly-once = FIFO queue with deduplication enabled. Standard queues can produce duplicates — consumers must be idempotent.
Message Grouping (MessageGroupID)
  • MessageGroupID partitions messages within a FIFO queue into ordered subsets
  • All messages with the same GroupID → delivered in order → to one consumer
  • Different GroupIDs → parallel processing by different consumers simultaneously
  • Ordering is guaranteed within a group, not across groups
  • Single GroupID = one consumer at a time = no parallelism (like a standard FIFO)
⚡ EXAM TIP "Process orders for each customer in order, but process different customers in parallel" → FIFO queue with CustomerID as MessageGroupID. Each customer gets their own ordered lane.
🔒Security
Encryption
  • In-flight: HTTPS API — enforced by default
  • At-rest: SSE using AWS KMS (SSE-KMS) or SSE-SQS (AWS-managed key)
  • Client-side: encrypt/decrypt in the application before sending to SQS
Access Control
  • IAM Policies — control which IAM principals can call SQS APIs
  • SQS Resource-Based Policy — attached to the queue itself; required for cross-account and service-to-service access
  • Use resource policies to allow S3 → SQS (S3 event notifications), SNS → SQS, Lambda → SQS
SQS Access Policy Examples

CROSS-ACCOUNT ACCESS

// Allow Account B to poll from Account A's queue
{
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::ACCOUNT-B:root" },
  "Action": "sqs:ReceiveMessage",
  "Resource": "arn:aws:sqs:us-east-1:ACCOUNT-A:queue1"
}

S3 → SQS NOTIFICATIONS

// Allow S3 to send event notifications to SQS
{
  "Effect": "Allow",
  "Principal": { "Service": "s3.amazonaws.com" },
  "Action": "sqs:SendMessage",
  "Resource": "arn:aws:sqs:us-east-1:123456:queue1",
  "Condition": {
    "ArnLike": { "aws:SourceArn": "arn:aws:s3:::bucket1" }
  }
}
↑SQS + Auto Scaling (DOP-C02 Key Pattern)
Scale Consumers Based on Queue Depth
SQS Queue depth rising → CW Metric:
ApproximateNumberOfMessages → CW Alarm → ASG Scale Out
  • Use Target Tracking Scaling on ASG with the SQS metric ApproximateNumberOfMessagesNotVisible per instance
  • Formula: backlog per instance = queue depth / number of instances
  • Set target backlog per instance (e.g., 1000 messages/instance) — ASG adjusts count automatically
  • Works with Lambda too — Lambda auto-scales based on queue depth natively
⚡ EXAM TIP The most-tested SQS+DevOps pattern: SQS queue depth → CloudWatch alarm → ASG scale-out. Use ApproximateNumberOfMessages for the alarm metric. This decouples the frontend load from backend processing capacity.
SQS as a Buffer — Classic Architecture
Client requests (spiky) → SQS Queue (buffer) → EC2 / Lambda consumers (steady) → DB / downstream

SQS absorbs traffic spikes, preventing downstream overload. Consumers process at a sustainable rate. Combined with ASG: queue grows → more consumers launch → queue drains.

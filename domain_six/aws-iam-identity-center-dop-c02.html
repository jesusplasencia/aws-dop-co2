01

Core Concept

Exam One-liner

IAM Identity Center (successor to AWS SSO) provides centralized single sign-on across all AWS accounts in an Organization and SAML 2.0 applications — using one set of credentials managed from one place, with access controlled via Permission Sets.

What it covers (one login for)

  • All AWS accounts in your AWS Organization
  • SAML 2.0 apps (Salesforce, Box, Microsoft 365, etc.)
  • EC2 Windows Instances (RDP via SSO)
  • Custom apps registered via SAML metadata

Key components

  • Identity Store — where users/groups live (built-in or synced)
  • Permission Sets — what access a user/group gets in an account
  • Assignments — binding of (user/group) + (permission set) + (account/app)
  • Access Portal — the web UI users log into to access everything
💡

Relationship with Control Tower: Control Tower automatically sets up IAM Identity Center as part of the Landing Zone. It is the recommended way to manage human access to multi-account environments — replacing per-account IAM users entirely.

02

Identity Providers (IdP)

Built-in Identity Store

  • Users/groups managed directly in IAM Identity Center
  • No external dependency
  • Best for AWS-native or small teams
  • Supports MFA natively

AWS Managed Microsoft AD

  • Connect to on-premises Active Directory via VPN/Direct Connect
  • 2-way trust between on-prem AD and AWS Managed AD
  • Users authenticate with AD credentials
  • Groups synced automatically

External IdP (SAML 2.0)

  • Supports Okta, Azure AD, OneLogin
  • SAML 2.0 handles authentication only
  • Cannot query IdP for users/groups
  • Use SCIM to sync users/groups automatically
⚠

Critical exam trap: SAML 2.0 alone cannot provision users/groups into IAM Identity Center — it only handles the login handshake. Without SCIM, you must manually create matching users/groups. The exam tests this distinction.

03

SCIM — Automatic User Provisioning

Exam Key

SCIM (System for Cross-domain Identity Management) automatically syncs user and group identities from an external IdP into IAM Identity Center. It solves the problem that SAML 2.0 alone cannot push user data.

SAML 2.0 + SCIM — how they work together
External IdP (Okta/Azure AD) ──SAML 2.0──▶ IAM Identity Center → Authenticate user ✓
External IdP (Okta/Azure AD) ──SCIM──▶ IAM Identity Center → Sync users & groups ✓

SCIM details

  • Syncs create, update, deactivate of users/groups
  • Must be supported and enabled in the external IdP
  • Runs continuously — near real-time sync
  • Okta, Azure AD, OneLogin all support SCIM natively
  • Deprovisioning in IdP → user automatically deactivated in Identity Center

Without SCIM (SAML only)

  • Must manually create every user and group in Identity Center
  • Users/groups must be identical to the external IdP
  • No automatic deprovisioning — security risk when employees leave
  • Only viable for small, static user populations

Permission Sets ★ Most Tested

Exam Key

A Permission Set is a collection of IAM policies (AWS-managed or inline) that defines what a user or group can do in a target AWS account. It is created once and assigned to multiple accounts — IAM Identity Center creates a corresponding IAM role in each target account.

User / Group

e.g. "DevTeam" group from Okta via SCIM

+
Permission Set

e.g. "ReadOnlyAccess" (AWS managed policy)

→
AWS Account(s)

IAM role auto-created in each assigned account

Permission Set contents

  • AWS Managed Policies (e.g. AdministratorAccess, ReadOnlyAccess)
  • Customer Managed Policies (reference by name — must exist in target account)
  • Inline Policy — JSON defined directly in the permission set
  • Permissions boundary — optional max permissions cap
  • Session duration configurable (default 1h, max 12h)

How it works at login

  • User logs in → sees list of accounts they have access to
  • Selects account + permission set → gets temporary STS credentials
  • Behind the scenes: Identity Center calls sts:AssumeRole on the auto-created IAM role
  • Role name pattern: AWSReservedSSO_<PermissionSetName>_<hash>
  • Users never need long-term IAM credentials

Multi-account assignment example

GroupPermission SetAccount(s)Result
DevelopersPowerUserAccessDev, TestFull access except IAM in Dev/Test
DevelopersReadOnlyAccessProdRead-only in Prod
OpsAdministratorAccessAll accountsFull admin everywhere
AuditorsSecurityAuditAll accountsSecurity read-only org-wide

Attribute-Based Access Control (ABAC)

Exam Key

ABAC grants permissions based on user attributes (tags on the principal) matching resource tags. You define the policy once — access automatically adjusts when attributes change, without updating permission sets.

ABAC policy — user can only control EC2s tagged with their project
{ "Statement": [ { "Effect": "Allow", "Action": "ec2:DescribeInstances", "Resource": "*" }, { "Effect": "Allow", "Action": ["ec2:StartInstances", "ec2:StopInstances"], "Resource": "*", "Condition": { "StringEquals": { // EC2 tag must match user's PrincipalTag "ec2:ResourceTag/AccessProject": "${aws:PrincipalTag/AccessProject}" } } } ] }

ABAC attributes (examples)

  • CostCenter — user can access resources in their cost center
  • Department — user can only manage their department's resources
  • Title / Role — e.g. "Admin" vs "Developer"
  • Locale — region-based access

ABAC vs RBAC — when to use

  • ABAC: many users/resources, dynamic teams, reduce permission set sprawl
  • RBAC (Permission Sets): stable teams, coarse-grained access, simpler orgs
  • ABAC requires consistent tagging discipline on both users and resources
  • Both can be combined — ABAC as a condition inside a Permission Set
⚠

Exam pattern: If a scenario says "hundreds of users across many projects and you want to avoid managing hundreds of permission sets" → the answer is ABAC. Define once, scale via attribute changes.

06

External IdP & SAML 2.0 Integration

External IdP SSO login flow
User browser → IAM Identity Center portal → SAML AuthnRequest → External IdP (Okta) → SAML assertion → Identity Center validates → Temp credentials issued

What you configure

  • IdP metadata XML URL or file (uploaded to Identity Center)
  • SP (Service Provider) metadata from Identity Center → uploaded to IdP
  • ACS (Assertion Consumer Service) URL
  • Attribute mappings (e.g. email → username)
  • SCIM endpoint + bearer token (for provisioning)

SAML 2.0 limitations to know

  • Cannot query IdP for user/group lists — SAML is one-way (assertion only)
  • No real-time deprovisioning via SAML — need SCIM for that
  • SAML assertions contain attributes that can be mapped to Identity Center attributes (used for ABAC)
  • Session validity tied to SAML assertion lifetime
07

Multi-Factor Authentication (MFA)

Always-On (Every Sign-in)

  • MFA required on every single login regardless of context
  • Highest security posture
  • Best for privileged/admin users
  • May cause friction for high-frequency users

Context-Aware (Adaptive MFA)

  • MFA triggered only when sign-in context changes
  • Analyzes: device, location, IP, browser fingerprint
  • Trusted context = no MFA prompt (smoother UX)
  • New device / unusual location = MFA required

Supported MFA methods

TOTP (Authenticator apps) FIDO2 / WebAuthn (hardware keys, biometrics) SMS (via external IdP only) Built-in authenticator (Touch ID, Face ID)
💡

When using an external IdP (Okta, Azure AD), MFA is typically enforced at the IdP level, not in IAM Identity Center. Identity Center MFA applies to the built-in identity store only.

08

Application Assignments

How to add a SAML 2.0 application

  • Add the application in the Identity Center console (pre-built catalog or custom)
  • Provide the ACS URL, Entity ID, and start URL from the application
  • Download Identity Center metadata / certificates → upload to the app
  • Assign users/groups to the application using the same Assignment model
  • Optionally map user attributes to SAML claims (e.g. email, department)

Supported application types

Salesforce Microsoft 365 Box Slack Custom SAML 2.0 apps AWS CLI v2 (via SSO profile) EC2 Windows (RDP SSO)
💡

AWS CLI v2 SSO: Developers run aws sso login --profile my-profile → browser opens Access Portal → selects account/role → temporary credentials cached locally. No long-term keys needed.

The exam tests when to use Identity Center vs traditional IAM users/roles.

01

Core Concept

Exam One-liner

Amazon Inspector runs automated, continuous vulnerability assessments on EC2 instances, ECR container images, and Lambda functions — scanning for known CVEs (Common Vulnerabilities and Exposures) and unintended network exposure. It is a pre-exploit, proactive tool, not a runtime threat detector.

What Inspector does

  • Scans for known software vulnerabilities (CVEs) in OS packages, application libraries, and code dependencies
  • Checks for unintended network reachability — open ports accessible from the internet
  • Runs continuously — re-evaluates when new CVEs are published or packages change
  • Assigns a risk score to each finding using CVSS + AWS context
  • Sends findings to Security Hub and EventBridge

What Inspector does NOT do

  • Does not detect active threats or suspicious behavior — use GuardDuty for that
  • Does not remediate — it identifies and reports; you patch via SSM Patch Manager
  • Does not scan S3 objects, RDS, or ECS tasks directly — only EC2, ECR, Lambda
  • Does not run on demand by default — it's triggered by resource events and CVE updates
⚠

Core exam trap: Inspector finds vulnerabilities (CVEs, open ports) — pre-exploit. GuardDuty finds active threats — runtime detection. If a question says "identify unpatched software" or "scan for CVEs" → Inspector. If it says "detect suspicious activity" → GuardDuty.

02

Supported Targets ★ Most Tested

Hard Boundary — Exam Tested

Inspector only runs on: EC2 instances, ECR container images, and Lambda functions. No other resource types. This boundary is explicitly tested.

EC2 Instances

EC2 Scanning

  • Requires SSM Agent running on the instance
  • Instance must be an SSM Managed Instance (IAM role or Default Host Mgmt Config)
  • Outbound port 443 to SSM endpoint required
  • SSM collects OS package inventory → Inspector analyzes against CVE database
  • Also checks network reachability — ports exposed to internet
  • Scans triggered on: agent launch, package installs, CVE database updates
ECR Container Images

ECR Scanning

  • Scans container images on push to ECR (no agent needed)
  • Scans OS packages and application libraries inside the image layers
  • Two modes: Basic scanning (free, on-push only) vs Enhanced scanning (Inspector, continuous)
  • Enhanced scanning re-evaluates when new CVEs are published — even for old images
  • Findings visible in both ECR and Inspector consoles
Lambda Functions

Lambda Scanning

  • Scans function code and package dependencies (npm, pip, etc.)
  • Triggered on function deployment (new version published)
  • Also re-scans when new CVEs affect a known dependency
  • No agent required — Inspector pulls function metadata directly
  • Covers: Python, Node.js, Java, .NET runtime dependencies
03

EC2 Setup & Data Flow

Exam Key

Inspector does not install its own agent on EC2 — it uses the SSM Agent already present. The SSM Agent collects package inventory, sends it to Systems Manager, and Inspector analyzes that inventory. No separate Inspector agent exists for EC2.

EC2 + SSM Agent + IAM Role → SSM: collect package inventory → Systems Manager → analyze → Amazon Inspector → Findings (CVEs / Network)

EC2 prerequisites checklist

  • SSM Agent — installed and running (pre-installed on Amazon Linux 2, Windows Server 2016+)
  • IAM Role attached — with AmazonSSMManagedInstanceCore policy OR Default Host Management Config enabled
  • Outbound 443 — to ssm.REGION.amazonaws.com (via internet or VPC endpoint)
  • Instance must appear in AWS Systems Manager → Fleet Manager

Scan triggers for EC2

  • Initial enablement of Inspector on the account
  • SSM Agent starts on an instance (new launch or restart)
  • Software package installed or removed on the instance
  • New CVE added to the NVD (National Vulnerability Database)
  • Network configuration change (SG rule modified)
04

Findings & Risk Score

Exam Key

Inspector uses an Amazon Inspector Score — a contextualized CVSS score adjusted by AWS-specific factors like network reachability and exploit availability. Higher score = higher priority to patch.

Finding Severity Scale (CVSS-based)

Informational 0
Low 0.1–3.9
Medium 4.0–6.9
High 7.0–8.9
Critical 9.0–10

Finding types

  • Software vulnerability — CVE in OS packages, libraries, application dependencies
  • Network reachability — EC2 port reachable from internet (0.0.0.0/0)
  • Code vulnerability — security issues in Lambda function code (via CodeGuru integration)

Score contextualization factors

  • Network exposure — is the port reachable? Increases score
  • Exploit available — is there a known exploit in the wild? Increases score
  • Package in use — is the affected package actually loaded?
  • Score differs from raw CVSS — two resources with same CVE can have different Inspector scores
05

Integrations & Outputs

Security Hub integration

  • All Inspector findings automatically sent to AWS Security Hub
  • Normalized to ASFF (Amazon Security Finding Format)
  • Enables org-wide compliance dashboard with findings from Inspector, GuardDuty, Macie, etc.
  • Security Hub can trigger automations on Inspector findings

EventBridge integration

  • Inspector publishes finding events to EventBridge
  • React to: new finding, finding state change, initial scan complete
  • Pattern: EventBridge → Lambda → open Jira ticket / notify Slack / trigger SSM Patch
  • Rule on Inspector2 event source, type Inspector2 Finding
💡

Automated patching pattern: Inspector finding (Critical CVE) → EventBridge → Lambda → SSM Run Command / Patch Manager → patch the affected EC2 instance automatically. This end-to-end pipeline is a common DOP-C02 scenario.

Multi-account with AWS Organizations

  • Designate a Delegated Administrator account for Inspector
  • Admin can enable/disable Inspector for all member accounts
  • Aggregated findings view across all accounts in one console
  • Member accounts cannot disable Inspector once enrolled via Organizations

Exam Cheat Sheet

⚠

Inspector only covers EC2, ECR container images, and Lambda. Not RDS, S3, ECS directly, or EKS nodes.

⚠

Inspector uses SSM Agent for EC2 — there is no separate Inspector agent. SSM must be working first.

✓

Inspector is continuous — re-scans when new CVEs are published, even for unchanged images/instances.

✓

ECR: Enhanced scanning (Inspector) vs Basic scanning. Enhanced = continuous + re-evaluated on new CVEs.

✓

Findings sent to Security Hub (ASFF format) and EventBridge for automation pipelines.

✓

Risk score = CVSS + network reachability + exploit availability. Not just raw CVE score.

✓

Automated response: Inspector finding → EventBridge → Lambda → SSM Patch Manager → remediate.

✓

Multi-account: Delegated Administrator account manages Inspector org-wide. Members can't opt out.

✓

"Unintended network accessibility" = Inspector. "Suspicious runtime behavior" = GuardDuty. Never swap these.

✓

EC2 requires: SSM Agent + IAM Role (SSMManagedInstanceCore) + outbound 443 to SSM endpoint.

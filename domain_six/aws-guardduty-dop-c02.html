01

Core Concept

Exam One-liner

GuardDuty is an intelligent threat detection service that continuously analyzes AWS data sources using ML, anomaly detection, and threat intelligence feeds — with no agents to install. It generates Findings for suspicious activity, which you act on via EventBridge.

Key characteristics

  • Enabled with one click — 30-day free trial, no software to install
  • Pulls data independently from CloudTrail, VPC Flow Logs, DNS Logs — does NOT read your existing log groups
  • Uses ML + anomaly detection + 3rd-party threat intel (e.g. CrowdStrike, Proofpoint IP/domain lists)
  • Dedicated finding type for CryptoCurrency mining attacks
  • Generate sample findings to test your EventBridge automations before real incidents

What GuardDuty does NOT do

  • Does not block traffic — it only detects and reports. Use WAF/SGs/NACLs to block.
  • Does not read your CloudWatch log groups — it gets an independent stream from the service APIs
  • Does not require CloudTrail to be enabled — GuardDuty creates its own independent feed
  • Does not auto-remediate — you build automation via EventBridge
02

Data Sources

GuardDuty pulls independent streams — enabling CloudTrail or Flow Logs in your account is not required for GuardDuty to analyze them.

Always-On (included by default)

CloudTrail Management Events

Unusual API calls, unauthorized deployments, CreateVPC, CreateTrail, StopLogging…

CloudTrail S3 Data Events

GetObject, PutObject, DeleteObject on S3 — detect unusual S3 access patterns

VPC Flow Logs

Unusual internal traffic, unexpected port scans, communication with known malicious IPs

DNS Logs

Compromised EC2 sending encoded data via DNS queries (DNS tunneling / exfiltration)

Optional Features (extra cost, enabled per account)

EKS Audit Logs

K8s API activity — detect malicious callers, privilege escalation in clusters

RDS & Aurora Login Activity

Anomalous database login attempts, unusual login patterns

EBS Malware Protection

On-demand malware scan of EBS volumes attached to EC2/ECS — scans on new finding

Lambda Network Activity

Network calls from Lambda functions — detect unusual outbound connections

S3 Data Events (extended)

All S3 API operations for deeper data access monitoring

⚠

Exam trap: GuardDuty pulls an independent stream of CloudTrail and Flow Log data. Even if you delete your CloudTrail trail, GuardDuty still receives events. Disabling GuardDuty itself (not CloudTrail) stops detection.

03

Findings

Exam Key

A Finding is a potential security issue. Every finding has a severity score 0.1–8+, a structured naming convention, and is published to both the administrator account and the originating member account.

Severity Scale

Low 0.1–3.9
Medium 4.0–6.9
High 7.0–8.9+

High severity → treat as active threat. Medium → investigate. Low → informational.

Finding Naming Convention

ThreatPurpose : ResourceTypeAffected / ThreatFamilyName . DetectionMechanism!Artifact
  • ThreatPurpose — primary goal: Backdoor, CryptoCurrency, Trojan, UnauthorizedAccess, Stealth, Policy
  • ResourceTypeAffected — target: EC2, IAMUser, S3, Kubernetes, RDS, ECS
  • ThreatFamilyName — activity: SSHBruteForce, NetworkPortUnusual, MaliciousIPCaller
  • DetectionMechanism — how detected: Tcp, Udp, DNS
  • Artifact — element used: DNS, CustomThreatIntel
CryptoCurrency:EC2/BitcoinToo.B!DNS
UnauthorizedAccess:EC2/SSHBruteForce
Stealth:IAMUser/CloudTrailLoggingDisabled

05

Multi-Account Strategy

Exam Key

In an AWS Organization, designate a member account as GuardDuty Delegated Administrator — not the management account. This admin account centrally manages GuardDuty across all org accounts and aggregates all findings.

Two ways to associate member accounts

  • Via AWS Organizations — auto-enrolls all current and future accounts; recommended
  • GuardDuty invitation — manually invite specific accounts; for accounts outside the org

Admin account capabilities

  • Add and remove member accounts
  • Enable/disable GuardDuty in member accounts
  • Manage suppression rules — suppress known-safe findings
  • Manage trusted IP lists — IPs that never trigger findings
  • Manage threat lists — custom malicious IP/domain lists

Findings aggregation

  • Findings published to both the admin account and the originating member account
  • Admin account sees findings from all member accounts in one pane
  • Member accounts can only see their own findings
  • EventBridge rules in the admin account can react to findings from all accounts
  • Use Finding Aggregation to pull findings from multiple regions into one region
⚠

Exam trap: The GuardDuty Delegated Administrator is a member account, not the management account. Member accounts cannot disable GuardDuty once enrolled via Organizations — only the admin account can.

06

Findings Automated Response

Standard response pipeline
GuardDuty Finding → EventBridge Rule → SNS → email/Slack/PagerDuty
GuardDuty Finding → EventBridge Rule → Lambda → auto-remediate

Common automated response patterns

  • EC2 compromise → Lambda isolates instance (remove from SG, snapshot EBS)
  • IAM credential leaked → Lambda revokes all sessions, disables access key
  • S3 bucket made public → Lambda re-enables Block Public Access
  • CryptoCurrency finding → Lambda terminates instance, notifies ops team
  • CloudTrail disabled → Lambda re-enables CloudTrail

Suppression rules

  • Filter out known-safe findings to reduce noise
  • Define by: finding type, resource, region, severity, tags
  • Suppressed findings are archived, not deleted — still visible if needed
  • Example: suppress Recon:EC2/PortProbeUnprotectedPort for dev accounts
  • Only the admin account can create org-wide suppression rules
💡

Sample findings: GuardDuty can generate synthetic sample findings for every finding type — use this to validate your EventBridge rules and Lambda automations before a real incident occurs. Critical for testing runbooks.

07

CloudFormation Deployment Pattern

Exam Key

Deploying GuardDuty via CloudFormation requires a Custom Resource (Lambda) to handle the idempotency problem — the native AWS::GuardDuty::Detector resource fails if GuardDuty is already enabled. Use StackSets to deploy org-wide.

CFN Stack / StackSet → Custom Resource → Lambda: check if enabled → Enable only if NOT already on → GuardDuty Detector ✓

Why Custom Resource is needed

  • AWS::GuardDuty::Detector fails with error if GuardDuty already enabled in that account
  • Using a Lambda Custom Resource allows conditional logic before creating
  • Lambda checks guardduty:ListDetectors → if empty, enable; if exists, skip
  • This makes the deployment idempotent — safe to re-run across all accounts

Org-wide deployment

  • Use CloudFormation StackSets to deploy the Custom Resource to all accounts
  • Deploy from the management or delegated admin account
  • Target by OU or all accounts in the org
  • Alternative: use Firewall Manager — simpler but less customizable

GuardDuty vs Related Security Services

ServiceWhat it detectsData sourcesKey distinction
GuardDuty Active threats, anomalous behavior, compromised resources CloudTrail, VPC Flow Logs, DNS, EKS, RDS, Lambda Runtime threat detection using ML. Finds active attacks.
AWS Inspector Vulnerabilities in EC2, Lambda, container images EC2 OS packages, ECR images, Lambda code Vulnerability scanning (CVEs). Not runtime — finds weaknesses before exploitation.
AWS Macie Sensitive data (PII, credentials) in S3 S3 object content Data classification. Finds where sensitive data lives, not threats.
Security Hub Aggregated findings from multiple services GuardDuty, Inspector, Macie, Config, Firewall Manager SIEM-like aggregation and normalization. Correlates findings, doesn't detect independently.
AWS Config Resource configuration drift / non-compliance Config recorder Configuration compliance over time. Not threat detection.

Exam Cheat Sheet

⚠

GuardDuty does NOT block. It detects and reports. You automate response via EventBridge → Lambda.

⚠

GuardDuty pulls independent data streams — disabling CloudTrail doesn't stop GuardDuty from seeing events.

✓

Findings published to both admin account and originating member account simultaneously.

✓

Delegated Admin = member account, not management account. Members can't disable GuardDuty once enrolled.

✓

CryptoCurrency attacks have a dedicated finding type — this is tested as a GuardDuty unique capability.

✓

DNS tunneling (data exfil via DNS) detected via DNS Logs data source — unique to GuardDuty.

✓

CFN deployment: use Custom Resource (Lambda) for idempotent enabling. Native resource fails if already enabled.

✓

Suppression rules: archive known-safe findings, not delete. Admin-only for org-wide rules.

✓

Sample findings = test your automations. Generate synthetically before real incidents.

✓

Severity: Low 0.1–3.9 / Medium 4.0–6.9 / High 7.0–8.9+. High = active threat requiring immediate action.

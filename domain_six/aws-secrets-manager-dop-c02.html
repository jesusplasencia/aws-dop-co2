01

Core Concept

Exam One-liner

AWS Secrets Manager stores, rotates, and manages access to secrets (passwords, API keys, DB credentials) with automatic rotation via Lambda, KMS encryption at rest, and native integration with RDS. It is the preferred service when automatic rotation is required.

Core capabilities

  • Encrypted storage — all secrets encrypted with AWS KMS (CMK or AWS-managed key)
  • Automatic rotation — schedule rotation every N days via Lambda
  • Versioning — secrets have versions: AWSCURRENT, AWSPENDING, AWSPREVIOUS
  • Resource-based policies — grant cross-account access to secrets
  • VPC endpoint support — access secrets privately without internet
  • Audit via CloudTrail — every GetSecretValue, RotateSecret call is logged

Supported secret types

  • RDS credentials — MySQL, PostgreSQL, Aurora (native rotation support)
  • Redshift credentials
  • DocumentDB credentials
  • Other databases — via custom Lambda rotation
  • API keys / OAuth tokens
  • Arbitrary JSON — any key-value secret structure
  • SSH keys, TLS certificates
02

Secret Rotation ★ Most Tested

Exam Key

Rotation is performed by a Lambda function — either AWS-provided (for RDS/Redshift/DocumentDB) or custom. The Lambda follows a 4-step rotation lifecycle using secret version staging labels. The application always reads AWSCURRENT — it never needs to know a rotation happened.

Step 1
createSecret

Lambda generates a new secret value, stores it as AWSPENDING version

Step 2
setSecret

Lambda updates the target system (e.g. RDS DB) with the new password from AWSPENDING

Step 3
testSecret

Lambda verifies the new credentials work by connecting to the database

Step 4
finishSecret

Lambda moves AWSPENDING → AWSCURRENT. Old version becomes AWSPREVIOUS

Rotation configuration

  • Schedule — fixed days (e.g. every 30 days) or cron expression
  • Immediately — option to rotate immediately when rotation is first enabled
  • AWS-managed Lambda — auto-created for RDS MySQL, PostgreSQL, Aurora, Redshift, DocumentDB
  • Custom Lambda — for any other secret type; you implement the 4 steps
  • Lambda must have network access to the target service and access to Secrets Manager endpoint

Application consumption pattern

  • App calls GetSecretValue — always returns AWSCURRENT
  • Cache the secret in your app — don't call API on every request
  • Use the Secrets Manager SDK cache client — auto-refreshes on rotation
  • On 401 / auth failure → force-refresh the cache → retry with new value
  • Never hardcode secrets — always retrieve dynamically
⚠

Rotation Lambda network access: If Lambda is in a VPC, it needs either a VPC endpoint for Secrets Manager or NAT Gateway to reach the Secrets Manager API. Without this, rotation silently fails. This is a frequent exam scenario.

03

KMS Encryption

How encryption works

  • Every secret is encrypted with a KMS key using envelope encryption
  • Default: AWS-managed key (aws/secretsmanager) — no cost for KMS
  • Custom: Customer Managed Key (CMK) — required for cross-account access
  • KMS key policy must grant Secrets Manager the ability to GenerateDataKey and Decrypt
  • All access to the secret value requires KMS kms:Decrypt permission on the caller

When to use CMK

  • Cross-account access — AWS-managed key cannot be shared; CMK is required
  • Key rotation control — manage rotation of the KMS key separately
  • Key policy customization — restrict who can decrypt even with SM access
  • Audit trail — CloudTrail logs KMS key usage separately from SM
04

RDS & Database Integration

Exam Key

For RDS/Aurora, Secrets Manager provides native managed rotation — AWS creates and manages the Lambda function automatically. The Lambda updates both Secrets Manager and the RDS instance password atomically.

Rotation schedule triggers → Secrets Manager invokes Lambda → Lambda: generate new password → Lambda: update RDS master password → Lambda: test connection → AWSPENDING → AWSCURRENT ✓

Supported database engines (native rotation)

RDS MySQL RDS PostgreSQL RDS Oracle RDS SQL Server Amazon Aurora (MySQL/PostgreSQL) Amazon Redshift Amazon DocumentDB
💡

ECS/EKS/Lambda consuming RDS secrets: Use the Secrets Manager VPC endpoint to retrieve DB credentials privately. Reference the secret ARN in your task definition (ECS) or environment variable and call GetSecretValue at startup — not at deploy time.

05

Multi-Region Secrets

us-east-1 (Primary)
  • Source of truth for secret value
  • Rotation happens here first
  • KMS key in this region encrypts primary
⟶ replicate ⟶
us-west-2 (Replica)
  • Read replica — kept in sync automatically
  • Encrypted with KMS key in this region
  • Can be promoted to standalone if needed

Multi-region behavior

  • Replicate the same secret to one or more regions
  • Replicas stay in sync with the primary — including after rotation
  • Each replica is encrypted with the KMS key of its own region — must pre-exist
  • Application in us-west-2 reads from the replica → lower latency, no cross-region call
  • Promote replica to standalone — decouples it for DR failover

Use cases

  • Multi-region active-active apps — each region reads its local replica
  • Disaster recovery — promote replica in DR region during failover
  • Multi-region RDS databases — replica secret used by apps in each region
  • Reduce latency — avoid cross-region GetSecretValue calls in production
06

Access Control

IAM policy (identity-based)

  • Grant secretsmanager:GetSecretValue to the IAM role of your EC2/Lambda/ECS task
  • Scope to specific secrets: "Resource": "arn:aws:secretsmanager:*:*:secret:prod/*"
  • Use secretsmanager:ResourceTag condition for ABAC

Resource-based policy (secret policy)

  • Attach a policy directly to the secret — enables cross-account access
  • Grant another account's role access without assuming a role in your account
  • Required when using CMK for cross-account: both SM policy + KMS key policy must allow
Cross-account access example
{ // Resource policy attached to the secret in Account A "Statement": [{ "Effect": "Allow", "Principal": { "AWS": "arn:aws:iam::ACCOUNT_B:role/AppRole" }, "Action": "secretsmanager:GetSecretValue", "Resource": "*" }] } // Also need KMS key policy in Account A to allow Account B's role to Decrypt

Secrets Manager vs SSM Parameter Store ★

This comparison is heavily tested — know exactly when to use each.

FeatureSecrets ManagerSSM Parameter Store
Primary purpose Secrets with automatic rotation Configuration + secrets (no native rotation)
Automatic rotation ✓ Native (Lambda-driven, scheduled) ✗ Manual only (EventBridge + Lambda workaround)
Cost $0.40/secret/month + $0.05/10K API calls Free (Standard). $0.05/advanced param/month
Encryption KMS — always encrypted KMS — optional (SecureString type)
Cross-account ✓ Resource-based policy + CMK ✗ Not supported natively
Secret versioning AWSCURRENT / AWSPENDING / AWSPREVIOUS Versions via history, no staging labels
RDS integration ✓ Native managed rotation for RDS/Aurora ✗ None
Hierarchy / config Flat key-value or JSON ✓ Hierarchical paths (e.g. /prod/db/password)
Use when DB passwords, API keys needing auto-rotation App config, feature flags, non-rotating parameters
⚠

Exam decision rule: Question mentions "automatic rotation" or "RDS password rotation" → Secrets Manager. Question mentions "application configuration", "feature flags", or "cost-sensitive" → SSM Parameter Store.

08

Exam Cheat Sheet

⚠

Rotation Lambda needs network access to both the secret target AND Secrets Manager. VPC endpoint or NAT required in VPC.

⚠

Cross-account secret access needs both a resource-based policy on the secret and a CMK key policy. AWS-managed key won't work cross-account.

✓

Rotation stages: AWSPENDING (being rotated) → AWSCURRENT (active) → AWSPREVIOUS (last value).

✓

4-step Lambda rotation: createSecret → setSecret → testSecret → finishSecret. All 4 must succeed.

✓

Multi-region replica uses the KMS key of the destination region — not the primary region's key.

✓

App always reads AWSCURRENT via GetSecretValue. Rotation is transparent to consumers.

✓

WAF + CloudFront origin secret rotation: Secrets Manager rotates → Lambda updates both CloudFront header and WAF rule atomically.

✓

"Automatic rotation needed" → Secrets Manager. "Config values + cheap" → SSM Parameter Store. Never swap these.

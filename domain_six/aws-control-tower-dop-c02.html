01

Core Concept

Exam One-liner

Control Tower is an opinionated orchestration layer on top of AWS Organizations that automates a secure, compliant multi-account environment (Landing Zone) using guardrails, Account Factory, SCPs, Config, IAM Identity Center, and CloudFormation StackSets — all in a few clicks.

What it automates

  • AWS Organizations setup with OU structure
  • SCPs for preventive guardrails
  • AWS Config for detective guardrails
  • CloudTrail enabled org-wide automatically
  • Centralized logging to Log Archive account
  • IAM Identity Center for SSO across accounts
  • CloudFormation StackSets to deploy resources org-wide

What it does NOT do

  • Does not replace Organizations — it builds on top of it
  • Cannot manage accounts outside its Landing Zone without enrollment
  • Does not prevent actions by itself — SCPs/guardrails do that
  • Manual customization beyond guardrails requires CfCT or AFT
02

Landing Zone

Exam Key

The Landing Zone is the automatically provisioned, secure, compliant multi-account environment. It is the output of setting up Control Tower — everything below is created for you.

Management Account

Owns the org, Control Tower console, billing payer

Log Archive Account

Centralized S3 for CloudTrail & Config history from all accounts. Read-only for auditors.

Audit Account

SNS notifications, cross-account audit access via read-only/admin IAM roles

AWS Organizations

Root OU, Security OU, Sandbox OU created automatically

Account Factory

Self-service portal to vend new accounts with config baseline

SCPs

Preventive guardrails applied to OUs via Organizations

IAM Identity Center

Centralized SSO access across all member accounts

Guardrails

Rules enforcing security, compliance, best practices

AWS Config

Detective guardrail backbone; Config Aggregator org-wide

Initial setup configuration choices

Enable / disable specific regions Security OU (Foundational) Sandbox OU (Additional) Log Archive Account Audit Account CloudTrail org-wide KMS encryption (optional) S3 storage (optional)
03

Guardrails ★ Most Tested

Exam Key — Two Types

Preventive guardrails use SCPs — they block the action before it happens.
Detective guardrails use AWS Config — they identify non-compliant resources after the fact and trigger remediation.

MANDATORY

Mandatory

  • Always enabled and enforced by Control Tower
  • Cannot be disabled
  • Ex: Disallow public read access to Log Archive S3
  • Ex: Disallow changes to CloudTrail in managed accounts
STRONGLY REC.

Strongly Recommended

  • Based on AWS best practices
  • Optional — you enable them
  • Ex: Enable encryption for EBS volumes on EC2 instances
  • Ex: Disallow creation of access keys for the root user
ELECTIVE

Elective

  • Commonly used by enterprises, optional
  • Ex: Disallow delete actions without MFA on S3
  • Ex: Disallow internet access for an Amazon VPC
Detective guardrail violation flow
Member Accounts → Config (Detective) → NON_COMPLIANT trigger → SNS → notify Admin + Lambda → remediate
⚠

Exam trap: Preventive guardrails = SCPs (block action). Detective guardrails = Config rules (detect after). Mixing these up is the #1 guardrail mistake on the exam.

04

Account Factory

Exam Key

Account Factory uses AWS Service Catalog under the hood to provision new accounts with pre-approved baselines. End users (e.g., developers) can self-service a new account through the Service Catalog portal.

What it provisions per account

  • Pre-configured VPC (subnets, CIDR, regions)
  • IAM roles for Control Tower access
  • Guardrails automatically applied from parent OU
  • Config recorder and delivery channel
  • CloudTrail org trail coverage
  • SSO access assigned via IAM Identity Center

Account Factory Customization (AFC)

  • Deploy a Custom Blueprint (CFN template) to new/existing accounts
  • Blueprint = Service Catalog Product stored in a Hub Account
  • Only one blueprint can be deployed per account
  • Customize VPC, security groups, tags, IAM roles at creation time
  • Different from CfCT — AFC is per-account, CfCT is org-wide GitOps
💡

EventBridge integration: Control Tower emits lifecycle events when accounts are created (CreateManagedAccount). Use EventBridge → SQS → Lambda to trigger post-provisioning automation on new accounts.

05

Customizations for Control Tower (CfCT)

Exam Key — CfCT vs AFC

CfCT is a GitOps-style framework for deploying CFN templates and SCPs org-wide via a CI/CD pipeline. AFC (Account Factory Customization) is per-account blueprint customization. They are different tools for different scopes.

CfCT pipeline flow

Code commit (CFN + SCPs) → CodePipeline → CodeBuild → Step Functions → StackSets → all accounts + SCPs → Organizations

CfCT features

  • GitOps: version-controlled CFN templates and SCP definitions
  • Automatically deploys to new accounts created via Account Factory
  • Uses EventBridge → SQS → Lambda for lifecycle event handling
  • Triggers CodePipeline to deploy via StackSets
  • Stored in S3 or CodeCommit as the source

CfCT vs AFC comparison

  • CfCT — org-wide, GitOps, CFN + SCPs, multiple accounts
  • AFC (Blueprint) — single account, Service Catalog, one blueprint max
  • CfCT handles ongoing customization; AFC handles initial baseline
  • Both can be used together in the same environment

Account Factory for Terraform (AFT)

Exam Key

AFT allows you to use Terraform to provision and customize accounts through Control Tower. You submit an account-request.tf file — AFT's pipeline creates the account in Control Tower and applies your customizations.

account-request.tf (minimal example)
module "aft" { source = "git@github.com:aws-ia/terraform-aws-control_tower_account_factory.git" # Required ct_management_account_id = "123412341234" log_archive_account_id = "234523452345" audit_account_id = "345634563456" aft_management_account_id = "456745674567" ct_home_region = "us-east-1" # Optional feature flags aft_feature_delete_default_vpcs_enabled = false aft_feature_cloudtrail_data_events = false aft_feature_enterprise_support = true }

AFT pipeline flow

  • ATF admins push account-request.tf to CodeCommit
  • CodePipeline triggers on push
  • CodeBuild runs Terraform, calls Control Tower Account Factory API
  • New account created → Conformance Packs deployed via StackSets

Built-in optional feature flags

  • CloudTrail Data Events — creates trail + enables data events
  • Enterprise Support Plan — turns on Enterprise Support
  • Delete Default VPCs — removes default VPCs in all regions
07

Config & CloudTrail Integration

AWS Config (Detective)

  • Control Tower auto-enables Config in all enabled regions
  • Config history + snapshots → S3 in Log Archive account
  • Config Aggregator created via StackSets org-wide
  • Detective guardrails = Config rules under the hood
  • You cannot disable Config in managed accounts without breaking Control Tower

CloudTrail (Audit)

  • Org-level CloudTrail enabled automatically covering all accounts
  • Logs delivered to Log Archive account S3 bucket
  • Trail is immutable — guardrail prevents disabling it
  • Management events captured by default; Data Events optional (AFT flag)
💡

Under the hood: Control Tower uses CloudFormation StackSets to deploy Config Aggregator, CloudTrail trail, IAM roles, and logging resources to every member account. This is what makes it "automated".

08

Enrolling Existing Accounts

Exam Key

Control Tower can enroll existing AWS accounts that were not created through Account Factory. This is a separate workflow that evaluates and applies guardrails to the existing account.

Enrollment process for existing accounts

1Move account into same Organization as Control Tower
2Create AWSControlTowerExecution IAM role in the account
3Control Tower creates Config Conformance Packs to evaluate the account
4Compliance results evaluated — how guardrails affect the account
5Config Delivery & Configuration Recorder created during evaluation are removed
6Move account into target OU — Control Tower setup complete

Control Tower vs Organizations vs Config

These three services are deeply interrelated — the exam tests whether you know which layer does what.

CapabilityAWS OrganizationsAWS ConfigControl Tower
Multi-account mgmt Yes — core service No Yes — built on Organizations
SCPs Yes — create/manage manually No Yes — auto-applied as guardrails
Compliance rules No Yes — Config Rules / Conformance Packs Yes — via Config (detective guardrails)
Account vending Manual via API/Console No Yes — Account Factory (automated)
Centralized logging No (you build it) Partial (you configure) Yes — Log Archive account auto-setup
SSO / Identity No No Yes — IAM Identity Center integrated
Setup effort Medium (manual) Medium (per service) Low — automated & opinionated

Exam Cheat Sheet

⚠

Preventive guardrail = SCP. Detective guardrail = Config. Never reverse these.

⚠

CfCT ≠ AFC. CfCT = GitOps org-wide pipeline. AFC = Service Catalog blueprint per account.

✓

3 mandatory accounts: Management, Log Archive, Audit — always created by Control Tower.

✓

Control Tower uses CloudFormation StackSets to deploy Config, CloudTrail, and IAM resources org-wide.

✓

Account Factory → Service Catalog under the hood. Self-service for developers.

✓

New account lifecycle event → EventBridge → SQS → Lambda for post-provisioning automation.

✓

AFT uses Terraform + CodePipeline. Trigger = push of account-request.tf to CodeCommit.

✓

Mandatory guardrails cannot be disabled. Strongly Recommended and Elective are optional.

✓

Config Aggregator created org-wide via StackSets — Log Archive account receives all Config data.

✓

To enroll an existing account: move to org → create AWSControlTowerExecution role → move to target OU.

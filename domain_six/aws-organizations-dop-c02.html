01

Core Concepts & Structure

Exam One-liner

AWS Organizations lets you centrally manage multiple AWS accounts — grouping them into Organizational Units (OUs) with a hierarchical structure, applying governance via SCPs, and consolidating billing.

🏢 Root OU
├─ Management Account
├─ OU (Dev)
├─ Member Account A
└─ Member Account B
└─ OU (Prod)
├─ OU (HR)
├─ Account C
└─ Account D
└─ OU (Finance)
├─ Account E
└─ Account F

Hierarchy rules

  • Root — top-level container; SCPs applied here affect all OUs and accounts
  • Management Account — creates the org, not affected by SCPs
  • OUs — can be nested; accounts inherit SCPs from all parent OUs
  • Member Accounts — can belong to only one OU at a time
  • OUs can contain both accounts and other OUs (up to 5 levels deep)

Why use multiple accounts?

Isolation per department / cost center Environment separation (Dev / Prod) Regulatory / compliance boundaries via SCP Per-account service limits Centralized logging (CloudTrail → S3) Blast radius reduction
02

OrganizationAccountAccessRole

Exam Key

An IAM role automatically created in every new member account. The Management Account assumes it via sts:AssumeRole to perform admin tasks across member accounts.

Behavior

  • Grants full administrator permissions in the member account to the Management Account
  • Assumed via the AssumeRole API from the Management Account
  • Auto-created for accounts created via Organizations
  • Must be created manually for existing invited accounts
  • Used for tasks like creating IAM users, setting up Config, enabling services

Cross-account access pattern

  • Management Account IAM user/role calls sts:AssumeRole
  • Target: arn:aws:iam::MEMBER_ID:role/OrganizationAccountAccessRole
  • Returns temporary credentials scoped to member account
  • This is the standard pattern for central account administration
⚠

Exam trap: If you invite an existing account into your organization, the role is NOT automatically created — you must create it manually in that account.

03

Feature Modes

Consolidated Billing Only

  • Single payment method for all accounts
  • Pricing benefits from aggregated usage (volume discounts)
  • Reserved Instance & Savings Plans sharing
  • No SCP support

All Features ★ (Default)

  • Includes everything in Consolidated Billing
  • Enables Service Control Policies (SCPs)
  • Invited accounts must explicitly approve
  • Can apply SCP to prevent accounts from leaving the org
  • Cannot switch back to Consolidated Billing only
04

Reserved Instances & Consolidated Billing

  • Consolidated billing treats all accounts as one for pricing tiers and RI discounts
  • All accounts can benefit from RIs or Savings Plans purchased by any account in the org
  • Management Account can turn off RI / Savings Plans sharing for any account
  • To share a discount, both accounts must have sharing turned on
  • Volume discounts (S3, Data Transfer) are computed across the entire organization
💡

Pattern: Purchase RIs in a central account and let all member accounts benefit — cost optimization without duplicating reservations.

05

Moving Accounts Between Organizations

1

The member account must leave the old organization (or the old org removes it)

2

The new organization sends an invite to the member account

3

The member account accepts the invite and joins the new organization

⚠

Remember: An account can only belong to one organization at a time. It must fully leave the old org before it can join another. An SCP can prevent accounts from leaving — use this to enforce membership.

06

Service Control Policies (SCP) ★ Most Tested

Exam One-liner

SCPs define the maximum permissions boundary for accounts and OUs. They do not grant permissions — they restrict what IAM can allow. An action must be permitted by both the SCP and an IAM policy to succeed.

What SCPs do

  • Define allowlist or blocklist of IAM actions
  • Applied at OU or Account level
  • Affect all Users, Roles, and root user in the account
  • Inherited hierarchically — child OUs/accounts get intersection of all parent SCPs
  • Require an explicit Allow from Root through each OU in the path

What SCPs do NOT do

  • Do not apply to the Management Account — ever
  • Do not affect service-linked roles (e.g. Config, GuardDuty integration roles)
  • Do not grant permissions — only restrict them
  • Do not affect the Organizations service itself in the management account
SCP Inheritance — how the hierarchy resolves
Root OU
SCP: FullAWSAccess (Allow *) — default; applies to everything below
Mgmt Account
No SCPs apply — management account is always exempt
OU (Sandbox)
Additional SCPs: Deny S3, Allow EC2
Account A (in Sandbox)
Effective: Everything allowed EXCEPT S3 (inherited from Sandbox OU) EXCEPT EC2 if Sandbox also denies
OU (Workloads → Prod)
FullAWSAccess — no additional restrictions
Accounts E & F (in Prod)
Can do anything — no deny SCPs in their path
Blocklist strategy (deny specific actions)
// Allow everything, then deny specific services { "Statement": [ { "Effect": "Allow", "Action": "*", "Resource": "*" }, { "Sid": "DenyDynamoDB", "Effect": "Deny", "Action": "dynamodb:*", "Resource": "*" } ] }
Allowlist strategy (permit only specific services)
// Remove FullAWSAccess, then allow only these { "Statement": [ { "Effect": "Allow", "Action": [ "ec2:*", "cloudwatch:*" ], "Resource": "*" } ] }
⚠

Default SCP: Every new account/OU inherits a FullAWSAccess allow-all policy. You must explicitly add deny SCPs or remove FullAWSAccess and replace with allowlist to restrict permissions.

Common SCP use cases for exam

  • Deny leaving org — Deny organizations:LeaveOrganization on all member OUs
  • Region lock — deny all actions unless aws:RequestedRegion matches allowed regions
  • Prevent disabling security services — deny cloudtrail:StopLogging, config:DeleteConfigurationRecorder, guardduty:DeleteDetector
  • Restrict to approved instance types — deny ec2:RunInstances unless instance type matches
  • Enforce PCI compliance — deny services not in PCI scope explicitly
  • Require MFA — deny all actions unless aws:MultiFactorAuthPresent is true
07

Multi-Account Strategies

Strategy OU Structure Best For
Business Unit Sales OU → Sales Acct 1 & 2; Retail OU → …; Finance OU → … Large enterprises with distinct business lines needing billing separation
Environmental Lifecycle Prod OU, Dev OU, Test OU — each with member accounts Software teams; enforces blast radius isolation between environments
Project-Based Project 1 OU, Project 2 OU, Project 3 OU Agencies / consultancies with client or project isolation requirements

Centralized logging architecture (exam pattern)

  • CloudTrail enabled on all accounts → logs sent to a central S3 account
  • CloudWatch Logs from all accounts → forwarded to a central logging account
  • Use S3 bucket policy in the logging account to allow cross-account writes
  • Security team has read access only to the central account — member accounts cannot delete logs
08

Services That Integrate with Organizations

Governance & Security

  • AWS Config — Aggregators, Org Rules, Conformance Packs
  • AWS CloudTrail — Org-level trail covers all accounts
  • AWS GuardDuty — Org-level threat detection, delegated admin
  • AWS Security Hub — Aggregated security findings across accounts
  • AWS IAM Access Analyzer — Org-level external access analysis

Operations & Deployment

  • CloudFormation StackSets — deploy stacks to multiple accounts/regions
  • AWS SSO / IAM Identity Center — centralized access management
  • AWS RAM — share resources (VPCs, Transit Gateways) org-wide
  • Service Catalog — share portfolios across accounts
  • Firewall Manager — org-level WAF and Security Group policies
💡

Delegated Administrator pattern: Instead of using the Management Account for every service, designate a member account as the admin for services like GuardDuty, Security Hub, Config, and Firewall Manager. This is AWS best practice — keep the Management Account clean.

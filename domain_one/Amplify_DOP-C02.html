The Two Distinct Pieces

The exam tests these as separate concepts. Most DOP-C02 questions focus on Amplify Hosting — the deployment and CI/CD mechanics.

Primary Exam Focus

Amplify Hosting

The CI/CD and hosting layer. Deploys your frontend, manages branch deployments, custom domains, pull request previews, rewrites and redirects.

Every branch is an environment. Every PR gets a preview URL. Custom domains are fully automated. This is what most DOP-C02 questions are about.

HIGH exam focus
Secondary Exam Focus

Amplify Studio / Libraries

The backend configuration layer. Visual interface to configure Auth, APIs, Storage, and Functions. Generates frontend code that integrates with AWS services automatically.

Know what underlying AWS service each capability maps to — when something breaks, the question asks about the real AWS service, not Amplify.

Branch-Based Deployments — The Core Feature

Every Git branch maps to a deployed environment automatically. This is the most tested Amplify concept in DOP-C02.

GitHub Repository
├── main → myapp.amplifyapp.com production
├── staging → staging.myapp.amplifyapp.com staging
├── feature/auth → feature-auth.myapp.amplifyapp.com dev
└── feature/ui → feature-ui.myapp.amplifyapp.com dev
Each branch gets: own URL · own build environment · own environment variables · own backend · independent deployments.
Pushing to feature/auth does NOT affect main.
main branch env vars
REACT_APP_API = https://api.myapp.com
REACT_APP_ENV = production
staging branch env vars
REACT_APP_API = https://staging-api.myapp.com
REACT_APP_ENV = staging
🎯
Exam Pattern

"Team needs the same codebase deployed to three environments with different API endpoints." Answer: Amplify branch deployments with per-branch environment variables — not three separate CodePipeline pipelines.

PR Previews
👁️

Pull Request Previews

Amplify auto-deploys a preview environment for every pull request. The critical exam detail: PR previews are configured on the target branch (main), NOT the source feature branch. This single distinction is used in trap answers.

1

Developer opens PR: feature/checkout → main

Amplify detects the PR via webhook configured on the main branch — the target branch, not the feature branch.

2

Amplify builds the PR branch automatically

Uses the same amplify.yml and build settings as the target branch. No manual trigger required.

3

Deploys to unique preview URL

pr-42.myapp.amplifyapp.com — completely isolated from all other environments.

4

Preview URL posted as PR comment on GitHub

Reviewers click the link directly from the PR — no manual sharing or communication needed.

5

PR merged → preview environment destroyed automatically

Zero cleanup required. Amplify tears down the preview on merge or close. The environment lifecycle is fully managed.

⚠️
Critical Trap — One Word Changes Everything

Configuring PR previews on the staging branch instead of main means PRs targeting main never get previewed. This single word swap is used in DOP-C02 trap answers. Always configure PR previews on the destination branch (main).

amplify.yml
📄

amplify.yml — The Build Specification

The equivalent of buildspec.yml for Amplify. Must be at the root of your repo. If absent, Amplify auto-detects the framework and uses sensible defaults.

amplify.yml Amplify Build Spec 
version: 1

applications:
  - frontend:
      phases:
        preBuild:                         # dependency installation
          commands:
            - npm ci
        build:                            # production bundle creation
          commands:
            - npm run build
      artifacts:
        baseDirectory: build             # React=build, Vue=dist, Next.js=.next
        files:
          - '**/*'
      cache:
        paths:
          - node_modules/**/*             # cache deps between builds

    backend:
      phases:
        build:
          commands:
            - amplifyPush --simple         # deploy backend (Cognito, AppSync, etc.)

  # Monorepo support — second app in same repository
  - appRoot: packages/admin            # path to sub-app root
    frontend:
      phases:
        preBuild:
          commands:
            - npm ci
        build:
          commands:
            - npm run build
      artifacts:
        baseDirectory: dist
        files:
          - '**/*'
artifacts.baseDirectory per framework
React (CRA) build Most common — exam default
Vue / Vite / Nuxt (static) dist Standard Vite output
Next.js (SSR) .next Server-side rendering output
Next.js (static export) out Static HTML export mode
Angular dist/project-name Project name from angular.json
Gatsby / Hugo public Static site generator output
Rewrites & Access Control
🔀

Rewrites, Redirects & Access Control

Single-page applications break without a rewrite rule. CloudFront looks for a file called /dashboard in S3 when a user navigates directly to myapp.com/dashboard. It doesn't exist. Result: 403 or 404.

Amplify Console → Rewrites and Redirects JSON Rule 
// SPA rewrite rule — fixes all deep link 404s
[
  {
    "source": "</^[^.]+$|\\.(?!(css|gif|ico|jpg|js|png|txt|svg|woff|ttf|map|json)$)([^.]+$)/>",
    "target": "/index.html",
    "status": "200",
    "condition": null
  }
]
// Plain English: any request that doesn't look like a known file extension
// → serve index.html with status 200
// → React Router / Vue Router handles routing client-side
Status Type Behavior Use Case
200 Rewrite URL in browser stays unchanged SPA routing — serve index.html, keep the URL
301 Permanent Redirect Browser updates URL permanently Old URL moved — SEO-safe redirect
302 Temporary Redirect Browser redirects, URL changes Maintenance pages, A/B testing
404 Custom Error Page Serves your branded error page Friendly error handling
⚠️
Exam Pattern — SPA 404 Fix

"Users get 404 errors when navigating directly to deep URLs in a React SPA deployed on Amplify." Answer: Rewrite rule → /index.html with status 200. Not CloudFront configuration. Not S3 settings. Not Route 53. The rewrite rule in Amplify is the correct and only answer.

Branch Access Control — Protecting Non-Production

Amplify provides native HTTP authentication per branch. No CloudFront, no Cognito, no infrastructure changes.

Branch Access Control Configuration Config 
Branch: main        → PUBLIC       (production — no restriction)
Branch: staging     → PROTECTED
                     Username: qa-reviewer
                     Password: SecurePass2024!
Branch: feature/*   → PROTECTED
                     Username: developer
                     Password: DevPreview2024!

# Result: browser shows native HTTP auth dialog before serving branch
# No AWS infrastructure changes — configured in Amplify Console
🎯
Exam Pattern

"Stakeholders need to preview staging but it must not be publicly accessible." Answer: Amplify basic access control on the staging branch. NOT CloudFront signed URLs (overkill). NOT Cognito User Pool (wrong layer for this use case).

Backend Capabilities
⚙️

Backend Capabilities — The Real AWS Services

Amplify doesn't invent new services — it configures existing AWS services on your behalf. When something breaks in an Amplify app, the exam asks about the underlying AWS service, not Amplify itself.

🔐
Authentication
Amazon Cognito
User Pools (sign-in, MFA, social) + Identity Pools (AWS resource access). Auth failures → diagnose Cognito, not Amplify.
🔷
GraphQL API
AWS AppSync + DynamoDB
AppSync manages the GraphQL endpoint and real-time subscriptions. DynamoDB backs the data model. Amplify GraphQL IS AppSync.
🔌
REST API
API Gateway + Lambda
Standard REST endpoints. Amplify generates the Lambda scaffold — you implement business logic.
📦
Storage
Amazon S3
Amplify configures bucket policies and Cognito Identity Pool access. Per-user vs shared storage paths handled automatically.
⚡
Functions
AWS Lambda
Amplify provisions, deploys, and wires Lambda functions to API Gateway. Version management handled automatically.
📊
Analytics
Amazon Pinpoint
Tracks user sessions, events, demographics, and revenue. Amplify adds the Pinpoint client with minimal configuration.
🤖
AI/ML Predictions
SageMaker · Rekognition · Polly
Also Translate and Comprehend. Amplify exposes these as high-level Predictions API calls from your frontend.
📡
PubSub
AWS IoT Core or AppSync
IoT Core for MQTT-based real-time messaging, or AppSync subscriptions for GraphQL-based real-time updates.
📈
Monitoring
Amazon CloudWatch
Build metrics, error rates, and custom application metrics. Set alarms on build failures or runtime errors.

DOP-C02 Exam Pattern Recognition

⚡
The Requirement Traceability Technique

DOP-C02 Amplify questions present 4 options that all sound plausible. Each wrong option violates exactly one requirement — usually with one subtle word change. Run requirements as filters, not confirmations. Ask: does this option VIOLATE any requirement? — not: does this option sound right?

"Deploy React/Vue/Angular/Next.js with zero infrastructure management" → Amplify Hosting
"Preview every feature branch at a unique URL" → Branch deployments — auto per branch
"Preview open PRs before merging to main" → PR Previews enabled on TARGET branch (main) — NOT staging
"Different API endpoints per environment" → Per-branch environment variables
"Staging must not be publicly accessible" → Basic access control on staging branch — NOT CloudFront signed URLs
"SPA shows 404 on direct URL navigation" → Rewrite rule → /index.html with status 200
"Amplify authentication sign-in is failing" → Diagnose Amazon Cognito User Pool — not Amplify itself
"Amplify GraphQL API issues" → Diagnose AWS AppSync — not API Gateway
"Multiple apps in a single repository" → Amplify monorepo with appRoot config in amplify.yml
"Custom domain + HTTPS automatically managed" → Amplify + Route 53 — fully automated cert, records, HTTPS
"Vercel or Netlify equivalent on AWS" → Amplify Hosting — not Elastic Beanstalk

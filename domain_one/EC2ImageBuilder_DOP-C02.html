01

The Four Building Blocks

Every Image Builder pipeline is assembled from four independent concepts. Understanding each individually makes the full architecture obvious.

1
📋
Recipe
What goes into the image
Combines a base image with an ordered list of components to apply. Image Recipe → AMI output. Container Recipe → Docker image for ECR.
Base image + [Component1, Component2, ...]
2
🔧
Component
Reusable unit of work
YAML document defining commands for build, validate, and test phases. AWS provides hundreds of managed components (AWS-UpdateLinux, AWS-InstallJava17, CIS hardening).
build → validate (same instance) → test (new instance from AMI)
3
🖥️
Infrastructure Config
Where the build runs
Defines the temp EC2 settings — instance type, IAM role, VPC, security group, SNS topic. You never see or manage this instance directly.
t3.medium · private subnet · EC2ImageBuilderRole
4
📡
Distribution Config
Where the AMI goes
Which regions and accounts receive the finished AMI after a successful build and test. Dynamic naming with buildDate and buildVersion variables.
us-east-1 + eu-west-1 + prod-account + staging-account
02

The Zero-Touch Pipeline Flow

1

Trigger fires

Schedule (cron), manual, or on new base image availability — when AWS updates Amazon Linux 2023 in their SSM public parameters, your pipeline detects it automatically.

2

Temp build EC2 launched

Image Builder spins up an instance using your Infrastructure Config. You never log in, never manage it — it appears, does its job, and disappears.

3

Components applied in order

Each Recipe component runs: build phase installs software, validate phase confirms it worked on the same instance. Components run sequentially — order matters.

4

AMI snapshot created

Build instance stops, EBS snapshot taken — candidate AMI created. Build instance is terminated. You never paid for it beyond the build window.

5

Test phase runs on new instance

A fresh instance launches FROM the candidate AMI. Component test phases execute here — real integration testing against the actual AMI. If tests fail, AMI is NOT distributed.

6

Distribution Config fires

AMI copied to configured regions, shared with target accounts. SSM Parameter Store updated with new AMI ID. SNS notification sent. Test instance terminated.

component-example.ymlYAML
name: MyNginxComponent
schemaVersion: 1.0

phases:
  - name: build              # runs on temp BUILD instance
    steps:
      - name: InstallNginx
        action: ExecuteBash
        inputs:
          commands:
            - amazon-linux-extras install nginx1 -y
            - systemctl enable nginx

  - name: validate          # runs on SAME build instance — verify install
    steps:
      - name: ValidateNginx
        action: ExecuteBash
        inputs:
          commands:
            - nginx -t
            - systemctl is-enabled nginx

  - name: test               # runs on SEPARATE test instance FROM the AMI
    steps:
      - name: TestNginxRunning
        action: ExecuteBash
        inputs:
          commands:
            - curl -s http://localhost | grep -i nginx
💡
The Test Phase Difference

The test phase is the key differentiator — it launches a fresh instance FROM the new AMI and runs your tests there. If tests fail, the AMI is not distributed. This is real integration testing against the actual image, not the build environment.

03

SSM Parameter Store — Tracking Latest AMIs

The most exam-tested pattern. How do ASGs, Launch Templates, and CloudFormation always use the latest AMI without manual updates?

# Image Builder pipeline completes successfully
→ New AMI created: ami-0newAMI12345
→ Image Builder writes to SSM Parameter Store:
   /myorg/amis/myapp/latest = ami-0newAMI12345
 
# All consumers read from that SSM path — always current
Launch Template: resolve:ssm:/myorg/amis/myapp/latest
CloudFormation: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
EventBridge: SSM change → Lambda → ASG Instance Refresh

CloudFormation SSM Type

stack.ymlYAML
Parameters:
  LatestAMI:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /myorg/amis/myapp/latest

Resources:
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref LatestAMI
      # Always uses current SSM value

ASG Zero-Touch Refresh

zero-touch-flowFlow
Image Builder writes new AMI to SSM
        ↓
EventBridge detects SSM parameter change
        ↓
Lambda creates new Launch Template version
  with updated AMI ID
        ↓
Lambda calls StartInstanceRefresh
  on Auto Scaling Group
        ↓
ASG rolling-replaces all instances
  with new patched AMI

# Human intervention: ZERO

AWS Public SSM Paths

AWS publishes the latest managed AMI IDs to SSM — reference them directly in Image Builder as the base image, so your pipeline auto-picks up the newest AWS base AMI on every run: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64

04

AWS RAM — Sharing AMIs & Components

Distribution Configuration

Copies the AMI

Copies AMI to target regions and accounts at build time. Each account gets their own independent copy of the AMI. Required for cross-region distribution.

AWS RAM Resource Share

Shares the reference

Shares access to the AMI without copying. Owner account pays for storage. Target accounts can launch instances. Same region only — no cross-region.

What RAM Can Share

Image Builder Resources

AMIs (launch permissions), Image Builder Components (reuse YAML across accounts), Image Builder Recipes (share entire build definitions).

DOP-C02 Exam Patterns

"EC2 instances always running the latest patched OS without manual AMI creation"→Image Builder monthly schedule + SSM tracking + ASG Instance Refresh
"Share golden AMI to 10 accounts in same region without copying"→AWS RAM Resource Share — not Distribution Config
"Share golden AMI to eu-west-1 for EU deployments"→Distribution Configuration — not AWS RAM (cross-region)
"CloudFormation always uses the latest golden AMI on deployment"→AWS::SSM::Parameter::Value<AWS::EC2::Image::Id> type
"Same CIS hardening applied to all team AMIs without duplicating scripts"→Image Builder Component shared via AWS RAM
"Pipeline succeeds but test fails — AMI not distributed"→Correct — test phase failure prevents distribution (safety gate)
"Trigger pipeline automatically when AWS releases new Amazon Linux"→On new base image availability trigger — via SSM public parameter change

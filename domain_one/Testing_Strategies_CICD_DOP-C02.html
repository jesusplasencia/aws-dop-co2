01
Fastest · Cheapest · Shift-Left
Unit Tests

Run inside CodeBuild's build phase. Test one function or method in complete isolation — all external dependencies are mocked. No database, no network, no other services involved.

Unit tests are the first gate. If they fail, no artifact is produced and nothing moves downstream. This is the shift-left principle — catch bugs as early and as cheaply as possible.

Shift-left defined: Moving testing earlier in the pipeline rather than waiting until staging or production. Every step to the left on the timeline is cheaper and faster to fix.

Characteristics
Runs in CodeBuild — pre_build or build phase
Speed Milliseconds to seconds — fastest of all test types
Dependencies All mocked — no external connections required
VPC needed No — no private resources accessed
On failure No artifact produced — pipeline stops immediately
Run on Every commit, every branch
buildspec-unit.yml Unit Tests 
version: 0.2

phases:
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - npm ci                              # install dependencies

  pre_build:
    commands:
      - npm run lint                        # syntax + style — fail fast
      - npm audit --audit-level high        # dependency vulnerability check

  build:
    commands:
      - npm test -- --ci                    # Jest unit tests
      # Python alternative:
      # pytest tests/unit/ --junitxml=results/unit-results.xml

reports:
  UnitTestResults:
    files:
      - "jest-results.xml"
    file-format: JUNITXML
  CoverageReport:
    files:
      - "coverage/lcov.info"
    file-format: CLOVERXML
    coverage-enabled: yes    # fail if coverage drops below threshold

artifacts:
  files:
    - '**/*'
  base-directory: build

cache:
  paths:
    - /root/.npm/**/*
🎯
Exam Pattern — Quality Gate

"Team wants to prevent deployments when code coverage drops below 80%." Answer: CodeBuild test reports with coverage threshold configured in the build project settings — not a separate Lambda checking coverage, not a manual code review step.

Integration Tests
02
Real Dependencies · VPC Required · Medium Speed
Integration Tests

Also run inside CodeBuild — but against real infrastructure. Components are tested working together with actual databases, APIs, and queues rather than mocks.

This is where VPC configuration becomes critical. If your tests need to hit a real RDS database or ElastiCache cluster, CodeBuild must be configured to run inside your VPC with access to those private resources.

The VPC + NAT Gateway pattern from cross-region pipelines applies here: CodeBuild in a private subnet needs a NAT Gateway to still reach public package registries while also accessing private resources.

Characteristics
Runs in CodeBuild — build or post_build phase
Speed Seconds to minutes — slower than unit
Dependencies Real — RDS, ElastiCache, internal APIs, queues
VPC needed Yes — if hitting private resources
On failure Artifact not promoted — pipeline stops
Run on PRs to main + main branch — not every feature commit
buildspec-integration.yml Integration Tests 
version: 0.2

# CodeBuild project must be configured with VPC settings:
# vpcId, subnetIds (private), securityGroupIds
# Private subnet must have NAT Gateway for internet access

env:
  parameter-store:
    DB_HOST: /myapp/test/db_host         # RDS endpoint in VPC
    DB_PASSWORD: /myapp/test/db_password
    REDIS_HOST: /myapp/test/redis_host

phases:
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - npm ci

  pre_build:
    commands:
      - echo "Running DB migration for test schema..."
      - npm run db:migrate:test               # set up test data

  build:
    commands:
      - npm run test:integration              # hits real RDS + ElastiCache

  post_build:
    finally:                               # always runs — even on failure
      - npm run db:teardown:test             # clean up test data

reports:
  IntegrationTestResults:
    files:
      - "test-results/integration-*.xml"
    file-format: JUNITXML
🎯
Exam Pattern — The VPC Trigger

"Tests need to access RDS during the CodeBuild build." This phrase triggers the VPC answer: CodeBuild configured with VPC settings (private subnet) + NAT Gateway for internet access. Not an internet gateway. Not a public subnet. This is the most common integration test scenario on DOP-C02.

⚠️
Always Use finally for Cleanup

Integration tests that create test data or spin up resources must clean up in a finally block — not in a regular commands block. Regular commands stop executing if a test fails. finally always runs regardless of failure. Without this, test environments accumulate stale data.

End-to-End Tests
03
Post-Staging · Full User Flows · Slowest
End-to-End Tests

E2E tests run after a deployment to staging — not inside CodeBuild build phases. They simulate real user behavior using browser automation (Cypress, Playwright, Selenium) or full API chains against a live deployed environment.

The E2E test stage is implemented as a separate CodeBuild project in the pipeline — it runs a different buildspec that launches the test framework against the staging URL passed as an environment variable.

If any E2E test fails — the pipeline stops. Production is never touched. The team investigates staging before the next deployment attempt.

Characteristics
Runs in Separate CodeBuild stage — after staging deploy
Speed Minutes to hours — slowest of all test types
Target Staging environment — live deployed application
VPC needed Only if staging is in a private network
On failure Pipeline stops — production never receives deployment
Run on PRs to main and main branch only — not every commit
buildspec-e2e.yml E2E Tests 
version: 0.2

# This buildspec runs AFTER deploy to staging
# STAGING_URL is injected as environment variable from the pipeline

env:
  variables:
    STAGING_URL: "https://staging.myapp.com"  # override per pipeline stage

phases:
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - npm ci
      - npx cypress verify                  # verify Cypress is installed correctly

  pre_build:
    commands:
      - echo "Running E2E against $STAGING_URL"
      - curl -f $STAGING_URL/health         # confirm staging is up before tests

  build:
    commands:
      - npx cypress run
          --config baseUrl=$STAGING_URL
          --reporter junit
          --reporter-options "mochaFile=results/e2e-[hash].xml"
      # Alternative — Playwright:
      # npx playwright test --reporter=junit

  post_build:
    finally:
      - echo "E2E tests complete — check reports"

reports:
  E2ETestResults:
    files:
      - "results/e2e-*.xml"
    file-format: JUNITXML

artifacts:
  files:
    - "cypress/screenshots/**/*"    # capture failure screenshots
    - "cypress/videos/**/*"          # capture failure videos
🎯
Exam Pattern — When to Run E2E

E2E tests are expensive and slow — they should NOT run on every feature branch commit. The correct pattern is: unit tests on every commit to every branch, E2E tests only on PRs targeting main or on a nightly schedule. Running E2E on every commit is the anti-pattern the exam identifies as the cause of slow pipelines.

Smoke Tests
04
Post-Production Deploy · ValidateService Hook · Auto Rollback
Smoke Tests

Run immediately after a production deployment via the ValidateService hook in CodeDeploy's appspec.yml. They're minimal — just enough to confirm the deployment didn't break the critical path.

Smoke tests are not comprehensive — they don't test every feature. They answer one question: is the application alive and are the most critical paths working?

If a smoke test script exits with a non-zero code, CodeDeploy automatically triggers a rollback to the previous working version. No human intervention required.

Characteristics
Runs in ValidateService hook in appspec.yml (CodeDeploy)
Speed Seconds — fastest possible health check
Target Production — the newly deployed version
On failure CodeDeploy automatic rollback — exit code 1 triggers it
Run on Every production deployment — always
Scope Critical path only — health endpoint + core business flow
appspec.yml CodeDeploy 
# appspec.yml — EC2 deployment
version: 0.0
os: linux

files:
  - source: /
    destination: /opt/myapp

hooks:
  ApplicationStart:
    - location: scripts/start.sh
      timeout: 60

  ValidateService:           # smoke tests run here
    - location: scripts/smoke-test.sh
      timeout: 120
      runas: ec2-user

# For Lambda deployments:
# hooks:
#   AfterAllowTraffic: SmokeTestLambdaFn
scripts/smoke-test.sh Bash 
#!/bin/bash
set -e

BASE_URL="https://api.myapp.com"

# Check 1 — health endpoint
STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  $BASE_URL/health)

if [ "$STATUS" != "200" ]; then
  echo "Health check FAILED — HTTP $STATUS"
  exit 1                # triggers rollback
fi

# Check 2 — critical business path
STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  $BASE_URL/api/products)

if [ "$STATUS" != "200" ]; then
  echo "Products endpoint FAILED — HTTP $STATUS"
  exit 1                # triggers rollback
fi

echo "All smoke tests PASSED"
exit 0                  # deployment succeeds
🎯
Exam Pattern — Auto Rollback on Bad Deploy

"How do you automatically rollback if a production deployment is unhealthy?" Answer: Smoke tests in the ValidateService hook — exit code 1 triggers CodeDeploy automatic rollback. Not CloudWatch alarms alone (reactive, not proactive). Not manual intervention. The ValidateService hook is the correct answer specifically because it runs before the deployment is marked successful.

The Testing Pyramid
The Testing Pyramid
The correct distribution of test types — and the most common anti-pattern the exam tests.
E2E Tests Few — expensive, slow, brittle
Integration Tests Some — medium cost, medium speed
Unit Tests Many — cheap, fast, reliable
Many Unit Tests

Hundreds or thousands. Milliseconds each. Run on every single commit. The backbone of fast, reliable CI. Catch 80% of bugs before anything else runs.

Some Integration Tests

Dozens to hundreds. Seconds to minutes each. Run on PRs to main. Validate service boundaries and real infrastructure interactions.

Few E2E Tests

Critical user flows only — login, checkout, core business path. Slow and expensive. Run on scheduled basis or pre-production only.

⚠️
The Inverted Pyramid Anti-Pattern

"A pipeline takes 45 minutes. The team has 500 E2E tests and 50 unit tests." This is the inverted pyramid — the most common pipeline performance anti-pattern on the exam. Answer: Invest in unit tests and reduce E2E tests to critical flows only. Not adding more CodeBuild instances. Not parallel test execution. The root cause is the wrong test distribution.

Static & Dynamic Analysis

SAST vs DAST — Security Testing

Two security testing approaches — the exam tests the distinction constantly.

Static Application Security Testing
SAST
When Before deployment — on source code
What Analyzes code without executing it — finds SQL injection, XSS, hardcoded secrets
Tools CodeGuru Reviewer, SonarQube, Checkmarx, Semgrep
Pipeline stage pre_build phase in CodeBuild — before compilation
Speed Fast — no deployment needed
Dynamic Application Security Testing
DAST
When After deployment — against running app on staging
What Sends attack payloads to live app — finds runtime vulnerabilities SAST misses
Tools OWASP ZAP, Burp Suite, AWS Security Hub integrations
Pipeline stage Post-staging deploy — separate CodeBuild stage
Speed Slower — requires live running environment
🎯
The Three Differentiators the Exam Tests

SAST vs GuardDuty: SAST = source code analysis. GuardDuty = runtime AWS API behavior monitoring.
SAST vs Inspector: SAST = source code. Inspector = EC2 instance and container image vulnerabilities.
SAST vs Macie: SAST = source code. Macie = sensitive data in S3 buckets.

Quality Gates — Stopping the Pipeline Automatically
CodeBuild test reports support thresholds that automatically fail the build if not met — no Lambda, no custom scripts.
80%
Minimum test pass rate — fail below this
70%
Minimum code coverage — fail below this
0
Critical security findings allowed — fail on any
💡
Manual Approval Gate — 7 Day Timeout

CodePipeline's native Manual Approval action pauses the pipeline until a reviewer approves or rejects. Default timeout: 7 days. If no action is taken within 7 days, the pipeline expires. This timeout detail appears in exam questions about compliance pipelines "timing out" after long weekends or holidays.

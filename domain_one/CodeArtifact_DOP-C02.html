Why Centralize
02

Why Centralize Dependencies

Without a central artifact repository, teams independently pull from public registries — creating a fragile, ungoverned, unaudited dependency chain. CodeArtifact resolves each problem systematically.

🔒
Security & Governance
Security teams cannot control which packages enter the build environment. A malicious package can be installed by any developer at any time. CodeArtifact acts as a gate — only packages that exist in your repositories are accessible.
📡
Availability & Reliability
Public packages can disappear (the left-pad incident). CodeArtifact caches every package it fetches — once cached, it is always available regardless of what happens upstream. Your builds never break because of a deleted public package.
🔍
Audit & Compliance
Every package request is logged to AWS CloudTrail automatically. You know exactly which package version was pulled, by which build, at what time — critical for compliance audits and security incident response.
🚀
Performance & Cost
Each package is fetched from the public registry only once. Subsequent requests are served from CodeArtifact's cache — faster builds, lower egress costs, and no rate limiting from public registries under heavy CI load.
🤝
Internal Package Sharing
Teams publish private packages (shared libraries, internal SDKs) to CodeArtifact and consume them exactly like public packages — same tooling, same workflow, no custom infrastructure required.
🛡️
Vulnerability Blocking
When a vulnerable package version is discovered, remove it from the approved repo. All downstream repositories can no longer access it — without touching a single developer machine or build script.
Upstream Repositories
03

Upstream Repositories — The Fallback Chain

When a package is requested and not found locally, CodeArtifact automatically looks upstream — and caches every package it finds at every layer it traverses.

# Developer runs: npm install lodash
 
team-frontend-npm → check local cache NOT FOUND
                       ↓ check upstream 1
internal-libs → check local cache NOT FOUND
                 ↓ check upstream 2
approved-npm → check local cache NOT FOUND
                 ↓ external connection
npmjs.com → FOUND — fetching lodash@4.17.21
 
# Cache propagates back up every layer:
approved-npm ← cached lodash@4.17.21
team-frontend-npm ← cached lodash@4.17.21 → returned to developer
 
# Second request — instant response from team-frontend-npm local cache
# No external call made. No upstream traversal. Zero latency.
—

The Five Rules

R1
1 external connection per repository
A repository can connect to only one public registry (e.g. npmjs.com). Need two public registries? Chain two repos as upstreams — each with its own external connection.
R2
Cached assets survive upstream removal
If you remove an upstream, packages already cached in your repo remain available permanently. You never lose what you already pulled — removing the upstream only stops future fetches.
R3
Priority order is explicit and deliberate
When a package exists in multiple upstreams, CodeArtifact returns it from the first upstream in the configured list. Order matters — security-vetted repos should be listed before open upstreams.
R4
No cross-domain upstreams
Upstream chaining only works within the same domain. You cannot configure a repo in domain A to use a repo in domain B as an upstream. Cross-account sharing requires resource policies, not upstream configuration.
R5
Asset caching is automatic at every layer
Every package fetched from an upstream is automatically cached locally at each layer it traverses. No configuration required — it happens transparently during every upstream fetch.

CodeBuild Integration
05

Integration with CodeBuild Pipelines

The pattern is consistent across all package managers: authenticate to CodeArtifact at the start of the build, then use your package manager normally — it transparently routes through CodeArtifact.

npm — buildspec.yml

buildspec-npm.yml YAML 
version: 0.2

phases:
  pre_build:
    commands:
      # Authenticate — token valid 12 hours
      - aws codeartifact login
          --tool npm
          --domain mycompany
          --domain-owner 123456789012
          --repository team-frontend-npm
          --region us-east-1
      # .npmrc now points to CodeArtifact

  build:
    commands:
      # npm install pulls from CodeArtifact
      - npm install
      - npm run build
      - npm test

artifacts:
  files:
    - '**/*'

Maven — buildspec.yml

buildspec-maven.yml YAML 
version: 0.2

phases:
  pre_build:
    commands:
      # Get auth token
      - export CODEARTIFACT_TOKEN=$(
          aws codeartifact get-authorization-token
          --domain mycompany
          --domain-owner 123456789012
          --query authorizationToken
          --output text)
      # Get repository endpoint
      - export REPO_URL=$(
          aws codeartifact get-repository-endpoint
          --domain mycompany
          --repository team-backend-maven
          --format maven
          --query repositoryEndpoint
          --output text)

  build:
    commands:
      - mvn -s settings.xml clean package
      - mvn -s settings.xml test

Required IAM Permissions — CodeBuild Service Role

codebuild-service-role-policy.json JSON 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "codeartifact:GetAuthorizationToken",
        "codeartifact:GetRepositoryEndpoint",
        "codeartifact:ReadFromRepository"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Action": "sts:GetServiceBearerToken",    // CRITICAL — commonly missed
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "sts:AWSServiceName": "codeartifact.amazonaws.com"
        }
      }
    }
  ]
}
⚠️
The Most Common Misconfiguration

sts:GetServiceBearerToken is NOT optional. The codeartifact:GetAuthorizationToken API internally calls STS to generate a bearer token. Without this permission, authentication fails even when all codeartifact:* permissions are correctly set. This is the single most common CodeArtifact misconfiguration — and a frequent exam trap.

⏱️
Auth Token Expiry

Authorization tokens are valid for 12 hours by default, configurable up to 12 hours. For long-running builds, refresh the token in a mid-build step. The aws codeartifact login command handles token refresh automatically when called again.

Publishing Internal Packages
06

Publishing Internal Packages

CodeArtifact is not just a proxy — it's the distribution channel for your own internally built packages. Publish once; every team consumes like any public package.

Publishing npm

publish-npm.sh Bash 
# 1. Authenticate to internal repo
aws codeartifact login \
  --tool npm \
  --domain mycompany \
  --domain-owner 123456789012 \
  --repository internal-libs

# 2. Publish — same as npmjs.com
npm publish

# Package now available to any repo
# that has internal-libs as an upstream

Publishing Maven

pom.xml + deploy Bash 
# Option 1 — CLI deploy
mvn deploy \
  -DaltDeploymentRepository=\
    codeartifact::default::${REPO_URL} \
  -s settings.xml

# Option 2 — pom.xml configuration
# <distributionManagement>
#   <repository>
#     <id>codeartifact</id>
#     <url>${REPO_URL}</url>
#   </repository>
# </distributionManagement>
Security & Governance
07

Security & Governance Model

Layer Mechanism What It Controls
Domain level Resource-based policy on domain Which AWS accounts can access any repo in the domain — cross-account sharing
Repository level Resource-based policy on repo Which principals can read, write, or administer a specific repository
Identity level IAM policies on users/roles Standard AWS IAM — which principals can call which CodeArtifact APIs
Network level VPC Endpoint + endpoint policy Restrict CodeArtifact access to within your VPC — traffic never leaves AWS network
Encryption AWS KMS (customer managed key) All packages encrypted at rest with your key — configured at domain level
Audit AWS CloudTrail (automatic) Every package request logged — who requested what, when, from which build
🔌
VPC Endpoints — Two Required

CodeArtifact itself is NOT inside your VPC. To keep traffic private, create two VPC Interface Endpoints: com.amazonaws.region.codeartifact.api for management API calls (get-authorization-token), and com.amazonaws.region.codeartifact.repositories for package upload/download traffic. Creating only one is the most common misconfiguration — authentication works but package downloads fail.

Exam Patterns
09

DOP-C02 Exam Pattern Recognition

🎯
The Governance Scenario Template

DOP-C02 CodeArtifact questions almost always appear inside a larger pipeline or security scenario. The tell is the word "governance", "audit", or "no direct internet access" from a build environment. Those three phrases point to CodeArtifact as part of the answer.

"Build environment has no internet access but needs npm packages" → CodeArtifact via VPC endpoint — no NAT Gateway needed for packages
"Security team needs to audit which packages entered the build" → CodeArtifact — every request logged to CloudTrail automatically
"Public package disappeared and broke all builds overnight" → CodeArtifact asset caching — cached packages survive upstream removal
"Block a vulnerable package version across all team builds" → Remove from approved-repo — all downstream repos blocked instantly
"CodeBuild authentication to CodeArtifact fails despite correct permissions" → Missing sts:GetServiceBearerToken in the CodeBuild service role
"Temporary unreviewed package — unblock build without bypassing governance" → Temp repo as additional upstream with lower priority — remove after review
"VPC endpoint created but package downloads still fail" → Missing second VPC endpoint — need both .api AND .repositories
"Multiple teams share the same internal authentication library" → Publish to internal-libs repo — teams consume via upstream chain

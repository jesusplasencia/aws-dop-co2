01
Root Cause #1 · AWS-Managed Key

The AWS-Managed Key Trap

Two accounts. Everything looks correctly configured. Deploy stage fails — CodeDeploy in Account B cannot read the artifact from S3.

✅Pipeline Service Role has sts:AssumeRole to ProdDeployRole
✅ProdDeployRole trust policy allows tooling pipeline role to assume it
✅ProdDeployRole has CodeDeploy permissions in Account B
✅S3 bucket policy grants ProdDeployRole GetObject access
❌Artifacts encrypted with AWS-managed KMS key
✗ Deploy stage FAILS — CodeDeploy in Account B cannot decrypt artifact

Diagnosis & Fix

Root Cause AWS-managed KMS keys cannot be used cross-account
AWS-managed keys are both region-scoped and account-scoped. Their key policy cannot be modified — you cannot add cross-account principals. Even though IAM, S3, and trust policies are perfect, the KMS decryption check fails before CodeDeploy even starts.
Fix — 3 Steps Replace with Customer Managed Key (CMK)
Step 1 — Create a CMK in Account A (tooling), us-east-1. Update CodePipeline to use it.
Step 2 — Add cross-account decrypt to the CMK key policy:
cmk-key-policy.json JSON 
{
  "Sid": "CrossAccountDecrypt",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::222222222222:role/ProdDeployRole"
  },
  "Action": ["kms:Decrypt", "kms:DescribeKey"],
  "Resource": "*"
}
Step 3 — Add KMS permissions to ProdDeployRole's IAM permission policy referencing the new CMK ARN.
⚠️
Why S3 fix alone doesn't work S3 and KMS are independent permission checks. S3 says "you can get this object" — but the object bytes are encrypted. KMS runs its own separate authorization check against the key policy. Passing S3 does not bypass KMS. Both gates must be open independently.
02
Root Cause #2 · Missing Regional Resources

Adding a Region — Two Missing Pieces

Scenario 1 is fixed. Now the business requires deploying to eu-west-1 as well. us-east-1 succeeds. eu-west-1 fails.

✅CodeDeploy app + deployment group created in eu-west-1 in Account B
✅Deploy action added with "region": "eu-west-1"
✅S3 bucket codepipeline-eu-west-1-artifacts created in eu-west-1
✅ProdDeployRole permissions updated for eu-west-1 CodeDeploy actions
❌No CMK created in eu-west-1
❌New eu-west-1 bucket has no bucket policy
✓ us-east-1 SUCCEEDS — existing config intact
✗ eu-west-1 FAILS — CodeDeploy cannot read artifact

Diagnosis & Fix

Missing Piece 1 No CMK in eu-west-1 (Tooling Account)
Every region needs its own CMK. The us-east-1 CMK is region-scoped — it cannot encrypt or decrypt artifacts stored in eu-west-1. Create a new CMK in eu-west-1 in the tooling account. Update its key policy to grant ProdDeployRole decrypt access.
Missing Piece 2 New eu-west-1 S3 bucket has no bucket policy
S3 bucket policies are per bucket — not per account. The policy on tooling-artifacts (us-east-1) has zero effect on the new eu-west-1 bucket. Every new bucket is a blank slate. Add a bucket policy to codepipeline-eu-west-1-artifacts granting ProdDeployRole read access.
💡
The Rule

New region = new bucket + new CMK. New bucket = new bucket policy. New CMK = new key policy. Every new region requires this same checklist — no shortcuts, no inheritance from other regions.

03
Root Cause #3 · SCP Precedence

The Invisible Wall — SCP Override

Everything working perfectly for months. Security team restructures Organization OUs and attaches an SCP. Next morning — both deployments fail. Nothing in the pipeline configuration changed.

Root
├── OU: Tooling
│ └── Account A (111111111111) — tooling
└── OU: Production ← SCP attached here
└── Account B (222222222222) — production
production-ou-scp.json SCP 
{
  "Effect": "Deny",
  "Action": [
    "codepipeline:*",
    "codebuild:*",
    "codecommit:*"
  ],
  "Resource": "*"
}
✗ us-east-1 FAILS — SCP denies CodeDeploy actions on CrossAccountDeployRole
✗ eu-west-1 FAILS — same SCP, same block

Diagnosis & Fix

Root Cause SCP Deny overrides all IAM — even correctly configured roles
SCPs operate above IAM in the AWS evaluation chain. An SCP Deny cannot be overridden by any IAM allow — regardless of how perfectly the roles, trust policies, KMS, and S3 are configured. This is why two hours of IAM debugging found nothing wrong — IAM was never the problem.
⚠️
Common Mistake Changing Effect: Deny to Effect: Allow does NOT fix this. SCPs don't grant permissions — they only restrict what IAM can grant. An SCP Allow is the default state (implicit). The fix is either removing the SCP or adding an exception.
Correct Fix — ArnNotLike Condition Carve out exception for the legitimate deploy role
Keep the governance intent (block production accounts from creating their own pipelines) but exclude the cross-account deploy role:
fixed-scp.json SCP 
{
  "Effect": "Deny",
  "Action": ["codepipeline:*", "codebuild:*", "codecommit:*"],
  "Resource": "*",
  "Condition": {
    "ArnNotLike": {
      "aws:PrincipalArn": "arn:aws:iam::222222222222:role/CrossAccountDeployRole"
    }
  }
}
Result: Everyone in the production account is blocked from creating pipelines — except CrossAccountDeployRole, which can still execute deployments.

Scale · StackSets · Tooling Account Ownership

5 Accounts → 15 Accounts — Scale Without Manual Work

Root
├── OU: Tooling
│ └── Account: tooling (111111111111) — us-east-1
└── OU: Production
├── prod-us (222222222222) → us-east-1
├── prod-eu (333333333333) → eu-west-1
├── prod-apac (444444444444) → ap-southeast-1
├── prod-sa (555555555555) → sa-east-1
└── prod-af (666666666666) → af-south-1
+ 10 more accounts incoming this quarter

The Junior vs Senior Approach

Junior Proposal Lambda running every night to check and create roles
Why it's architecturally wrong: Accounts aren't created every night — most executions are wasted compute. It's reactive polling instead of event-driven response. Operationally expensive to maintain. No single source of truth for what the role should look like.
Senior Solution CloudFormation StackSets with Automatic Deployments
Automatic Deployments: ACTIVATED — when a new account joins the Production OU, the StackSet deploys the CrossAccountDeployRole automatically. No EventBridge. No Lambda. No trigger of any kind. It just happens.
cross-account-role-stackset.yaml CloudFormation StackSet 
# Targets Production OU — deployed to all member accounts
Resources:
  CrossAccountDeployRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: CrossAccountDeployRole
      AssumeRolePolicyDocument:
        Statement:
          - Effect: Allow
            Principal:
              AWS: !Sub "arn:aws:iam::${ToolingAccountId}:role/CodePipelineServiceRole"
            Action: sts:AssumeRole
      Policies:
        - PolicyName: DeployPolicy
          PolicyDocument:
            Statement:
              - Effect: Allow
                Action: [codedeploy:*, s3:GetObject*, kms:Decrypt]
                Resource: "*"
💡
The Part Most People Miss StackSets solve the target account side automatically. But CMKs and S3 artifact buckets live in the tooling account — and must be created manually per new region. The tooling account is always the bottleneck. Wrap the tooling-side setup in a CloudFormation template so adding a new region is one stack deployment, not three manual steps.
05
Final Boss · All Three Root Causes · One Red Herring

Four Events. Three Failures. One Red Herring.

Everything working perfectly. Four things happen on the same day. Midnight pipeline runs — all three deployments fail.

Event 1 — SCP Added

Security team attaches SCP to Production OU: deny kms:* for any region NOT in [us-east-1, eu-west-1]

Event 2 — New Account: prod-apac

Added to Production OU. Team creates CMK + S3 bucket in ap-southeast-1 in tooling account. Adds cross-region deploy action to pipeline.

Event 3 — CMK Rotation

Team creates a new CMK in us-east-1. Updates CodePipeline to use it. Updates new CMK key policy for both prod-us and prod-eu. Deletes old CMK.

Event 4 — prod-eu Moves OU

prod-eu migrated from Production OU to new EU-Compliance OU which has no SCPs attached.

✗prod-us, us-east-1 — FAILS
✗prod-eu, eu-west-1 — FAILS
✗prod-apac, ap-southeast-1 — FAILS

Three Failures — Three Distinct Root Causes

Failure 1 — prod-us, us-east-1 CMK rotation — old ARN hardcoded in IAM policy
Event 3 created a new CMK with a different ARN. The CMK key policy was updated correctly for both accounts. But the IAM permission policy on CrossAccountDeployRole in prod-us still points to the deleted old CMK ARN:
prod-us-role-policy.json — BROKEN JSON 
{
  "Action": ["kms:Decrypt"],
  "Resource": "arn:aws:kms:us-east-1:111111111111:key/OLD_CMK_ID"
  // OLD_CMK_ID was deleted — this permission now points to nothing
}
Fix: Update CrossAccountDeployRole IAM permission policy in prod-us (and prod-eu) to reference the new CMK ARN.
Failure 2 — prod-eu, eu-west-1 Same CMK rotation issue — Event 4 is a red herring
Event 4 (moving prod-eu to EU-Compliance OU) looks significant but changes nothing for eu-west-1 deployments. The EU-Compliance OU has no SCPs — so prod-eu actually has MORE freedom now. The SCP from Production OU that allowed eu-west-1 is gone, but that SCP was only relevant to ap-southeast-1 anyway since eu-west-1 was explicitly permitted.

The real cause is identical to Failure 1 — CrossAccountDeployRole in prod-eu has the deleted old CMK ARN hardcoded in its IAM permission policy.

Fix: Update CrossAccountDeployRole IAM permission policy in prod-eu to reference the new CMK ARN.
Failure 3 — prod-apac, ap-southeast-1 SCP blocks KMS in ap-southeast-1 — region not in allowlist
The SCP from Event 1 denies kms:* for any region not in [us-east-1, eu-west-1]. prod-apac is in the Production OU. When CrossAccountDeployRole tries to call kms:Decrypt on the ap-southeast-1 CMK — the SCP blocks it. The StackSet correctly deployed CrossAccountDeployRole to prod-apac — role assumption is not the problem.
fixed-scp-with-apac.json SCP Fix 
{
  "Effect": "Deny",
  "Action": ["kms:*"],
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "aws:RequestedRegion": [
        "us-east-1",
        "eu-west-1",
        "ap-southeast-1"  // ← add the new region
      ]
    }
  }
}
🎯
The Red Herring Event 4 (prod-eu moving to EU-Compliance OU) is a complete red herring. It's designed to make you think the OU change caused the eu-west-1 failure and spend time analyzing SCP inheritance. The real cause was always the CMK rotation from Event 3. On the exam — when something "structural" changes but the real failure is elsewhere, eliminate the structural change first by asking: does this change affect the specific permission that's failing?
🔗
The Hidden Connection

All three failures trace to just two events: Event 3 (CMK rotation) caused Failures 1 and 2 via stale ARNs in IAM policies. Event 1 + Event 2 interaction (SCP allowlist + new region) caused Failure 3. Event 4 was noise.

Every failure in this lab — one of three root causes
1

Wrong Key Type

AWS-managed key used cross-account. Replace with Customer Managed Key — its policy can be modified to grant cross-account access.

2

Missing Permission

KMS key policy, S3 bucket policy, or IAM policy missing or pointing to a stale ARN. All three layers must be open simultaneously.

3

SCP Blocking

SCP Deny overrides all IAM. Fix with ArnNotLike exception or remove the SCP. Never fix by changing Deny to Allow.

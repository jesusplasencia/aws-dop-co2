Multi-Account &
Cross-Region Pipelines

Three problems. Three solutions. Every exam question is a variation.

1 Identity — Cross-Account IAM Roles
2 Encryption — KMS Customer Managed Key
3 Geography — Regional Artifact Buckets
🏗️

Why This Architecture Exists

In mature organizations, AWS accounts are separated by purpose for security, blast radius reduction, and cost tracking. The pipeline lives in a central Tooling Account and must deploy into isolated target accounts — without any account having standing access to another.

Tooling Account (Shared Services)
  • CodePipeline lives here
  • CodeBuild lives here
  • Artifact S3 bucket lives here
  • KMS Customer Managed Key lives here
Dev Account
  • Deploy targets: EC2, ECS, Lambda
Staging Account
  • Deploy targets: EC2, ECS, Lambda
Production Account
  • Deploy targets: EC2, ECS, Lambda
⚡

The Three Problems

1
Identity

Cross-Account Access

CodePipeline in Account A needs to DO things in Account B. How does it prove it's allowed?

→ Cross-account IAM Role chain
2
Encryption

KMS Key Access

Artifacts flow through S3 encrypted with a KMS key. Account B can't read artifacts encrypted with Account A's key.

→ CMK with cross-account key policy
3
Geography

Cross-Region Artifacts

CodePipeline is in us-east-1. You need to deploy to eu-west-1. Artifacts can't travel across regions without a bucket.

→ Regional artifact S3 buckets per region

The Exam Pattern

Every multi-account and cross-region question is a variation of one or more of these three problems. Identify which problem the scenario describes — you already know the answer.

🔑

Problem 1 — Identity: The IAM Role Chain

Every cross-account action in CodePipeline works through role assumption. Two roles. Always two roles. One in the tooling account. One in each target account.

Tooling Account
CodePipeline Service Role
sts:AssumeRole
→
Has permission to assume the cross-account role in target accounts. Must list each target role ARN explicitly.
Target Account (Prod/Staging/Dev)
CrossAccountDeployRole
Trust Policy
←
Trusts the tooling account pipeline role ARN. Has permissions to run CodeDeploy, ECS, Lambda actions in the target account.
Result
Actions in Target Account
executes
→
CodeDeploy, ECS update service, Lambda update function — all executed in the target account via the assumed role.

Role 1 — Pipeline Service Role (Tooling)

pipeline-service-role-policy.json JSON 
// Key permissions needed
{
  "Action": "sts:AssumeRole",
  "Resource": [
    "arn:aws:iam::PROD:role/CrossAccountDeployRole",
    "arn:aws:iam::STAGING:role/CrossAccountDeployRole"
  ]
},
{
  "Action": [
    "s3:GetObject", "s3:PutObject"
  ],
  "Resource": "arn:aws:s3:::artifact-bucket/*"
},
{
  "Action": [
    "kms:GenerateDataKey",
    "kms:Decrypt"
  ],
  "Resource": "arn:aws:kms:::key/CMK_ID"
}

Role 2 — Cross-Account Role (Target)

cross-account-role-trust.json JSON 
// Trust policy — who can assume this role
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::TOOLING:role/CodePipelineServiceRole"
    },
    "Action": "sts:AssumeRole"
  }]
}

// Permission policy — what it can DO
{
  "Action": [
    "codedeploy:CreateDeployment",
    "s3:GetObject",
    "kms:Decrypt"
  ]
}

Problem 2 — Encryption: KMS & S3 Policies

⚠️
Critical — AWS-Managed Keys Cannot Be Used Cross-Account

This is a hard AWS restriction. You must use a Customer Managed Key (CMK) for all cross-account pipelines. If the scenario says "the S3 and IAM configs look correct but it still fails" — the answer is always the AWS-managed key limitation.

KMS CMK Key Policy

cmk-key-policy.json JSON 
// Tooling account — full access
{
  "Sid": "ToolingFullAccess",
  "Principal": {
    "AWS": "arn:aws:iam::TOOLING:root"
  },
  "Action": "kms:*"
},
// Target accounts — decrypt only
{
  "Sid": "CrossAccountDecrypt",
  "Principal": {
    "AWS": [
      "arn:aws:iam::PROD:role/CrossAccountDeployRole",
      "arn:aws:iam::STAGING:role/CrossAccountDeployRole"
    ]
  },
  "Action": [
    "kms:Decrypt",
    "kms:DescribeKey"
  ]
}

S3 Bucket Policy

artifact-bucket-policy.json JSON 
// Allow target accounts to read artifacts
{
  "Sid": "CrossAccountRead",
  "Effect": "Allow",
  "Principal": {
    "AWS": [
      "arn:aws:iam::PROD:role/CrossAccountDeployRole",
      "arn:aws:iam::STAGING:role/CrossAccountDeployRole"
    ]
  },
  "Action": [
    "s3:GetObject",
    "s3:GetObjectVersion",
    "s3:GetBucketVersioning",
    "s3:ListBucket"
  ],
  "Resource": [
    "arn:aws:s3:::artifact-bucket",
    "arn:aws:s3:::artifact-bucket/*"
  ]
}
⚠️
Dual Permission Requirement — The Most Tested Concept

KMS key policies are resource-based policies — separate from IAM. Even if the IAM role has kms:Decrypt in its permission policy, the KMS key must also grant access in its own key policy. Both must allow the action. This is the most common cross-account failure on the exam.

🔧

The Three-Fix Rule — Memorize This

When a cross-account deployment fails, the complete fix always touches all three layers simultaneously. Fix one or two — still broken. Fix all three — works.

1

IAM Role Chain

Tooling pipeline role: sts:AssumeRole to target.
Target role: trust policy + deploy actions.

2

KMS Key Policy

CMK (not AWS-managed). Key policy grants target role kms:Decrypt + kms:DescribeKey.

3

S3 Bucket Policy

Bucket policy grants target role s3:GetObject + s3:GetObjectVersion.

Fix 1 of 3 → still broken · Fix 2 of 3 → still broken · Fix all 3 → works

Problem 3 — Geography: Cross-Region Pipelines

CodePipeline is a regional service. When you need to deploy to a different region, artifacts in us-east-1 S3 cannot be read by CodeDeploy in eu-west-1 — the bucket is in a different region. For every region you deploy to: dedicated S3 bucket + dedicated CMK in that region.

cross-region-pipeline-action.json JSON 
// Cross-region deploy action — "region" field is the key
{
  "name": "Deploy-EU",
  "actions": [{
    "name": "DeployToEU",
    "actionTypeId": {
      "category": "Deploy",
      "owner": "AWS",
      "provider": "CodeDeploy",
      "version": "1"
    },
    "region": "eu-west-1",          // ← tells CodePipeline to use eu-west-1 artifact bucket
    "inputArtifacts": [{ "name": "BuildArtifact" }],
    "configuration": {
      "ApplicationName": "MyApp-EU",
      "DeploymentGroupName": "MyApp-EU-Production"
    }
  }]
}
us-east-1 (primary)
  • ✓ S3: codepipeline-us-east-1-artifacts
  • ✓ KMS CMK in us-east-1
  • ✓ CodePipeline + CodeBuild here
eu-west-1 (secondary)
  • → S3: codepipeline-eu-west-1-artifacts (must pre-exist)
  • → KMS CMK in eu-west-1 (must pre-exist)
  • ✓ CodeDeploy target here
⚠️
Exam Trap — Missing Regional Resources

Cross-region deploy action is configured correctly but the pipeline fails. The cause is a missing S3 bucket or KMS CMK in the target region. The regional bucket and key must be pre-created — CodePipeline does not create them automatically.

🏢

AWS Organizations — Scale Without Manual Work

When you have dozens of accounts, managing cross-account roles one by one doesn't scale. Two Organization features solve this at scale.

CloudFormation StackSets

Deploy cross-account role to all accounts in an OU automatically

Define the CrossAccountDeployRole once as a CloudFormation template. Deploy via StackSet targeting the entire Organization or specific OUs. New accounts added to the OU automatically get the role.

Exam Answer for 50+ accounts
Service Control Policies (SCPs)

Prevent production accounts from creating their own pipelines

SCP on the Production OU denies codepipeline:* and codebuild:*. Production accounts can only receive deployments — they cannot create their own pipelines. SCPs override IAM — even correctly configured roles are blocked.

Exam Trap — SCP blocks correct IAM
cross-account-role-stackset.yaml CloudFormation StackSet 
# Deployed to all target accounts via StackSet — one template, all accounts
Resources:
  CrossAccountDeployRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: CrossAccountDeployRole
      AssumeRolePolicyDocument:
        Statement:
          - Effect: Allow
            Principal:
              AWS: !Sub "arn:aws:iam::${ToolingAccountId}:role/CodePipelineServiceRole"
            Action: sts:AssumeRole
      Policies:
        - PolicyName: DeployPolicy
          PolicyDocument:
            Statement:
              - Effect: Allow
                Action:
                  - codedeploy:*
                  - s3:GetObject*
                  - kms:Decrypt
                  - kms:DescribeKey
                Resource: "*"

DOP-C02 Exam Scenarios

Scenario 01

Classic Cross-Account Failure

"CodePipeline in the tooling account fails deploying to production. The cross-account IAM role exists and has CodeDeploy permissions. What is missing?"
Answer

KMS key policy or S3 bucket policy (or both). The IAM role existing is not enough — resource-based policies on KMS and S3 must also grant access. The question tells you the role is fine — so the gap is always KMS or S3.

Scenario 02

Cross-Region Artifact Failure

"CodePipeline deploys to us-east-1 successfully but fails deploying the same artifact to eu-west-1. The cross-region action is configured. What is the most likely cause?"
Answer

Missing S3 artifact bucket or KMS CMK in eu-west-1. Cross-region actions require regional buckets and regional CMKs pre-created. The pipeline cannot replicate artifacts to a region with no artifact bucket.

Scenario 03

AWS-Managed Key Failure

"A team sets up cross-account CodePipeline. S3 bucket policy and IAM roles are correctly configured but the target account cannot read build artifacts. What is the issue?"
Answer

The pipeline uses an AWS-managed KMS key, which cannot be used cross-account. Fix: create a Customer Managed Key, update the pipeline to use it, and grant the target role decrypt permissions in the CMK key policy.

Scenario 04

Organization-Scale Role Deployment

"A company has 50 AWS accounts in an Organization. They want all accounts to have a cross-account deploy role consistently without visiting each account manually."
Answer

CloudFormation StackSets targeting the Organization or OU. Not Lambda SDK loops. Not manual IAM visits. Not AWS Config rules.

Scenario 05

SCP Blocking Deployments

"A pipeline deployed to production for months. After moving the production account to a new OU, deployments started failing. No pipeline config changed. What happened?"
Answer

A Service Control Policy on the new OU is denying the actions the cross-account role needs. SCPs override even correctly configured IAM roles — SCP deny beats IAM allow, always.

Scenario 06

Cross-Account AND Cross-Region

"Pipeline in tooling account (us-east-1) must deploy to production account in eu-west-1. Deployment fails. Everything else is correctly configured except one thing."
Answer

Check all 6 requirements: IAM roles (both), KMS policy (us-east-1), S3 policy (us-east-1), S3 bucket in eu-west-1, KMS CMK in eu-west-1, cross-account role access to eu-west-1 resources.

Deployment Velocities — Memorize the Intervals

EC2 / On-Premises

4 Strategies

HalfAtATime — 50% capacity always available
OneAtATime — slowest, safest
Custom % — user-defined percentage

Canary + Linear

Canary: 5, 10, 15, 30 min intervals
Linear: Every 1, 2, 3, 10 min — 10% each
AllAtOnce — instant shift
4 Canary options · 4 Linear options
ECS

Fewer Options

AllAtOnce — instant shift
Only 2 Canary · Only 2 Linear
⚠️
Trap #3 — ECS vs Lambda Intervals

ECS Canary only has 5 and 15 minute options. Lambda has 5, 10, 15, 30. If a scenario requires 10-minute Canary for ECS — that doesn't exist. Lambda would be the right service for that requirement.

Traps #4 & #5

Health checks → ValidateService not AfterInstall.
DB migrations → BeforeInstall not ApplicationStop (instance still serving traffic at ApplicationStop).

EC2 appspec.yml

appspec.yml (EC2)YAML
version: 0.0
os: linux

files:
  - source: /app
    destination: /opt/myapp

hooks:
  BeforeInstall:
    - location: scripts/stop_server.sh
      timeout: 60
      runas: root
  AfterInstall:
    - location: scripts/configure.sh
      timeout: 120
  ApplicationStart:
    - location: scripts/start.sh
      timeout: 60
  ValidateService:       # health check here
    - location: scripts/healthcheck.sh
      timeout: 180

Lambda appspec.yml

appspec.yml (Lambda)YAML
version: 0.0
Resources:
  - MyLambdaFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: MyFunction
        Alias: live
        CurrentVersion: !Ref VersionA
        TargetVersion: !Ref VersionB

Hooks:
  BeforeAllowTraffic: SmokeTestFunction
  AfterAllowTraffic: PostValidateFunction

ECS appspec.yml

appspec.yml (ECS)YAML
version: 0.0
Resources:
  - TargetService:
      Type: AWS::ECS::Service
      Properties:
        TaskDefinition: arn:aws:ecs:...:myapp:5
        LoadBalancerInfo:
          ContainerName: myapp
          ContainerPort: 8080
Hooks:
  BeforeAllowTraffic: ValidateBeforeFn
  AfterAllowTraffic: ValidateAfterFn
04

ECS Blue/Green — Deep Mechanics

# ECS Blue/Green deployment flow
Step 1 → New ECS task set (Green) created alongside Blue
Step 2 → ALB test listener (port 8080) routes to Green for validation
Step 3 → BeforeAllowTraffic hook validates Green via test listener
Step 4 → Traffic shifts Blue → Green on production listener (port 443)
Step 5 → AfterAllowTraffic hook validates production traffic
Step 6 → Blue kept alive during termination wait time (default: 1 hour)
Step 7 → Blue task set terminated after wait period
⚠️
Trap #6 — Termination Wait Time

Blue tasks are NOT immediately terminated after traffic shifts. Default wait: 1 hour. During this window Blue can serve as instant rollback — traffic shifts back in seconds because Blue is still running.

05

Rollbacks

💡
Rollback is a redeploy, not an undo

CodeDeploy redeploys the last known good revision as a new deployment. It does not restore files in-place. For EC2 In-Place this means running old hooks again — slower. For Lambda/ECS, rollback = fast traffic shift back to Blue/previous alias.

🎯
Trap #7 — Instant Rollback

"Scenario requires instant rollback capability." Always favor Blue/Green (Lambda or ECS) over EC2 In-Place — Blue environment is still live, traffic shift takes seconds.

Rollback TriggerBehavior
Deployment failureCodeDeploy automatically triggers rollback to last successful revision
CloudWatch Alarm breachAlarm configured on deployment metric triggers automatic rollback
Manual rollbackYou trigger via console, CLI, or SDK

DOP-C02 Exam Patterns

"In-Place deployment on Lambda"→Does NOT exist — eliminate immediately
"CodeDeploy Agent on ECS"→Not required — wrong answer
"ECS Canary at 10-minute intervals"→Not available for ECS — Lambda only
"Post-deployment health check"→ValidateService hook — NOT AfterInstall
"DB migrations before install"→BeforeInstall — NOT ApplicationStop
"Instant rollback capability needed"→Blue/Green Lambda or ECS — not EC2 In-Place
"Blue ECS tasks terminated immediately after shift"→False — 1 hour termination wait (default)
"Agent can't pull deployment bundle"→Check EC2 Instance Profile S3 permissions first
